    React Styling Techniques

    React is unopinionated about how you style your components — it just renders DOM. That freedom means there are many valid approaches, from plain CSS files to runtime CSS-in-JS to utility-first frameworks to full component libraries. This chapter walks through each major option with modern, runnable examples (React 18/19 functional components), and ends with a decision table to help you pick.

    📚 The examples assume a Vite + React project. Create one with npm create vite@latest my-app -- --template react-ts.

    1. CSS in React

    Inline Styles

    In React, inline styles are passed as a JavaScript object to the style prop. Property names are camelCased (backgroundColor, not background-color), and values are usually strings. Numeric values for length properties get px appended automatically.

    function Badge() {
  return (
    <span
      style={{
        backgroundColor: '#2563eb',
        color: 'white',
        padding: '4px 8px',
        borderRadius: 6,        // becomes 6px
        fontWeight: 600,
      }}
    >
      New
    </span>
  );
}

    Handling Dynamic Styles

    Because the style is just an object, you can compute it from props or state — this is where inline styles shine.

    function ProgressBar({ percent }) {
  return (
    <div style={{ width: '100%', background: '#e5e7eb', borderRadius: 8 }}>
      <div
        style={{
          width: `${percent}%`,
          height: 8,
          background: percent < 30 ? '#dc2626' : '#16a34a',
          borderRadius: 8,
          transition: 'width 0.3s ease',
        }}
      />
    </div>
  );
}

    Pros and Cons

    Inline styles at a glance

    Pros: scoped to the element, trivially dynamic, no extra build tooling, great for one-off computed values.

    Cons: no pseudo-classes (:hover, :focus), no media queries, no keyframes, no @supports. Styles can't be cached or reused, and re-creating the object on every render can hurt performance in hot paths. Use inline styles for dynamic values, not as your primary styling strategy.

    External CSS Stylesheets

    The simplest "real" approach: write a .css file and import it. With Vite (or CRA), the bundler injects it into the page.

    /* Button.css */
.btn {
  padding: 8px 16px;
  border: none;
  border-radius: 6px;
  cursor: pointer;
}
.btn--primary {
  background: #2563eb;
  color: #fff;
}

    import './Button.css';

function Button({ children }) {
  return <button className="btn btn--primary">{children}</button>;
}

    Global scope

    Plain CSS imports are global. A class named .btn defined in one file applies everywhere on the page. As projects grow, name collisions become a real risk — which is exactly why naming conventions and CSS Modules exist.

    Structuring CSS for Large Projects

    A common pattern is to colocate a component's CSS with its component file, plus a few global files:

    src/
├── styles/
│   ├── reset.css        # normalize/reset
│   ├── variables.css    # CSS custom properties (design tokens)
│   └── global.css       # base typography, body, etc.
└── components/
    ├── Button/
    │   ├── Button.jsx
    │   └── Button.css
    └── Card/
        ├── Card.jsx
        └── Card.css

    Using CSS custom properties as design tokens keeps things consistent:

    /* variables.css */
:root {
  --color-primary: #2563eb;
  --color-text: #111827;
  --space-2: 8px;
  --radius: 6px;
}

    Naming Conventions (BEM, SMACSS)

    BEM (Block, Element, Modifier) is the most popular convention. It encodes the component structure into the class name so styles stay flat and predictable:

    .block            /* the component         → .card        */
.block__element   /* a part of it          → .card__title */
.block--modifier  /* a variant/state       → .card--featured */

    .card { border: 1px solid #e5e7eb; border-radius: var(--radius); }
.card__title { font-weight: 700; }
.card__body { color: #4b5563; }
.card--featured { border-color: var(--color-primary); }

    function Card({ featured, title, children }) {
  return (
    <div className={`card ${featured ? 'card--featured' : ''}`}>
      <h3 className="card__title">{title}</h3>
      <div className="card__body">{children}</div>
    </div>
  );
}

    SMACSS (Scalable and Modular Architecture for CSS) is less about naming and more about categorizing rules into five buckets — Base, Layout (l-), Module, State (is-active, is-hidden), and Theme. It pairs well with BEM rather than competing with it.

    Tip

    Conventions like BEM are a discipline, not a tool — nothing enforces them. If you want the compiler to guarantee no collisions, reach for CSS Modules.

    CSS Modules

    What Are CSS Modules?

    A CSS Module is a regular CSS file whose class names are locally scoped by default. At build time, each class is renamed to a unique hash (e.g. .btn → .Button_btn__a1b2c), so it can never collide with a class of the same name elsewhere. Vite supports them out of the box — just name the file *.module.css.

    Using CSS Modules in React

    /* Button.module.css */
.btn {
  padding: 8px 16px;
  border-radius: 6px;
}
.primary {
  background: #2563eb;
  color: white;
}

    import styles from './Button.module.css';

function Button({ children }) {
  // styles.btn is the hashed, locally-scoped class string
  return <button className={`${styles.btn} ${styles.primary}`}>{children}</button>;
}

    For conditional classes, the tiny clsx library is the standard helper:

    import clsx from 'clsx';
import styles from './Button.module.css';

function Button({ variant = 'primary', disabled, children }) {
  return (
    <button
      className={clsx(styles.btn, styles[variant], { [styles.disabled]: disabled })}
      disabled={disabled}
    >
      {children}
    </button>
  );
}

    Scope Isolation and Benefits

    Why CSS Modules?

    • Guaranteed isolation — class names are hashed, so no global leakage and no naming wars.
    • Plain CSS — full power of CSS (pseudo-classes, media queries, keyframes) with zero new syntax to learn.
    • Dead-code friendly — unused classes are easy to spot since they're imported by name.
    • Zero runtime — it's all resolved at build time, so there's no performance cost at render.

    They're an excellent default when you like writing real CSS but want safety. Composition is supported via the composes keyword.

    2. Styled Components (CSS-in-JS)

    Introduction

    styled-components lets you write actual CSS inside template literals and attach it to a component. Each styled component generates a unique class name, so styles are automatically scoped. It's the archetypal CSS-in-JS library.

    Installing and Setting Up

    npm install styled-components

    For TypeScript, the types ship with the package as of v6 (no separate @types package needed).

    Writing Styled Components

    import styled from 'styled-components';

const Button = styled.button`
  padding: 8px 16px;
  border: none;
  border-radius: 6px;
  background: #2563eb;
  color: white;
  cursor: pointer;

  &:hover {
    background: #1d4ed8;
  }
`;

function App() {
  return <Button>Click me</Button>;
}

    The & refers to the generated class itself — so nesting, pseudo-classes, and media queries all work as in Sass.

    Using Props to Modify Styles Dynamically

    Interpolated functions receive the component's props, letting styles react to data.

    v6 transient props

    In styled-components v6, props used only for styling should be prefixed with $ (transient props). This stops them from leaking onto the underlying DOM node as invalid HTML attributes.

    import styled, { css } from 'styled-components';

const Button = styled.button<{ $variant?: 'primary' | 'ghost'; $size?: 'sm' | 'md' }>`
  border-radius: 6px;
  cursor: pointer;
  padding: ${(props) => (props.$size === 'sm' ? '4px 10px' : '8px 16px')};

  ${(props) =>
    props.$variant === 'ghost'
      ? css`
          background: transparent;
          color: #2563eb;
          border: 1px solid #2563eb;
        `
      : css`
          background: #2563eb;
          color: white;
          border: none;
        `}
`;

function App() {
  return (
    <>
      <Button $variant="primary">Primary</Button>
      <Button $variant="ghost" $size="sm">Ghost</Button>
    </>
  );
}

    Theming with Styled Components

    Wrap your app in a ThemeProvider and the theme becomes available in every styled component's interpolations.

    import styled, { ThemeProvider } from 'styled-components';

const theme = {
  colors: { primary: '#2563eb', text: '#111827' },
  radius: '6px',
};

const Card = styled.div`
  color: ${(props) => props.theme.colors.text};
  border: 1px solid ${(props) => props.theme.colors.primary};
  border-radius: ${(props) => props.theme.radius};
  padding: 16px;
`;

function App() {
  return (
    <ThemeProvider theme={theme}>
      <Card>Themed card</Card>
    </ThemeProvider>
  );
}

    Type the theme

    In TypeScript, augment the library's DefaultTheme so props.theme is fully typed: 

    // styled.d.ts
import 'styled-components';
declare module 'styled-components' {
  export interface DefaultTheme {
    colors: { primary: string; text: string };
    radius: string;
  }
}

    Global Styles

    Use createGlobalStyle for resets and base styles. Render it once near the root.

    import { createGlobalStyle } from 'styled-components';

const GlobalStyle = createGlobalStyle`
  * { box-sizing: border-box; margin: 0; }
  body {
    font-family: system-ui, sans-serif;
    color: ${(props) => props.theme.colors.text};
  }
`;

function Root() {
  return (
    <ThemeProvider theme={theme}>
      <GlobalStyle />
      <App />
    </ThemeProvider>
  );
}

    Best Practices

    Styled-components best practices

    • Define styled components at module scope, never inside render — defining them in render recreates the component on every render, breaking memoization and causing DOM remounts.
    • Use transient props ($) for styling-only props (v6).
    • Keep a single theme source of truth and type it.
    • Use the Babel/SWC plugin for better debugging class names and dead-code elimination.
    • Be aware of the runtime cost: styles are computed in the browser. For very large or render-heavy apps, a zero-runtime option (CSS Modules, Tailwind, or a compile-time CSS-in-JS lib) may perform better.

    3. Tailwind CSS with React

    What Is Tailwind CSS?

    Tailwind CSS is a utility-first framework. Instead of writing custom CSS, you compose small single-purpose classes (flex, pt-4, text-center, bg-blue-600) directly in your markup. It generates only the CSS you actually use, so the final bundle stays tiny.

    Installing Tailwind in a Vite Project

    Tailwind's setup differs by major version. Both are shown below.

    v4 ships a first-party Vite plugin and removes the JS config file in favor of CSS-based configuration.

    npm install tailwindcss @tailwindcss/vite

    // vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import tailwindcss from '@tailwindcss/vite';

export default defineConfig({
  plugins: [react(), tailwindcss()],
});

    /* src/index.css — a single import replaces the old @tailwind directives */
@import "tailwindcss";

    npm install -D tailwindcss postcss autoprefixer
npx tailwindcss init -p

    // tailwind.config.js — content tells Tailwind where to scan for classes
export default {
  content: ['./index.html', './src/**/*.{js,jsx,ts,tsx}'],
  theme: { extend: {} },
  plugins: [],
};

    /* src/index.css */
@tailwind base;
@tailwind components;
@tailwind utilities;

    Make sure your CSS entry file is imported in main.jsx:

    import './index.css';

    Using Utility Classes for Styling

    function Card({ title, children }) {
  return (
    <div className="max-w-sm rounded-lg border border-gray-200 p-4 shadow-sm">
      <h3 className="mb-2 text-lg font-bold text-gray-900">{title}</h3>
      <p className="text-gray-600">{children}</p>
      <button className="mt-4 rounded-md bg-blue-600 px-4 py-2 text-white hover:bg-blue-700">
        Action
      </button>
    </div>
  );
}

    Responsive design and states are prefixes: md:flex, hover:bg-blue-700, dark:bg-gray-800, focus:ring-2.

    For conditional classes, combine clsx with tailwind-merge (which resolves conflicting utilities like p-2 p-4):

    import { twMerge } from 'tailwind-merge';
import clsx from 'clsx';

const cn = (...inputs) => twMerge(clsx(inputs));

function Button({ active, className, children }) {
  return (
    <button className={cn('rounded px-4 py-2', active && 'bg-blue-600 text-white', className)}>
      {children}
    </button>
  );
}

    Customizing Tailwind Configurations

    @import "tailwindcss";

@theme {
  --color-brand: #2563eb;
  --font-display: "Inter", sans-serif;
  --spacing-18: 4.5rem;
}

    These tokens automatically create utilities like bg-brand, font-display, and p-18.

    export default {
  content: ['./index.html', './src/**/*.{js,jsx,ts,tsx}'],
  theme: {
    extend: {
      colors: { brand: '#2563eb' },
      fontFamily: { display: ['Inter', 'sans-serif'] },
      spacing: { 18: '4.5rem' },
    },
  },
  plugins: [],
};

    Performance: The Built-in JIT Engine

    PurgeCSS is no longer needed

    Modern Tailwind (v3.4+ and v4) uses a built-in JIT (Just-In-Time) engine. It scans your source files and generates only the classes you actually use, on demand. There is no separate purge step and no purge config — the old PurgeCSS workflow from Tailwind v1/v2 is obsolete. Production bundles are typically just a few kilobytes of CSS. In v3 you point the engine at your files via the content array; in v4 detection is largely automatic.

    Tailwind vs Traditional CSS

    Trade-offs

    In favor: rapid development, no context-switching to a .css file, no naming decisions, consistent design tokens, tiny production CSS, styles colocated with markup.

    Against: verbose className strings ("class soup"), a learning curve for the utility vocabulary, and markup that mixes structure with styling. Extract repeated patterns into components (not @apply-heavy CSS) to keep it manageable.

    4. Emotion.js

    Introduction

    Emotion is a flexible, high-performance CSS-in-JS library. It offers two APIs: a styled API nearly identical to styled-components (@emotion/styled), and a css prop API (@emotion/react) that lets you attach styles inline without creating a named component. Emotion is the styling engine under the hood of MUI v5/v6.

    Setting Up Emotion in React

    npm install @emotion/react @emotion/styled

    To use the css prop with the automatic JSX runtime in a Vite project, add the Emotion preset to the React plugin:

    // vite.config.js
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react({ jsxImportSource: '@emotion/react' })],
});

    Writing CSS with Emotion

    The css prop — define styles inline as an object or template literal:

    /** @jsxImportSource @emotion/react */
import { css } from '@emotion/react';

const card = css`
  padding: 16px;
  border-radius: 6px;
  border: 1px solid #e5e7eb;
  &:hover { border-color: #2563eb; }
`;

function Card() {
  return <div css={card}>Hello from Emotion</div>;
}

    The styled API — identical ergonomics to styled-components:

    import styled from '@emotion/styled';

const Button = styled.button<{ primary?: boolean }>`
  padding: 8px 16px;
  border-radius: 6px;
  background: ${(props) => (props.primary ? '#2563eb' : '#e5e7eb')};
  color: ${(props) => (props.primary ? 'white' : '#111827')};
`;

    Theming with Emotion

    import { ThemeProvider, css } from '@emotion/react';

const theme = { colors: { primary: '#2563eb' } };

function Title() {
  return (
    <h1 css={(t) => css`color: ${t.colors.primary};`}>Themed</h1>
  );
}

function App() {
  return (
    <ThemeProvider theme={theme}>
      <Title />
    </ThemeProvider>
  );
}

    Performance Considerations

    Emotion performance

    • Emotion caches serialized styles, so repeated identical styles are computed once.
    • Like all runtime CSS-in-JS, there is a per-render cost; define static styles outside the component to avoid re-serializing them every render.
    • For SSR, Emotion provides extraction utilities so critical CSS ships in the initial HTML.
    • Emotion vs styled-components: very similar feature sets. Emotion's css prop is more flexible (no need to name every styled element) and it's the default engine for MUI — making it a natural choice in MUI-based projects.

    5. Bootstrap & Material UI

    Sometimes you don't want to build a design system from scratch — you want ready-made, accessible components. That's what UI libraries provide.

    Installing and Using Bootstrap in React

    Don't use plain Bootstrap's jQuery plugins in React. Use React Bootstrap, which reimplements Bootstrap components as real React components.

    npm install react-bootstrap bootstrap

    // import the stylesheet once, e.g. in main.jsx
import 'bootstrap/dist/css/bootstrap.min.css';

    import { Button, Card } from 'react-bootstrap';

function Profile() {
  return (
    <Card style={{ width: '18rem' }}>
      <Card.Body>
        <Card.Title>Jane Doe</Card.Title>
        <Card.Text>Frontend developer.</Card.Text>
        <Button variant="primary">Follow</Button>
      </Card.Body>
    </Card>
  );
}

    Using Material UI Components

    Material UI (MUI) is a comprehensive React component library implementing Google's Material Design. The current major versions are v5/v6, both built on Emotion by default.

    npm install @mui/material @emotion/react @emotion/styled

    import { Button, Card, CardContent, Typography } from '@mui/material';

function Profile() {
  return (
    <Card sx={{ maxWidth: 320 }}>
      <CardContent>
        <Typography variant="h6">Jane Doe</Typography>
        <Typography color="text.secondary">Frontend developer.</Typography>
        <Button variant="contained" sx={{ mt: 2 }}>Follow</Button>
      </CardContent>
    </Card>
  );
}

    The sx prop is MUI's shorthand styling system — it accepts theme-aware values (mt: 2 → 16px from the spacing scale) and is the idiomatic way to do one-off overrides.

    Customizing Themes

    MUI uses a central theme via ThemeProvider and createTheme:

    import { createTheme, ThemeProvider } from '@mui/material/styles';

const theme = createTheme({
  palette: {
    mode: 'light',
    primary: { main: '#2563eb' },
  },
  shape: { borderRadius: 8 },
});

function App() {
  return (
    <ThemeProvider theme={theme}>
      {/* every MUI component now reads this theme */}
    </ThemeProvider>
  );
}

    Bootstrap is customized either by overriding CSS variables (Bootstrap 5 exposes many --bs-* custom properties) or by recompiling its Sass with your own variable values before importing.

    Performance Considerations for UI Libraries

    Bundle size

    Component libraries add weight. To keep bundles lean:

    • Import only what you use — import Button from '@mui/material/Button' (named imports from the package root are tree-shaken in modern bundlers, but path imports are the safest guarantee).
    • With Bootstrap, import individual components from react-bootstrap rather than the whole library.
    • MUI's runtime CSS-in-JS (Emotion) has a render cost; for static-heavy pages consider its zero-runtime Pigment CSS option (used in MUI's newer offerings) or extract critical CSS for SSR.

    When to Use a Component Library vs Custom Styling

    Decision guidance

    Reach for a component library (MUI, Bootstrap) when:

    • You need to ship fast (dashboards, internal tools, MVPs, admin panels).
    • You want accessibility, keyboard support, and complex widgets (date pickers, autocomplete, data grids) for free.
    • A consistent, conventional look is acceptable or desired.

    Prefer custom styling (CSS Modules, Tailwind, CSS-in-JS) when:

    • You have a bespoke / branded design that would fight a library's defaults.
    • Bundle size and full control matter more than speed of delivery.
    • You want to build your own design system.

    A common hybrid: a headless library (Radix UI, Headless UI, React Aria) for behavior + accessibility, styled with Tailwind for full visual control. This gives you the best of both worlds.

    Comparison & Decision Table

    Approach Type Scope isolation Dynamic styles Runtime cost Best for
    Inline styles JS object Per-element ✅ Excellent Low (object churn) One-off computed values, quick prototypes
    External CSS + BEM Plain CSS ❌ Global (by convention) ⚠️ via class toggling None Simple sites, full CSS power, no build extras
    CSS Modules Compiled CSS ✅ Auto (hashed) ⚠️ via class toggling None Real CSS with guaranteed safety — a great default
    styled-components CSS-in-JS ✅ Auto ✅ Excellent (props/theme) Medium (runtime) Component-scoped dynamic styling, theming
    Emotion CSS-in-JS ✅ Auto ✅ Excellent Medium (cached) Same as above; css prop flexibility, MUI projects
    Tailwind CSS Utility classes ✅ (atomic) ⚠️ via conditional classes None (build-time JIT) Rapid dev, design consistency, tiny CSS bundle
    Bootstrap / MUI Component library ✅ ✅ (sx, props, theme) Medium–High Fast delivery, accessibility, prebuilt widgets

    Quick recommendations

    TL;DR

    • Just starting / want plain CSS safely → CSS Modules.
    • Move fast with a consistent design + small bundle → Tailwind CSS.
    • Heavily dynamic, theme-driven component styling → styled-components or Emotion.
    • Need accessible, prebuilt components yesterday → MUI (or React Bootstrap).
    • Bespoke design + accessibility → headless components + Tailwind.

    There's no single "right" answer — most production apps mix two or three of these (e.g. Tailwind for layout + MUI for complex widgets). Choose based on team familiarity, design requirements, and performance budget.
