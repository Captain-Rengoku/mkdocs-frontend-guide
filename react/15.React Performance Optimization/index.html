    React Performance Optimization

    React is fast by default. Most apps never need any of the techniques in this chapter, and reaching for them too early usually makes code harder to read without making it faster. The golden rule of performance work is:

    Measure first, optimize second.

    This chapter covers the tools React gives you to control re-rendering and bundle size — React.memo, useMemo, useCallback, React.lazy, code splitting, virtualization, and the concurrent features (useTransition, useDeferredValue). It also covers the React Compiler, which (in React 19) auto-memoizes your components and changes how often you should reach for manual memoization at all.

    Premature optimization is the enemy

    Wrapping everything in useMemo/useCallback/React.memo is a common anti-pattern. Each adds CPU and memory overhead (a comparison + a cached value), and clutters the code. Only optimize a path after you have profiled it and confirmed it is a real bottleneck. Read When NOT to use them below.

    1. React.memo for Component Optimization

    What is React.memo?

    By default, when a component re-renders, all of its children re-render too — even if their props did not change. React.memo is a higher-order component that memoizes the rendered output of a component. React skips re-rendering it when its props are shallowly equal to the previous props.

    import { memo } from "react";

interface AvatarProps {
  name: string;
  src: string;
}

// Re-renders only when `name` or `src` change identity.
const Avatar = memo(function Avatar({ name, src }: AvatarProps) {
  console.log("Avatar rendered:", name);
  return <img src={src} alt={name} className="h-10 w-10 rounded-full" />;
});

export default Avatar;

    When to Use React.memo

    React.memo pays off when all of these are true:

    • The component re-renders often with the same props.
    • The component is expensive to render (big subtree, heavy markup, costly calculations).
    • Its props are stable (primitives, or memoized objects/functions/callbacks).

    A classic case: a parent holds frequently-changing state (a text input, a timer) but renders an expensive child that does not depend on that state.

    import { memo, useState } from "react";

const ExpensiveList = memo(function ExpensiveList({ items }: { items: string[] }) {
  console.log("ExpensiveList rendered"); // does NOT log on every keystroke
  return (
    <ul>
      {items.map((item) => (
        <li key={item}>{item}</li>
      ))}
    </ul>
  );
});

function Page() {
  const [query, setQuery] = useState("");
  const [items] = useState(() => ["Apple", "Banana", "Cherry"]);

  return (
    <div>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      {/* `items` keeps the same reference, so memo skips the re-render */}
      <ExpensiveList items={items} />
    </div>
  );
}

    Preventing Unnecessary Re-renders

    React.memo only stops a re-render if the props are referentially stable. The most common reason memo "doesn't work" is passing a new object, array, or function on every render:

    // ❌ memo is useless here — a brand-new function is created every render,
//    so `onClick` is never shallow-equal to the previous one.
<Avatar name="Ada" src="/ada.png" onClick={() => doThing()} />

// ✅ Pair memo with useCallback / useMemo to keep references stable.
const handleClick = useCallback(() => doThing(), []);
<Avatar name="Ada" src="/ada.png" onClick={handleClick} />

    memo and useCallback/useMemo are a team

    React.memo checks props; useCallback/useMemo keep those props referentially stable across renders. Using memo without stabilizing object/function props usually does nothing.

    Custom Comparison Function

    React.memo accepts a second argument — a comparison function (prevProps, nextProps) => boolean. Return true to skip the render (props are "equal"), false to re-render. This is the opposite polarity of class shouldComponentUpdate.

    interface UserCardProps {
  user: { id: number; name: string };
  theme: string;
}

const UserCard = memo(
  function UserCard({ user }: UserCardProps) {
    return <div>{user.name}</div>;
  },
  (prev, next) => prev.user.id === next.user.id // re-render only when id changes
);

    Use custom comparison sparingly

    A custom comparator runs on every render and is easy to get wrong (forget a prop and you ship stale UI). It also tempts deep comparisons, which can be slower than just re-rendering. Prefer fixing the data flow (stable references, smaller props) over a hand-written comparator.

    Limitations of React.memo

    • It performs a shallow comparison only. New object/array/function identities defeat it.
    • It does not prevent re-renders caused by useState/useReducer inside the component, or by a useContext value changing.
    • It adds overhead: a props comparison plus memory to cache the result. For cheap components this is a net loss.
    • It is not a semantic guarantee — React may still choose to re-render. Treat it as a performance hint, never as a way to skip side effects.

    2. useCallback & useMemo Hooks

    Both hooks cache a value between renders so its identity stays stable as long as the dependencies don't change.

    • useMemo(fn, deps) caches the return value of fn.
    • useCallback(fn, deps) caches the function itself. useCallback(fn, deps) is exactly useMemo(() => fn, deps).

    How useMemo Optimizes Computation-heavy Operations

    Use useMemo to avoid re-running an expensive calculation on every render when its inputs haven't changed.

    import { useMemo, useState } from "react";

function ProductTable({ products }: { products: Product[] }) {
  const [filter, setFilter] = useState("");

  // Only re-sorts/filters when `products` or `filter` change,
  // not when some unrelated state elsewhere triggers a render.
  const visible = useMemo(() => {
    console.log("recomputing visible products");
    return products
      .filter((p) => p.name.toLowerCase().includes(filter.toLowerCase()))
      .sort((a, b) => a.price - b.price);
  }, [products, filter]);

  return (
    <>
      <input value={filter} onChange={(e) => setFilter(e.target.value)} />
      <ul>{visible.map((p) => <li key={p.id}>{p.name}</li>)}</ul>
    </>
  );
}

    How useCallback Prevents Unnecessary Function Recreation

    Functions are recreated on every render. That's usually fine — but it matters when the function is:

    1. passed as a prop to a React.memo-wrapped child, or
    2. listed in another hook's dependency array (e.g. a useEffect that would otherwise re-run every render).
    import { memo, useCallback, useState } from "react";

const Button = memo(function Button({ onClick, label }: { onClick: () => void; label: string }) {
  console.log("Button rendered:", label);
  return <button onClick={onClick}>{label}</button>;
});

function Toolbar() {
  const [count, setCount] = useState(0);

  // Same function identity across renders → memoized Button won't re-render.
  const increment = useCallback(() => setCount((c) => c + 1), []);

  return (
    <div>
      <span>{count}</span>
      <Button onClick={increment} label="Increment" />
    </div>
  );
}

    Use the updater form to shrink dependencies

    setCount((c) => c + 1) lets the dependency array stay []. If you wrote setCount(count + 1) you'd need [count], recreating the callback on every change and defeating the point.

    Practical Examples Together

    A typical "real" use: a memoized child receiving both a derived list and a stable handler.

    function Dashboard({ rows }: { rows: Row[] }) {
  const [selectedId, setSelectedId] = useState<number | null>(null);

  const totals = useMemo(
    () => rows.reduce((sum, r) => sum + r.amount, 0),
    [rows]
  );

  const handleSelect = useCallback((id: number) => setSelectedId(id), []);

  return <MemoizedTable rows={rows} total={totals} onSelect={handleSelect} />;
}

    When NOT to use useMemo and useCallback

    This is the part people skip. Most useMemo/useCallback calls in real codebases are unnecessary and slightly harmful.

    Skip them when:

    • The calculation is cheap (string concat, simple arithmetic, mapping a short array). The memo machinery costs more than the work it saves.
    • The value/function is not passed to a memoized child and not a hook dependency. Stabilizing identity buys you nothing.
    • You're memoizing a value used only in JSX of the same component (it re-renders anyway).
    // ❌ Pointless — the work is trivial and `full` isn't a dependency anywhere.
const full = useMemo(() => `${first} ${last}`, [first, last]);

// ✅ Just compute it.
const full = `${first} ${last}`;

    The mental model

    useMemo/useCallback are caches. A cache only helps if (a) recomputing is expensive, or (b) reference identity is load-bearing. If neither is true, you're paying for a cache you never read.

    React Compiler may make these obsolete

    If you adopt the React Compiler (see §7), you should stop adding useMemo/useCallback by hand for performance — the compiler inserts the right memoization automatically. The hooks remain valid React; you just rarely need to write them yourself.

    3. Lazy Loading & React Suspense

    What is Lazy Loading?

    Lazy loading means deferring the download of code (or data, or images) until it is actually needed. Instead of shipping one giant JavaScript bundle on first load, you split it and fetch pieces on demand. This shrinks the initial bundle, improving time-to-interactive.

    React.lazy for Component Splitting

    React.lazy takes a function that returns a dynamic import() and gives you back a component you render normally. The bundler creates a separate chunk for that module.

    import { lazy, Suspense } from "react";

// `SettingsPanel` is NOT in the main bundle — it's fetched when first rendered.
const SettingsPanel = lazy(() => import("./SettingsPanel"));

function App() {
  const [showSettings, setShowSettings] = useState(false);
  return (
    <div>
      <button onClick={() => setShowSettings(true)}>Open settings</button>
      <Suspense fallback={<Spinner />}>
        {showSettings && <SettingsPanel />}
      </Suspense>
    </div>
  );
}

    React.lazy needs a default export

    The imported module must export default the component. For a named export, re-map it inside the import:

    const Chart = lazy(() =>
  import("./charts").then((m) => ({ default: m.Chart }))
);

    Using <Suspense> for Fallback UI

    A lazy component must be rendered inside a <Suspense> boundary. While the chunk is loading, React shows the fallback; once it arrives, it swaps in the real UI. One boundary can wrap many lazy components, and boundaries can nest for granular loading states.

    <Suspense fallback={<PageSkeleton />}>
  <Header />
  <Suspense fallback={<ChartSpinner />}>
    <LazyChart />
  </Suspense>
</Suspense>

    Best Practices for Lazy Loading

    • Split at natural boundaries: routes, modals, tabs, heavy widgets (rich text editors, charts, maps) the user may never open.
    • Don't over-split: dozens of tiny chunks add HTTP overhead and waterfalls. Split things that are large or rarely used.
    • Give meaningful fallbacks: skeletons that match layout prevent layout shift better than a bare spinner.
    • Handle load failures with an Error Boundary around <Suspense> (a chunk can fail on flaky networks / after a redeploy).
    • Consider preloading on intent — e.g. start the import on onMouseEnter of the button that reveals it, so the chunk is ready by click time.
    // Preload pattern: kick off the import early, render the lazy component later.
const importPanel = () => import("./SettingsPanel");
const SettingsPanel = lazy(importPanel);
// ...
<button onMouseEnter={importPanel} onClick={open}>Open settings</button>

    4. Code Splitting & Dynamic Imports

    Introduction to Code Splitting

    Code splitting is the bundler-level mechanism behind lazy loading. Tools like Vite, webpack, and Rspack see a dynamic import("...") and emit that module as a separate chunk that loads at runtime. React.lazy is just the React-aware wrapper around this.

    // Dynamic import returns a Promise — works for any module, not just components.
async function exportToPdf(data: Report) {
  const { generatePdf } = await import("./pdf"); // heavy lib stays out of main bundle
  return generatePdf(data);
}

    Route-based Code Splitting with React Router

    The highest-value split is per route: a user landing on / shouldn't download the code for /admin/reports. With React Router v6.4+ (the data router API):

    import { lazy, Suspense } from "react";
import { createBrowserRouter, RouterProvider } from "react-router-dom";

const Home = lazy(() => import("./routes/Home"));
const Dashboard = lazy(() => import("./routes/Dashboard"));
const Settings = lazy(() => import("./routes/Settings"));

const router = createBrowserRouter([
  {
    path: "/",
    element: (
      <Suspense fallback={<PageSpinner />}>
        <Home />
      </Suspense>
    ),
  },
  {
    path: "/dashboard",
    element: (
      <Suspense fallback={<PageSpinner />}>
        <Dashboard />
      </Suspense>
    ),
  },
  {
    path: "/settings",
    element: (
      <Suspense fallback={<PageSpinner />}>
        <Settings />
      </Suspense>
    ),
  },
]);

export default function App() {
  return <RouterProvider router={router} />;
}

    Route splitting with lazy route config

    React Router also supports a per-route lazy property that loads the route module (component and its loader/action) on demand, which keeps loaders out of the main bundle too:

    {
  path: "/dashboard",
  lazy: () => import("./routes/Dashboard"), // module exports `Component`, `loader`, etc.
}

    Performance Benefits of Code Splitting

    • Smaller initial bundle → faster first paint and time-to-interactive.
    • Pay-as-you-go: users download only the code for screens they visit.
    • Better caching: a change to one route invalidates only that route's chunk, not the whole app.
    • Parallelism: the browser can fetch route chunks alongside data.

    Watch for request waterfalls

    Naïve splitting can create a chain: load route chunk → then component renders → then it starts its data fetch. Prefer router data loaders (React Router) or framework conventions (Next.js) that start data fetching in parallel with the chunk download, and preload the next route on link hover.

    5. Avoiding Unnecessary Re-renders

    Memoization treats symptoms. Often the better fix is structuring state and components so the re-render never happens. To do that, you need to know why React re-renders.

    Understanding React's Reconciliation Process

    When state changes, React re-renders the component and, by default, all of its descendants. Rendering produces a new virtual tree; React then reconciles it against the previous tree (a "diff") and applies the minimal set of real DOM mutations.

    Key facts that drive most performance decisions:

    • A component re-renders when: its state changes, its parent re-renders, or a context it consumes changes.
    • A component re-rendering does not mean the DOM changes — reconciliation may find nothing to update. But the render function (and its child renders) still runs, which can be the expensive part.
    • Re-rendering is top-down. Props being "the same value" does not stop it unless the child is wrapped in React.memo.

    Key Techniques to Reduce Re-renders

    a) Stable references

    Don't create new objects/arrays/functions inline when they're passed to memoized children or used as hook dependencies. Hoist constants out of the component, or memoize them.

    // ❌ new array + new style object every render
<Chart data={[1, 2, 3]} style={{ height: 300 }} />

// ✅ stable references
const CHART_DATA = [1, 2, 3];            // module scope
const CHART_STYLE = { height: 300 };
<Chart data={CHART_DATA} style={CHART_STYLE} />

    b) State colocation

    Keep state as close as possible to where it's used. If only one small component needs a piece of state, don't lift it to a big parent — that would re-render the whole subtree on every change. Move the state into the component that uses it.

    // ❌ `hover` lives in Parent, so Parent + ALL children re-render on hover.
function Parent() {
  const [hover, setHover] = useState(false);
  return (
    <>
      <HugeList />
      <Tooltip visible={hover} onEnter={() => setHover(true)} />
    </>
  );
}

// ✅ Move hover state into the only component that needs it.
function Parent() {
  return (
    <>
      <HugeList />
      <HoverableTooltip />
    </>
  );
}

    c) "Lifting state down" / push state into children

    The mirror image of "lifting state up": when frequently-changing state has crept up to a parent that renders expensive siblings, extract a small component that owns the state, so only it re-renders.

    // The expensive <Report /> re-renders on every keystroke because `text` lives here.
function Page() {
  const [text, setText] = useState("");
  return (
    <>
      <input value={text} onChange={(e) => setText(e.target.value)} />
      <Report /> {/* expensive, doesn't use `text` */}
    </>
  );
}

// ✅ Isolate the input + its state. <Report /> now never re-renders on typing.
function SearchBox() {
  const [text, setText] = useState("");
  return <input value={text} onChange={(e) => setText(e.target.value)} />;
}
function Page() {
  return (
    <>
      <SearchBox />
      <Report />
    </>
  );
}

    d) Pass children as props (the "slot" trick)

    A component that re-renders does not re-render children passed to it from above — those elements were created by the parent and keep their identity.

    // <ExpensiveChild /> is created in App, so Ticker re-rendering won't re-render it.
function Ticker({ children }: { children: React.ReactNode }) {
  const [n, setN] = useState(0);
  useEffect(() => {
    const id = setInterval(() => setN((x) => x + 1), 1000);
    return () => clearInterval(id);
  }, []);
  return <div>{n}{children}</div>;
}

function App() {
  return (
    <Ticker>
      <ExpensiveChild />
    </Ticker>
  );
}

    e) Context splitting

    Every consumer of a context re-renders when any part of that context value changes. Two fixes:

    • Split contexts by update frequency: a rarely-changing AuthContext and a fast-changing CursorContext should be separate providers, so cursor updates don't re-render auth consumers.
    • Memoize the provider value so it isn't a new object every render.
    • Separate state and dispatch: put dispatch (stable) in one context and the changing state in another, so components that only dispatch never re-render.
    const StateCtx = createContext<State | null>(null);
const DispatchCtx = createContext<Dispatch | null>(null);

function Provider({ children }: { children: React.ReactNode }) {
  const [state, dispatch] = useReducer(reducer, initial);
  // dispatch is referentially stable; only StateCtx consumers re-render on state change.
  return (
    <DispatchCtx.Provider value={dispatch}>
      <StateCtx.Provider value={state}>{children}</StateCtx.Provider>
    </DispatchCtx.Provider>
  );
}

    f) Correct, stable keys

    In lists, key tells React which item is which across renders. Bad keys cause extra DOM work, lost state, and bugs.

    // ❌ Index keys break when the list reorders/inserts — React reuses the wrong DOM nodes,
//    so input focus, animations, and component state get mismatched.
{items.map((item, i) => <Row key={i} item={item} />)}

// ✅ Stable, unique id from the data.
{items.map((item) => <Row key={item.id} item={item} />)}

    Never use Math.random() (or a fresh id) as a key

    A key that changes every render forces React to unmount and remount the element each time — the opposite of optimization, and it destroys component state.

    Class-Component Equivalents (for legacy code)

    The functional techniques above replace the older class APIs you'll still see in existing code:

    • shouldComponentUpdate(nextProps, nextState) — return false to skip the update. The class-era ancestor of React.memo's comparator (note the inverted return value).
    • React.PureComponent — a base class that implements shouldComponentUpdate with a shallow props/state comparison. It is the class equivalent of wrapping a function component in React.memo.
    import { PureComponent } from "react";

class Row extends PureComponent<{ item: Item }> {
  render() {
    return <li>{this.props.item.name}</li>; // skips render when item is shallow-equal
  }
}

    6. List Virtualization for Large Lists

    What is Virtualization?

    Rendering 10,000 rows means 10,000 DOM nodes — slow to mount, heavy on memory, janky to scroll. Virtualization (a.k.a. windowing) renders only the handful of items currently visible in the viewport (plus a small overscan buffer) and recycles them as you scroll. The list appears full-height via a spacer, but only ~20 rows exist in the DOM at any moment.

    Choosing a Library

    Don't build this by hand. The current recommendations:

    Library Notes
    @tanstack/react-virtual Modern, headless (you render the markup), tiny, hook-based, great TS support, supports dynamic/variable sizes. A strong default for new code.
    react-window Small, battle-tested, component-based API. Good when you want a simple drop-in for fixed/variable-size lists and grids.
    react-virtualized The older, heavier predecessor of react-window (same author). More features (tables, autosizers) but a larger bundle. Prefer react-window/@tanstack/react-virtual for new projects.

    Example: @tanstack/react-virtual

    import { useRef } from "react";
import { useVirtualizer } from "@tanstack/react-virtual";

function VirtualList({ rows }: { rows: string[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count: rows.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 35, // estimated row height in px
    overscan: 5,
  });

  return (
    <div ref={parentRef} style={{ height: 400, overflow: "auto" }}>
      {/* full-height spacer so the scrollbar reflects the real list length */}
      <div style={{ height: virtualizer.getTotalSize(), position: "relative" }}>
        {virtualizer.getVirtualItems().map((virtualRow) => (
          <div
            key={virtualRow.key}
            style={{
              position: "absolute",
              top: 0,
              left: 0,
              width: "100%",
              height: virtualRow.size,
              transform: `translateY(${virtualRow.start}px)`,
            }}
          >
            {rows[virtualRow.index]}
          </div>
        ))}
      </div>
    </div>
  );
}

    Example: react-window

    import { FixedSizeList } from "react-window";

function VirtualList({ rows }: { rows: string[] }) {
  return (
    <FixedSizeList height={400} width="100%" itemCount={rows.length} itemSize={35}>
      {({ index, style }) => <div style={style}>{rows[index]}</div>}
    </FixedSizeList>
  );
}

    When to Use Virtualization

    • Use it for long, scrollable lists/grids/tables — hundreds to thousands of items (chat logs, feeds, data grids, file trees).
    • Don't use it for short lists (a few dozen items) — the complexity isn't worth it, and you lose native browser features.
    • Trade-offs to know: Ctrl+F find-in-page only sees rendered rows; SEO crawlers see only what's in the DOM; sticky headers, variable heights, and horizontal scroll need extra care.

    Combine with pagination / infinite scroll

    Virtualization controls DOM size; it doesn't reduce data size. For huge datasets, pair it with server-side pagination or infinite loading so you never hold the entire dataset in memory.

    7. The React Compiler (React 19)

    What it Does

    The React Compiler is a build-time tool (a Babel/SWC plugin) that automatically memoizes your components and hooks. It analyzes your code and inserts the equivalent of useMemo, useCallback, and React.memo precisely where they help — far more thoroughly and correctly than humans typically do by hand.

    It became stable in 2025 and works with React 17+ via a runtime helper, though it shines with React 19.

    // You write plain, readable React:
function ProductTable({ products, filter }: Props) {
  const visible = products
    .filter((p) => p.name.includes(filter))
    .sort((a, b) => a.price - b.price);
  return <List items={visible} />;
}

// The compiler transforms this so `visible` is recomputed only when
// `products` or `filter` change, and <List /> is skipped when `items` is stable —
// no manual useMemo/memo required.

    How to Think About Manual Memoization With the Compiler

    The new default with the compiler

    With the React Compiler enabled, stop writing useMemo/useCallback/React.memo for performance. Write clean, idiomatic React and let the compiler do the memoization. Manual memoization becomes redundant (and the compiler often does it better).

    A few nuances:

    • Existing useMemo/useCallback keep working — you don't have to rip them out, and the compiler won't double-memoize. But new code shouldn't add them for perf.
    • useMemo for semantics (not perf) is still occasionally meaningful — e.g. guaranteeing a referential identity that an effect's dependency relies on — though the compiler usually handles this too.
    • The compiler requires your components to follow the Rules of React (pure render, no mutation of props/state during render). Enable the eslint-plugin-react-hooks rules; the compiler will bail out (and just leave code un-optimized) on components it can't prove are safe, rather than break them.
    • You can adopt it incrementally — enable it for a directory, or opt specific files in/out — so there's no big-bang migration.

    It does not replace the structural techniques

    The compiler memoizes; it does not restructure your component tree, split your contexts, fix bad keys, or virtualize long lists. The lessons in §5 and §6 still apply.

    8. Keeping the UI Responsive: useTransition & useDeferredValue

    React 18 introduced concurrent features that let you mark some updates as non-urgent, so React can keep the UI responsive (typing, clicking) while heavy work happens in the background. These complement memoization — they're about prioritizing work, not skipping it.

    useTransition

    Wrap a state update you're willing to interrupt in startTransition. The urgent part (the input value) updates immediately; the expensive part (filtering a huge list) is marked as a transition and won't block typing. isPending lets you show a subtle loading hint.

    import { useState, useTransition } from "react";

function Search({ allItems }: { allItems: Item[] }) {
  const [query, setQuery] = useState("");
  const [results, setResults] = useState<Item[]>(allItems);
  const [isPending, startTransition] = useTransition();

  function handleChange(value: string) {
    setQuery(value); // urgent: keeps the input snappy

    startTransition(() => {
      // non-urgent: can be interrupted by faster keystrokes
      setResults(allItems.filter((i) => i.name.includes(value)));
    });
  }

  return (
    <>
      <input value={query} onChange={(e) => handleChange(e.target.value)} />
      {isPending && <span>Updating…</span>}
      <ResultList items={results} />
    </>
  );
}

    useDeferredValue

    A simpler alternative when you don't own the state setter: derive a deferred copy of a value that "lags behind" during heavy renders. Great for feeding an expensive child off a fast-changing input.

    import { useDeferredValue, useMemo, useState } from "react";

function Search({ allItems }: { allItems: Item[] }) {
  const [query, setQuery] = useState("");
  const deferredQuery = useDeferredValue(query); // lags during heavy work

  const results = useMemo(
    () => allItems.filter((i) => i.name.includes(deferredQuery)),
    [allItems, deferredQuery]
  );

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <ResultList items={results} />
    </>
  );
}

    Which one?

    Use useTransition when you control the state update (you can wrap the setter). Use useDeferredValue when the value comes from props or you can't wrap the update. Both keep the input responsive while the expensive subtree catches up.

    9. Measuring Performance

    You cannot optimize what you don't measure. Always profile before and after a change to confirm it actually helped.

    React DevTools Profiler

    The React Developer Tools browser extension has a Profiler tab:

    1. Click Record, interact with the app, then stop.
    2. The flamegraph shows which components rendered and how long each took.
    3. The ranked chart lists components by render time — your worst offenders.
    4. Enable "Highlight updates when components render" (in DevTools settings) to see re-renders flash on screen — the fastest way to spot a component that re-renders when it shouldn't.

    DevTools can also tell you why a component rendered (props changed, state changed, parent rendered, hooks changed) — turn on "Record why each component rendered" in the Profiler settings.

    The <Profiler> API

    For programmatic measurement (e.g. logging to analytics, or in tests), wrap any subtree in <Profiler>:

    import { Profiler, type ProfilerOnRenderCallback } from "react";

const onRender: ProfilerOnRenderCallback = (
  id,          // the `id` prop of the Profiler tree
  phase,       // "mount" | "update" | "nested-update"
  actualDuration, // time spent rendering this update
  baseDuration,   // estimated time to render the whole subtree without memoization
  startTime,
  commitTime
) => {
  console.log(`${id} [${phase}] took ${actualDuration.toFixed(2)}ms`);
};

function App() {
  return (
    <Profiler id="Sidebar" onRender={onRender}>
      <Sidebar />
    </Profiler>
  );
}

    <Profiler> adds overhead — keep it out of hot production paths

    The <Profiler> component itself costs a little CPU and memory. React strips it from production builds by default unless you opt into the profiling build. Use it for investigation and synthetic measurement, not as always-on instrumentation in production-critical paths.

    Don't forget the basics

    Browser DevTools' Performance panel, Lighthouse, and the Web Vitals (LCP, INP, CLS) measure what the user actually feels. A component that renders in 3 ms but ships a 2 MB bundle is still slow to the user — code splitting (§3–4) often beats render-time micro-optimization.

    Summary & Checklist

    A practical order of operations when something feels slow:

    1. Profile first. Use the React DevTools Profiler and browser Performance panel to find the actual bottleneck. Don't guess.
    2. Fix structure before adding memo. Colocate state, lift state down, split contexts, pass children as props, and use stable keys. These remove re-renders instead of skipping them.
    3. Reach for useMemo/useCallback/React.memo only for proven-expensive renders or load-bearing reference identity — and skip them entirely if you've enabled the React Compiler.
    4. Keep the UI responsive during heavy updates with useTransition / useDeferredValue.
    5. Shrink the bundle with React.lazy + <Suspense> and route-based code splitting.
    6. Virtualize long lists with @tanstack/react-virtual or react-window.
    7. Measure again to confirm the change actually helped — and revert it if it didn't.

    Remember

    The fastest code is the code that never runs. Most "React performance" wins come from not re-rendering and not shipping code, not from clever memoization.
