    React Testing and Debugging

    Tests give you the confidence to change code without breaking it, and good debugging tools turn "why is this broken?" into a five-minute fix instead of an afternoon. This chapter is the general testing reference for the guide: how to set up a test runner, write unit and integration tests, mock the network, run end-to-end (E2E) tests, and debug a running app.

    Form-specific tests live in the Forms chapter

    The React Forms chapter has a dedicated Testing Forms section (controlled forms, React Hook Form, validation/error assertions). This chapter covers the general toolkit and patterns; head there for form-focused examples instead of duplicating them here.

    1. Introduction to Testing

    Why Test?

    • Catch regressions early — a failing test points at the broken commit instead of an angry user.
    • Document behavior — a test is an executable spec of what a component should do.
    • Enable refactoring — you can restructure internals freely as long as behavior stays green.
    • Speed up reviews — reviewers trust a PR with tests more than one without.

    The Testing Pyramid

    Think of your test suite as a pyramid: lots of fast, cheap tests at the bottom, fewer slow, expensive tests at the top.

            ▲   E2E  (Playwright / Cypress)   ← few, slow, high confidence
       ▲▲▲  Integration  (RTL + MSW)      ← some, medium speed
      ▲▲▲▲▲ Unit  (Vitest / Jest)         ← many, fast, isolated
    Layer Scope Tools Speed
    Unit A single function or component in isolation Vitest / Jest Milliseconds
    Integration Several units working together (component + state + DOM) React Testing Library Fast
    E2E The whole app in a real browser, like a user Playwright / Cypress Seconds

    In practice the line between "unit" and "integration" in React is blurry — most component tests with React Testing Library are really small integration tests, and that's a good thing.

    Test Behavior, Not Implementation

    The single most important testing principle in React:

    The more your tests resemble the way your software is used, the more confidence they can give you. — Kent C. Dodds, author of React Testing Library

    • Do assert on what the user sees and does: visible text, roles, what happens after a click.
    • Don't assert on internal state variables, the names of useState hooks, or how many times a function rendered.

    If you rename a state variable or swap useState for useReducer and your tests break even though the UI behaves identically, your tests were coupled to the implementation.

    // ❌ Coupled to implementation — breaks on harmless refactors
expect(wrapper.state("count")).toBe(1);

// ✅ Coupled to behavior — survives refactors
expect(screen.getByText("Count: 1")).toBeInTheDocument();

    2. Unit Testing

    Choosing a Runner: Vitest vs. Jest

    This guide uses Vite, so Vitest is the natural choice — it reuses your existing vite.config (including path aliases and plugins), starts fast, and has a Jest-compatible API. Jest remains the most common runner overall and is the default in Create React App / many Next.js setups, so you'll see it everywhere.

    Vitest (recommended here) Jest (alternative)
    Config Reuses vite.config.ts Separate jest.config + Babel/ts-jest
    Speed Very fast (esbuild, native ESM) Fast, but heavier ESM setup
    API describe / it / expect / vi describe / it / expect / jest
    Mock fn vi.fn(), vi.mock() jest.fn(), jest.mock()

    The test code is almost identical

    Apart from the mocking namespace (vi vs jest) and config, the assertions and queries are the same. Examples below use Vitest; the Jest equivalent is noted inline.

    Setting Up Vitest

    npm install -D vitest @testing-library/react @testing-library/user-event \
  @testing-library/jest-dom jsdom

    Configure Vitest inside your Vite config (or a vitest.config.ts):

    // vite.config.ts
/// <reference types="vitest/config" />
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  test: {
    globals: true,            // use describe/it/expect without importing
    environment: "jsdom",     // a fake DOM so components can render
    setupFiles: "./src/test/setup.ts",
    css: true,
  },
});

    // src/test/setup.ts
import "@testing-library/jest-dom"; // adds matchers like toBeInTheDocument()

    Add a script and run it:

    // package.json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run",
    "coverage": "vitest run --coverage"
  }
}

    npm test          # watch mode
npm run test:run  # single run (CI)

    Jest setup, for comparison

    With Jest you install jest @testing-library/* jest-environment-jsdom, set testEnvironment: "jsdom" and setupFilesAfterEnv: ["@testing-library/jest-dom"] in jest.config.js, and run npx jest.

    Writing a Basic Unit Test

    Pure functions are the easiest thing to test — no DOM, no rendering.

    // math.ts
export const add = (a: number, b: number) => a + b;
export const slugify = (s: string) =>
  s.trim().toLowerCase().replace(/\s+/g, "-");

    // math.test.ts
import { describe, it, expect } from "vitest"; // omit with `globals: true`
import { add, slugify } from "./math";

describe("add", () => {
  it("adds two numbers", () => {
    expect(add(2, 3)).toBe(5);
  });
});

describe("slugify", () => {
  it("lowercases and dashes spaces", () => {
    expect(slugify("  Hello World ")).toBe("hello-world");
  });
});

    Anatomy: describe groups related tests, it (alias test) is one case, and expect(...).matcher(...) makes the assertion. Common matchers: toBe, toEqual (deep), toContain, toThrow, toHaveBeenCalledWith, and the jest-dom DOM matchers (toBeInTheDocument, toHaveTextContent, toBeDisabled).

    Mocking Functions and Modules

    A mock replaces a real dependency with a controllable stand-in so a test stays fast, isolated, and deterministic.

    Spy / mock function — verify it was called, and control its return value:

    import { vi, expect, it } from "vitest"; // jest.fn() / jest.spyOn() with Jest

it("calls the callback with the new value", () => {
  const onChange = vi.fn();           // a mock function
  onChange("hello");

  expect(onChange).toHaveBeenCalledTimes(1);
  expect(onChange).toHaveBeenCalledWith("hello");
});

it("controls a return value", () => {
  const getUser = vi.fn().mockReturnValue({ id: 1, name: "Ada" });
  expect(getUser().name).toBe("Ada");
});

    Module mock — replace an entire imported module (e.g. an API client):

    // userService.ts
export async function fetchUser(id: number) {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

    // profile.test.ts
import { vi, expect, it, beforeEach } from "vitest";
import { fetchUser } from "./userService";

vi.mock("./userService", () => ({
  fetchUser: vi.fn(),
}));

beforeEach(() => {
  vi.mocked(fetchUser).mockResolvedValue({ id: 1, name: "Ada" });
});

it("returns the mocked user", async () => {
  await expect(fetchUser(1)).resolves.toEqual({ id: 1, name: "Ada" });
});

    Reset mocks between tests

    Call vi.clearAllMocks() (or set clearMocks: true in config) in a beforeEach/afterEach so call history from one test never leaks into the next. Mock timers with vi.useFakeTimers() to test setTimeout/setInterval/debounce logic without real waiting.

    Snapshot Testing

    A snapshot records a component's rendered output to a .snap file; later runs compare against it and fail if the output changed.

    import { render } from "@testing-library/react";
import { expect, it } from "vitest";
import Badge from "./Badge";

it("matches the snapshot", () => {
  const { asFragment } = render(<Badge label="New" />);
  expect(asFragment()).toMatchSnapshot();
});

    The first run writes the snapshot; commit it. When output changes intentionally, update with vitest -u (jest -u).

    Use snapshots sparingly

    Large snapshots are noisy — reviewers rubber-stamp diffs they don't understand, and every styling tweak turns them red. Prefer small, focused snapshots or explicit toBeInTheDocument assertions for most cases. Use inline snapshots (toMatchInlineSnapshot()) for tiny serializable values.

    3. React Testing Library (RTL)

    Why RTL (and Not Enzyme)?

    React Testing Library renders a component into a real (jsdom) DOM and lets you query and interact with it the way a user would. It deliberately gives you no access to component internals — that's what keeps tests resilient.

    Enzyme is effectively dead — do not use it for new projects

    Enzyme (Airbnb) pioneered shallow rendering and internal-state inspection, but it relies on private React internals and has no official adapter for React 18 or 19. It is unmaintained for modern React. The react-list outline mentions Enzyme, Shallow rendering, and wrapper.state() for historical context only — all new tests should use React Testing Library. If you inherit an Enzyme suite, migrate it to RTL over time.

    React Testing Library Enzyme (legacy)
    Philosophy Test behavior via the DOM Test implementation / internals
    React 18/19 ✅ First-class ❌ No official adapter
    Shallow render ✗ (intentionally) ✓ (shallow)
    Reads state directly ✗ ✓ (wrapper.state())
    Status Actively maintained, recommended Unmaintained

    Rendering a Component

    // Counter.tsx
import { useState } from "react";

export default function Counter({ start = 0 }: { start?: number }) {
  const [count, setCount] = useState(start);
  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount((c) => c + 1)}>Increment</button>
    </div>
  );
}

    // Counter.test.tsx
import { render, screen } from "@testing-library/react";
import { describe, it, expect } from "vitest";
import Counter from "./Counter";

describe("Counter", () => {
  it("renders the initial count", () => {
    render(<Counter start={3} />);
    expect(screen.getByText("Count: 3")).toBeInTheDocument();
  });
});

    render mounts the component; screen is the global query object scoped to the rendered output. RTL auto-cleans the DOM between tests (with globals: true).

    Queries: Prefer Roles and Labels

    Queries come in three variants and several "by" suffixes:

    Variant Returns Throws if not found? Use for
    getBy* element yes element that should already be present
    queryBy* element or null no asserting absence
    findBy* Promise<element> yes (after timeout) element that appears async

    Prefer queries in this priority order (most accessible first): getByRole → getByLabelText → getByPlaceholderText → getByText → … → getByTestId (last resort).

    screen.getByRole("button", { name: /increment/i }); // ✅ best — role + accessible name
screen.getByLabelText(/email/i);                     // ✅ form fields
screen.getByText(/count: 3/i);                       // ✅ visible text
screen.queryByText(/error/i);                        // ✅ assert it's NOT there
screen.getByTestId("counter");                       // ⚠️ only when nothing else fits

    // Asserting absence with queryBy*
expect(screen.queryByRole("alert")).not.toBeInTheDocument();

    Simulating User Events with user-event

    @testing-library/user-event fires realistic event sequences (a click fires pointerdown, mousedown, focus, mouseup, click) — far closer to reality than the low-level fireEvent. Always await it and create a session with setup().

    import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { describe, it, expect } from "vitest";
import Counter from "./Counter";

describe("Counter interactions", () => {
  it("increments when the button is clicked", async () => {
    const user = userEvent.setup();
    render(<Counter />);

    await user.click(screen.getByRole("button", { name: /increment/i }));
    await user.click(screen.getByRole("button", { name: /increment/i }));

    expect(screen.getByText("Count: 2")).toBeInTheDocument();
  });
});

    Common actions: user.type(input, "text"), user.clear(input), user.selectOptions(select, "value"), user.keyboard("{Enter}"), user.tab(), user.upload(input, file).

    fireEvent still exists

    fireEvent.click(el) dispatches a single raw event and is synchronous. Use it for edge cases (e.g. firing a scroll or change directly), but reach for user-event by default — it catches bugs fireEvent misses (like clicking a disabled button).

    Asserting DOM Changes

    jest-dom adds expressive matchers:

    expect(el).toBeInTheDocument();
expect(el).toBeVisible();
expect(el).toHaveTextContent("Saved");
expect(button).toBeDisabled();
expect(input).toHaveValue("hello");
expect(checkbox).toBeChecked();
expect(el).toHaveClass("active");
expect(el).toHaveAttribute("aria-expanded", "true");

    Testing Asynchronous Behavior

    For UI that updates after a fetch, timeout, or promise, use findBy* (retries until the element appears) or waitFor (retries an assertion). Never use a bare setTimeout to "wait."

    // UserCard.tsx
import { useEffect, useState } from "react";

export default function UserCard({ id }: { id: number }) {
  const [name, setName] = useState<string | null>(null);

  useEffect(() => {
    fetch(`/api/users/${id}`)
      .then((r) => r.json())
      .then((u) => setName(u.name));
  }, [id]);

  if (!name) return <p>Loading…</p>;
  return <h2>{name}</h2>;
}

    // UserCard.test.tsx
import { render, screen, waitFor } from "@testing-library/react";
import { describe, it, expect, vi, beforeEach } from "vitest";
import UserCard from "./UserCard";

beforeEach(() => {
  vi.spyOn(globalThis, "fetch").mockResolvedValue({
    ok: true,
    json: async () => ({ name: "Ada Lovelace" }),
  } as Response);
});

describe("UserCard", () => {
  it("shows the loading state then the user", async () => {
    render(<UserCard id={1} />);

    // Present immediately
    expect(screen.getByText(/loading/i)).toBeInTheDocument();

    // Appears after the fetch resolves — findBy* awaits it
    expect(await screen.findByRole("heading", { name: /ada lovelace/i }))
      .toBeInTheDocument();

    // Loading text is now gone
    await waitFor(() =>
      expect(screen.queryByText(/loading/i)).not.toBeInTheDocument()
    );
  });
});

    Wrap shared providers in a custom render

    If components need Router, theme, or store context, write a renderWithProviders helper that wraps render with your providers and re-export it from a single test-utils file. See the State Management and React Router chapters for the providers you'll need.

    4. Mocking the Network with MSW

    Spying on fetch (as above) works for one call, but it gets brittle fast. Mock Service Worker (MSW) intercepts requests at the network layer using handlers, so your component code runs unchanged — it makes a real fetch, and MSW answers. The same handlers work in tests, in Storybook, and in the browser during development.

    npm install -D msw

    Define handlers and a server:

    // src/test/handlers.ts
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("/api/users/:id", ({ params }) =>
    HttpResponse.json({ id: Number(params.id), name: "Ada Lovelace" })
  ),

  http.post("/api/users", async ({ request }) => {
    const body = (await request.json()) as { name: string };
    return HttpResponse.json({ id: 99, ...body }, { status: 201 });
  }),
];

    // src/test/server.ts
import { setupServer } from "msw/node";
import { handlers } from "./handlers";

export const server = setupServer(...handlers);

    Wire it into your test setup so the server starts once and resets between tests:

    // src/test/setup.ts
import "@testing-library/jest-dom";
import { beforeAll, afterEach, afterAll } from "vitest";
import { server } from "./server";

beforeAll(() => server.listen({ onUnhandledRequest: "error" }));
afterEach(() => server.resetHandlers()); // undo per-test overrides
afterAll(() => server.close());

    Now UserCard from the previous section needs no fetch spy — MSW handles it. To test an error or empty state, override a handler for just that test:

    import { http, HttpResponse } from "msw";
import { server } from "../test/server";
import { render, screen } from "@testing-library/react";
import UserCard from "./UserCard";

it("handles a server error", async () => {
  server.use(
    http.get("/api/users/:id", () =>
      HttpResponse.json({ message: "Boom" }, { status: 500 })
    )
  );

  render(<UserCard id={1} />);
  expect(await screen.findByText(/loading/i)).toBeInTheDocument();
  // ...assert your component's error UI here
});

    MSW v2 uses http and HttpResponse

    MSW v2 (current) replaced the v1 rest.get / res(ctx.json(...)) API with the http namespace and the standard Response-based HttpResponse. If you find rest-based examples online, they're for the old v1.

    5. End-to-End (E2E) Testing

    E2E tests drive a real browser against your running app, exercising the full stack the way a user does. They're slower and pricier than unit tests, so keep them for critical flows (login, checkout, the happy path of each major feature).

    Playwright (recommended)

    Playwright (Microsoft) is the modern default: one tool runs Chromium, Firefox, and WebKit; it has built-in auto-waiting (no flaky manual sleeps), parallelism, tracing, and a codegen recorder. Its assertions auto-retry until they pass or time out.

    npm init playwright@latest
# scaffolds playwright.config.ts, a tests/ folder, and installs browsers

    // playwright.config.ts (essentials)
import { defineConfig, devices } from "@playwright/test";

export default defineConfig({
  testDir: "./e2e",
  use: { baseURL: "http://localhost:5173", trace: "on-first-retry" },
  webServer: {
    command: "npm run dev",        // Playwright starts your app for you
    url: "http://localhost:5173",
    reuseExistingServer: !process.env.CI,
  },
  projects: [{ name: "chromium", use: { ...devices["Desktop Chrome"] } }],
});

    Basic test + navigation:

    // e2e/home.spec.ts
import { test, expect } from "@playwright/test";

test("navigates from home to the about page", async ({ page }) => {
  await page.goto("/");
  await expect(page.getByRole("heading", { name: /welcome/i })).toBeVisible();

  await page.getByRole("link", { name: /about/i }).click();
  await expect(page).toHaveURL(/.*about/);
  await expect(page.getByRole("heading", { name: /about us/i })).toBeVisible();
});

    Filling and submitting a form:

    test("logs in and lands on the dashboard", async ({ page }) => {
  await page.goto("/login");

  await page.getByLabel(/email/i).fill("user@example.com");
  await page.getByLabel(/password/i).fill("secret123");
  await page.getByRole("button", { name: /log in/i }).click();

  await expect(page).toHaveURL(/.*dashboard/);
  await expect(page.getByText(/welcome back/i)).toBeVisible();
});

    Mocking an API response with page.route:

    test("shows users from a mocked API", async ({ page }) => {
  await page.route("**/api/users", (route) =>
    route.fulfill({
      status: 200,
      contentType: "application/json",
      body: JSON.stringify([{ id: 1, name: "Ada Lovelace" }]),
    })
  );

  await page.goto("/users");
  await expect(page.getByText("Ada Lovelace")).toBeVisible();
});

    Playwright queries feel like RTL

    getByRole, getByLabel, and getByText exist in Playwright too, so the accessibility-first habits you build with RTL carry straight over. Run npx playwright codegen localhost:5173 to record clicks into a starter test, and npx playwright test --ui for the time-travel debugger.

    Cypress (alternative)

    Cypress is the long-standing, popular E2E tool with an excellent interactive runner (watch commands replay live in a real browser) and a gentle learning curve. It runs in Chromium-family browsers, Firefox, and WebKit (experimental). Its API is chainable and also auto-retries.

    npm install -D cypress
npx cypress open   # interactive runner

    // cypress/e2e/login.cy.ts
describe("login flow", () => {
  it("logs in and reaches the dashboard", () => {
    // Mock the API
    cy.intercept("GET", "/api/users", {
      statusCode: 200,
      body: [{ id: 1, name: "Ada Lovelace" }],
    }).as("getUsers");

    cy.visit("/login");
    cy.findByLabelText(/email/i).type("user@example.com");
    cy.findByLabelText(/password/i).type("secret123");
    cy.findByRole("button", { name: /log in/i }).click();

    cy.url().should("include", "/dashboard");
    cy.contains(/welcome back/i).should("be.visible");
  });
});
    Playwright (recommended) Cypress
    Browsers Chromium, Firefox, WebKit (all native) Chromium-family, Firefox, WebKit (experimental)
    Architecture Out-of-process, true parallelism Runs in the browser event loop
    Multi-tab / multi-origin ✅ Native Limited
    Auto-wait ✅ ✅
    Best feature Tracing + codegen + multi-browser Interactive time-travel runner

    Add @testing-library/cypress for RTL-style queries

    Cypress's default selectors are CSS-based; install @testing-library/cypress to get cy.findByRole / cy.findByLabelText, keeping your accessibility-first habits.

    6. Debugging React Applications

    React Developer Tools

    Install React Developer Tools as a browser extension (Chrome, Firefox, Edge) — or use the standalone react-devtools package for React Native / non-browser targets. It adds two tabs to your browser DevTools: Components and Profiler.

    Components Tab — Inspect State & Props

    • Browse the live component tree and search by component name.
    • Select a component to see its props, state, hooks, and context in the right panel.
    • Edit values inline — change a prop or state value and watch the UI update, no code change needed.
    • Hover a component to highlight the DOM it renders; click the inspector to go the other way (DOM → component).
    • Right-click → "Show source" jumps to where the component is defined; the bug icon logs the selected component's data to the console.

    Find why a value is what it is

    With a component selected, the Hooks section lists every hook in order. This is how you spot a stale closure, an unexpected context value, or a state that never updated — without scattering console.log everywhere.

    Profiler Tab — Find Slow Renders

    The Profiler records why and how long components render:

    1. Open Profiler → click Record → interact with your app → Stop.
    2. The flamegraph shows each commit; wider/colored bars rendered longer.
    3. The ranked chart lists components by render time, worst first.
    4. Select a component to see "Why did this render?" (props changed, state changed, hooks changed, or parent re-rendered).

    Use it to confirm an optimization actually helped — e.g. before/after adding React.memo, useMemo, or useCallback. (See the Performance Optimization chapter for those techniques.)

    Highlight updates

    In the Components tab settings, enable "Highlight updates when components render" to get a flashing outline on every re-render. Lots of flashing on an idle screen is a smell — something is re-rendering more than it should.

    Common Debugging Techniques

    • console.log / console.table — quick and underrated; console.table(arr) prints arrays of objects as a grid.
    • debugger statement — pauses execution at that line when DevTools is open; far better than logging for stepping through logic.
    • screen.debug() — inside an RTL test, prints the current DOM so you can see what the test actually rendered. screen.logTestingPlaygroundURL() suggests the best query.
    • Source maps + breakpoints — set breakpoints directly in your .tsx source in the browser's Sources panel.
    • Strict Mode double-invoke — in development, <StrictMode> intentionally renders components and runs effects twice to surface impurities and missing cleanup. Don't "fix" the double log; fix the impure code it reveals.
    • The dependency-array lint rule — eslint-plugin-react-hooks catches missing useEffect/useCallback dependencies, the source of countless stale-data bugs.

    Error Boundaries

    A JavaScript error during rendering will, by default, unmount your whole React tree — the user sees a blank page. An error boundary is a component that catches errors in its child tree and renders a fallback UI instead, while logging the error for you.

    // ErrorBoundary.tsx
import { Component, type ErrorInfo, type ReactNode } from "react";

interface Props {
  fallback: ReactNode;
  children: ReactNode;
}
interface State {
  hasError: boolean;
}

export class ErrorBoundary extends Component<Props, State> {
  state: State = { hasError: false };

  // Render the fallback on the next render
  static getDerivedStateFromError(): State {
    return { hasError: true };
  }

  // Side effect: log to your monitoring service (Sentry, etc.)
  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error("Caught by boundary:", error, info.componentStack);
  }

  render() {
    return this.state.hasError ? this.props.fallback : this.props.children;
  }
}

    // Usage — wrap a risky subtree
<ErrorBoundary fallback={<p>Something went wrong. Please refresh.</p>}>
  <Dashboard />
</ErrorBoundary>

    Error boundaries must be class components — but you don't have to write one

    There is no hook equivalent of getDerivedStateFromError / componentDidCatch, so the boundary itself is a class. In practice most teams use react-error-boundary, a small library that gives you a <ErrorBoundary> with a FallbackComponent, a resetErrorBoundary callback to retry, and a useErrorBoundary hook — no class needed.

    What error boundaries do NOT catch

    They only catch errors thrown during rendering, in lifecycle methods, and in constructors of the components below them. They do not catch:

    • Errors in event handlers (use a try/catch there).
    • Asynchronous errors (setTimeout, promises, fetch rejections).
    • Errors in server-side rendering.
    • Errors thrown in the boundary itself.

    Place boundaries at strategic points (per route, per widget) so one broken section doesn't take down the entire page.

    Summary

    • Test behavior, not implementation — assert on what the user sees, not internal state.
    • Follow the pyramid: many fast unit tests, some integration tests, a few E2E tests.
    • Vitest is the runner of choice for Vite projects (Jest is the common alternative); both share a near-identical API.
    • React Testing Library + user-event render and drive components like a user; query by role/label and handle async with findBy* / waitFor.
    • Enzyme is dead for modern React — use RTL for all new tests.
    • MSW mocks the network at the request layer, so component code runs unchanged across tests, Storybook, and dev.
    • Playwright is the modern E2E default (Cypress the popular alternative); both auto-wait and can mock APIs.
    • React DevTools (Components + Profiler) is your primary debugging tool; pair it with debugger, screen.debug(), and error boundaries to keep failures contained and diagnosable.
    • For form-specific tests, see the Testing Forms section of the React Forms chapter.
