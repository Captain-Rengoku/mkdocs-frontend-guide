    React SSR & SSG

    Rendering strategy decides where and when your HTML is generated: in the browser, on a server at request time, or ahead of time at build. Server-Side Rendering (SSR) and Static Site Generation (SSG) move work off the user's device, producing real HTML that ships fast and is fully crawlable—two things plain client-side React struggles with.

    This chapter starts with the concepts (CSR / SSR / SSG / ISR), then goes deep on Next.js, which is the dominant React framework for all of these. The modern App Router (Next.js 13.4+, default in 14/15) is presented as the primary approach, with the legacy Pages Router APIs covered afterward as the older model you'll still meet in existing codebases.

    📚 For more: https://nextjs.org/docs · https://react.dev/reference/rsc/server-components

    1. Introduction to SSR, SSG, CSR & ISR

    A React app is ultimately HTML + JavaScript. The rendering strategy is the answer to: who turns my components into HTML, and at what moment?

    CSR — Client-Side Rendering

    The browser downloads a near-empty HTML shell plus a JavaScript bundle, then React builds the DOM on the user's device. This is the classic SPA model (Vite + React, Create React App).

    <!-- What a crawler / slow device first receives with CSR -->
<div id="root"></div>
<script src="/bundle.js"></script>
    • ✅ Cheap to host (static files + an API), great for highly interactive, auth-gated dashboards where SEO is irrelevant.
    • ❌ Blank screen until JS downloads, parses, and runs; weaker SEO; performance scales with the user's device.

    SSR — Server-Side Rendering

    The server runs your React components on every request and returns fully-formed HTML. The browser shows content immediately, then React hydrates it to make it interactive.

    • ✅ Personalized, always-fresh content (per-user dashboards, request-time data); good SEO.
    • ❌ Server compute on every request → higher cost and latency than serving a static file; needs a running server/serverless runtime.

    SSG — Static Site Generation

    HTML is generated once at build time and served from a CDN. Every visitor gets the same pre-built file.

    • ✅ Fastest possible delivery (a file on a CDN), cheapest hosting, excellent SEO.
    • ❌ Data is frozen at build time; rebuilding a huge site can be slow.

    ISR — Incremental Static Regeneration

    SSG's pragmatic upgrade: pages are static but regenerate in the background on a schedule (or on demand) without a full rebuild. You get CDN speed with data that stays reasonably fresh—ideal for blogs, docs, and product catalogs that change occasionally.

    These are per-route, not per-app

    In modern Next.js you don't pick one strategy for the whole site. Each route chooses its own rendering mode, and you can even mix static and dynamic regions inside a single page using streaming. Think of CSR/SSR/SSG/ISR as a spectrum, not four separate buckets.

    Side-by-side

    Strategy HTML generated Data freshness Hosting cost Best for
    CSR In the browser Live (client fetch) Lowest (static + API) Auth-only apps, internal dashboards
    SSR Per request on the server Always fresh Higher (compute/request) Personalized, request-time data
    SSG Once at build Frozen at build Lowest (CDN) Marketing, docs, rarely-changing content
    ISR At build, refreshed in background Stale-then-fresh Low (CDN + occasional compute) Blogs, catalogs, content that changes sometimes

    SEO, performance & Core Web Vitals

    The whole reason these strategies exist is that what the server sends first matters.

    • SEO — Crawlers do execute JavaScript, but rendering it is slower, sometimes deferred, and unreliable for social-share/Open Graph previews. Pre-rendered HTML (SSR/SSG/ISR) gives crawlers complete content immediately.
    • Performance — Pre-rendered HTML produces meaningful content before any app JS runs, and SSG/ISR can be cached at the CDN edge.
    • Core Web Vitals — The metrics Google measures map directly onto rendering choices:
      • LCP (Largest Contentful Paint): SSR/SSG paint real content fast; CSR delays it behind the JS bundle.
      • INP (Interaction to Next Paint, the metric that replaced FID in 2024): hurt by large hydration bundles—shipping less client JS (see Server Components) helps directly.
      • CLS (Cumulative Layout Shift): pre-rendering exact markup, including sized images, avoids reflow.

    Less JavaScript is the throughline

    Across SEO and Core Web Vitals, the recurring win is shipping less JavaScript to the client. React Server Components (below) make that the default rather than an optimization you bolt on later.

    2. Benefits of SSR & SSG

    Improved SEO with pre-rendering

    Search engines and social scrapers receive the actual <h1>, copy, and <meta>/Open Graph tags in the initial response—no JS execution required. This is the single biggest reason content-driven sites adopt SSR/SSG.

    Faster initial page load

    The user sees real content as soon as the HTML arrives, instead of staring at a spinner while a bundle downloads and executes. This improves perceived performance and LCP, especially on slow networks and low-end devices.

    Caching & performance optimization

    Static and ISR pages live on a CDN as plain files—served from an edge location near the user, infinitely scalable, and cheap. SSR responses can also be cached (full-page or fragment) to absorb traffic spikes.

    Reduced client-side JavaScript

    With React Server Components, components that only render data never ship to the browser at all. Only genuinely interactive ("use client") components are hydrated. Less JS → faster parse/execute → better INP and lower bandwidth.

    SSR is not automatically faster end-to-end

    SSR adds server work on every request and a hydration step on the client. A poorly-cached SSR page with a heavy data fetch can have a worse TTFB than CSR. SSR wins on first contentful paint and SEO, not necessarily on total cost. Prefer SSG/ISR whenever the data doesn't truly need to be per-request.

    3. The modern approach: Next.js App Router

    What is Next.js?

    Next.js is a React framework (by Vercel) that provides file-system routing, server rendering, data fetching, bundling, image optimization, and serverless deployment out of the box. As of v13.4 the App Router (the app/ directory) is stable and is the default and recommended model in Next.js 14 and 15. It is built on React Server Components (RSC).

    Setting up a project

    npx create-next-app@latest my-app --typescript --app
cd my-app
npm run dev

    This scaffolds the app/ directory with TypeScript. Key conventions:

    File Purpose
    app/layout.tsx Shared shell (required root layout, wraps <html>/<body>)
    app/page.tsx UI for the / route
    app/blog/[slug]/page.tsx Dynamic route /blog/:slug
    app/loading.tsx Streaming fallback shown while a segment loads
    app/error.tsx Error boundary for a segment ("use client")
    app/api/.../route.ts A Route Handler (API endpoint)

    Server Components are the default

    In the App Router, every component is a Server Component unless you opt out. Server Components run only on the server, can be async, can await data directly, and send zero JavaScript to the browser.

    // app/page.tsx — a Server Component (no "use client" directive)
// Runs on the server. The DB call, the secret, and this code never reach the browser.
import { db } from "@/lib/db";

export default async function HomePage() {
  const posts = await db.post.findMany({ take: 10 }); // direct data access

  return (
    <main>
      <h1>Latest posts</h1>
      <ul>
        {posts.map((p) => (
          <li key={p.id}>{p.title}</li>
        ))}
      </ul>
    </main>
  );
}

    There's no getServerSideProps here—fetching is just await inside the component. This is the headline difference from the Pages Router.

    Data fetching & caching with fetch

    Next.js extends the native fetch with caching controls. The cache option you choose is what makes a route SSR vs SSG vs ISR.

    // SSR: re-fetch on every request (dynamic, request-time data)
const res = await fetch("https://api.example.com/feed", { cache: "no-store" });

// SSG: fetched once at build and cached indefinitely (static)
const res = await fetch("https://api.example.com/page", { cache: "force-cache" });

// ISR: cached, but revalidated in the background at most every 60 seconds
const res = await fetch("https://api.example.com/products", {
  next: { revalidate: 60 },
});

    Caching defaults changed in Next.js 15

    In Next.js 14, fetch was cached by default. In Next.js 15, the defaults flipped to uncached (fetch and route segments are dynamic unless you opt in), which is far less surprising. Always set cache/revalidate explicitly rather than relying on the version's default. You can also tag fetches with next: { tags: ["products"] } for targeted on-demand invalidation.

    The "use client" boundary

    When a component needs interactivity—state, effects, event handlers, browser-only APIs—mark it with "use client" at the top of the file. Everything imported into a client component becomes part of the client bundle.

    // app/components/LikeButton.tsx
"use client";
import { useState } from "react";

export default function LikeButton({ initial }: { initial: number }) {
  const [likes, setLikes] = useState(initial);
  return <button onClick={() => setLikes((n) => n + 1)}>♥ {likes}</button>;
}

    A Server Component can render a Client Component and pass it serializable props (and even pass Server Components as children):

    // Server Component composing a Client Component
import LikeButton from "./components/LikeButton";

export default async function Post({ id }: { id: string }) {
  const post = await getPost(id); // server-only
  return (
    <article>
      <h2>{post.title}</h2>
      <LikeButton initial={post.likes} /> {/* interactive island */}
    </article>
  );
}

    Server/client boundary rules

    • You cannot pass functions, class instances, or Dates-as-behaviour from a Server Component to a Client Component—only serializable props.
    • You cannot use useState/useEffect/onClick/window in a Server Component.
    • You cannot import a server-only module (DB client, fs, secrets) into a Client Component. Mark such modules with import "server-only" so an accidental import fails the build instead of leaking to the browser.
    • Push "use client" down the tree to the smallest interactive leaf. Marking a layout "use client" drags its whole subtree into the client bundle.

    Streaming with <Suspense> and loading.tsx

    The App Router can stream HTML: send the shell immediately and flush slower parts as they resolve. Wrap a slow Server Component in <Suspense> to show a fallback without blocking the rest of the page.

    // app/dashboard/page.tsx
import { Suspense } from "react";
import Stats from "./Stats";        // slow async Server Component
import Sidebar from "./Sidebar";    // fast

export default function Dashboard() {
  return (
    <div>
      <Sidebar />
      <Suspense fallback={<p>Loading stats…</p>}>
        <Stats /> {/* streamed in when its data resolves */}
      </Suspense>
    </div>
  );
}

    Adding an app/<segment>/loading.tsx file gives the whole route segment an instant loading UI automatically (it wraps the segment in a Suspense boundary for you).

    // app/dashboard/loading.tsx
export default function Loading() {
  return <p>Loading dashboard…</p>;
}

    4. SSG & ISR with the App Router

    Static generation by default

    If a route has no dynamic data sources (no cookies(), headers(), no-store fetches, or dynamic functions), Next.js statically generates it at build time automatically. No special export needed.

    generateStaticParams for dynamic SSG

    To statically pre-render a set of dynamic routes (e.g. one page per blog post), export generateStaticParams. It's the App Router replacement for getStaticPaths.

    // app/blog/[slug]/page.tsx

// 1. Tell Next.js which slugs to pre-render at build time
export async function generateStaticParams() {
  const posts = await fetch("https://api.example.com/posts").then((r) => r.json());
  return posts.map((post: { slug: string }) => ({ slug: post.slug }));
}

// 2. Render each one (this Server Component runs per slug at build)
export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params; // Next.js 15: params is async
  const post = await fetch(`https://api.example.com/posts/${slug}`).then((r) =>
    r.json()
  );
  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  );
}

    params and searchParams are async in Next.js 15

    In Next.js 15, params, searchParams, cookies(), and headers() are Promises and must be awaited (or read with use()). In Next.js 14 they were synchronous objects. The examples here use the v15 async form.

    Incremental Static Regeneration (ISR)

    Add a segment-level revalidate (or a per-fetch revalidate) to make a static page refresh in the background.

    // app/products/page.tsx
export const revalidate = 3600; // regenerate at most once per hour

export default async function Products() {
  const products = await fetch("https://api.example.com/products").then((r) =>
    r.json()
  );
  return <ProductList products={products} />;
}

    How it works: the first request after the window serves the existing (stale) page instantly, then triggers a background regeneration so the next visitor gets fresh HTML—the stale-while-revalidate pattern.

    For unknown slugs, dynamicParams (default true) lets routes not listed in generateStaticParams be generated on first request and then cached—on-demand ISR.

    On-demand revalidation

    Instead of (or alongside) time-based revalidation, invalidate exactly when data changes—e.g. from a CMS webhook in a Route Handler:

    // app/api/revalidate/route.ts
import { revalidatePath, revalidateTag } from "next/cache";
import { NextResponse } from "next/server";

export async function POST(request: Request) {
  const { path, tag } = await request.json();
  if (path) revalidatePath(path);       // refresh a specific route
  if (tag) revalidateTag(tag);          // refresh all fetches with this tag
  return NextResponse.json({ revalidated: true });
}

    unstable_cache for non-fetch data

    fetch caching only covers HTTP calls. To cache a direct DB query or any async function with the same revalidate/tag semantics, wrap it:

    import { unstable_cache } from "next/cache";
import { db } from "@/lib/db";

export const getCachedProducts = unstable_cache(
  async () => db.product.findMany(),
  ["products-list"],                 // cache key parts
  { revalidate: 3600, tags: ["products"] }
);

    Mixing static and dynamic on one page

    You don't have to choose per page. Render the static shell statically and stream the dynamic part inside <Suspense>—the page is cached, but the personalized region (which reads cookies()/no-store) is rendered dynamically and streamed in. This partial approach (formalized as Partial Prerendering, experimental in 14/15) is the App Router's answer to "SSG + client-side rendering for the dynamic bits."

    Best practices for static sites

    • Default to static; opt into dynamic only where data is truly per-request.
    • Use revalidate for content that changes on a schedule; use tags + on-demand revalidation for content that changes on events (CMS publish).
    • Keep generateStaticParams lists bounded; use dynamicParams for the long tail.
    • Set <meta>/Open Graph via the metadata export or generateMetadata so pre-rendered pages are share- and SEO-ready.

    5. API Routes & serverless functions

    Route Handlers (App Router)

    In the App Router, API endpoints are Route Handlers: a route.ts file exporting functions named after HTTP verbs. They use the Web Request/Response APIs.

    // app/api/posts/route.ts
import { NextResponse } from "next/server";

export async function GET() {
  const posts = await db.post.findMany();
  return NextResponse.json(posts);
}

export async function POST(request: Request) {
  const body = await request.json();
  const created = await db.post.create({ data: body });
  return NextResponse.json(created, { status: 201 });
}

    Dynamic segments work here too—app/api/posts/[id]/route.ts receives params:

    // app/api/posts/[id]/route.ts
export async function GET(
  _request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;
  const post = await db.post.findUnique({ where: { id } });
  if (!post) return new Response("Not found", { status: 404 });
  return Response.json(post);
}

    Do you even need an API route?

    Server Components can query your database directly, and Server Actions ("use server" functions) handle mutations from forms without a hand-written endpoint. Reserve Route Handlers for things that genuinely need an HTTP endpoint: webhooks, third-party callbacks, public APIs, or non-React clients (mobile apps).

    Serverless functions & runtimes

    On platforms like Vercel, each Route Handler (and SSR render) is deployed as a serverless function—it spins up on demand, scales automatically, and you pay per invocation. You can pick the runtime per route:

    // Run on the lightweight Edge runtime (fast cold starts, geographically distributed)
export const runtime = "edge"; // or "nodejs" (default, full Node APIs)
    • Node.js runtime — full Node APIs, database drivers, larger libraries. Default.
    • Edge runtime — a limited Web-standard environment with near-zero cold start, great for auth checks, redirects, and lightweight personalization close to the user.

    Middleware

    middleware.ts at the project root runs before a request reaches a route—ideal for auth gating, redirects, A/B routing, and header rewriting. It runs on the Edge runtime.

    // middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const token = request.cookies.get("session")?.value;
  if (!token && request.nextUrl.pathname.startsWith("/dashboard")) {
    return NextResponse.redirect(new URL("/login", request.url));
  }
  return NextResponse.next();
}

export const config = { matcher: ["/dashboard/:path*"] };

    Fetching from your own API

    A Server Component can call your Route Handler over HTTP, but it's often simpler (and skips a network hop) to call the shared data function directly. From the client, fetch the Route Handler as you would any API:

    "use client";
import { useState } from "react";

export default function NewPostForm() {
  const [title, setTitle] = useState("");
  async function submit() {
    await fetch("/api/posts", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ title }),
    });
  }
  return (
    <>
      <input value={title} onChange={(e) => setTitle(e.target.value)} />
      <button onClick={submit}>Create</button>
    </>
  );
}

    Deployment considerations

    • Vercel is zero-config for Next.js (SSR/ISR/Route Handlers become serverless/edge functions automatically). Other hosts (Netlify, AWS, Cloudflare) have adapters; self-hosting uses next start behind Node, or output: "export" for a pure-static build (no SSR/ISR/Route Handlers).
    • Cold starts affect serverless functions; the Edge runtime mitigates this for latency-sensitive routes.
    • Secrets belong in server-only env vars; only NEXT_PUBLIC_-prefixed vars are exposed to the browser.
    • CDN caching of ISR/static output is what gives you scale—make sure cache headers and revalidation are configured intentionally.

    6. The legacy Pages Router

    Older model — know it, but prefer the App Router for new code

    The pages/ directory predates Server Components. It's still supported and you'll find it across existing projects, but new Next.js apps should use the App Router. The data-fetching functions below do not exist in the App Router—they're replaced by async components, fetch caching, generateStaticParams, and revalidate.

    SSR with getServerSideProps

    Runs on the server on every request; its return value becomes the page's props.

    // pages/dashboard.tsx
import type { GetServerSideProps } from "next";

export const getServerSideProps: GetServerSideProps = async (context) => {
  const res = await fetch(`https://api.example.com/user/${context.params?.id}`);
  const user = await res.json();
  return { props: { user } }; // can also return { notFound: true } / { redirect } 
};

export default function Dashboard({ user }: { user: { name: string } }) {
  return <h1>Welcome, {user.name}</h1>;
}

    SSG with getStaticProps + getStaticPaths

    getStaticProps runs at build time; getStaticPaths enumerates dynamic routes to pre-render. Adding revalidate to getStaticProps enables ISR in the Pages Router.

    // pages/blog/[slug].tsx
import type { GetStaticPaths, GetStaticProps } from "next";

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch("https://api.example.com/posts").then((r) => r.json());
  return {
    paths: posts.map((p: { slug: string }) => ({ params: { slug: p.slug } })),
    fallback: "blocking", // generate unlisted slugs on first request (on-demand ISR)
  };
};

export const getStaticProps: GetStaticProps = async ({ params }) => {
  const post = await fetch(
    `https://api.example.com/posts/${params?.slug}`
  ).then((r) => r.json());
  return {
    props: { post },
    revalidate: 60, // ISR: rebuild this page at most once per minute
  };
};

export default function BlogPost({ post }: { post: { title: string } }) {
  return <h1>{post.title}</h1>;
}

    API routes (pages/api)

    A file in pages/api/ exporting a default handler becomes an endpoint—the precursor to Route Handlers.

    // pages/api/hello.ts
import type { NextApiRequest, NextApiResponse } from "next";

export default function handler(req: NextApiRequest, res: NextApiResponse) {
  res.status(200).json({ message: "Hello from a serverless function" });
}

    App Router ↔ Pages Router cheat sheet

    Goal Pages Router (legacy) App Router (modern)
    SSR (per-request) getServerSideProps async component + fetch(..., { cache: "no-store" })
    SSG (build-time) getStaticProps async component + fetch(..., { cache: "force-cache" }) (default for static routes)
    Dynamic SSG paths getStaticPaths generateStaticParams
    ISR revalidate in getStaticProps export const revalidate / per-fetch revalidate
    API endpoint pages/api/* (default handler) app/api/*/route.ts (verb exports)
    Mutations from a form API route + client fetch Server Actions ("use server")

    7. Hydration & common pitfalls

    Hydration is the step where React attaches event listeners to the server-rendered HTML and "takes over" the static markup, turning it interactive. The DOM the client produces on first render must match what the server sent.

    Hydration mismatch

    If the server-rendered HTML and the first client render differ, React throws a hydration mismatch and may discard and re-render that subtree. Common causes:

    // ❌ Mismatch: server and client compute different values
function Clock() {
  return <span>{new Date().toLocaleTimeString()}</span>; // differs server vs client
}

// ❌ Mismatch: window/localStorage exist only on the client
function Theme() {
  return <div>{localStorage.getItem("theme")}</div>; // crashes/differs on server
}

    // ✅ Render the stable value first, then update after mount
"use client";
import { useState, useEffect } from "react";

function Clock() {
  const [time, setTime] = useState<string | null>(null);
  useEffect(() => setTime(new Date().toLocaleTimeString()), []);
  return <span>{time ?? "—"}</span>; // same output on server & first client render
}

    Typical mismatch triggers to watch for: Date.now()/Math.random(), reading window/localStorage during render, locale/timezone-dependent formatting, and invalid HTML nesting (e.g. a <div> inside a <p>). For unavoidable client-only content, suppressHydrationWarning (on a single element) or a mounted-flag pattern are escape hatches.

    Server/client boundary mistakes

    • Importing a Server Component into a Client Component to render it directly (instead, pass it as children/props).
    • Adding "use client" too high in the tree, ballooning the client bundle.
    • Using useState/useEffect/event handlers in a Server Component (the build will error).
    • Leaking secrets by importing server-only modules into client code—guard with import "server-only".

    Other gotchas

    • Forgetting cache/revalidate and being surprised by stale or always-dynamic pages (especially across the Next.js 14→15 default change).
    • Calling browser APIs at module scope in code that also runs on the server.
    • Double data fetching when you fetch in a Server Component and re-fetch the same thing on the client—lift it to the server.

    8. Beyond Next.js (briefly)

    Next.js dominates, but it isn't the only option:

    • Vite SSR — Vite ships a low-level SSR API (renderToString / renderToPipeableStream on the server, hydrateRoot on the client). Maximum control, but you assemble routing, data loading, and caching yourself. Good when you need a custom setup.
    • React Router v7 (framework mode) — the merger of Remix into React Router. Provides SSR, nested routes with loader/action data APIs, and streaming as a framework, as an alternative to Next.js.
    • Remix — the same lineage, web-standards-first (built on Request/Response), now largely converging into React Router v7.
    • Astro — content-focused; renders React (and other frameworks) as static HTML with selective hydration ("islands"), excellent for content sites.

    The common foundation

    All of these build on the same React primitives: renderToPipeableStream/renderToReadableStream (streaming SSR), hydrateRoot (client takeover), and increasingly React Server Components. Frameworks differ mainly in ergonomics—routing conventions, data-loading APIs, and caching—not in the underlying rendering mechanism.

    📚 Summary

    🔹 Concept 🧠 Description
    CSR Browser renders React from a JS bundle; weakest SEO/LCP, cheapest hosting
    SSR HTML built per request on the server; fresh + SEO-friendly, higher cost
    SSG HTML built once at build, served from CDN; fastest + cheapest, frozen data
    ISR Static pages regenerated in the background (time-based or on-demand)
    Hydration Client attaches interactivity to server HTML; output must match exactly
    "use client" Opt a component into the client bundle (state, effects, events)
    Server Component Default in App Router; runs server-only, async, ships zero JS
    fetch cache/revalidate Per-request caching that selects SSR vs SSG vs ISR
    generateStaticParams App Router: enumerate dynamic routes to pre-render (replaces getStaticPaths)
    revalidate / unstable_cache App Router ISR: time-based + non-fetch caching
    revalidatePath / revalidateTag On-demand cache invalidation (e.g. CMS webhooks)
    Route Handler (route.ts) App Router API endpoint using Web Request/Response
    Server Actions ("use server") Server mutations from forms without a hand-written endpoint
    Middleware Edge code before a route — auth, redirects, rewrites
    getServerSideProps Legacy Pages Router SSR
    getStaticProps / getStaticPaths Legacy Pages Router SSG (+ revalidate for ISR)
    pages/api/* Legacy Pages Router API routes
