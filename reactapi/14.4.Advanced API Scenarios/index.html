    Advanced API Scenarios

    This chapter expands on §4 of the API Integration outline. The basic CRUD patterns get you fetching and mutating data; the patterns here keep that data fetching fast, resilient, and cheap under real-world conditions — flaky networks, rate limits, rapid user input, large datasets, and offline devices.

    Two stacks run side by side throughout:

    • Plain axios v1 + hooks — the manual, "know what's happening" version.
    • TanStack Query v5 — the declarative version that bakes most of these scenarios in for free.

    Stack assumptions

    • React 18/19 functional components and hooks only.
    • axios v1 — we use the standard AbortController / signal, not the deprecated CancelToken.
    • TanStack Query v5 — note the renamed/changed APIs: gcTime (was cacheTime), object-form filters, the new useInfiniteQuery signature, and placeholderData: keepPreviousData (which replaced keepPreviousData: true).

    All examples assume the shared axios instance from the previous chapters:

    // src/api/axiosClient.js
import axios from "axios";

const api = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL || "https://api.example.com",
  timeout: 10000,
  headers: { "Content-Type": "application/json" },
});

export default api;

    1. Retry Logic with Exponential Backoff

    Transient failures — a dropped connection, a 503 during a deploy, a brief 429 — usually succeed on a second try. Retrying blindly hammers the server, so we add exponential backoff (each wait doubles) plus jitter (a small random offset) so that many clients retrying at once don't synchronize into a thundering herd.

    A reusable retry helper

    // src/api/withRetry.ts
import axios, { AxiosError, AxiosRequestConfig } from "axios";
import api from "./axiosClient";

const sleep = (ms: number) => new Promise((r) => setTimeout(r, ms));

// Only retry on transient conditions, never on 4xx (except 429).
function isRetryable(error: unknown): boolean {
  if (!axios.isAxiosError(error)) return false;
  const err = error as AxiosError;
  if (err.code === "ERR_CANCELED") return false; // respect cancellation
  const status = err.response?.status;
  if (status === undefined) return true; // network error / timeout
  return status === 429 || status === 408 || (status >= 500 && status <= 599);
}

export async function requestWithRetry<T>(
  config: AxiosRequestConfig,
  { retries = 3, baseDelay = 500, maxDelay = 8000 } = {}
): Promise<T> {
  let attempt = 0;
  // eslint-disable-next-line no-constant-condition
  while (true) {
    try {
      const res = await api.request<T>(config);
      return res.data;
    } catch (error) {
      if (attempt >= retries || !isRetryable(error)) throw error;

      // Honour a server-provided Retry-After header if present.
      const retryAfter = axios.isAxiosError(error)
        ? Number(error.response?.headers?.["retry-after"])
        : NaN;

      const backoff = Math.min(baseDelay * 2 ** attempt, maxDelay);
      const jitter = Math.random() * backoff * 0.25; // ±25% jitter
      const wait = Number.isFinite(retryAfter) ? retryAfter * 1000 : backoff + jitter;

      await sleep(wait);
      attempt += 1;
    }
  }
}

    Usage:

    const users = await requestWithRetry<User[]>({ url: "/users", method: "get" });

    Never retry non-idempotent writes blindly

    GET, PUT, DELETE are idempotent — safe to retry. A POST may create a duplicate. Retry writes only when the endpoint supports an idempotency key (a unique header the server uses to dedupe), e.g. headers: { "Idempotency-Key": crypto.randomUUID() }.

    With TanStack Query

    Query handles retries declaratively. retry is a count or a predicate, and retryDelay computes the backoff.

    import { useQuery } from "@tanstack/react-query";

useQuery({
  queryKey: ["users"],
  queryFn: () => api.get("/users").then((r) => r.data),
  retry: (failureCount, error) => {
    const status = axios.isAxiosError(error) ? error.response?.status : undefined;
    if (status && status >= 400 && status < 500 && status !== 429) return false;
    return failureCount < 3;
  },
  retryDelay: (attempt) => Math.min(1000 * 2 ** attempt, 30_000),
});

    2. Canceling Requests with AbortController

    When a component unmounts mid-flight, or a user fires a new search before the last finishes, the stale response is at best wasted work and at worst a race condition that overwrites fresh data. axios v1 supports the web-standard AbortController through the signal option.

    Cancel on unmount

    import { useEffect, useState } from "react";
import axios from "axios";
import api from "../api/axiosClient";

function UserList() {
  const [users, setUsers] = useState([]);

  useEffect(() => {
    const controller = new AbortController();

    api
      .get("/users", { signal: controller.signal })
      .then((res) => setUsers(res.data))
      .catch((err) => {
        if (axios.isCancel(err)) return; // expected on unmount
        console.error(err);
      });

    return () => controller.abort(); // cleanup cancels in-flight request
  }, []);

  return <ul>{users.map((u) => <li key={u.id}>{u.name}</li>)}</ul>;
}

    Detecting a cancellation in axios v1

    Use axios.isCancel(err), or check err.code === "ERR_CANCELED" / err.name === "CanceledError". Always treat it as a non-error — bail out silently rather than showing the user a failure.

    Cancel the previous request (live search)

    Keep a ref to the active controller and abort it before starting the next call, so only the latest query can win:

    import { useRef, useState } from "react";
import axios from "axios";
import api from "../api/axiosClient";

function useLiveSearch() {
  const [results, setResults] = useState([]);
  const controllerRef = useRef(null);

  const search = async (query) => {
    controllerRef.current?.abort();          // cancel the previous request
    const controller = new AbortController();
    controllerRef.current = controller;

    try {
      const res = await api.get("/search", {
        params: { q: query },
        signal: controller.signal,
      });
      setResults(res.data);
    } catch (err) {
      if (!axios.isCancel(err)) throw err;
    }
  };

  return { results, search };
}

    TanStack Query cancels for you

    Query passes a signal into your queryFn. Forward it to axios and Query will automatically abort superseded or unmounted queries: 

    useQuery({
  queryKey: ["search", q],
  queryFn: ({ signal }) => api.get("/search", { params: { q }, signal }).then((r) => r.data),
});

    3. Debouncing & Throttling API Calls

    Both limit how often a function runs, but with different timing:

    Technique Fires… Use for
    Debounce once, after activity stops for N ms search-as-you-type, autosave
    Throttle at most once per N ms during activity scroll/resize handlers, rapid clicks

    A debounce hook

    Debounce the value, then drive the request off the debounced value — this plays nicely with React's render model and avoids stale closures.

    import { useEffect, useState } from "react";

function useDebouncedValue(value, delay = 400) {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id); // reset timer on every keystroke
  }, [value, delay]);

  return debounced;
}

    import { useEffect, useState } from "react";
import axios from "axios";
import api from "../api/axiosClient";

function SearchBox() {
  const [term, setTerm] = useState("");
  const debouncedTerm = useDebouncedValue(term, 400);
  const [results, setResults] = useState([]);

  useEffect(() => {
    if (!debouncedTerm) {
      setResults([]);
      return;
    }
    const controller = new AbortController();

    api
      .get("/search", { params: { q: debouncedTerm }, signal: controller.signal })
      .then((res) => setResults(res.data))
      .catch((err) => {
        if (!axios.isCancel(err)) console.error(err);
      });

    return () => controller.abort(); // cancel if term changes again
  }, [debouncedTerm]);

  return (
    <>
      <input value={term} onChange={(e) => setTerm(e.target.value)} />
      <ul>{results.map((r) => <li key={r.id}>{r.title}</li>)}</ul>
    </>
  );
}

    Debounce + cancel together

    Debouncing cuts the number of requests; AbortController cancels any that are still in flight when a newer one starts. Use both for live search — they solve different halves of the problem.

    A throttle helper

    function throttle(fn, limit = 1000) {
  let waiting = false;
  let lastArgs = null;

  return function throttled(...args) {
    if (waiting) {
      lastArgs = args; // remember the latest call to run on the trailing edge
      return;
    }
    fn.apply(this, args);
    waiting = true;
    setTimeout(() => {
      waiting = false;
      if (lastArgs) {
        throttled.apply(this, lastArgs);
        lastArgs = null;
      }
    }, limit);
  };
}

    With TanStack Query, debounce only the query key input (state), not the query itself — Query then dedupes and caches per debounced key automatically.

    4. Parallel, Batch & Dependent Requests

    Parallel — Promise.all vs Promise.allSettled

    Run independent requests concurrently instead of one after another.

    // Fail-fast: rejects as soon as ANY request fails.
export const getDashboard = async () => {
  const [users, posts, products] = await Promise.all([
    api.get("/users"),
    api.get("/posts"),
    api.get("/products"),
  ]);
  return { users: users.data, posts: posts.data, products: products.data };
};

    // Resilient: never rejects — inspect each result individually.
export const getDashboardSafe = async () => {
  const results = await Promise.allSettled([
    api.get("/users"),
    api.get("/posts"),
    api.get("/products"),
  ]);

  return results.map((r) =>
    r.status === "fulfilled"
      ? { ok: true, data: r.value.data }
      : { ok: false, error: r.reason }
  );
};

    Choosing between them

    Use Promise.all when the page is useless without every piece (fail fast). Use Promise.allSettled for dashboards where each widget can succeed or fail on its own.

    Dependent / sequential requests

    When request B needs data from request A, you must await A first.

    export const getUserWithLatestOrder = async (userId) => {
  const { data: user } = await api.get(`/users/${userId}`);
  const { data: orders } = await api.get("/orders", {
    params: { userId: user.id, _sort: "createdAt", _order: "desc", _limit: 1 },
  });
  return { ...user, latestOrder: orders[0] ?? null };
};

    In TanStack Query, express dependency with the enabled flag — the second query stays idle until the first resolves:

    const { data: user } = useQuery({
  queryKey: ["user", userId],
  queryFn: () => api.get(`/users/${userId}`).then((r) => r.data),
});

const { data: orders } = useQuery({
  queryKey: ["orders", user?.id],
  queryFn: () => api.get("/orders", { params: { userId: user.id } }).then((r) => r.data),
  enabled: !!user?.id, // gate on the dependency
});

    For many parallel queries that vary, use useQueries so each gets its own cache entry:

    import { useQueries } from "@tanstack/react-query";

const results = useQueries({
  queries: ids.map((id) => ({
    queryKey: ["product", id],
    queryFn: () => api.get(`/products/${id}`).then((r) => r.data),
  })),
});

    5. Conditional Requests

    Don't fetch when you don't need to. Two flavours:

    Client-side gating — skip the call until prerequisites are met. With raw hooks, guard inside the effect; with Query, use enabled:

    useQuery({
  queryKey: ["profile", token],
  queryFn: () => api.get("/me").then((r) => r.data),
  enabled: !!token, // only fetch once authenticated
});

    HTTP conditional requests — let the server skip the work. Send the ETag/Last-Modified you got last time; a 304 Not Modified means "use your cached copy," saving bandwidth.

    let lastETag = null;

export const getConfigIfChanged = async () => {
  const res = await api.get("/config", {
    headers: lastETag ? { "If-None-Match": lastETag } : {},
    validateStatus: (s) => s === 200 || s === 304, // treat 304 as success
  });
  if (res.status === 304) return null; // unchanged — keep current data
  lastETag = res.headers.etag;
  return res.data;
};

    6. Pagination & Infinite Scroll

    Two server contracts dominate. Pick based on how the API exposes "the next set."

    • Offset / page-based — ?page=2&limit=20 or ?_start=20&_limit=20. Simple, jumpable, but can skip/duplicate rows if data shifts between pages.
    • Cursor-based — ?cursor=abc123&limit=20. The server returns an opaque nextCursor; stable under inserts/deletes and the right choice for feeds.

    Offset pagination with "keep previous data"

    When the user clicks "next," show the current page until the new one arrives instead of flashing a spinner. In v5 this is placeholderData: keepPreviousData.

    import { useState } from "react";
import { useQuery, keepPreviousData } from "@tanstack/react-query";
import api from "../api/axiosClient";

function getPosts(page: number, limit = 10) {
  return api
    .get("/posts", { params: { _page: page, _limit: limit } })
    .then((r) => ({
      items: r.data,
      total: Number(r.headers["x-total-count"]) || 0,
    }));
}

export function PostsTable() {
  const [page, setPage] = useState(1);

  const { data, isPlaceholderData, isPending } = useQuery({
    queryKey: ["posts", { page }],          // object filter — v5 style
    queryFn: () => getPosts(page),
    placeholderData: keepPreviousData,      // v5 replacement for keepPreviousData: true
    staleTime: 30_000,
  });

  if (isPending) return <p>Loading…</p>;
  const hasMore = page * 10 < (data?.total ?? 0);

  return (
    <>
      <ul>{data.items.map((p) => <li key={p.id}>{p.title}</li>)}</ul>
      <button disabled={page === 1} onClick={() => setPage((p) => p - 1)}>
        Prev
      </button>
      <button
        disabled={isPlaceholderData || !hasMore}
        onClick={() => setPage((p) => p + 1)}
      >
        Next
      </button>
    </>
  );
}

    Cursor pagination + infinite scroll (useInfiniteQuery v5)

    The v5 signature requires initialPageParam and gives getNextPageParam the signature (lastPage, allPages, lastPageParam, allPageParams).

    import { useInfiniteQuery } from "@tanstack/react-query";
import api from "../api/axiosClient";

type FeedPage = { items: Post[]; nextCursor: string | null };

function fetchFeed({ pageParam }: { pageParam: string | null }): Promise<FeedPage> {
  return api
    .get("/feed", { params: { cursor: pageParam, limit: 20 } })
    .then((r) => r.data); // { items, nextCursor }
}

export function useFeed() {
  return useInfiniteQuery({
    queryKey: ["feed"],
    queryFn: fetchFeed,
    initialPageParam: null as string | null,   // required in v5
    getNextPageParam: (lastPage) => lastPage.nextCursor ?? undefined, // undefined = no more pages
  });
}

    Wire it to an IntersectionObserver sentinel so scrolling loads the next page:

    import { useEffect, useRef } from "react";
import { useFeed } from "./useFeed";

export function Feed() {
  const { data, fetchNextPage, hasNextPage, isFetchingNextPage } = useFeed();
  const sentinelRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    const node = sentinelRef.current;
    if (!node) return;

    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting && hasNextPage && !isFetchingNextPage) {
          fetchNextPage();
        }
      },
      { rootMargin: "200px" } // prefetch slightly before the user reaches the end
    );

    observer.observe(node);
    return () => observer.disconnect();
  }, [hasNextPage, isFetchingNextPage, fetchNextPage]);

  return (
    <>
      {data?.pages.flatMap((page) =>
        page.items.map((post) => <article key={post.id}>{post.title}</article>)
      )}
      <div ref={sentinelRef} />
      {isFetchingNextPage && <p>Loading more…</p>}
    </>
  );
}

    7. Optimistic Updates & Rollback

    Apply the change to the UI immediately, before the server confirms, then roll back if the request fails. This makes likes, toggles, and reorders feel instant.

    Manual optimistic update

    import { useState } from "react";
import api from "../api/axiosClient";

function LikeButton({ post }) {
  const [likes, setLikes] = useState(post.likes);
  const [pending, setPending] = useState(false);

  const like = async () => {
    const previous = likes;
    setLikes(previous + 1); // optimistic
    setPending(true);
    try {
      await api.patch(`/posts/${post.id}`, { likes: previous + 1 });
    } catch (err) {
      setLikes(previous); // rollback on failure
      console.error(err);
    } finally {
      setPending(false);
    }
  };

  return <button disabled={pending} onClick={like}>♥ {likes}</button>;
}

    Optimistic mutation with TanStack Query

    The canonical pattern: cancel in-flight queries, snapshot the cache, write the optimistic value, restore the snapshot in onError, and resync in onSettled.

    import { useMutation, useQueryClient } from "@tanstack/react-query";
import api from "../api/axiosClient";

function useToggleLike(postId: number) {
  const qc = useQueryClient();

  return useMutation({
    mutationFn: (liked: boolean) =>
      api.patch(`/posts/${postId}`, { liked }).then((r) => r.data),

    onMutate: async (liked) => {
      await qc.cancelQueries({ queryKey: ["post", postId] }); // avoid races
      const previous = qc.getQueryData(["post", postId]);     // snapshot
      qc.setQueryData(["post", postId], (old: any) => ({ ...old, liked })); // optimistic
      return { previous };                                    // context for rollback
    },

    onError: (_err, _liked, context) => {
      qc.setQueryData(["post", postId], context?.previous);   // rollback
    },

    onSettled: () => {
      qc.invalidateQueries({ queryKey: ["post", postId] });   // resync with server
    },
  });
}

    Always cancel before snapshotting

    Without cancelQueries, a refetch that resolves after your optimistic write can clobber it with stale server data. Cancelling first guarantees your optimistic value is the latest write to the cache.

    8. Request Deduplication & Caching

    If three components mount at once and each asks for /users, you want one network request shared by all three. This is deduplication; caching extends it across time.

    Manual in-flight dedup

    Cache the promise, not just the result, so concurrent callers await the same request:

    const inflight = new Map<string, Promise<any>>();

export function dedupedGet<T>(url: string, config = {}): Promise<T> {
  const key = url + JSON.stringify(config);
  if (inflight.has(key)) return inflight.get(key)!;

  const promise = api
    .get<T>(url, config)
    .then((r) => r.data)
    .finally(() => inflight.delete(key)); // clear once settled

  inflight.set(key, promise);
  return promise;
}

    TanStack Query does this by default

    Query dedupes by queryKey automatically, then caches results. The two timing knobs to understand:

    useQuery({
  queryKey: ["users"],
  queryFn: () => api.get("/users").then((r) => r.data),
  staleTime: 60_000, // data is "fresh" for 60s — no refetch on remount/focus
  gcTime: 5 * 60_000, // unused data is garbage-collected after 5min (was cacheTime in v4)
});

    staleTime vs gcTime

    • staleTime — how long data is considered fresh; while fresh, Query serves the cache and skips refetching. Default 0.
    • gcTime — how long an inactive (unobserved) cache entry survives before being garbage-collected. Default 5 minutes. Renamed from cacheTime in v5.

    9. Polling vs Long-Polling

    Both keep data fresh without WebSockets.

    • Polling (short polling) — ask on a fixed interval ("any updates?"). Simple, but wastes requests when nothing changed.
    • Long-polling — the server holds the request open until there's data (or a timeout), then you immediately reconnect. Fewer empty round-trips, near real-time, but ties up a connection.

    Polling with TanStack Query

    useQuery({
  queryKey: ["job", jobId],
  queryFn: () => api.get(`/jobs/${jobId}`).then((r) => r.data),
  // Poll every 3s until the job finishes, then stop.
  refetchInterval: (query) =>
    query.state.data?.status === "done" ? false : 3000,
  refetchIntervalInBackground: false, // pause when the tab is hidden
});

    Long-polling loop

    import axios from "axios";
import api from "../api/axiosClient";

export async function longPoll(onMessage, { signal } = {}) {
  let cursor = 0;
  while (!signal?.aborted) {
    try {
      const { data } = await api.get("/updates", {
        params: { since: cursor },
        timeout: 30_000, // longer than usual — the server holds the request
        signal,
      });
      if (data.events?.length) {
        onMessage(data.events);
        cursor = data.cursor;
      }
    } catch (err) {
      if (axios.isCancel(err)) break;
      await new Promise((r) => setTimeout(r, 1000)); // back off before reconnecting
    }
  }
}

    When to reach for WebSockets/SSE instead

    Polling is fine for low-frequency updates (job status, notifications). For high-frequency, bidirectional streams (chat, live cursors), use WebSockets; for one-way server pushes, Server-Sent Events. See the Real-Time Data chapter.

    10. Rate Limiting & 429 Handling

    A 429 Too Many Requests means you've exceeded the server's quota. The response usually carries a Retry-After header (seconds, or an HTTP date) telling you when to try again — respect it.

    import axios from "axios";
import api from "./axiosClient";

api.interceptors.response.use(
  (res) => res,
  async (error) => {
    if (axios.isAxiosError(error) && error.response?.status === 429 && error.config) {
      const header = error.response.headers["retry-after"];
      const waitMs = parseRetryAfter(header) ?? 1000;
      await new Promise((r) => setTimeout(r, waitMs));
      return api.request(error.config); // replay the original request once
    }
    return Promise.reject(error);
  }
);

function parseRetryAfter(value?: string): number | null {
  if (!value) return null;
  const seconds = Number(value);
  if (!Number.isNaN(seconds)) return seconds * 1000; // "120"
  const date = Date.parse(value);                    // "Wed, 21 Oct 2025 07:28:00 GMT"
  return Number.isNaN(date) ? null : Math.max(0, date - Date.now());
}

    To avoid hitting the limit, cap concurrency on the client with a small semaphore so you never have more than N requests in flight:

    function pLimit(concurrency) {
  let active = 0;
  const queue = [];

  const next = () => {
    if (active >= concurrency || queue.length === 0) return;
    active++;
    const { fn, resolve, reject } = queue.shift();
    fn().then(resolve, reject).finally(() => {
      active--;
      next();
    });
  };

  return (fn) =>
    new Promise((resolve, reject) => {
      queue.push({ fn, resolve, reject });
      next();
    });
}

// Usage: at most 4 concurrent uploads
const limit = pLimit(4);
await Promise.all(files.map((f) => limit(() => api.post("/upload", f))));

    Combine 429 handling with backoff

    A single retry on 429 can still fail under sustained load. For robustness, route 429 through the exponential-backoff helper from §1, which already honours Retry-After.

    11. Error Normalization & a Global Error Handler

    axios errors come in three shapes — error.response (server replied with a non-2xx), error.request (sent, no reply), and a setup error (neither). Code shouldn't repeat that triage everywhere. Normalize once into a predictable shape, in a response interceptor.

    // src/api/errors.ts
import axios, { AxiosError } from "axios";

export type ApiError = {
  status: number | null;       // HTTP status, or null for network/setup errors
  code: string;                // machine-readable: "UNAUTHORIZED", "NETWORK", …
  message: string;             // human-readable, safe to display
  details?: unknown;           // raw server payload, for debugging
  isNetworkError: boolean;
};

export function normalizeError(error: unknown): ApiError {
  if (!axios.isAxiosError(error)) {
    return { status: null, code: "UNKNOWN", message: String(error), isNetworkError: false };
  }
  const err = error as AxiosError<any>;

  if (err.response) {
    const status = err.response.status;
    const serverMsg = err.response.data?.message;
    const codeByStatus: Record<number, string> = {
      400: "BAD_REQUEST",
      401: "UNAUTHORIZED",
      403: "FORBIDDEN",
      404: "NOT_FOUND",
      409: "CONFLICT",
      422: "VALIDATION",
      429: "RATE_LIMITED",
    };
    return {
      status,
      code: codeByStatus[status] ?? (status >= 500 ? "SERVER_ERROR" : "HTTP_ERROR"),
      message: serverMsg || `Request failed with status ${status}`,
      details: err.response.data,
      isNetworkError: false,
    };
  }

  if (err.request) {
    return {
      status: null,
      code: err.code === "ECONNABORTED" ? "TIMEOUT" : "NETWORK",
      message:
        err.code === "ECONNABORTED"
          ? "Request timed out."
          : "Network error — check your connection.",
      isNetworkError: true,
    };
  }

  return { status: null, code: "SETUP", message: err.message, isNetworkError: false };
}

    Hook it into the instance so every rejection is already normalized:

    import api from "./axiosClient";
import { normalizeError } from "./errors";

api.interceptors.response.use(
  (res) => res,
  (error) => {
    const normalized = normalizeError(error);

    // Cross-cutting reactions:
    if (normalized.code === "UNAUTHORIZED") {
      window.dispatchEvent(new CustomEvent("auth:logout")); // global logout
    }
    // Report unexpected server errors to your monitoring tool.
    if (normalized.status && normalized.status >= 500) {
      // Sentry.captureException(error);
    }

    return Promise.reject(normalized); // callers now receive ApiError, not AxiosError
  }
);

    12. Request Queuing & Offline Handling

    On a flaky or offline connection, mutations should not just fail — queue them and flush when connectivity returns. The browser exposes navigator.onLine plus online/offline events.

    A simple persisted mutation queue

    // src/api/offlineQueue.ts
import api from "./axiosClient";

type QueuedRequest = { id: string; method: string; url: string; data?: unknown };
const STORAGE_KEY = "offline-queue";

function load(): QueuedRequest[] {
  return JSON.parse(localStorage.getItem(STORAGE_KEY) || "[]");
}
function save(queue: QueuedRequest[]) {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(queue));
}

export function enqueue(req: Omit<QueuedRequest, "id">) {
  const queue = load();
  queue.push({ ...req, id: crypto.randomUUID() });
  save(queue);
}

export async function flushQueue() {
  if (!navigator.onLine) return;
  let queue = load();

  for (const req of [...queue]) {
    try {
      await api.request({ method: req.method, url: req.url, data: req.data });
      queue = queue.filter((q) => q.id !== req.id); // drop on success
      save(queue);
    } catch {
      break; // stop on first failure — preserve order, retry later
    }
  }
}

// Flush automatically when the connection returns.
window.addEventListener("online", flushQueue);

    // Call site: try now, fall back to the queue when offline.
export async function saveDraft(draft: unknown) {
  if (!navigator.onLine) {
    enqueue({ method: "post", url: "/drafts", data: draft });
    return;
  }
  try {
    return await api.post("/drafts", draft);
  } catch (err) {
    enqueue({ method: "post", url: "/drafts", data: draft });
    throw err;
  }
}

    Don't reinvent this for production

    TanStack Query ships first-class offline support: set networkMode: "offlineFirst" and use its persistence plugins (persistQueryClient + an async storage persister) to pause mutations offline and resume them on reconnect. For true background sync that survives a page close, a Service Worker with the Background Sync API is the robust option.

    13. Interceptors for Cross-Cutting Concerns

    Interceptors are the single best place for behaviour that should apply to every request or response — so you don't sprinkle it across call sites. Common uses: auth tokens, request IDs, logging/timing, and centralized error handling.

    // src/api/axiosClient.ts
import axios from "axios";

const api = axios.create({ baseURL: import.meta.env.VITE_API_BASE_URL, timeout: 10_000 });

// --- Request interceptor: auth + tracing ---
api.interceptors.request.use((config) => {
  const token = localStorage.getItem("accessToken");
  if (token) config.headers.Authorization = `Bearer ${token}`;
  config.headers["X-Request-Id"] = crypto.randomUUID();   // correlate logs
  (config as any).metadata = { start: performance.now() }; // for timing below
  return config;
});

// --- Response interceptor: timing + logging ---
api.interceptors.response.use(
  (response) => {
    const start = (response.config as any).metadata?.start ?? 0;
    const ms = performance.now() - start;
    if (import.meta.env.DEV) {
      console.debug(
        `${response.config.method?.toUpperCase()} ${response.config.url} → ${response.status} (${ms.toFixed(0)}ms)`
      );
    }
    return response;
  },
  (error) => Promise.reject(error)
);

export default api;

    Refresh-token interceptor (single-flight)

    A common cross-cutting need: when a 401 arrives, refresh the token once and replay every request that failed during the refresh — not one refresh call per failed request.

    let refreshing: Promise<string> | null = null; // shared promise for concurrent 401s

api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const original = error.config;
    if (error.response?.status === 401 && original && !original._retried) {
      original._retried = true;

      refreshing ??= api
        .post("/auth/refresh")
        .then((r) => r.data.accessToken as string)
        .finally(() => {
          refreshing = null;
        });

      try {
        const token = await refreshing;          // all 401'd requests await the same call
        localStorage.setItem("accessToken", token);
        original.headers.Authorization = `Bearer ${token}`;
        return api.request(original);            // replay the original request
      } catch (e) {
        window.dispatchEvent(new CustomEvent("auth:logout"));
        return Promise.reject(e);
      }
    }
    return Promise.reject(error);
  }
);

    Interceptor ordering & loops

    Request interceptors run in reverse registration order; response interceptors run in registration order. Guard refresh logic with a flag (_retried above) so a failed refresh can't trigger an infinite retry loop, and make sure the refresh call itself is exempt from the auth interceptor.

    Rule of thumb

    Reach for TanStack Query for server-state in any non-trivial app — it gives you caching, dedup, retries, and cancellation for free. Keep the manual axios patterns for one-off scripts, non-React code, and the rare case where you need precise control over the request lifecycle.
