    Real-Time Data

    Most of what we have covered so far is request/response: the client asks, the server answers, the connection closes. Real-time data flips this around — the server pushes updates to the client as soon as they happen. Chat messages, live dashboards, stock tickers, notifications, collaborative editing, and presence ("X is typing…") all need this.

    This chapter covers the transport choices (polling, WebSockets, Socket.IO, SSE), how to wrap each one in clean React hooks with correct useEffect cleanup, and how real-time fits alongside TanStack Query v5 so your cache stays the single source of truth.

    0️⃣ The Mental Model

    There are four common ways to get "fresh" data into a React app:

    Approach Direction Connection Best for
    Polling Client pulls New HTTP request each time Simple, low-frequency updates
    Long polling Client pulls HTTP held open until data Fallback when WS unavailable
    WebSockets Bidirectional One persistent TCP socket Chat, games, collaboration
    Server-Sent Events (SSE) Server pushes One persistent HTTP stream Feeds, notifications, live logs

    Real-time is a spectrum, not a switch

    "Real-time" rarely means zero latency. A 2-second poll feels real-time on a dashboard; a chat app needs sub-100ms. Pick the cheapest transport that meets the latency your feature actually requires.

    1️⃣ Polling vs Real-Time Tradeoffs

    Polling (the simplest option)

    Polling means re-fetching on an interval. With TanStack Query v5 you get this for free via refetchInterval — no extra infrastructure required.

    import { useQuery } from "@tanstack/react-query";

function fetchPrices(): Promise<Price[]> {
  return fetch("/api/prices").then((res) => {
    if (!res.ok) throw new Error("Failed to load prices");
    return res.json();
  });
}

export function PriceTicker() {
  const { data, isPending } = useQuery({
    queryKey: ["prices"],
    queryFn: fetchPrices,
    refetchInterval: 5000,              // poll every 5s
    refetchIntervalInBackground: false, // pause when tab is hidden
  });

  if (isPending) return <p>Loading…</p>;
  return <PriceList prices={data} />;
}

    refetchInterval can also be a function, so you can back off or stop polling based on the latest data:

    useQuery({
  queryKey: ["job", jobId],
  queryFn: () => fetchJob(jobId),
  // poll while running, stop once finished
  refetchInterval: (query) =>
    query.state.data?.status === "done" ? false : 2000,
});

    When polling is the right call

    • Updates are infrequent (minutes) or staleness of a few seconds is acceptable.
    • You don't control the backend, or it only speaks REST.
    • You want zero new infrastructure and trivial scaling (it's just HTTP).

    When to reach for real-time push instead

    Symptom of "polling is wrong" Better fit
    You poll every 1–2s to feel "live" WebSockets / SSE
    Most polls return no change (wasted requests) SSE / WS
    You need to send data continuously too WebSockets / Socket.IO
    Server-to-client only (feeds, logs) SSE
    Many tabs/users hammering the API Push (one connection, server fans out)

    Polling cost is multiplicative

    1,000 users polling every 2s = 500 requests/second of mostly-empty responses. A single push connection per user delivers updates only when they exist. Below a few seconds of interval, push is usually both faster and cheaper.

    2️⃣ WebSockets — The Raw Browser API

    WebSockets give you a single, full-duplex (two-way) persistent connection. After an HTTP handshake that "upgrades" the connection, both sides can send messages at any time with very low overhead.

    The bare API

    const socket = new WebSocket("wss://api.example.com/ws");

socket.onopen = () => socket.send(JSON.stringify({ type: "hello" }));
socket.onmessage = (event) => console.log("received", JSON.parse(event.data));
socket.onerror = (event) => console.error("ws error", event);
socket.onclose = (event) => console.log("closed", event.code, event.reason);

// later
socket.close(1000, "done"); // 1000 = normal closure

    Always use wss:// in production

    ws:// is unencrypted and will be blocked by browsers on HTTPS pages (mixed content). Use wss:// — the TLS-secured variant — everywhere except local development.

    Naive React usage (and why it breaks)

    A first attempt usually looks like this — and it has a subtle bug:

    // ⚠️ Buggy: no cleanup, reconnects on every render that changes `url`
useEffect(() => {
  const ws = new WebSocket(url);
  ws.onmessage = (e) => setMessages((m) => [...m, JSON.parse(e.data)]);
}, [url]);

    Problems:

    1. No cleanup — the socket is never closed, so you leak connections on unmount and in React 18 Strict Mode (which mounts → unmounts → mounts).
    2. Stale closures — handlers capture old state.
    3. No reconnection — a dropped connection stays dead.

    The fix is a dedicated hook.

    3️⃣ A Reusable useWebSocket Hook

    This hook handles connection lifecycle, auto-reconnection with exponential backoff, a heartbeat to detect dead connections, and clean teardown on unmount.

    // useWebSocket.ts
import { useCallback, useEffect, useRef, useState } from "react";

type Status = "connecting" | "open" | "closing" | "closed";

interface Options {
  /** Called for every parsed message. */
  onMessage?: (data: unknown) => void;
  /** Max reconnection attempts before giving up. */
  maxRetries?: number;
  /** Send a ping every N ms to keep the connection alive. */
  heartbeatMs?: number;
}

export function useWebSocket(url: string, options: Options = {}) {
  const { onMessage, maxRetries = 10, heartbeatMs = 25_000 } = options;

  const [status, setStatus] = useState<Status>("closed");
  const wsRef = useRef<WebSocket | null>(null);
  const retriesRef = useRef(0);
  const reconnectTimer = useRef<ReturnType<typeof setTimeout> | null>(null);
  const heartbeatTimer = useRef<ReturnType<typeof setInterval> | null>(null);
  // Keep the latest callback without re-running the connect effect.
  const onMessageRef = useRef(onMessage);
  onMessageRef.current = onMessage;
  // Lets cleanup distinguish an intentional close from a dropped connection.
  const shouldReconnect = useRef(true);

  const connect = useCallback(() => {
    setStatus("connecting");
    const ws = new WebSocket(url);
    wsRef.current = ws;

    ws.onopen = () => {
      retriesRef.current = 0; // reset backoff on success
      setStatus("open");

      // Heartbeat: keep proxies/load balancers from killing an idle socket.
      heartbeatTimer.current = setInterval(() => {
        if (ws.readyState === WebSocket.OPEN) {
          ws.send(JSON.stringify({ type: "ping" }));
        }
      }, heartbeatMs);
    };

    ws.onmessage = (event) => {
      try {
        const data = JSON.parse(event.data);
        if (data?.type === "pong") return; // ignore heartbeat replies
        onMessageRef.current?.(data);
      } catch {
        onMessageRef.current?.(event.data); // non-JSON payloads
      }
    };

    ws.onclose = () => {
      setStatus("closed");
      if (heartbeatTimer.current) clearInterval(heartbeatTimer.current);

      // Reconnect with exponential backoff + jitter, unless we closed on purpose.
      if (shouldReconnect.current && retriesRef.current < maxRetries) {
        const attempt = retriesRef.current++;
        const base = Math.min(1000 * 2 ** attempt, 30_000); // cap at 30s
        const jitter = Math.random() * 1000; // avoid thundering herd
        reconnectTimer.current = setTimeout(connect, base + jitter);
      }
    };

    ws.onerror = () => ws.close(); // force onclose → triggers reconnect path
  }, [url, maxRetries, heartbeatMs]);

  const send = useCallback((payload: unknown) => {
    const ws = wsRef.current;
    if (ws?.readyState === WebSocket.OPEN) {
      ws.send(typeof payload === "string" ? payload : JSON.stringify(payload));
      return true;
    }
    return false; // not connected — caller can queue/ignore
  }, []);

  useEffect(() => {
    shouldReconnect.current = true;
    connect();

    // Cleanup: stop reconnecting, clear timers, close the socket.
    return () => {
      shouldReconnect.current = false;
      if (reconnectTimer.current) clearTimeout(reconnectTimer.current);
      if (heartbeatTimer.current) clearInterval(heartbeatTimer.current);
      wsRef.current?.close(1000, "component unmounted");
      wsRef.current = null;
    };
  }, [connect]);

  return { status, send };
}

    Using it

    function ChatRoom() {
  const [messages, setMessages] = useState<Message[]>([]);

  const { status, send } = useWebSocket("wss://api.example.com/chat", {
    onMessage: (data) => setMessages((prev) => [...prev, data as Message]),
  });

  return (
    <div>
      <ConnectionBadge status={status} />
      <MessageList messages={messages} />
      <button
        disabled={status !== "open"}
        onClick={() => send({ type: "message", text: "Hi!" })}
      >
        Send
      </button>
    </div>
  );
}

    Why these patterns matter

    • onMessageRef — we store the callback in a ref so a changing onMessage prop doesn't tear down and rebuild the socket on every render.
    • Exponential backoff + jitter — 1s, 2s, 4s, 8s… capped at 30s, plus random jitter so a thousand clients don't all reconnect on the same tick.
    • Heartbeat — load balancers and proxies often drop sockets idle for ~30–60s. A periodic ping keeps the connection (and detects half-open ones).
    • Cleanup — the effect's return function flips shouldReconnect off, clears timers, and closes the socket. This is what makes Strict Mode and unmount safe.

    4️⃣ Socket.IO

    The raw WebSocket API is low-level: no rooms, no automatic reconnection, no fallback transport, no event names — you build all of that yourself. Socket.IO is a battle-tested library (client + server) that adds:

    • Automatic reconnection with backoff (built in).
    • Named events instead of one big onmessage switch.
    • Rooms and namespaces for grouping clients (e.g. one room per chat).
    • Acknowledgements (request/response over the socket).
    • Automatic fallback to HTTP long-polling when WebSockets are blocked.

    Socket.IO is not plain WebSocket

    A Socket.IO client cannot talk to a raw WebSocket server and vice versa — they use different framing/handshake protocols. Use the Socket.IO client only with a Socket.IO server.

    Client setup

    npm install socket.io-client

    // socket.ts — a single shared instance
import { io, Socket } from "socket.io-client";

export const socket: Socket = io("https://api.example.com", {
  autoConnect: false,          // connect manually after auth
  transports: ["websocket"],   // skip the polling handshake if WS is fine
  reconnectionAttempts: 10,
  reconnectionDelay: 1000,     // backoff is handled for you
});

    Events and rooms

    socket.connect();

// listen
socket.on("message", (msg) => console.log("new message", msg));
socket.on("connect", () => console.log("connected", socket.id));
socket.on("disconnect", (reason) => console.log("disconnected", reason));

// emit
socket.emit("join_room", { roomId: "general" });
socket.emit("message", { roomId: "general", text: "Hello room!" });

// emit with acknowledgement (server calls the callback)
socket.emit("message", payload, (ack) => {
  if (ack.ok) console.log("delivered", ack.id);
});

    Rooms are a server-side concept; the client just asks to join/leave and the server decides who receives a broadcast:

    // server.js (Node) — for reference
io.on("connection", (socket) => {
  socket.on("join_room", ({ roomId }) => socket.join(roomId));
  socket.on("message", ({ roomId, text }) => {
    // broadcast to everyone in the room except the sender
    socket.to(roomId).emit("message", { text, from: socket.id });
  });
});

    React integration

    Wrap the shared instance in a hook so components subscribe to events with proper cleanup (otherwise listeners stack up on every render/remount):

    // useSocketEvent.ts
import { useEffect } from "react";
import { socket } from "./socket";

export function useSocketEvent<T>(event: string, handler: (data: T) => void) {
  useEffect(() => {
    socket.on(event, handler);
    return () => {
      socket.off(event, handler); // remove THIS listener on unmount
    };
  }, [event, handler]);
}

    function Room({ roomId }: { roomId: string }) {
  const [messages, setMessages] = useState<Message[]>([]);

  // join/leave the room with the component lifecycle
  useEffect(() => {
    socket.connect();
    socket.emit("join_room", { roomId });
    return () => {
      socket.emit("leave_room", { roomId });
      // Don't disconnect the shared socket here if other components use it.
    };
  }, [roomId]);

  useSocketEvent<Message>("message", (msg) =>
    setMessages((prev) => [...prev, msg]),
  );

  const send = (text: string) => socket.emit("message", { roomId, text });

  return <Chat messages={messages} onSend={send} />;
}

    Use a single shared socket, not one per component

    Creating io() inside a component opens a new connection every mount. Define one instance in a module (or a Context provider) and have components subscribe to events on it. Disconnect once — typically on logout or app unmount.

    5️⃣ Server-Sent Events (SSE)

    SSE is the often-forgotten middle ground. It's a one-way stream from server to client over a normal HTTP connection, using the built-in EventSource API. The browser handles reconnection and event IDs for you.

    const es = new EventSource("/api/notifications");

es.onmessage = (e) => console.log("data:", e.data);                // default event
es.addEventListener("price", (e) => console.log("price", e.data)); // named event
es.onerror = () => console.log("connection lost (browser will retry)");

es.close(); // stop listening

    A useEventSource hook

    // useEventSource.ts
import { useEffect, useRef, useState } from "react";

export function useEventSource<T>(
  url: string,
  onMessage: (data: T) => void,
) {
  const [connected, setConnected] = useState(false);
  const onMessageRef = useRef(onMessage);
  onMessageRef.current = onMessage;

  useEffect(() => {
    const es = new EventSource(url, { withCredentials: true });

    es.onopen = () => setConnected(true);
    es.onmessage = (event) => {
      try {
        onMessageRef.current(JSON.parse(event.data) as T);
      } catch {
        onMessageRef.current(event.data as unknown as T);
      }
    };
    es.onerror = () => setConnected(false); // EventSource auto-reconnects

    return () => es.close(); // cleanup on unmount
  }, [url]);

  return { connected };
}

    function Notifications() {
  const [items, setItems] = useState<Notice[]>([]);
  const { connected } = useEventSource<Notice>("/api/notifications", (n) =>
    setItems((prev) => [n, ...prev]),
  );
  return <NoticeFeed items={items} connected={connected} />;
}

    When to prefer SSE over WebSockets

    Prefer SSE when… Prefer WebSockets when…
    Data flows server → client only You need to send data too (chat, cursors)
    You want auto-reconnect for free You need binary frames or sub-protocols
    Updates are text/JSON (feeds, logs, progress) You need very high message rates both ways
    You want to ride existing HTTP/2 infra You're already on Socket.IO

    SSE quirks to remember

    • SSE is text only (UTF-8) — no binary.
    • Over HTTP/1.1, browsers cap connections per domain (~6), so many SSE streams can starve other requests. HTTP/2 removes this limit — strongly preferred.
    • The native EventSource cannot send custom headers (e.g. Authorization). Pass auth via cookies (withCredentials) or a query param, or use a polyfill like event-source-polyfill that supports headers.

    6️⃣ Real-Time + TanStack Query v5

    Here's the key insight: TanStack Query is already your cache. Don't keep a separate useState array of live data that drifts out of sync with your queries. Instead, let the socket write into the Query cache, and your components keep reading from useQuery exactly as before.

    Two patterns: surgically update the cache (setQueryData) or invalidate and let Query refetch.

    Pattern A — setQueryData (no refetch)

    Apply the pushed update directly to the cache. Fast and bandwidth-free.

    import { useQuery, useQueryClient } from "@tanstack/react-query";
import { useWebSocket } from "./useWebSocket";

function useLiveMessages(roomId: string) {
  const queryClient = useQueryClient();

  useWebSocket(`wss://api.example.com/rooms/${roomId}`, {
    onMessage: (data) => {
      const msg = data as Message;
      queryClient.setQueryData<Message[]>(["messages", roomId], (old = []) => [
        ...old,
        msg,
      ]);
    },
  });
}

function Room({ roomId }: { roomId: string }) {
  useLiveMessages(roomId); // wires the socket into the cache
  const { data: messages } = useQuery({
    queryKey: ["messages", roomId],
    queryFn: () => fetchMessages(roomId),
  });
  return <MessageList messages={messages ?? []} />;
}

    Pattern B — invalidateQueries (refetch source of truth)

    When the push only signals "something changed" (not the full new value), or you don't trust the payload to be complete, invalidate and let Query refetch:

    queryClient.invalidateQueries({ queryKey: ["orders"] });

    Which pattern?

    Use setQueryData when the message contains the full, trustworthy new state — it's instant and avoids a round trip. Use invalidateQueries when the message is just a notification, or when computing the new cache value is error-prone (pagination, sorting, server-side filtering).

    Streaming updates with streamedQuery (v5.x)

    TanStack Query v5 ships a streamedQuery helper that accumulates results from an async iterable (e.g. a streamed LLM response or a chunked feed) into the query's data over time:

    import { useQuery } from "@tanstack/react-query";
import { streamedQuery } from "@tanstack/react-query";

const { data } = useQuery({
  queryKey: ["chat-stream", chatId],
  queryFn: streamedQuery({
    streamFn: () => fetchChatChunks(chatId), // returns an AsyncIterable<Chunk>
  }),
});
// `data` is an array that grows as chunks arrive

    This is ideal for token-by-token AI responses or progressively loaded lists, while still benefiting from Query's caching, status flags, and devtools.

    7️⃣ Optimistic UI for Real-Time

    Real-time apps feel instant when the sender's own action shows immediately, before the server echoes it back. With a socket you typically:

    1. Add the message to the cache right away with a temporary id and pending flag.
    2. Emit it to the server.
    3. Reconcile when the server's broadcast (with the real id) arrives.
    function useSendMessage(roomId: string) {
  const queryClient = useQueryClient();
  const { send } = useWebSocket(`wss://api.example.com/rooms/${roomId}`, {
    onMessage: (data) => {
      const incoming = data as Message;
      queryClient.setQueryData<Message[]>(["messages", roomId], (old = []) => {
        // Replace the optimistic placeholder if its tempId matches; else append.
        const exists = old.some((m) => m.tempId === incoming.tempId);
        return exists
          ? old.map((m) => (m.tempId === incoming.tempId ? incoming : m))
          : [...old, incoming];
      });
    },
  });

  return (text: string) => {
    const tempId = crypto.randomUUID();
    const optimistic: Message = { id: tempId, tempId, text, pending: true };

    // 1. show immediately
    queryClient.setQueryData<Message[]>(["messages", roomId], (old = []) => [
      ...old,
      optimistic,
    ]);
    // 2. send (server should echo back the same tempId)
    send({ type: "message", text, tempId });
  };
}

    Always plan the rollback

    If the send fails or times out, mark the optimistic message as failed (or remove it) so users can retry. Optimistic UI that silently loses a message is worse than a spinner. With useMutation you get onError rollback for free; for raw sockets, track a timeout and reconcile manually.

    8️⃣ Presence & Typing Indicators

    Presence ("who's online") and typing indicators are classic socket features. The trick is throttling outbound events and expiring stale state.

    Typing indicator

    function useTyping(roomId: string, send: (p: unknown) => void) {
  const lastSent = useRef(0);

  // Throttle: emit "typing" at most once per second.
  const onType = () => {
    const now = Date.now();
    if (now - lastSent.current > 1000) {
      send({ type: "typing", roomId });
      lastSent.current = now;
    }
  };
  return onType;
}

    On the receiving side, show the indicator and auto-clear it if no new typing event arrives within a few seconds (the sender may have closed the tab):

    function TypingIndicator({ roomId }: { roomId: string }) {
  const [typers, setTypers] = useState<Record<string, number>>({});

  useSocketEvent<{ userId: string }>("typing", ({ userId }) => {
    setTypers((prev) => ({ ...prev, [userId]: Date.now() }));
  });

  // Expire entries older than 3s.
  useEffect(() => {
    const id = setInterval(() => {
      setTypers((prev) => {
        const now = Date.now();
        return Object.fromEntries(
          Object.entries(prev).filter(([, t]) => now - t < 3000),
        );
      });
    }, 1000);
    return () => clearInterval(id); // cleanup the interval
  }, []);

  const names = Object.keys(typers);
  if (names.length === 0) return null;
  return <p className="typing">{names.join(", ")} typing…</p>;
}

    Presence

    Presence usually relies on the server detecting connect/disconnect. Socket.IO's disconnect event (and rooms) makes this straightforward:

    // server.js (reference)
io.on("connection", (socket) => {
  io.emit("presence", { userId: socket.userId, online: true });
  socket.on("disconnect", () =>
    io.emit("presence", { userId: socket.userId, online: false }),
  );
});

    Heartbeats power presence

    A user who loses Wi-Fi doesn't send a clean disconnect. Rely on the transport's heartbeat/ping timeout (Socket.IO does this automatically) to flip them offline, rather than trusting an explicit "goodbye" message.

    9️⃣ Scaling & Auth for Sockets

    Authenticating the handshake (token-based)

    You cannot rely on Authorization headers for the browser WebSocket / EventSource APIs (they don't accept custom headers). The common, robust options:

    1. Cookie-based — if you use HttpOnly cookies, the browser sends them on the handshake automatically. Best for same-site apps.
    2. Token in the connect payload — send a short-lived token after the socket opens and let the server validate before allowing any events.
    3. Token via auth/query field — Socket.IO exposes an auth option; raw WS often uses a query param (kept short-lived because URLs get logged).
    // Socket.IO: token via the `auth` field (validated in server middleware)
export const socket = io("https://api.example.com", {
  autoConnect: false,
  auth: { token: getAccessToken() },
});

// Refresh the token before reconnecting (tokens expire):
socket.on("connect_error", async (err) => {
  if (err.message === "auth_expired") {
    socket.auth = { token: await refreshAccessToken() };
    socket.connect();
  }
});

    // server.js — Socket.IO auth middleware (reference)
io.use((socket, next) => {
  const token = socket.handshake.auth.token;
  try {
    socket.userId = verifyJwt(token).sub;
    next();
  } catch {
    next(new Error("auth_expired"));
  }
});

    Short-lived tokens for sockets

    Long-lived sockets outlive access tokens. Use short-lived access tokens and re-authenticate on reconnect, or validate the token once at handshake and trust the established connection until it drops. Never put refresh tokens in a URL.

    Scaling across multiple servers

    A single Node process holds sockets in memory, so a broadcast only reaches clients on that process. When you run multiple instances (you will), you need:

    • Sticky sessions at the load balancer (a client must reach the same instance for the duration of its connection — especially for Socket.IO's HTTP polling fallback).
    • A pub/sub adapter (e.g. the Socket.IO Redis adapter) so a message emitted on instance A is fanned out to clients connected to instances B and C.
    // server.js — Redis adapter for horizontal scaling (reference)
import { createAdapter } from "@socket.io/redis-adapter";
import { createClient } from "redis";

const pub = createClient({ url: process.env.REDIS_URL });
const sub = pub.duplicate();
await Promise.all([pub.connect(), sub.connect()]);
io.adapter(createAdapter(pub, sub));

    Frontend takeaways for scale

    You rarely write this backend code, but it affects you: design messages to be idempotent (clients may receive a message twice after a reconnect), include a monotonic id or timestamp so the client can dedupe/reorder, and never assume strict ordering across reconnects.

    🔟 Cleanup — The Non-Negotiable

    Every example above closes its connection in a useEffect cleanup. This is the single most common real-time bug source. Get it wrong and you leak sockets, double up event listeners, and break in React 18/19 Strict Mode.

    useEffect(() => {
  const ws = new WebSocket(url);
  // …wire up handlers…

  return () => {
    ws.close(1000, "unmount"); // ← REQUIRED
  };
}, [url]);

    Checklist for any real-time hook:

    • [x] Close the connection (ws.close(), es.close(), or socket.off(...)) in the cleanup function.
    • [x] Clear all timers — reconnect timeouts and heartbeat intervals.
    • [x] Remove the specific listener with off, not a blanket teardown of a shared socket other components still use.
    • [x] Stop reconnection on intentional unmount (the shouldReconnect flag).
    • [x] Test in Strict Mode — if it survives mount → unmount → mount cleanly, it'll survive production.

    Strict Mode is a feature, not a bug

    React 18/19 deliberately double-invokes effects in development to surface missing cleanup. If your socket "connects twice" in dev, that is the warning that your cleanup is incomplete — fix the hook, don't disable Strict Mode.

    1️⃣1️⃣ Summary

    Need Reach for
    Occasional updates, REST-only backend TanStack Query refetchInterval (polling)
    Two-way, low-latency (chat, games, cursors) Socket.IO (or raw WS + useWebSocket)
    One-way server → client (feeds, logs, progress) SSE (useEventSource)
    Keeping live data and cache in sync setQueryData / invalidateQueries
    Streaming/growing results (AI tokens, chunks) streamedQuery
    Instant sender feedback Optimistic update + reconcile on echo

    Golden rules:

    • Pick the cheapest transport that meets your real latency need — don't reach for WebSockets when a 5s poll or SSE will do.
    • Let TanStack Query own the data; sockets just write into the cache.
    • Always clean up connections, listeners, and timers on unmount.
    • For sockets at scale: short-lived auth tokens, sticky sessions, a pub/sub adapter, and idempotent, ordered messages.
