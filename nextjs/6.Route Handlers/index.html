    Route Handlers (API)

    A Route Handler is how you build an HTTP API inside the App Router. You create a route.ts file and export functions named after HTTP verbs (GET, POST, …); Next.js wires each one up as an endpoint. Handlers are built on the Web Platform Request/Response APIs, so the same code you'd write for a service worker, Deno, or Cloudflare Workers mostly carries over—Next.js just adds a couple of ergonomic helpers (NextRequest/NextResponse).

    This is the App Router's replacement for pages/api/*. The mental model is different too: because Server Components can read your database directly and Server Actions handle form mutations, you no longer reach for an API route for every server interaction. Route Handlers are for things that genuinely need a public HTTP endpoint—webhooks, third-party callbacks, OAuth, public/mobile APIs, file downloads, and streaming.

    📚 For more: https://nextjs.org/docs/app/building-your-application/routing/route-handlers

    1. route.ts basics

    A Route Handler lives at app/<segment>/route.ts and exports one function per HTTP method it supports. The file takes the place of page.tsx for that segment—you can't have both page.tsx and route.ts in the same folder, because both claim the same URL.

    // app/api/health/route.ts → responds at GET /api/health
export async function GET() {
  return Response.json({ status: "ok", time: Date.now() });
}

    You're not limited to an api/ folder—that's just a convention. app/sitemap-data/route.ts answers at /sitemap-data. The api/ prefix is purely organizational (and keeps API URLs visually distinct from pages).

    Supported HTTP methods

    You can export any of these named functions. Each receives the incoming request and returns a Response (or a Promise<Response>):

    // app/api/items/route.ts
export async function GET(request: Request) { /* read */ }
export async function POST(request: Request) { /* create */ }
export async function PUT(request: Request) { /* replace */ }
export async function PATCH(request: Request) { /* partial update */ }
export async function DELETE(request: Request) { /* remove */ }
export async function HEAD(request: Request) { /* headers only */ }
export async function OPTIONS(request: Request) { /* preflight / capabilities */ }
    • If a method isn't exported, Next.js automatically returns 405 Method Not Allowed with a correct Allow header.
    • HEAD is handled automatically if you export GET (it runs GET and strips the body), but you can export it explicitly to optimize.
    • OPTIONS is auto-generated for the methods you export, though you'll often write your own for CORS preflight (see §5).

    Verbs are uppercase named exports

    They must be the exact uppercase verb names. export async function get() won't be picked up. They can be async or sync; returning a Promise<Response> is the norm since you usually await data.

    What a handler returns

    A handler must resolve to a Web Response. You have three common ways to build one:

    // 1. The native Response constructor — full control over body + init
return new Response("Hello", { status: 200, headers: { "X-Custom": "1" } });

// 2. Response.json() — a static helper that sets Content-Type for you (Web standard)
return Response.json({ ok: true }, { status: 201 });

// 3. NextResponse — Next's subclass, adds cookies(), redirect(), rewrite(), next()
import { NextResponse } from "next/server";
return NextResponse.json({ ok: true }, { status: 201 });

    Response.json() and NextResponse.json() produce the same JSON response; reach for NextResponse when you need its extras (cookie mutation, redirects, rewrites).

    2. NextRequest/NextResponse vs Request/Response

    Everything in a Route Handler is plain Web Request/Response by default. next/server exports two subclasses that add Next-specific conveniences—you opt in by typing your parameter as NextRequest and/or returning NextResponse.

    NextRequest — extends Request

    import type { NextRequest } from "next/server";

export async function GET(request: NextRequest) {
  request.nextUrl.pathname;           // parsed URL (no manual `new URL(request.url)`)
  request.nextUrl.searchParams;       // a URLSearchParams you can read directly
  request.cookies.get("session");     // typed cookie helpers (get/getAll/has)
  request.headers.get("user-agent");  // standard Headers (same as native Request)
}

    Without NextRequest, you'd reconstruct the URL yourself: const { searchParams } = new URL(request.url). Both work—NextRequest is just less boilerplate.

    NextResponse — extends Response

    import { NextResponse } from "next/server";

// JSON (typed via a generic)
NextResponse.json<{ id: string }>({ id: "1" }, { status: 201 });

// Redirect (must be an absolute URL)
NextResponse.redirect(new URL("/login", request.url));

// Set cookies on the response
const res = NextResponse.json({ ok: true });
res.cookies.set("token", "abc", { httpOnly: true, secure: true, path: "/" });
return res;

    When to use which

    Use bare Request/Response for simple, portable handlers (and to keep the Web-standard mental model). Reach for NextRequest/NextResponse when you want nextUrl, the cookie helpers, or redirect/rewrite—i.e. the things plain Web APIs make awkward. You can mix them: accept a NextRequest and return a plain Response, or vice versa.

    3. Reading request data

    Dynamic route segments (async params)

    Dynamic segments work exactly like pages. app/api/posts/[id]/route.ts receives a second argument with params. In Next.js 15, params is a Promise and must be awaited.

    // app/api/posts/[id]/route.ts → /api/posts/123
export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;            // Next 15: params is async
  const post = await db.post.findUnique({ where: { id } });
  if (!post) return new Response("Not found", { status: 404 });
  return Response.json(post);
}

    Catch-all segments give you a string array—app/api/files/[...path]/route.ts yields params: Promise<{ path: string[] }> (so /api/files/a/b/c → ["a", "b", "c"]).

    params is a Promise in Next 15

    In Next.js 14 params was a plain object. In 15 it's a Promise (aligning with the async-request-APIs change). Always destructure with await params. The same applies to cookies() and headers() from next/headers.

    Query / search params

    Read them off nextUrl.searchParams (a standard URLSearchParams):

    // /api/search?q=next&page=2&tag=ssr&tag=rsc
export async function GET(request: NextRequest) {
  const sp = request.nextUrl.searchParams;
  const q = sp.get("q") ?? "";          // "next"
  const page = Number(sp.get("page") ?? "1");  // 2
  const tags = sp.getAll("tag");        // ["ssr", "rsc"] — repeated keys
  const has = sp.has("q");              // true

  const results = await search(q, { page, tags });
  return Response.json(results);
}

    With a plain Request, do const { searchParams } = new URL(request.url) instead.

    Reading searchParams makes the route dynamic

    Touching the query string opts the handler out of static rendering (there's nothing to statically cache—the answer depends on the URL). That's usually what you want for a search endpoint.

    JSON body

    request.json() parses the body. It throws on malformed JSON, so guard it:

    export async function POST(request: Request) {
  let body: unknown;
  try {
    body = await request.json();
  } catch {
    return Response.json({ error: "Invalid JSON" }, { status: 400 });
  }

  // Validate before trusting it (Zod recommended — never trust the client)
  const parsed = CreatePostSchema.safeParse(body);
  if (!parsed.success) {
    return Response.json({ errors: parsed.error.flatten() }, { status: 422 });
  }

  const created = await db.post.create({ data: parsed.data });
  return Response.json(created, { status: 201 });
}

    A Zod schema for the example above:

    import { z } from "zod";

const CreatePostSchema = z.object({
  title: z.string().min(1),
  body: z.string().min(1),
  published: z.boolean().default(false),
});

    A request body can only be read once

    request.json(), request.text(), request.formData(), and request.arrayBuffer() all consume the stream. Call exactly one. If you need the raw bytes and the parsed value (e.g. webhook signature verification, §5), read the raw form first and parse from that.

    Form data (and file uploads)

    For application/x-www-form-urlencoded or multipart/form-data bodies, use request.formData():

    export async function POST(request: Request) {
  const form = await request.formData();

  const email = form.get("email");      // string | File | null
  const avatar = form.get("avatar");    // a File for <input type="file">

  if (avatar instanceof File) {
    const bytes = Buffer.from(await avatar.arrayBuffer());
    await saveToStorage(avatar.name, bytes);
  }

  return Response.json({ email });
}

    form.get() returns string | File | null; form.getAll("key") returns every value for a repeated field.

    Cookies

    Two ways, both valid:

    import { cookies } from "next/headers";

export async function GET(request: NextRequest) {
  // (a) From the request object (NextRequest)
  const sessionA = request.cookies.get("session")?.value;

  // (b) From next/headers — async in Next 15, works across the App Router
  const cookieStore = await cookies();
  const sessionB = cookieStore.get("session")?.value;

  return Response.json({ session: sessionA ?? sessionB });
}

    To set a cookie, mutate the response (you cannot set a request cookie):

    const res = NextResponse.json({ ok: true });
res.cookies.set("theme", "dark", {
  httpOnly: true,
  secure: process.env.NODE_ENV === "production",
  sameSite: "lax",
  path: "/",
  maxAge: 60 * 60 * 24 * 7, // one week
});
return res;

    Headers

    import { headers } from "next/headers";

export async function GET(request: NextRequest) {
  // (a) Off the request
  const auth = request.headers.get("authorization");

  // (b) From next/headers (read-only, async in Next 15)
  const h = await headers();
  const ua = h.get("user-agent");

  if (!auth?.startsWith("Bearer ")) {
    return new Response("Unauthorized", { status: 401 });
  }
  return Response.json({ ua });
}

    Reading cookies/headers forces dynamic rendering

    Like searchParams, calling cookies() or headers() (or reading request.cookies) marks the handler dynamic—it can't be cached because the response depends on the incoming request.

    4. Caching & dynamic behavior

    This is the part that trips people up most, and it changed between major versions.

    GET handlers are uncached by default in Next.js 15

    In Next.js 14, a GET Route Handler with no dynamic inputs was cached at build time by default (it behaved like a static asset). In Next.js 15, that default flipped: Route Handlers are not cached by default—every GET runs per request unless you opt in.

    Next.js 14 Next.js 15
    GET with no dynamic inputs Cached (static) by default Uncached (dynamic) by default
    POST/PUT/PATCH/DELETE Always dynamic Always dynamic
    Reading searchParams/cookies/headers Forces dynamic Forces dynamic 
    // Next.js 15: this runs on every request by default
export async function GET() {
  const data = await getExpensiveData();
  return Response.json(data);
}

    Opting back into caching

    Use the route segment config exports to control behavior explicitly. They're plain export const values Next.js reads at build time:

    // app/api/feed/route.ts

// Force this GET to be statically cached (build-time / shared)
export const dynamic = "force-static";

// …or revalidate the cached response at most every 60s (ISR for the endpoint)
export const revalidate = 60;

export async function GET() {
  const posts = await db.post.findMany();
  return Response.json(posts);
}

    Key segment-config options for handlers:

    Export Effect
    export const dynamic = "force-static" Cache the GET response; dynamic functions return empty/default values
    export const dynamic = "force-dynamic" Never cache; run per request (the Next 15 default for handlers)
    export const dynamic = "auto" Default — Next.js decides based on what you use
    export const revalidate = 60 Time-based revalidation (seconds) for a cached handler
    export const runtime = "edge" Run on the Edge runtime instead of Node.js
    export const fetchCache = "..." Override the default caching of fetch calls inside the handler

    You can also control caching at the data layer instead of the whole route—per-fetch options still apply inside a handler:

    export async function GET() {
  // This specific fetch is cached & revalidated, independent of the route
  const res = await fetch("https://api.example.com/rates", {
    next: { revalidate: 3600, tags: ["rates"] },
  });
  return Response.json(await res.json());
}

    …and invalidate it on demand from another handler with revalidateTag("rates").

    Be explicit

    Because the default changed across versions—and is the opposite of what fetch does inside Server Components—don't rely on it. State dynamic/revalidate (or per-fetch cache/revalidate) intentionally so the behavior is obvious to the next reader and stable across upgrades.

    5. Cross-cutting concerns

    CORS

    By default a Route Handler only answers same-origin requests in a browser. To allow cross-origin calls, set CORS headers yourself—and handle the OPTIONS preflight:

    // app/api/public/route.ts
import { NextResponse } from "next/server";

const ALLOWED_ORIGIN = "https://app.example.com";

const corsHeaders = {
  "Access-Control-Allow-Origin": ALLOWED_ORIGIN,
  "Access-Control-Allow-Methods": "GET, POST, OPTIONS",
  "Access-Control-Allow-Headers": "Content-Type, Authorization",
  "Access-Control-Max-Age": "86400",
};

// Browsers send a preflight OPTIONS for non-simple requests
export async function OPTIONS() {
  return new NextResponse(null, { status: 204, headers: corsHeaders });
}

export async function GET() {
  return NextResponse.json({ ok: true }, { headers: corsHeaders });
}

    CORS for all routes belongs in middleware or next.config

    Per-route headers are fine for one endpoint. To apply CORS across many routes, set them once in middleware.ts or via the headers() config in next.config.ts. Avoid Access-Control-Allow-Origin: * for authenticated APIs—pin it to known origins.

    Streaming responses

    Return a ReadableStream to stream a response incrementally—ideal for LLM token streams, long-running jobs, or large exports. The browser/consumer receives bytes as they're produced instead of waiting for the whole body.

    // app/api/stream/route.ts
export async function GET() {
  const encoder = new TextEncoder();

  const stream = new ReadableStream({
    async start(controller) {
      for (let i = 1; i <= 5; i++) {
        controller.enqueue(encoder.encode(`chunk ${i}\n`));
        await new Promise((r) => setTimeout(r, 500));
      }
      controller.close();
    },
  });

  return new Response(stream, {
    headers: { "Content-Type": "text/plain; charset=utf-8" },
  });
}

    For Server-Sent Events (a long-lived text/event-stream the browser consumes via EventSource), the shape is the same—just format each chunk as data: ...\n\n and set the SSE content type:

    return new Response(stream, {
  headers: {
    "Content-Type": "text/event-stream",
    "Cache-Control": "no-cache, no-transform",
    Connection: "keep-alive",
  },
});

    AI streaming

    For LLM apps, the Vercel AI SDK (ai package) wraps this pattern—result.toDataStreamResponse() returns exactly such a streaming Response from a Route Handler, and a matching client hook consumes it. Under the hood it's the ReadableStream shown above.

    Webhooks (raw body + signature verification)

    Webhooks (Stripe, GitHub, Clerk, etc.) sign the request so you can prove it really came from them. The signature is computed over the exact raw bytes, so you must verify against request.text()/arrayBuffer() before parsing—request.json() would re-serialize and could change the bytes, breaking the signature.

    // app/api/webhooks/stripe/route.ts
import { NextResponse } from "next/server";
import Stripe from "stripe";

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);
const webhookSecret = process.env.STRIPE_WEBHOOK_SECRET!;

export async function POST(request: Request) {
  // 1. Read the RAW body — do not call request.json() first
  const rawBody = await request.text();
  const signature = request.headers.get("stripe-signature");

  if (!signature) {
    return NextResponse.json({ error: "Missing signature" }, { status: 400 });
  }

  // 2. Verify the signature against the raw bytes
  let event: Stripe.Event;
  try {
    event = stripe.webhooks.constructEvent(rawBody, signature, webhookSecret);
  } catch (err) {
    return NextResponse.json({ error: "Invalid signature" }, { status: 400 });
  }

  // 3. Now it's safe to act on the (verified) event
  switch (event.type) {
    case "checkout.session.completed":
      await fulfillOrder(event.data.object);
      break;
    // …other event types
  }

  // 4. Return 2xx quickly so the provider doesn't retry
  return NextResponse.json({ received: true });
}

    Webhook gotchas

    • Read the raw body first. Calling request.json() consumes the stream and reshapes the bytes—signature verification will fail. In the App Router you get the raw body simply via request.text() (no bodyParser config needed, unlike the old Pages Router).
    • Return 2xx fast. Providers retry on non-2xx or slow responses, which can double-process events. Acknowledge first, then do heavy work (queue it, or use after() from next/server).
    • Make handlers idempotent. Retries and at-least-once delivery mean the same event can arrive twice—key off the event ID.

    6. Route Handlers vs Server Actions — when to use which

    Both run on the server and both can mutate data, so it's easy to confuse them. The distinction: a Server Action is an RPC-style function you call from your own React UI; a Route Handler is a real, addressable HTTP endpoint anyone can call.

    // Server Action — "use server"; invoked from a form/component, no URL
"use server";
export async function createPost(formData: FormData) {
  const title = formData.get("title");
  await db.post.create({ data: { title: String(title) } });
}

    // Route Handler — a URL (POST /api/posts) callable by anything that speaks HTTP
export async function POST(request: Request) {
  const body = await request.json();
  const post = await db.post.create({ data: body });
  return Response.json(post, { status: 201 });
}

    How they differ

    Server Action Route Handler
    Invocation Called like a function from your React app (form action, useActionState, event handler) A public URL — fetch/curl/any HTTP client
    Has a stable URL? No (internal RPC endpoint, obfuscated id) Yes — you design the path & method
    Progressive enhancement Yes — forms work before JS hydrates No — needs a client to make the request
    External callers No (only your app) Yes — webhooks, mobile apps, third parties
    HTTP semantics Hidden (always a POST under the hood) Full control — status codes, headers, methods, CORS
    Return type Serializable JS value A Response (any body/status/headers)
    Typical use Form submits & mutations from your UI Public/3rd-party APIs, webhooks, OAuth callbacks, downloads, streaming

    Rules of thumb

    Pick the right tool

    • Mutation triggered by your own UI? Use a Server Action—less boilerplate, progressive enhancement, built-in pending/optimistic state via useActionState/useFormStatus/useOptimistic, and easy revalidatePath/revalidateTag after.
    • Reading data for your own UI? Often you need neither—fetch directly in a Server Component (await db…). Skip the network hop.
    • Something external must call you over HTTP? Use a Route Handler: webhooks, OAuth/payment callbacks, public REST/JSON APIs, mobile/native clients, file downloads/uploads, RSS/sitemaps, and streaming responses (SSE, AI token streams).
    • You need precise HTTP control (custom status codes, Cache-Control, content negotiation, CORS): Route Handler.

    In short: Server Actions for your app's own mutations; Route Handlers when the caller isn't your React app or when you need true HTTP control. They coexist happily—most apps use both.
