    Optimizations (Image, Metadata/SEO, Scripts)

    Next.js ships a set of built-in optimizations that you'd otherwise wire up by hand: an image pipeline that resizes and reformats on demand, a typed Metadata API that produces correct SEO and social-share tags, file conventions for sitemaps and robots.txt, controlled loading of third-party scripts, and code-splitting via lazy components. The throughline is the same one from the rendering chapter—ship less to the client, and ship it at the right moment—because that is what moves Core Web Vitals.

    This chapter is App Router-first, targeting Next.js 15 with TypeScript. Everything here is the framework's default toolkit; you reach for these on essentially every production app.

    📚 For more: https://nextjs.org/docs/app/building-your-application/optimizing

    1. Images with next/image

    <img> is easy to get wrong: oversized files, no responsive variants, no modern formats, and layout shift while the image loads. The next/image component fixes all of those by optimizing images on demand—resizing to the requested dimensions, serving modern formats (AVIF/WebP), lazy-loading off-screen images, and reserving space to prevent layout shift.

    Basic usage with a local import

    For images in your project, import them. Next.js reads the intrinsic width/height at build time, so it can reserve the correct space automatically (no width/height props needed) and even generate a blur placeholder.

    // app/page.tsx
import Image from "next/image";
import hero from "@/public/hero.jpg"; // a real file in /public or anywhere importable

export default function Home() {
  return (
    <Image
      src={hero}
      alt="A mountain range at sunrise"
      placeholder="blur" // auto blur-up using a generated low-res preview
      // width/height inferred from the imported file
    />
  );
}

    For remote images (a URL string), Next.js can't know the dimensions, so you must provide width and height (the intrinsic aspect ratio, not a CSS size) or use fill:

    <Image
  src="https://cdn.example.com/photos/123.jpg"
  alt="Product photo"
  width={800}
  height={600} // sets the aspect ratio; CSS can still scale it responsively
/>

    width/height are the aspect ratio, not the display size

    The width and height props tell Next.js the image's proportions so it can reserve space and avoid layout shift. Control the rendered size with CSS (e.g. className="w-full h-auto"). Setting width={800} does not force the image to render at 800px.

    Responsive images and sizes

    By default next/image is lazy and generates a srcset of candidate widths. The browser picks the right one—but only if you tell it how wide the image will be on screen via the sizes prop. Without sizes, a full-width responsive image may download a desktop-sized file on a phone.

    import Image from "next/image";
import banner from "@/public/banner.jpg";

export default function ResponsiveBanner() {
  return (
    <Image
      src={banner}
      alt="Seasonal sale banner"
      // The image is full-viewport on mobile, half on tablet, a third on desktop.
      sizes="(max-width: 640px) 100vw, (max-width: 1024px) 50vw, 33vw"
      className="w-full h-auto"
    />
  );
}

    sizes is a media-query list read left to right; the first match wins, and the final value is the fallback. Next.js uses it to generate an appropriate srcset so the browser fetches the smallest sufficient variant.

    sizes is mandatory whenever you use fill or a fluid width

    If the image scales with the viewport (most responsive layouts), set sizes. The default (100vw) is correct for true full-bleed images but wasteful for anything narrower.

    fill for unknown-dimension containers

    When you don't know the intrinsic size—or you want the image to cover a styled box (hero sections, card thumbnails, art-directed crops)—use fill. The image absolutely positions itself to fill the nearest positioned ancestor, so the parent must be position: relative (or absolute/fixed) and have a size.

    import Image from "next/image";

export default function Card() {
  return (
    <div className="relative aspect-video w-full overflow-hidden rounded-lg">
      <Image
        src="https://cdn.example.com/cover.jpg"
        alt="Article cover"
        fill
        sizes="(max-width: 768px) 100vw, 50vw"
        className="object-cover" // control cropping with object-fit
      />
    </div>
  );
}

    With fill, omit width/height and use object-fit (object-cover/object-contain) plus object-position to control cropping.

    priority for the LCP image

    Off-screen images lazy-load, which is what you want—except for the largest image above the fold, which is usually your Largest Contentful Paint element. Mark it priority to preload it (and skip lazy-loading), improving LCP.

    <Image src={hero} alt="..." priority sizes="100vw" />

    Use priority sparingly

    priority adds a <link rel="preload">, competing for early bandwidth. Apply it to the one LCP image per page (typically the hero), not to every image. Next.js warns in dev if a large above-the-fold image lacks priority.

    Remote images and next.config

    By design, Next.js will only optimize remote images from hosts you explicitly allow—otherwise your image endpoint could be abused as an open proxy. Configure allowed hosts with remotePatterns:

    // next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "cdn.example.com",
        pathname: "/photos/**", // optional path scoping
      },
      {
        protocol: "https",
        hostname: "*.supabase.co", // wildcard subdomains are allowed in hostname
      },
    ],
    // Opt into format negotiation; AVIF is smaller but slower to encode.
    formats: ["image/avif", "image/webp"],
  },
};

export default nextConfig;

    remotePatterns replaced domains

    Older guides use images.domains: [...]. That option is deprecated in favor of remotePatterns, which is more precise (protocol, port, pathname) and therefore safer. Prefer remotePatterns in new code.

    Modern formats and quality

    Next.js negotiates the best format the browser accepts. With formats: ["image/avif", "image/webp"], modern browsers get AVIF (smallest), others fall back to WebP, then the original. Tune compression per-image with quality (default 75):

    <Image src={hero} alt="..." quality={90} priority sizes="100vw" />

    When to skip optimization

    For SVGs, tiny icons, or images you don't control the pipeline for, you can pass unoptimized (per-image or globally). On a fully static export (output: "export") the default optimizer isn't available—use unoptimized or a custom loader pointing at an external CDN.

    2. The Metadata API

    Correct <head> tags drive SEO and social-share previews. Instead of manually rendering <title>/<meta> (and fighting duplicates across nested layouts), the App Router gives you a typed Metadata API: export a metadata object or a generateMetadata function from a layout.tsx or page.tsx, and Next.js renders the tags for you—deduplicated and merged down the route tree.

    Static metadata

    For tags known at build time, export a metadata object typed as Metadata:

    // app/about/page.tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: "About Us",
  description: "Who we are and what we build.",
  keywords: ["company", "team", "mission"],
  alternates: { canonical: "/about" },
};

export default function AboutPage() {
  return <h1>About Us</h1>;
}

    Metadata merges down the tree: a root layout sets site-wide defaults, and each page overrides or extends them. The title field supports a template so child titles get a consistent suffix:

    // app/layout.tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: {
    default: "Acme Inc", // used when a page sets no title
    template: "%s · Acme Inc", // "About Us" → "About Us · Acme Inc"
  },
  description: "Tools for building on the web.",
};

    metadataBase for absolute URLs

    Open Graph and Twitter card URLs must be absolute when scrapers fetch them. Set metadataBase once (usually in the root layout) and then use relative paths everywhere—Next.js resolves them against the base.

    // app/layout.tsx
export const metadata: Metadata = {
  metadataBase: new URL(
    process.env.NEXT_PUBLIC_SITE_URL ?? "https://acme.example.com"
  ),
  openGraph: { images: ["/og.png"] }, // resolved to https://acme.example.com/og.png
};

    Forgetting metadataBase

    Without metadataBase, relative OG/Twitter image paths resolve to localhost in dev and may be dropped or broken in production. Set it once at the root; everything below inherits it.

    Open Graph and Twitter cards

    Social previews come from openGraph and twitter fields. Provide a title, description, and a 1200×630 image for rich link previews:

    // app/blog/[slug]/page.tsx (excerpt)
export const metadata: Metadata = {
  title: "Shipping faster with the App Router",
  openGraph: {
    title: "Shipping faster with the App Router",
    description: "How we cut our build times in half.",
    url: "/blog/app-router-speed",
    siteName: "Acme Blog",
    type: "article",
    images: [
      {
        url: "/blog/app-router-speed/og.png", // resolved via metadataBase
        width: 1200,
        height: 630,
        alt: "App Router performance",
      },
    ],
  },
  twitter: {
    card: "summary_large_image",
    title: "Shipping faster with the App Router",
    description: "How we cut our build times in half.",
    images: ["/blog/app-router-speed/og.png"],
  },
};

    Dynamic metadata with generateMetadata

    When tags depend on data (a blog post's title, a product's name), export an async generateMetadata function instead of a static object. It receives the same params/searchParams as the page.

    // app/blog/[slug]/page.tsx
import type { Metadata } from "next";
import { notFound } from "next/navigation";

type Props = { params: Promise<{ slug: string }> };

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { slug } = await params; // Next.js 15: params is a Promise
  const post = await getPost(slug);
  if (!post) return {}; // page itself will handle notFound()

  return {
    title: post.title,
    description: post.excerpt,
    openGraph: {
      title: post.title,
      description: post.excerpt,
      type: "article",
      publishedTime: post.date,
      images: [{ url: post.coverImage, width: 1200, height: 630 }],
    },
  };
}

export default async function BlogPost({ params }: Props) {
  const { slug } = await params;
  const post = await getPost(slug);
  if (!post) notFound();
  return <article>{/* ... */}</article>;
}

    Data fetches are deduplicated

    Calling getPost(slug) in both generateMetadata and the page looks wasteful, but fetch requests are memoized per request, so the same call runs once. For non-fetch data, wrap the loader in React's cache() to get the same deduplication.

    metadata and "use client" don't mix

    The Metadata API only works in Server Components. You can't export metadata/generateMetadata from a file marked "use client". Keep metadata in the page.tsx/layout.tsx (which stay server components) and push interactivity into child client components.

    3. File-based metadata

    Some metadata is better expressed as files than as exports. Next.js recognizes special filenames in app/ and turns them into the right routes and tags. These can be static assets or dynamic .ts files that generate content.

    sitemap.ts

    A sitemap.ts (or .xml) in app/ produces /sitemap.xml. Export a default function returning a typed MetadataRoute.Sitemap:

    // app/sitemap.ts
import type { MetadataRoute } from "next";

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const baseUrl = "https://acme.example.com";

  // Dynamic entries from your data source
  const posts = await getAllPosts();
  const postEntries: MetadataRoute.Sitemap = posts.map((post) => ({
    url: `${baseUrl}/blog/${post.slug}`,
    lastModified: post.updatedAt,
    changeFrequency: "weekly",
    priority: 0.7,
  }));

  return [
    { url: baseUrl, lastModified: new Date(), changeFrequency: "daily", priority: 1 },
    { url: `${baseUrl}/about`, changeFrequency: "monthly", priority: 0.5 },
    ...postEntries,
  ];
}

    robots.ts

    Similarly, robots.ts generates /robots.txt:

    // app/robots.ts
import type { MetadataRoute } from "next";

export default function robots(): MetadataRoute.Robots {
  return {
    rules: [
      { userAgent: "*", allow: "/", disallow: ["/admin", "/api"] },
    ],
    sitemap: "https://acme.example.com/sitemap.xml",
  };
}

    opengraph-image (and twitter-image)

    Drop an opengraph-image.png (or .jpg) into any route segment and Next.js wires up the correct og:image meta tags automatically—no manual URLs. Even better, generate the image dynamically with opengraph-image.tsx using next/og's ImageResponse, which renders JSX to a PNG at the edge:

    // app/blog/[slug]/opengraph-image.tsx
import { ImageResponse } from "next/og";

export const runtime = "edge";
export const alt = "Blog post cover";
export const size = { width: 1200, height: 630 };
export const contentType = "image/png";

export default async function Image({ params }: { params: { slug: string } }) {
  const post = await getPost(params.slug);

  return new ImageResponse(
    (
      <div
        style={{
          width: "100%",
          height: "100%",
          display: "flex",
          flexDirection: "column",
          justifyContent: "center",
          padding: 80,
          background: "linear-gradient(135deg, #0f172a, #1e293b)",
          color: "white",
          fontSize: 64,
        }}
      >
        <div style={{ fontSize: 28, opacity: 0.7 }}>Acme Blog</div>
        <div style={{ fontWeight: 700 }}>{post?.title ?? "Untitled"}</div>
      </div>
    ),
    { ...size }
  );
}

    Other file conventions

    The same pattern covers favicon.ico, icon.png/icon.tsx, apple-icon.png, and manifest.ts (a web app manifest). Place them in app/; Next.js emits the matching <link>/<meta> tags. A static opengraph-image.png in a segment is the zero-code option; the .tsx version is for per-page, data-driven cards.

    4. Scripts with next/script

    Third-party scripts (analytics, chat widgets, payment SDKs) are a common performance drag. The next/script component lets you load them with an explicit strategy so they don't block your page from becoming interactive, and it deduplicates a script even if multiple components request it.

    // app/layout.tsx
import Script from "next/script";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        {children}

        {/* Analytics: load once the page is idle, never blocking interactivity */}
        <Script
          src="https://example.com/analytics.js"
          strategy="afterInteractive"
        />
      </body>
    </html>
  );
}

    The strategy prop controls when the script loads:

    Strategy When it loads Use for
    beforeInteractive Before any hydration, injected into the initial HTML Critical scripts needed before the page runs (rare): bot detection, consent managers. Must live in the root layout.
    afterInteractive (default) Right after the page becomes interactive Most third-party scripts: analytics, tag managers.
    lazyOnload During browser idle time, after everything else Low-priority widgets: chat, social embeds, support tools.
    worker (experimental) In a web worker via Partytown Offloading heavy third-party JS off the main thread.

    You can also run inline code and react to load events:

    <Script id="show-banner" strategy="afterInteractive">
  {`window.dataLayer = window.dataLayer || []; dataLayer.push({ ready: true });`}
</Script>

<Script
  src="https://maps.example.com/sdk.js"
  strategy="lazyOnload"
  onLoad={() => console.log("Maps SDK ready")} // requires "use client" parent
  onError={(e) => console.error("Maps SDK failed", e)}
/>

    Inline scripts need an id

    Give inline <Script> blocks a unique id so Next.js can track and deduplicate them. onLoad/onError/onReady are client callbacks, so the file using them must be a Client Component.

    5. Lazy loading with next/dynamic

    Code-splitting keeps the initial bundle small by loading heavy components only when needed. next/dynamic is a wrapper around React.lazy + Suspense tailored to Next.js. Use it for components that are large, rarely shown, or below the fold (charts, rich editors, modals, maps).

    // app/dashboard/page.tsx
import dynamic from "next/dynamic";

// HeavyChart and its dependencies are split into a separate chunk,
// fetched only when this component actually renders.
const HeavyChart = dynamic(() => import("@/components/HeavyChart"), {
  loading: () => <p>Loading chart…</p>, // shown while the chunk downloads
});

export default function Dashboard() {
  return (
    <section>
      <h1>Dashboard</h1>
      <HeavyChart />
    </section>
  );
}

    Disabling SSR with ssr: false

    Some components only work in the browser—they touch window, use a non-SSR-safe library, or are purely interactive. Skip server rendering with ssr: false so they render only on the client (avoiding hydration mismatches and server-side crashes).

    "use client";
import dynamic from "next/dynamic";

// A map widget that reads `window` and isn't SSR-safe.
const Map = dynamic(() => import("@/components/Map"), {
  ssr: false,
  loading: () => <div className="h-96 animate-pulse bg-gray-200" />,
});

export default function LocationPicker() {
  return <Map />;
}

    ssr: false requires a Client Component in Next.js 15

    In Next.js 15, calling dynamic(..., { ssr: false }) inside a Server Component is not allowed—it throws a build error. Move the dynamic() call into a file marked "use client" (as above). Without ssr: false, next/dynamic works fine in Server Components.

    Server Components don't need next/dynamic to split code

    RSCs are already streamed and code-split by the framework. Reach for next/dynamic mainly to lazy-load Client Components, to attach a loading fallback, or to opt out of SSR. For deferring server work, prefer <Suspense> with streaming (see the rendering chapter).

    6. Performance: bundle analysis & Core Web Vitals

    Optimization is measurement-driven. Two habits matter most: knowing what's in your JavaScript bundle, and watching the field metrics Google ranks on.

    Bundle analysis with @next/bundle-analyzer

    The analyzer produces an interactive treemap of your bundles, exposing the heavy dependencies worth lazy-loading, replacing, or removing.

    npm install --save-dev @next/bundle-analyzer

    // next.config.ts
import type { NextConfig } from "next";
import bundleAnalyzer from "@next/bundle-analyzer";

const withBundleAnalyzer = bundleAnalyzer({
  enabled: process.env.ANALYZE === "true",
});

const nextConfig: NextConfig = {
  // ...your config
};

export default withBundleAnalyzer(nextConfig);

    # Build with analysis enabled; opens treemaps in your browser
ANALYZE=true npm run build

    What to look for

    A few common wins: a large date/utility library imported wholesale (switch to per-function imports or a lighter alternative), an editor/charting lib loaded eagerly (lazy-load it), duplicated dependencies across chunks, and big client components that could be Server Components. The treemap makes the offenders obvious.

    Core Web Vitals (including INP)

    The three metrics Google reports as Core Web Vitals:

    • LCP — Largest Contentful Paint (loading): time to render the largest above-the-fold element. Levers: next/image with priority on the LCP image, SSR/SSG so real content paints early, next/font to avoid font-swap delays.
    • INP — Interaction to Next Paint (responsiveness): replaced FID as a Core Web Vital in March 2024. It measures the latency of all interactions across the visit, capturing slow event handlers and long tasks—not just the first input. Levers: ship less client JS (Server Components), code-split with next/dynamic, and avoid expensive synchronous work in event handlers.
    • CLS — Cumulative Layout Shift (visual stability): how much the layout jumps. Levers: always size images/embeds (next/image reserves space), reserve space for ads/widgets, and use next/font to prevent font-driven reflow.

    Why INP rewards the App Router

    INP is dominated by main-thread work and hydration cost. Because Server Components send zero JavaScript for non-interactive UI, an App Router app naturally ships a smaller client bundle—directly improving INP. Pushing "use client" to the smallest leaves and lazy-loading heavy interactive widgets compounds the win.

    Measuring vitals in your app

    For lab data, use Lighthouse (Chrome DevTools) and next build's per-route bundle report. For real-user (field) data, report vitals from a Client Component with useReportWebVitals, then send them to your analytics endpoint:

    // app/_components/web-vitals.tsx
"use client";
import { useReportWebVitals } from "next/web-vitals";

export function WebVitals() {
  useReportWebVitals((metric) => {
    // metric.name is "LCP" | "INP" | "CLS" | "FCP" | "TTFB"
    navigator.sendBeacon("/api/vitals", JSON.stringify(metric));
  });
  return null;
}

    // app/layout.tsx (mount it once)
import { WebVitals } from "./_components/web-vitals";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <WebVitals />
        {children}
      </body>
    </html>
  );
}

    On Vercel, use Speed Insights

    If you deploy to Vercel, @vercel/speed-insights collects field Core Web Vitals (including INP) with one component—no custom endpoint needed. Either way, optimize against field data: lab scores don't always reflect what real users on real devices experience.

    📚 Summary

    🔹 Feature 🧠 What it does
    next/image On-demand resize, modern formats, lazy-load, no layout shift
    sizes Tells the browser the on-screen width so it fetches the right variant
    priority Preloads the LCP image (one per page) instead of lazy-loading it
    fill Image fills a positioned parent; pair with object-fit + sizes
    images.remotePatterns Allowlist of remote hosts the optimizer may fetch (replaces domains)
    metadata export Static, typed <head> tags merged down the route tree
    generateMetadata Async metadata from data (deduped via fetch/cache() memoization)
    metadataBase Base URL so relative OG/Twitter image paths resolve absolutely
    openGraph / twitter Social-share previews (1200×630 image, summary_large_image)

    opengraph-image Static or ImageResponse-generated per-route OG images
    next/script Load third-party JS with a strategy (afterInteractive, lazyOnload, …)
    next/dynamic Lazy-load (mainly Client) components; ssr: false for browser-only
    @next/bundle-analyzer Treemap of bundle contents to find heavy dependencies
    LCP / INP / CLS Core Web Vitals; INP replaced FID in 2024 and rewards shipping less JS
