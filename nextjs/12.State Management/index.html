    State Management in Next.js

    "State management" in a React SPA usually means one big question: which library holds all my data? In the App Router that question mostly dissolves. With React Server Components, the server already holds your data and renders it to HTML, so a huge category of "state" — the server's data — never needs to live in a client store at all. What remains is genuinely client-side: ephemeral UI, the URL, and a cache of server data you fetch from the client.

    This chapter sorts state into the few buckets that actually exist in an App Router app, then shows the right tool for each: the URL for shareable state, Context/Zustand for client-only UI state (with the per-request provider pattern that keeps server renders safe), and TanStack Query / SWR for caching server data on the client. It closes with the patterns for handing initial server data across the server/client boundary.

    📚 For more: https://nextjs.org/docs/app/building-your-application/rendering · https://tanstack.com/query/latest · https://zustand.docs.pmnd.rs

    1. Server state vs client state

    The first and most important move is to stop treating all "state" as one thing. In an App Router app there are really two species, and they want different tools.

    Server state is data that lives somewhere else — a database, a CMS, a third-party API — and your UI only borrows a copy. It's the post list, the current user's profile, the product catalog. Its source of truth is the server. Your job is to fetch it, display it, and keep the copy reasonably fresh.

    Client state is data that is born in the browser and only matters there: is this dropdown open, what's typed in this input, which tab is active, is the sidebar collapsed. It has no server source of truth.

    Server state                      Client state
-----------------------------     -----------------------------
source of truth = server          source of truth = browser
posts, user, catalog, settings    modal open?, form draft, tab
needs fetching + freshness        needs nothing — it just is

    Why RSC means less client state

    In a classic SPA, server state gets dragged into client state: you useEffect + fetch, drop the result into useState or Redux, then manage loading flags, error flags, and refetching by hand. That's the bulk of what state libraries were doing — managing a cache of server data.

    React Server Components remove most of that. A Server Component can await your data directly and render it to HTML. There's no loading flag, no client store, no hydration of that data into a reducer — it's just rendered output.

    // app/posts/page.tsx — a Server Component
// The data is "state" in the abstract sense, but it never enters a client store.
import { db } from "@/lib/db";

export default async function PostsPage() {
  const posts = await db.post.findMany({ take: 20 }); // server state, rendered server-side
  return (
    <ul>
      {posts.map((p) => (
        <li key={p.id}>{p.title}</li>
      ))}
    </ul>
  );
}

    So before reaching for any state tool, ask where the source of truth is:

    Question Use
    Is it server data displayed on first load? Server Component — await it, no store
    Should it be shareable / bookmarkable / survive refresh? The URL (searchParams)
    Is it data fetched from the client (after interaction, polling, infinite scroll)? TanStack Query / SWR
    Is it client-only UI state for one component? useState
    Is it client-only UI state shared across many components? Context (rarely changing) or Zustand (frequently changing)

    Reach for a global client store last, not first

    The most common App Router mistake migrating from an SPA is recreating a giant Redux/Zustand store full of server data. Most of that data should be fetched on the server (RSC) or cached by TanStack Query — both of which handle freshness for you. A global client store should hold only genuinely client-owned state.

    2. The URL as state

    A surprising amount of "state" should live in the URL query string, not in a JS variable: the current search term, active filters, sort order, page number, the open tab. Putting it in the URL gives you four things a useState can't: it's shareable (paste the link, get the same view), bookmarkable, survives a refresh, and works with the back/forward buttons.

    Reading searchParams

    On the server, a page receives searchParams as a prop (a Promise in Next.js 15). Because it's a prop the Server Component reads, changing the URL re-runs the server render with the new value — no client fetching required.

    // app/products/page.tsx — Server Component reads the URL
import { db } from "@/lib/db";

export default async function ProductsPage({
  searchParams,
}: {
  searchParams: Promise<{ q?: string; sort?: string }>;
}) {
  const { q = "", sort = "newest" } = await searchParams; // Next.js 15: async
  const products = await db.product.findMany({
    where: q ? { name: { contains: q } } : undefined,
    orderBy: sort === "price" ? { price: "asc" } : { createdAt: "desc" },
  });
  return <ProductGrid products={products} />;
}

    On the client, read the same values with useSearchParams:

    "use client";
import { useSearchParams } from "next/navigation";

export function SortLabel() {
  const params = useSearchParams();
  return <span>Sorted by: {params.get("sort") ?? "newest"}</span>;
}

    Writing to the URL

    A client component updates the URL by building a new query string and pushing it. Wrap URLSearchParams so you only change the keys you mean to:

    // app/products/SearchBox.tsx
"use client";
import { usePathname, useRouter, useSearchParams } from "next/navigation";
import { useCallback } from "react";

export function SearchBox() {
  const router = useRouter();
  const pathname = usePathname();
  const searchParams = useSearchParams();

  const setQuery = useCallback(
    (q: string) => {
      const params = new URLSearchParams(searchParams.toString());
      if (q) params.set("q", q);
      else params.delete("q");
      router.replace(`${pathname}?${params.toString()}`); // replace = no history spam
    },
    [router, pathname, searchParams],
  );

  return (
    <input
      defaultValue={searchParams.get("q") ?? ""}
      onChange={(e) => setQuery(e.target.value)}
      placeholder="Search products…"
    />
  );
}

    When the page above is a Server Component reading searchParams, router.replace triggers a fresh server render with the new query — the product list re-filters on the server.

    replace vs push for transient state

    Use router.replace for state that changes rapidly (a search box, a slider): it updates the URL without adding a history entry, so the back button doesn't have to click through every keystroke. Use router.push for navigations the user should be able to go back from (changing pages, applying a distinct filter set). Debounce rapid writes so you're not pushing on every keypress.

    The nuqs pattern: URL state with a useState-like API

    Hand-rolling URLSearchParams parsing, type coercion, and defaults for every filter gets tedious. nuqs is a small library that gives you a useState-style hook backed by the URL, with parsers for numbers, booleans, arrays, and more — and it works in both the App and Pages routers.

    npm install nuqs

    nuqs needs an adapter at the root so it knows how to talk to the router:

    // app/layout.tsx
import { NuqsAdapter } from "nuqs/adapters/next/app";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <NuqsAdapter>{children}</NuqsAdapter>
      </body>
    </html>
  );
}

    Now state reads and writes look just like useState, but the value lives in ?q=…&page=…:

    // app/products/Filters.tsx
"use client";
import { useQueryState, parseAsInteger, parseAsString } from "nuqs";

export function Filters() {
  const [q, setQ] = useQueryState("q", parseAsString.withDefault(""));
  const [page, setPage] = useQueryState(
    "page",
    parseAsInteger.withDefault(1),
  );

  return (
    <div>
      <input value={q} onChange={(e) => setQ(e.target.value || null)} />
      <button onClick={() => setPage((p) => p + 1)}>Next page (now {page})</button>
    </div>
  );
}

    nuqs handles serialization, parsing, defaults (a value equal to its default is removed from the URL to keep it clean), and batching multiple updates into one navigation. Setting a value to null removes the key.

    Shallow routing

    By default, changing searchParams in the App Router re-runs the matching Server Component (it's a navigation). Sometimes you want to change the URL without re-rendering on the server — for example, syncing a purely client-side tab to the URL for shareability while a TanStack Query result drives the actual content.

    nuqs exposes this directly with shallow (default true = client-only update):

    "use client";
import { useQueryState } from "nuqs";

export function Tabs() {
  // shallow: true (the default) updates the URL on the client only — no server round-trip
  const [tab, setTab] = useQueryState("tab", { defaultValue: "overview" });
  // …render based on `tab`, no Server Component re-render
}

export function ServerSyncedFilter() {
  // shallow: false forces a server round-trip so a Server Component re-reads searchParams
  const [q, setQ] = useQueryState("q", { shallow: false, defaultValue: "" });
}

    Shallow updates don't re-run the server

    With a shallow URL update, your Server Component does not re-render, so any server-side searchParams logic won't run. Choose shallow: false when a Server Component must react to the change; keep the default shallow: true when the URL is just a bookmark for client-driven UI.

    3. Client state: Context and Zustand in the App Router

    For state that genuinely belongs to the browser and is shared across components — a theme, a shopping-cart drawer, a multi-step wizard — you need a client store. The catch in the App Router is that providers and stores are client concepts, and the server renders your tree too. Getting the boundary right is the whole game.

    Context with the "use client" provider pattern

    React Context requires a component that calls createContext/useContext, which only works in Client Components. But your root layout is a Server Component, and you want it to stay that way. The pattern is to extract the provider into its own "use client" file and render it from the (server) layout, passing children through.

    // app/theme-provider.tsx
"use client";
import { createContext, useContext, useState } from "react";

type Theme = "light" | "dark";
const ThemeContext = createContext<{
  theme: Theme;
  toggle: () => void;
} | null>(null);

export function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<Theme>("light");
  const toggle = () => setTheme((t) => (t === "light" ? "dark" : "light"));
  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error("useTheme must be used within ThemeProvider");
  return ctx;
}

    // app/layout.tsx — stays a Server Component
import { ThemeProvider } from "./theme-provider";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        {/* children passed through the client provider stay Server Components */}
        <ThemeProvider>{children}</ThemeProvider>
      </body>
    </html>
  );
}

    The key insight: {children} passed into a Client Component are still rendered on the server. Wrapping your tree in a client provider does not turn the whole tree into client code — only the provider component itself is a Client Component.

    Context is great for low-frequency shared state (theme, locale, the current user object). For state that updates often, every consumer of the context re-renders on every change, which gets expensive — that's where Zustand shines.

    Zustand v5 — the per-request store

    Zustand is a tiny, hook-based store with no provider required in a pure SPA. You'd normally create the store once at module scope and import it anywhere:

    // ⚠️ This SPA-style pattern is UNSAFE on the server — see the warning below
import { create } from "zustand";

export const useCounterStore = create<{ count: number; inc: () => void }>(
  (set) => ({ count: 0, inc: () => set((s) => s.count + 1) }),
);

    Never use a module-level store on the server

    A module-level store is created once per server process and shared across every request. On the server that means one user's state can leak into another user's render — the same store instance is reused for concurrent requests. It also can't be initialized with per-request data (like the logged-in user). In the App Router you must create the store per request, behind a provider, so each render gets its own isolated instance.

    The fix is to create the store inside a "use client" provider, hold the instance in a useRef (so it's created exactly once per component instance), and read it through Context. Define the store as a factory, not a singleton:

    // app/stores/counter-store.ts
import { createStore } from "zustand/vanilla";

export type CounterState = { count: number };
export type CounterActions = {
  increment: () => void;
  setCount: (count: number) => void;
};
export type CounterStore = CounterState & CounterActions;

// A factory — call it to get a fresh store. `initState` lets the server seed it.
export const createCounterStore = (
  initState: CounterState = { count: 0 },
) =>
  createStore<CounterStore>()((set) => ({
    ...initState,
    increment: () => set((state) => ({ count: state.count + 1 })),
    setCount: (count) => set({ count }),
  }));

    // app/stores/counter-store-provider.tsx
"use client";
import { createContext, useContext, useRef, type ReactNode } from "react";
import { useStore } from "zustand";
import {
  createCounterStore,
  type CounterStore,
  type CounterState,
} from "./counter-store";

type CounterStoreApi = ReturnType<typeof createCounterStore>;
const CounterStoreContext = createContext<CounterStoreApi | null>(null);

export function CounterStoreProvider({
  children,
  initialState,
}: {
  children: ReactNode;
  initialState?: CounterState;
}) {
  // useRef ensures the store is created exactly ONCE per component instance,
  // and a new instance per request on the server.
  const storeRef = useRef<CounterStoreApi | null>(null);
  if (storeRef.current === null) {
    storeRef.current = createCounterStore(initialState);
  }
  return (
    <CounterStoreContext.Provider value={storeRef.current}>
      {children}
    </CounterStoreContext.Provider>
  );
}

// Selector hook — `selector` keeps re-renders scoped to the slice you use.
export function useCounterStore<T>(selector: (store: CounterStore) => T): T {
  const store = useContext(CounterStoreContext);
  if (!store) {
    throw new Error("useCounterStore must be used within CounterStoreProvider");
  }
  return useStore(store, selector);
}

    Wrap the relevant part of the tree — and seed it with server data if you have it:

    // app/dashboard/layout.tsx — a Server Component
import { CounterStoreProvider } from "@/app/stores/counter-store-provider";
import { getInitialCount } from "@/lib/data";

export default async function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  const count = await getInitialCount(); // per-request server data
  return (
    <CounterStoreProvider initialState={{ count }}>
      {children}
    </CounterStoreProvider>
  );
}

    Selecting state with useShallow

    The hook above takes a selector, so a component only re-renders when its slice changes. But if a selector returns a new object or array each call (e.g. picking multiple fields), the default referential (Object.is) equality sees a new reference every time and re-renders on every store change. Zustand v5's useShallow compares the selected object shallowly to prevent that:

    "use client";
import { useShallow } from "zustand/react/shallow";
import { useCounterStore } from "@/app/stores/counter-store-provider";

export function CounterControls() {
  // Selecting multiple values returns a NEW object each render.
  // useShallow does a shallow compare so this only re-renders when count/increment change.
  const { count, increment } = useCounterStore(
    useShallow((s) => ({ count: s.count, increment: s.increment })),
  );

  return <button onClick={increment}>Count: {count}</button>;
}

    Selector discipline keeps Zustand fast

    Select the smallest slice each component needs. A single-value selector ((s) => s.count) needs no useShallow because it returns a primitive. Reach for useShallow only when a selector returns a fresh object/array. Selecting the whole store (useCounterStore((s) => s)) re-renders on every change — avoid it.

    4. Caching server data on the client: TanStack Query & SWR

    RSC handles data on first render. But some data is fetched from the client: results that update after a user interaction, infinite scroll, polling/live data, optimistic mutations, or any page where re-running a Server Component would be too coarse. For that, you want a client-side cache of server state — and that's exactly what TanStack Query and SWR are. They give you caching, deduplication, background refetching, stale-while-revalidate, and request states (loading/error/data) so you stop hand-rolling useEffect + useState.

    Provider setup (TanStack Query v5)

    QueryClientProvider is a Client Component, so it follows the same "use client" provider pattern. Critically, you must create the QueryClient so that it isn't shared across requests on the server but is stable across re-renders on the client:

    // app/get-query-client.ts
import {
  isServer,
  QueryClient,
  defaultShouldDehydrateQuery,
} from "@tanstack/react-query";

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        // With SSR, set staleTime > 0 so data isn't refetched on the client immediately
        staleTime: 60 * 1000,
        // gcTime (v5; renamed from cacheTime) — how long unused cache is kept before GC
        gcTime: 5 * 60 * 1000,
      },
      dehydrate: {
        // Also dehydrate pending queries so streamed/prefetched data hydrates correctly
        shouldDehydrateQuery: (query) =>
          defaultShouldDehydrateQuery(query) ||
          query.state.status === "pending",
      },
    },
  });
}

let browserQueryClient: QueryClient | undefined;

export function getQueryClient() {
  if (isServer) {
    // Server: ALWAYS a fresh client per request (no cross-request leakage)
    return makeQueryClient();
  }
  // Browser: reuse a single client so React state survives Suspense/re-renders
  if (!browserQueryClient) browserQueryClient = makeQueryClient();
  return browserQueryClient;
}

    // app/providers.tsx
"use client";
import { QueryClientProvider } from "@tanstack/react-query";
import { getQueryClient } from "./get-query-client";

export function Providers({ children }: { children: React.ReactNode }) {
  // NB: call getQueryClient() during render, NOT in useState's initializer with a new client,
  // and do not wrap in useMemo — getQueryClient handles the singleton-per-environment logic.
  const queryClient = getQueryClient();
  return (
    <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
  );
}

    // app/layout.tsx
import { Providers } from "./providers";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  );
}

    One QueryClient per server request, one per browser tab

    On the server, a module-level QueryClient would be shared by every concurrent request — the same data-leak hazard as a module-level Zustand store. getQueryClient returns a fresh client on the server every call but memoizes a single client in the browser (where one tab = one user). Never put new QueryClient() at module scope.

    Using a query

    // app/posts/PostList.tsx
"use client";
import { useQuery } from "@tanstack/react-query";

async function fetchPosts() {
  const res = await fetch("/api/posts");
  if (!res.ok) throw new Error("Failed to load posts");
  return res.json() as Promise<{ id: string; title: string }[]>;
}

export function PostList() {
  const { data, isPending, isError } = useQuery({
    queryKey: ["posts"],
    queryFn: fetchPosts,
  });

  if (isPending) return <p>Loading…</p>;
  if (isError) return <p>Something went wrong.</p>;
  return (
    <ul>
      {data.map((p) => (
        <li key={p.id}>{p.title}</li>
      ))}
    </ul>
  );
}

    Prefetch on the server, hydrate on the client (HydrationBoundary)

    The best of both worlds: fetch the initial data on the server (no client loading spinner, SEO-friendly) and hand it to TanStack Query so the same useQuery finds it already cached. You prefetch into a server-side QueryClient, dehydrate it, and wrap the client subtree in HydrationBoundary.

    // app/posts/page.tsx — Server Component
import {
  dehydrate,
  HydrationBoundary,
} from "@tanstack/react-query";
import { getQueryClient } from "@/app/get-query-client";
import { getPosts } from "@/lib/data";
import { PostList } from "./PostList";

export default async function PostsPage() {
  const queryClient = getQueryClient();

  // Prefetch on the server using the SAME queryKey the client uses.
  // (Call the data function directly — no need to HTTP-fetch your own API.)
  await queryClient.prefetchQuery({
    queryKey: ["posts"],
    queryFn: getPosts,
  });

  return (
    // dehydrate() serializes the cache; HydrationBoundary rehydrates it on the client
    <HydrationBoundary state={dehydrate(queryClient)}>
      <PostList />
    </HydrationBoundary>
  );
}

    The client PostList uses useQuery({ queryKey: ["posts"], queryFn: fetchPosts }). Because the cache was hydrated with that exact queryKey, it renders the data immediately — no loading flash — and only refetches in the background once the data goes stale (per staleTime). The query keys must match exactly for hydration to connect.

    staleTime is what prevents a double fetch

    If staleTime is 0 (the default), hydrated data is considered stale instantly and TanStack Query refetches it on mount — you prefetched on the server and fetched again on the client. Set a staleTime (e.g. 30–60s) so the freshly-prefetched data is trusted for a while. This is the single most common SSR-hydration mistake.

    SWR, briefly

    SWR (also from Vercel) is the lighter-weight alternative with the same stale-while-revalidate philosophy. Setup mirrors TanStack Query — a client provider and a hook — and you seed server data via fallback (SWR's equivalent of hydration):

    // app/posts/Posts.tsx
"use client";
import useSWR, { SWRConfig } from "swr";

const fetcher = (url: string) => fetch(url).then((r) => r.json());

function PostList() {
  const { data, isLoading } = useSWR("/api/posts", fetcher);
  if (isLoading) return <p>Loading…</p>;
  return <ul>{data.map((p: any) => <li key={p.id}>{p.title}</li>)}</ul>;
}

// A Server Component renders <SWRConfig value={{ fallback: { "/api/posts": posts } }}>
// to seed the cache, and PostList reads it without a client fetch.
export function Posts({ fallback }: { fallback: Record<string, unknown> }) {
  return (
    <SWRConfig value={{ fallback }}>
      <PostList />
    </SWRConfig>
  );
}

    Do you even need a client cache?

    For data shown on first load that rarely changes, plain RSC (await in a Server Component) is simpler and ships less JS. Reach for TanStack Query / SWR when the client drives fetching: pagination, infinite scroll, polling, dependent queries, optimistic updates, or shared cached data across many components that mutate frequently.

    5. Crossing the server/client boundary

    The recurring theme above — seeding a store, hydrating a query cache — is one pattern: fetch on the server, pass the initial value to a Client Component as a prop. Worth stating its rules directly.

    Pass initial data as a serializable prop

    A Server Component fetches and passes the result down. The Client Component takes it as initialData and owns it from there.

    // app/cart/page.tsx — Server Component
import { getCart } from "@/lib/data";
import { CartWidget } from "./CartWidget";

export default async function CartPage() {
  const cart = await getCart(); // server fetch
  return <CartWidget initialItems={cart.items} />; // hand off across the boundary
}

    // app/cart/CartWidget.tsx
"use client";
import { useState } from "react";

type Item = { id: string; name: string };

export function CartWidget({ initialItems }: { initialItems: Item[] }) {
  // Seed client state from the server once; client owns it afterward.
  const [items, setItems] = useState(initialItems);
  // …add/remove interactively without a round-trip
  return <div>{items.length} items</div>;
}

    Only serializable props cross the boundary

    Props passed from a Server to a Client Component must be serializable: plain objects, arrays, strings, numbers, booleans, null, and (with RSC's superset of JSON) Date, Map, Set, and Promises. You cannot pass functions, class instances, or DB connections. If you need behavior on the client, define it in client code; pass only the data.

    Choosing the hand-off mechanism

    The three sections above are really three flavors of the same boundary crossing — pick by what owns the data afterward:

    Initial server data lands in… Mechanism When
    Local component state initialItems prop → useState(initialItems) One component owns it; no refetching
    A Zustand store <Provider initialState={…}> → useRef factory Shared client state seeded from the server
    A query cache prefetchQuery → dehydrate → <HydrationBoundary> Server data the client will refetch/keep fresh
    The URL render with ?key=value already set Shareable, bookmarkable, refresh-proof

    Don't double-fetch across the boundary

    The classic mistake is fetching in a Server Component to render initial HTML and fetching the same thing again in a Client Component on mount. Either render it server-side and pass it as a prop, or hydrate the client cache (so the client treats the server's copy as fresh via staleTime/fallback). Pick one path for each piece of data.

    📚 Summary

    🔹 Concept 🧠 Description
    Server state Data owned by the server (DB/CMS/API); prefer fetching it in a Server Component
    Client state Browser-born UI state (open/closed, drafts, tabs); needs a client tool
    RSC → less client state Server Components render data directly, so most "state" never enters a client store
    URL as state searchParams for shareable/bookmarkable/refresh-proof state
    nuqs useState-like hooks backed by the URL, with parsers and defaults
    Shallow routing shallow: true updates the URL without re-running the Server Component
    Context provider pattern Put providers in a "use client" file; render from the (server) layout
    Zustand per-request store A factory + useRef provider — never a module-level singleton on the server
    useShallow Shallow-compares multi-field selectors to avoid extra re-renders (Zustand v5)
    TanStack Query / SWR Client-side cache of server state for client-driven fetching
    getQueryClient Fresh QueryClient per server request, one memoized client in the browser
    HydrationBoundary Prefetch on the server, dehydrate, hydrate the client cache (set staleTime!)
    Boundary hand-off Fetch on the server, pass serializable initial data to a Client Component
