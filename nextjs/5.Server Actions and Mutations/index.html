    Server Actions & Mutations

    Reading data in the App Router is just await inside a Server Component. Writing data—creating a post, updating a profile, deleting a row—is the job of Server Actions: async functions that are guaranteed to run on the server, callable directly from your components (including straight from a <form>) without you hand-writing an API route, a fetch, or any request plumbing.

    This chapter covers what Server Actions are and how the "use server" directive works, wiring them to forms with progressive enhancement, managing form state and UX with React 19's useActionState / useFormStatus / useOptimistic, validating input with Zod and returning typed errors, finishing a mutation with revalidation and redirect, the security rules that make actions safe, and calling actions outside of forms from Client Components.

    📚 For more: https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations · https://react.dev/reference/rsc/server-actions

    1. What are Server Actions?

    A Server Action is an async function marked with the "use server" directive. Next.js compiles it into a secure RPC endpoint: when a Client Component "calls" the action, React serializes the arguments, POSTs them to the server, runs the real function there, and serializes the result back. Your client code never sees the function body, the database client, or any secret it touches—only the call.

    Three things make them distinct from a Route Handler:

    • No endpoint to write or name — there's no URL, no route.ts, no fetch. You import the function and call it (or hand it to a form's action prop).
    • They work without JavaScript — when used as a form action, the form submits over plain HTTP and the action still runs (see progressive enhancement below).
    • They compose with the cache — an action can call revalidatePath / revalidateTag and redirect to update the UI after a write, all in one round trip.

    The "use server" directive

    There are two ways to mark code as a Server Action. File level — put "use server" at the very top of a module, and every export becomes a Server Action:

    // app/actions.ts
"use server";

import { db } from "@/lib/db";

export async function createPost(title: string, body: string) {
  return db.post.create({ data: { title, body } });
}

export async function deletePost(id: string) {
  await db.post.delete({ where: { id } });
}

    Inline — put "use server" as the first statement inside an async function, which is handy for small actions defined right in a Server Component:

    // app/page.tsx (a Server Component)
export default function Page() {
  async function create(formData: FormData) {
    "use server";
    await db.post.create({ data: { title: String(formData.get("title")) } });
  }

  return (
    <form action={create}>
      <input name="title" />
      <button type="submit">Create</button>
    </form>
  );
}

    \"use server\" is not \"use client\"

    They are unrelated despite the rhyme. "use client" marks a component boundary that ships to the browser. "use server" marks a function as a server-only RPC endpoint that is exposed to the network. Putting "use server" at the top of a Client Component file is a mistake—a Client Component imports actions from a separate "use server" module instead.

    A typical organization

    A common, scalable pattern is to keep actions in dedicated files (often colocated near the feature, or under a lib//actions/ folder) so both Server and Client Components can import them:

    // app/posts/actions.ts
"use server";

import { db } from "@/lib/db";
import { revalidatePath } from "next/cache";

export async function toggleLike(postId: string) {
  // ...mutation...
  revalidatePath(`/posts/${postId}`);
}

    2. Forms & progressive enhancement

    The headline use of Server Actions is the <form action={...}> prop. Pass an action directly, and Next.js wires the form to it. The action receives a single FormData argument:

    // app/posts/new/page.tsx — a Server Component
import { db } from "@/lib/db";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";

export default function NewPostPage() {
  async function createPost(formData: FormData) {
    "use server";
    const title = String(formData.get("title") ?? "");
    const body = String(formData.get("body") ?? "");

    const post = await db.post.create({ data: { title, body } });

    revalidatePath("/posts");        // refresh the list
    redirect(`/posts/${post.id}`);   // go to the new post
  }

  return (
    <form action={createPost}>
      <input name="title" placeholder="Title" required />
      <textarea name="body" placeholder="Write something…" />
      <button type="submit">Publish</button>
    </form>
  );
}

    Note the inputs use the native name attribute—that's how values land in FormData. No useState, no onChange, no controlled inputs required.

    Progressive enhancement

    Because the form's action is a real server endpoint, the form works before (or without) JavaScript. If the user submits while the page's JS is still downloading—or if JS is disabled entirely—the browser performs a normal HTML form POST and the action still runs. Once React hydrates, the same form upgrades to a client-side submission (no full-page reload, with pending state and optimistic UI available). You get the resilience of classic server-rendered forms and the polish of an SPA from the same code.

    Passing extra arguments with .bind

    Forms only carry their fields. To pass additional data (an id, a context value) to an action, use .bind—the bound arguments are sent securely as part of the action call, not as hidden form fields the client can tamper with as easily:

    // Bind the post id; formData is appended as the last argument
import { updatePost } from "./actions";

export function EditForm({ id }: { id: string }) {
  const updatePostWithId = updatePost.bind(null, id);
  return (
    <form action={updatePostWithId}>
      <input name="title" />
      <button type="submit">Save</button>
    </form>
  );
}

    // app/posts/actions.ts
"use server";
export async function updatePost(id: string, formData: FormData) {
  // id came from .bind, formData from the form
}

    Server vs client form handling

    With Server Actions you reach for name + FormData and let the platform submit the form. This is the opposite of the controlled-input + fetch pattern you'd use against a Route Handler. Both are valid—Server Actions are usually less code and give you progressive enhancement for free.

    3. Form state & UX

    Raw <form action> works, but real apps need to show errors, disable the button while submitting, and feel instant. React 19 (which ships with Next.js 15) provides three hooks for exactly this.

    useActionState — state + the action in one

    useActionState wraps an action so it can return a value (validation errors, a success message) that React stores as state and re-renders with. It's the backbone of form UX.

    React 19 renamed useFormState → useActionState

    In React 18 / Next.js 14 this hook lived in react-dom and was called useFormState. In React 19 it moved to react and was renamed useActionState, gaining a third element in its return tuple: the pending flag. Use useActionState for all new code; useFormState is deprecated.

    The action's signature changes: it now receives the previous state as its first argument and FormData as its second.

    // app/posts/new-form.tsx
"use client";

import { useActionState } from "react";
import { createPost, type FormState } from "./actions";

const initialState: FormState = { ok: false, errors: {} };

export function NewPostForm() {
  const [state, formAction, isPending] = useActionState(createPost, initialState);

  return (
    <form action={formAction}>
      <input name="title" aria-describedby="title-error" />
      {state.errors?.title && (
        <p id="title-error" role="alert">
          {state.errors.title}
        </p>
      )}

      <textarea name="body" />
      {state.errors?.body && <p role="alert">{state.errors.body}</p>}

      <button type="submit" disabled={isPending}>
        {isPending ? "Publishing…" : "Publish"}
      </button>

      {state.ok && <p>Saved!</p>}
    </form>
  );
}

    The matching action receives prevState then formData and returns the next state:

    // app/posts/actions.ts
"use server";

export type FormState = {
  ok: boolean;
  errors?: { title?: string; body?: string };
};

export async function createPost(
  _prevState: FormState,
  formData: FormData
): Promise<FormState> {
  const title = String(formData.get("title") ?? "");
  if (!title) return { ok: false, errors: { title: "Title is required" } };

  await db.post.create({ data: { title } });
  return { ok: true };
}

    useFormStatus — pending state for a submit button

    useFormStatus (from react-dom) reads the status of the nearest parent <form>. Its main use is a reusable submit button that disables and relabels itself while the form is in flight—without the parent needing to pass a prop down.

    // app/components/submit-button.tsx
"use client";

import { useFormStatus } from "react-dom";

export function SubmitButton({ children }: { children: React.ReactNode }) {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? "Submitting…" : children}
    </button>
  );
}

    useFormStatus must live inside the form

    The hook reads context from the surrounding <form>, so the component that calls it has to be rendered as a child of that form, not the component that renders the form. If you already have isPending from useActionState, prefer that for the same form; reach for useFormStatus when you want a self-contained button component that any form can drop in.

    useOptimistic — instant UI before the server responds

    useOptimistic lets you show the expected result immediately, then reconcile with the real server state when the action finishes (or roll back automatically if it fails). Perfect for likes, todos, and chat messages.

    // app/todos/todo-list.tsx
"use client";

import { useOptimistic } from "react";
import { addTodo } from "./actions";

type Todo = { id: string; text: string; pending?: boolean };

export function TodoList({ todos }: { todos: Todo[] }) {
  const [optimisticTodos, addOptimistic] = useOptimistic(
    todos,
    (current, newText: string) => [
      ...current,
      { id: crypto.randomUUID(), text: newText, pending: true },
    ]
  );

  return (
    <>
      <ul>
        {optimisticTodos.map((t) => (
          <li key={t.id} style={{ opacity: t.pending ? 0.5 : 1 }}>
            {t.text}
          </li>
        ))}
      </ul>

      <form
        action={async (formData: FormData) => {
          const text = String(formData.get("text") ?? "");
          addOptimistic(text);     // update UI instantly
          await addTodo(text);     // then run the real mutation
        }}
      >
        <input name="text" />
        <SubmitButton>Add</SubmitButton>
      </form>
    </>
  );
}

    When addTodo resolves and the page revalidates, optimisticTodos is replaced by the authoritative todos prop. If the action throws, React discards the optimistic entry automatically.

    4. Validation & typed errors

    Never trust the client. FormData is just strings the browser sent—anyone can craft a request that bypasses your required attributes and HTML5 constraints. Validate every input inside the action with a schema library. Zod pairs especially well with useActionState because it produces structured, field-level errors you can return as typed state.

    // app/posts/actions.ts
"use server";

import { z } from "zod";
import { db } from "@/lib/db";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";

const PostSchema = z.object({
  title: z.string().min(1, "Title is required").max(120, "Title is too long"),
  body: z.string().min(10, "Body must be at least 10 characters"),
});

// The shape our form state takes — strongly typed for the client to consume
export type FormState = {
  ok: boolean;
  message?: string;
  // Per-field error arrays, keyed by the schema's field names
  errors?: Partial<Record<keyof z.infer<typeof PostSchema>, string[]>>;
};

export async function createPost(
  _prevState: FormState,
  formData: FormData
): Promise<FormState> {
  const parsed = PostSchema.safeParse({
    title: formData.get("title"),
    body: formData.get("body"),
  });

  if (!parsed.success) {
    // flatten() gives you { fieldErrors: { title: [...], body: [...] } }
    return {
      ok: false,
      message: "Please fix the errors below.",
      errors: parsed.error.flatten().fieldErrors,
    };
  }

  await db.post.create({ data: parsed.data }); // parsed.data is fully typed

  revalidatePath("/posts");
  redirect("/posts");
}

    On the client, the state.errors object is typed, so reading state.errors?.title is checked at compile time:

    "use client";
import { useActionState } from "react";
import { createPost, type FormState } from "./actions";

const initial: FormState = { ok: false };

export function PostForm() {
  const [state, action, pending] = useActionState(createPost, initial);

  return (
    <form action={action}>
      <input name="title" />
      {state.errors?.title?.map((e) => (
        <p key={e} role="alert">{e}</p>
      ))}

      <textarea name="body" />
      {state.errors?.body?.map((e) => (
        <p key={e} role="alert">{e}</p>
      ))}

      {state.message && <p>{state.message}</p>}
      <button disabled={pending}>Save</button>
    </form>
  );
}

    Validate once, on the server

    Client-side validation (required, type="email", a Zod check before submit) is a UX nicety—it gives faster feedback. The server-side validation inside the action is the one that actually protects your data. If you only validate one place, validate on the server. safeParse keeps you in control of the error shape; parse throws, which surfaces as an unhandled action error instead of friendly field messages.

    5. After a mutation: revalidation & redirect

    A write usually invalidates something you've already cached or rendered. Next.js gives you two cache-busting functions and a navigation helper, all callable from inside an action.

    revalidatePath and revalidateTag

    After mutating, tell Next.js which cached data is now stale so the next render fetches fresh data:

    "use server";
import { revalidatePath, revalidateTag } from "next/cache";

export async function publishPost(id: string) {
  await db.post.update({ where: { id }, data: { published: true } });

  revalidatePath("/posts");          // re-render the /posts route
  revalidatePath(`/posts/${id}`);    // and this specific post's page
  revalidateTag("posts");            // and every fetch tagged "posts"
}
    • revalidatePath(path) clears the cache for a specific route (or a layout/page). Use it when you know exactly which URL displays the changed data.
    • revalidateTag(tag) clears every cached fetch that was tagged with that string (fetch(url, { next: { tags: ["posts"] } })). Use it when the same data appears across many routes.

    Without revalidation, the user would submit successfully but keep seeing the old, cached UI.

    redirect after a write

    To send the user somewhere after a successful mutation, call redirect from next/navigation:

    "use server";
import { redirect } from "next/navigation";

export async function createPost(formData: FormData) {
  const post = await db.post.create({ data: { title: String(formData.get("title")) } });
  revalidatePath("/posts");
  redirect(`/posts/${post.id}`); // must come AFTER revalidation
}

    redirect throws on purpose

    redirect() works by throwing a special internal signal that Next.js catches, so code after it never runs, and you should not call it inside a try/catch that swallows errors (it would catch the redirect signal and break navigation). Call redirect after your mutation and revalidation, at the end of the action. If you catch errors around a DB call, put redirect outside that try/catch. The same applies to notFound().

    Returning errors instead of throwing

    When you're using useActionState, you typically return an error state rather than redirect-ing, so the form can display field messages and stay on the page. Reserve redirect for the success path (or for navigation that should happen regardless, like sending an unauthenticated user to /login).

    6. Security

    This is the most important section. Because a Server Action compiles to a publicly reachable POST endpoint, you must treat it with the same suspicion as any API route. The convenience of "just call a function" can hide the fact that anyone on the internet can invoke it with any arguments.

    Server Actions are public endpoints — secure every one

    A "use server" function is exposed to the network. An attacker can call it directly with forged arguments, skipping your form, your .bind values, and your client-side checks entirely. Every action must authenticate and authorize on its own. Never assume an action can only be reached "from the right page" or "by the right user"—there is no such guarantee.

    Always authenticate and authorize inside the action

    Do the auth check at the top of the action body, not in a wrapper component or middleware you hope ran:

    // app/posts/actions.ts
"use server";

import { auth } from "@/lib/auth";
import { db } from "@/lib/db";
import { z } from "zod";
import { redirect } from "next/navigation";

const UpdateSchema = z.object({
  id: z.string().uuid(),
  title: z.string().min(1).max(120),
});

export async function updatePost(_prev: unknown, formData: FormData) {
  // 1. AUTHENTICATE — who is calling?
  const session = await auth();
  if (!session?.user) redirect("/login");

  // 2. VALIDATE — never trust the payload
  const parsed = UpdateSchema.safeParse({
    id: formData.get("id"),
    title: formData.get("title"),
  });
  if (!parsed.success) {
    return { ok: false, errors: parsed.error.flatten().fieldErrors };
  }

  // 3. AUTHORIZE — is THIS user allowed to touch THIS resource?
  const post = await db.post.findUnique({ where: { id: parsed.data.id } });
  if (!post) return { ok: false, message: "Not found" };
  if (post.authorId !== session.user.id) {
    return { ok: false, message: "Forbidden" }; // ownership check
  }

  // 4. Only now perform the mutation
  await db.post.update({
    where: { id: parsed.data.id },
    data: { title: parsed.data.title },
  });

  return { ok: true };
}

    Never trust client input — including hidden fields and .bind args

    Treat every value from FormData (and even arguments passed via .bind) as attacker-controlled. A hidden <input name="role" value="admin"> or a bound userId can be swapped by anyone replaying the request. Re-derive identity from the server-side session, validate the shape with a schema, and check ownership/permissions against the database. Authorization based on the resource's actual owner—not a value the client sent—is what stops one user from editing another's data.

    Other hardening notes

    • Don't leak internals in error messages. Return generic messages to the client; log the real error server-side.
    • Rate-limit sensitive actions (login, password reset, anything that sends email or money) the same way you would an API route.
    • CSRF — Server Actions only accept POST and Next.js enforces same-origin checks (the Origin/Host comparison, configurable via serverActions.allowedOrigins), which mitigates CSRF for the common case—but this does not replace your own auth checks.
    • Use a Data Access Layer. Centralize auth + ownership checks in functions your actions call, so the rules live in one audited place rather than copy-pasted into every action.
    • Dead-code elimination is not security. An unused/unreferenced action can still be present in the deployed bundle's action manifest; assume any "use server" export is reachable.

    7. Calling actions from Client Components

    Forms are the common case, but Server Actions are just functions—you can call them from event handlers, effects, or anywhere in a Client Component. The catch: a call that updates the UI should be wrapped so React knows it's a transition.

    Non-form usage with startTransition

    Outside a <form action>, wrap the action call in startTransition. This keeps the UI responsive (the pending state is tracked, the page stays interactive) and is the supported way to invoke an action imperatively:

    // app/posts/like-button.tsx
"use client";

import { useTransition } from "react";
import { likePost } from "./actions";

export function LikeButton({ postId }: { postId: string }) {
  const [isPending, startTransition] = useTransition();

  return (
    <button
      disabled={isPending}
      onClick={() =>
        startTransition(async () => {
          await likePost(postId); // a Server Action, called like a normal function
        })
      }
    >
      {isPending ? "…" : "♥ Like"}
    </button>
  );
}

    // app/posts/actions.ts
"use server";
import { auth } from "@/lib/auth";
import { revalidatePath } from "next/cache";

export async function likePost(postId: string) {
  const session = await auth();
  if (!session?.user) throw new Error("Unauthorized"); // still authenticate!
  await db.like.create({ data: { postId, userId: session.user.id } });
  revalidatePath(`/posts/${postId}`);
}

    Importing actions into Client Components

    A Client Component cannot define a "use server" action (its file is "use client"), but it can import one from a separate "use server" module and call it. This is the standard pattern: actions live in actions.ts, and any component—server or client—imports them.

    "use client";
import { deletePost } from "./actions"; // imported from a "use server" file

export function DeleteButton({ id }: { id: string }) {
  return <button onClick={() => deletePost(id)}>Delete</button>;
}

    Reading the return value with useActionState vs startTransition

    If you call an action imperatively and need its returned value (e.g. an error to display), useActionState is still the cleanest path—pass its formAction to a <form> or invoke it inside startTransition. For fire-and-forget mutations that just revalidate, a plain startTransition call is enough. Either way, don't await an action directly in render or an effect without a transition—you'll lose pending tracking and risk waterfalls.

    🔹 Concept 🧠 Description
    Server Action async function marked "use server"; compiles to a secure RPC endpoint
    "use server" (file) Top of a module — every export becomes an action
    "use server" (inline) First line in an async function — that function becomes an action

    Progressive enhancement Form submits over plain HTTP pre-hydration, upgrades to client submit after
    .bind(null, arg) Pass extra arguments to an action securely (not as form fields)
    useActionState React 19 hook: action returns state (errors/success) + pending flag
    useFormStatus react-dom hook read inside a form for a self-contained submit button
    useOptimistic Show the expected result instantly, reconcile/rollback on completion
    Zod + safeParse Validate FormData in the action; return typed fieldErrors
    revalidatePath Bust the cache for a specific route after a write
    revalidateTag Bust every fetch tagged with a string after a write
    redirect Navigate after a mutation; throws (keep it out of try/catch)
    Auth in the action Authenticate + authorize inside every action — they're public endpoints
    Never trust input Re-derive identity from the session; validate; check ownership in the DB
    startTransition Call an action imperatively from a Client Component, keeping pending state
