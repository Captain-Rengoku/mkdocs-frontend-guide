    Database Integration

    A real app eventually needs a database. The great thing about the App Router is that there's no special data layer to learn first: a Server Component can await a query directly, and a Server Action can write to the database from a form—no hand-written API endpoint required. The hard parts aren't the queries; they're the things that don't show up in a tutorial: keeping database code off the client, centralizing auth checks so you can't forget one, and not exhausting your connection pool the first time traffic spikes on a serverless host.

    This chapter uses Prisma as the primary ORM (with a quick look at Drizzle), on Next.js 15 / TypeScript / PostgreSQL. Everything is App Router-first and the code is meant to run.

    📚 For more: Prisma + Next.js · Next.js — Data Security

    1. Choosing an ORM

    You can talk to a database with the raw driver (pg, postgres, mysql2), but for most apps an ORM/query builder gives you type-safe queries, schema management, and migrations. In the Next.js world the two dominant choices are Prisma and Drizzle.

    Prisma vs Drizzle

    Prisma Drizzle
    Model Schema-first: you write schema.prisma, it generates a typed client Code-first: you define tables in TypeScript
    Query style High-level, fluent (db.user.findMany) SQL-like, close to the metal (db.select().from(users))
    Type safety Generated client types Inferred straight from your TS schema
    Migrations prisma migrate (mature, robust) drizzle-kit (lightweight)
    Bundle / runtime Heavier; query engine (Rust) or pure-JS via driver adapters Very thin; just a query builder over a driver
    Edge runtime Works via driver adapters First-class, very edge-friendly
    Learning curve Gentle; great DX and docs Steeper if you don't know SQL, but transparent

    Rule of thumb:

    • Reach for Prisma when you want the smoothest DX, a clear schema file, and powerful migrations—the default "just works" choice for most product teams.
    • Reach for Drizzle when you want minimal overhead, SQL-level control, the lightest possible Edge footprint, or you simply prefer writing queries that look like SQL.

    This chapter goes deep on Prisma because it's the most common starting point, but the patterns—singleton client, data-access layer, server-only, connection pooling—apply to Drizzle and even the raw driver too.

    The patterns matter more than the tool

    Whatever you pick, the architectural advice in sections 6–8 (centralize access, keep it server-only, pool connections) is what keeps a Next.js app maintainable and production-safe. The ORM is an implementation detail you could swap.

    2. Setting up Prisma

    Install

    # Prisma CLI (dev) + the generated client (runtime)
npm install prisma --save-dev
npm install @prisma/client

# Scaffold prisma/schema.prisma and a .env with DATABASE_URL
npx prisma init --datasource-provider postgresql

    prisma init creates:

    • prisma/schema.prisma — your data model and generator/datasource config.
    • .env with a DATABASE_URL placeholder. Keep this server-only; do not prefix it with NEXT_PUBLIC_.

    Define schema.prisma

    The schema is the single source of truth. From it Prisma generates a fully-typed client.

    // prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  posts     Post[]
  createdAt DateTime @default(now())
}

model Post {
  id        String   @id @default(cuid())
  title     String
  content   String?
  published Boolean  @default(false)
  author    User     @relation(fields: [authorId], references: [id], onDelete: Cascade)
  authorId  String
  createdAt DateTime @default(now())

  @@index([authorId])
}

    Generate the client

    npx prisma generate

    This writes the typed client into node_modules/@prisma/client. You re-run it whenever the schema changes (and prisma migrate runs it for you).

    Generate on install

    Add a postinstall script so deployments always have a fresh client: 

    // package.json
{ "scripts": { "postinstall": "prisma generate" } }
    Hosts like Vercel cache node_modules, so without this your build can ship a stale or missing client.

    3. The Prisma Client singleton

    This is the single most common Prisma-in-Next.js mistake, so it gets its own section.

    In development, Next.js hot-reloads your modules on every save. If you do new PrismaClient() at module scope, each reload creates a new client with its own connection pool. After a few edits you'll see Postgres reject connections with too many clients already. The fix is to cache one instance on globalThis, which survives hot reloads.

    // lib/prisma.ts
import { PrismaClient } from "@prisma/client";

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined;
};

export const prisma =
  globalForPrisma.prisma ??
  new PrismaClient({
    // log: ["query"], // uncomment to see SQL in dev
  });

// In dev, keep the single instance across hot-reloads.
// In production each serverless instance gets exactly one, so don't cache.
if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}

    Import this prisma everywhere—never call new PrismaClient() in route files, components, or actions.

    Always use the singleton

    Instantiating PrismaClient per-request (or per-render) opens a fresh connection pool each time. In dev this throws too many clients already after a few hot reloads. In serverless production it's worse: every cold-started function instance holds its own pool, and concurrent invocations multiply them until the database hits max_connections. One cached client per process is the rule. (Section 9 covers what to do when even one pool per instance is too many.)

    4. Migrations & seeding

    Migrations

    A migration is a versioned SQL file that takes your database from one schema state to the next. During development:

    # Create a migration from schema changes, apply it, and regenerate the client
npx prisma migrate dev --name init

    In CI/CD or production you apply already-committed migrations without generating new ones:

    # Apply pending migrations only — safe for deploy pipelines
npx prisma migrate deploy

    A handy fast path while prototyping is prisma db push, which syncs the schema to the database without creating migration files. Use it for spikes; switch to migrate once the schema matters, because only migrate gives you a reviewable, replayable history.

    migrate dev is a dev-only command

    prisma migrate dev can reset the database and is interactive. Never run it against production. Production pipelines use prisma migrate deploy, which only applies existing migrations and never prompts or drops data.

    Seeding

    A seed script fills a fresh database with baseline/dev data. Configure it in package.json:

    // package.json
{
  "prisma": {
    "seed": "tsx prisma/seed.ts"
  }
}

    // prisma/seed.ts
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient(); // a standalone script — singleton not needed here

async function main() {
  // upsert is idempotent: safe to run the seed repeatedly
  const ada = await prisma.user.upsert({
    where: { email: "ada@example.com" },
    update: {},
    create: {
      email: "ada@example.com",
      name: "Ada Lovelace",
      posts: {
        create: [
          { title: "Hello world", content: "First post", published: true },
        ],
      },
    },
  });
  console.log("Seeded:", ada.email);
}

main()
  .then(() => prisma.$disconnect())
  .catch(async (e) => {
    console.error(e);
    await prisma.$disconnect();
    process.exit(1);
  });

    # Run on demand; migrate reset also runs it automatically
npx prisma db seed

    Inspect data with Prisma Studio

    npx prisma studio opens a local GUI to browse and edit your tables—great for sanity-checking seeds and migrations.

    5. Querying data in Server Components

    Because Server Components run only on the server, you can import the Prisma client and await a query directly inside the component. No getServerSideProps, no API hop.

    // app/posts/page.tsx — a Server Component
import { prisma } from "@/lib/prisma";

export default async function PostsPage() {
  const posts = await prisma.post.findMany({
    where: { published: true },
    include: { author: { select: { name: true } } },
    orderBy: { createdAt: "desc" },
    take: 20,
  });

  return (
    <main>
      <h1>Published posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>
            {post.title} — <em>{post.author.name ?? "Anonymous"}</em>
          </li>
        ))}
      </ul>
    </main>
  );
}

    The query, the connection string, and the result all stay on the server; only the rendered HTML reaches the browser.

    Caching a direct DB query

    fetch caching doesn't apply to Prisma calls (they aren't HTTP). To cache a query with revalidate/tag semantics, wrap it in unstable_cache (or the newer 'use cache' directive):

    // lib/queries.ts
import { unstable_cache } from "next/cache";
import { prisma } from "@/lib/prisma";

export const getPublishedPosts = unstable_cache(
  async () => prisma.post.findMany({ where: { published: true } }),
  ["published-posts"],            // cache key parts
  { revalidate: 60, tags: ["posts"] }
);

    Then revalidateTag("posts") after a mutation invalidates it on demand.

    Direct DB access is dynamic by default

    A route that queries the database renders dynamically unless you opt into caching. If the data is largely static, wrap it in unstable_cache/'use cache' and revalidate on writes, so you get CDN-fast reads instead of hitting the database on every request.

    6. Mutating data in Server Actions

    A Server Action ("use server") lets a form submit straight to a server function that writes to the database—then revalidate the affected route. This replaces the old "client fetch → API route → DB" round trip for most mutations.

    // app/posts/actions.ts
"use server";

import { prisma } from "@/lib/prisma";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
import { z } from "zod";
import { getCurrentUser } from "@/lib/dal"; // see section 7

const CreatePost = z.object({
  title: z.string().min(1, "Title is required").max(120),
  content: z.string().max(10_000).optional(),
});

export async function createPost(_prev: unknown, formData: FormData) {
  // 1. Authenticate — NEVER trust the client to be authorized
  const user = await getCurrentUser();
  if (!user) return { error: "Not authenticated" };

  // 2. Validate input on the server
  const parsed = CreatePost.safeParse({
    title: formData.get("title"),
    content: formData.get("content"),
  });
  if (!parsed.success) {
    return { error: parsed.error.flatten().fieldErrors };
  }

  // 3. Write, scoping the row to the authenticated user
  await prisma.post.create({
    data: { ...parsed.data, authorId: user.id },
  });

  // 4. Refresh the cache and navigate
  revalidatePath("/posts");
  redirect("/posts");
}

    Wire it to a form with useActionState for typed errors and pending state:

    // app/posts/new-post-form.tsx
"use client";

import { useActionState } from "react";
import { createPost } from "./actions";

export default function NewPostForm() {
  const [state, formAction, pending] = useActionState(createPost, null);

  return (
    <form action={formAction}>
      <input name="title" placeholder="Title" />
      <textarea name="content" placeholder="Content" />
      <button disabled={pending}>{pending ? "Saving…" : "Create"}</button>
      {state?.error && (
        <p role="alert">{JSON.stringify(state.error)}</p>
      )}
    </form>
  );
}

    Server Actions are public endpoints

    Every Server Action compiles to a callable HTTP endpoint. Anyone can invoke it with arbitrary input—it is not protected just because the UI only shows the form to logged-in users. Always re-check authentication and authorization, and re-validate input, inside the action. The auth check in step 1 is not optional.

    7. The data-access layer (DAL) pattern

    Scattering prisma.* calls and auth checks across dozens of components and actions is how authorization bugs happen—you forget one check, or two callers enforce ownership differently. The data-access layer centralizes all database access behind a small set of functions that own the auth rules. Components and actions call the DAL; they never touch Prisma directly.

    // lib/dal.ts
import "server-only"; // ← makes this module impossible to import into client code
import { cache } from "react";
import { redirect } from "next/navigation";
import { prisma } from "@/lib/prisma";
import { auth } from "@/auth"; // your session source (e.g. Auth.js, section 10)

// Resolve the session once per request; `cache` dedupes within a render pass.
export const getCurrentUser = cache(async () => {
  const session = await auth();
  if (!session?.user?.email) return null;
  return prisma.user.findUnique({ where: { email: session.user.email } });
});

// Use in pages/layouts that must be private — redirect if not logged in.
export const requireUser = cache(async () => {
  const user = await getCurrentUser();
  if (!user) redirect("/login");
  return user;
});

// Authorization lives WITH the data: a user only ever sees their own posts.
export async function getMyPosts() {
  const user = await requireUser();
  return prisma.post.findMany({
    where: { authorId: user.id },
    orderBy: { createdAt: "desc" },
  });
}

// Ownership is enforced in the query, not assumed by the caller.
export async function getPostForUser(postId: string) {
  const user = await requireUser();
  return prisma.post.findFirst({
    where: { id: postId, authorId: user.id }, // scoped → can't read others' rows
  });
}

    Now a private page is trivially correct—the DAL guarantees the auth check and the ownership scoping:

    // app/dashboard/page.tsx
import { getMyPosts } from "@/lib/dal";

export default async function Dashboard() {
  const posts = await getMyPosts(); // redirects if logged out; only my rows
  return (
    <ul>
      {posts.map((p) => (
        <li key={p.id}>{p.title}</li>
      ))}
    </ul>
  );
}

    Why import "server-only"

    The server-only package has no runtime behavior—it exists to fail the build if a server module is ever imported into a Client Component. Without it, an innocent refactor (importing a DAL function into a "use client" file for a type) would bundle your Prisma client, queries, and DATABASE_URL into the browser.

    npm install server-only

    Don't leak the database into the client

    Put import "server-only" at the top of every module that touches the database, secrets, or the session (lib/prisma.ts, lib/dal.ts, etc.). If a Client Component accidentally imports one, you get a clear build error instead of silently shipping your connection string and query logic to every visitor. This pairs with the boundary rules: never pass a raw Prisma model to a client component if it contains fields the user shouldn't see—select only what the UI needs.

    Treat the DAL as your security boundary

    Auth in middleware.ts is an optimization (it short-circuits obvious cases fast), not your real defense. The authoritative checks belong in the DAL, right next to the data, so every code path—page, action, route handler—goes through the same gate. This is exactly the model Next.js recommends after the 2025 middleware-bypass advisories.

    8. Transactions & a quick note on relations

    For multi-step writes that must all succeed or all fail, use a transaction so you never end up half-written:

    // Interactive transaction: both succeed or both roll back
await prisma.$transaction(async (tx) => {
  const post = await tx.post.create({ data: { title, authorId: user.id } });
  await tx.user.update({
    where: { id: user.id },
    data: { /* e.g. increment a counter */ },
  });
  return post;
});

    Prisma also accepts an array form (prisma.$transaction([queryA, queryB])) for independent queries you want batched. For reads, prefer select/include to fetch exactly the columns and relations you render—this keeps payloads small and avoids accidentally serializing sensitive fields toward the client.

    9. Serverless connection-pooling caveats

    This is the issue that bites apps in production, and it deserves real attention.

    The problem. Traditional databases (PostgreSQL especially) allow a limited number of concurrent connections (often ~100). A long-running Node server opens one pool and reuses it. Serverless flips this: each function instance is its own short-lived process with its own pool, and a traffic spike can spin up hundreds of instances simultaneously. Multiply pools by instances and you blow past max_connections—even though your singleton is correct per instance.

    There are three production-grade fixes, often combined:

    a) Prisma Accelerate (or Prisma Postgres)

    A managed connection pooler + global cache that sits in front of your database. You point Prisma at an Accelerate URL and add an extension; Prisma multiplexes many serverless callers over a small, healthy set of real connections.

    // lib/prisma.ts (with Accelerate)
import { PrismaClient } from "@prisma/client";
import { withAccelerate } from "@prisma/extension-accelerate";

const globalForPrisma = globalThis as unknown as {
  prisma: ReturnType<typeof makeClient> | undefined;
};

function makeClient() {
  return new PrismaClient().$extends(withAccelerate());
}

export const prisma = globalForPrisma.prisma ?? makeClient();
if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;

    # DATABASE_URL points to your Accelerate endpoint (prisma://...)
npm install @prisma/extension-accelerate

    Accelerate also lets you add per-query caching (cacheStrategy: { ttl, swr }) at the data layer.

    b) An external pooler (PgBouncer / Supabase / Neon)

    Many managed Postgres providers expose a PgBouncer pooling endpoint (and a separate direct endpoint for migrations). The trick is that transaction-mode poolers don't support prepared statements, so you flag the URL accordingly and give Prisma a directUrl for migrations:

    // prisma/schema.prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")     // pooled (PgBouncer) → pgbouncer=true&connection_limit=1
  directUrl = env("DIRECT_URL")       // direct connection → used by prisma migrate
}

    # .env
DATABASE_URL="postgresql://user:pass@host:6543/db?pgbouncer=true&connection_limit=1"
DIRECT_URL="postgresql://user:pass@host:5432/db"

    migrate and db push need a real (direct) connection; runtime queries go through the pooler. The directUrl field is what lets both coexist.

    c) Driver adapters (serverless / Edge drivers)

    For Edge runtimes—or serverless databases with HTTP/WebSocket drivers (Neon, PlanetScale, Turso, Cloudflare)—use a Prisma driver adapter. This swaps Prisma's default engine for a driver designed for fan-out connections, and is what makes Prisma work on the Edge runtime at all.

    // lib/prisma.ts (Neon serverless driver adapter)
import { PrismaClient } from "@prisma/client";
import { PrismaNeon } from "@prisma/adapter-neon";

const adapter = new PrismaNeon({ connectionString: process.env.DATABASE_URL });
export const prisma = new PrismaClient({ adapter });

    npm install @prisma/adapter-neon @neondatabase/serverless
# enable in schema.prisma generator:  previewFeatures = ["driverAdapters"]  (older versions)

    Don't connect serverless functions straight to raw Postgres

    Pointing many serverless/Edge functions directly at a vanilla Postgres max_connections is the classic production outage: it works in dev and on launch day, then falls over under load with remaining connection slots are reserved / too many clients. Always put a pooler (Accelerate, PgBouncer, or a serverless driver) between serverless compute and the database. Also note that Prisma's default engine can't run on the Edge runtime—use a driver adapter, or keep DB-touching code on the Node runtime.

    Picking an approach

    Situation Use
    Vercel + standard Postgres, want least setup Prisma Accelerate (pooling + caching in one)
    Managed Postgres that ships a pooler (Supabase/Neon) PgBouncer URL + directUrl for migrations
    Querying from the Edge runtime Driver adapter (Neon/PlanetScale/Turso/D1)
    Long-running Node server (self-hosted, not serverless) The plain singleton is usually fine

    📚 Summary

    🔹 Concept 🧠 Description
    Prisma vs Drizzle Prisma = schema-first, best DX; Drizzle = code-first, SQL-like, Edge-light
    schema.prisma Single source of truth → generates the typed client
    Client singleton Cache one PrismaClient on globalThis to survive hot reload & limit pools
    migrate dev / migrate deploy Create+apply migrations in dev; apply-only in production
    Seeding Idempotent prisma/seed.ts (use upsert); run via prisma db seed
    Server Component query await prisma.* directly; result HTML only reaches the client
    unstable_cache / 'use cache' Cache non-fetch DB queries with revalidate/tags
    Server Action mutation "use server" + Zod validate + auth check + revalidatePath
    DAL Centralize DB access + auth/ownership in lib/dal.ts
    import "server-only" Build-time guard so DB/secrets can't bundle into the client
    $transaction All-or-nothing multi-step writes
    Connection pooling Serverless multiplies pools → use Accelerate / PgBouncer / driver adapter
    directUrl Direct connection for migrations alongside a pooled runtime URL
