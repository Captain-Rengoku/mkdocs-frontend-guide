    Deployment

    Building a Next.js app is only half the job — it has to run somewhere. Next.js is built by Vercel, so a zero-config push-to-deploy on Vercel is the smoothest path, but the framework is deliberately portable: the same app runs on any Node server, inside a Docker container, or behind your own reverse proxy. The trade-off is what you have to wire up yourself — ISR caching, image optimization, and observability all come free on Vercel and become your responsibility when you self-host.

    This chapter covers deploying to Vercel, the standalone build output, self-hosting with a Node server and a production Dockerfile, how environment variables behave (build-time vs runtime, the NEXT_PUBLIC_ exposure rule), CDN/caching considerations, and monitoring/analytics. Examples target Next.js 15 with the App Router and TypeScript.

    📚 For more: https://nextjs.org/docs/app/building-your-application/deploying

    1. Deploying to Vercel

    Vercel is the reference platform for Next.js. It detects the framework, runs the build, and maps your app's features onto its infrastructure automatically — there is no adapter, plugin, or config file to write for a standard app.

    What "zero-config" actually does

    When you connect a Next.js repo, Vercel infers everything from package.json and the framework:

    Your code Vercel maps it to
    Static routes / SSG / ISR pages Static files on the global CDN
    async Server Components / SSR routes Serverless (or Edge) Functions
    Route Handlers (route.ts) Serverless / Edge Functions
    next/image The built-in Image Optimization service
    middleware.ts Runs at the edge, before the cache
    revalidate / revalidateTag Managed ISR + the Data Cache

    You do not run next start yourself on Vercel — there is no long-lived server. Each dynamic render and Route Handler is an on-demand function invocation.

    Connecting a repository

    The normal flow is Git-driven:

    1. Push your project to GitHub, GitLab, or Bitbucket.
    2. In the Vercel dashboard, Add New → Project and import the repo.
    3. Vercel auto-detects "Next.js" as the framework preset; the default build settings are correct for almost all apps.
    4. Add environment variables (below), then Deploy.

    From then on, every push to the production branch (usually main) triggers a production deployment, and every push to any other branch or pull request triggers a preview deployment.

    You can also deploy from the terminal with the Vercel CLI, which is handy for one-off deploys and debugging the build locally:

    npm i -g vercel

vercel          # creates a preview deployment from the current directory
vercel --prod   # promotes to production
vercel env pull # download project env vars into a local .env.local

    Environment variables on Vercel

    Set these under Project → Settings → Environment Variables. Each variable is scoped to one or more environments:

    • Production — used for production deployments.
    • Preview — used for preview (branch/PR) deployments.
    • Development — pulled down by vercel env pull for local next dev.
    # Example values set in the dashboard (never commit real secrets)
DATABASE_URL=postgres://user:pass@host:5432/app   # Production + Preview
NEXT_PUBLIC_SITE_URL=https://example.com          # Production only
STRIPE_SECRET_KEY=sk_live_...                      # Production only

    NEXT_PUBLIC_ variables are inlined into the browser bundle

    Any variable prefixed with NEXT_PUBLIC_ is embedded into the client JavaScript at build time and is fully visible to anyone who opens DevTools. Never put secrets (API keys, database URLs, tokens) behind that prefix. See section 5.

    Preview deployments

    Every branch and pull request gets its own immutable URL (e.g. my-app-git-feature-x-team.vercel.app). This is the single most useful Vercel feature for teams:

    • Reviewers click a live URL instead of pulling the branch.
    • Each preview is fully isolated, with its own Preview environment variables.
    • The Vercel GitHub app comments the preview URL on the PR automatically.

    Because previews use the Preview env scope, point them at a staging database — not production — so a test PR can't mutate live data.

    Build & function settings

    The defaults rarely need changing, but the knobs exist:

    • Build command / Output directory / Install command — override only for monorepos or custom toolchains. For a monorepo, set the Root Directory to the app's subfolder.
    • Node.js version — set in Project Settings or pinned via the engines field in package.json.
    • Function region & runtime — choose a region close to your database to cut latency; pick the runtime per route with export const runtime = "edge" | "nodejs".
    • Function max duration — extend the execution limit for long-running Route Handlers.

    A small amount can be committed as vercel.json so it lives in the repo:

    {
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "regions": ["iad1"],
  "functions": {
    "app/api/report/route.ts": { "maxDuration": 60 }
  },
  "crons": [{ "path": "/api/cron/cleanup", "schedule": "0 3 * * *" }]
}

    2. The build output

    Running the production build compiles your app and prints a per-route breakdown:

    next build

    Route (app)                      Size     First Load JS
┌ ○ /                            1.2 kB         92 kB
├ ● /blog/[slug]                 0.8 kB         91 kB
├ ƒ /dashboard                   2.4 kB         96 kB
└ ƒ /api/posts                   0 B             0 B

○  (Static)    prerendered as static content
●  (SSG)       prerendered using generateStaticParams
ƒ  (Dynamic)   server-rendered on demand

    Read this output before every deploy — it tells you exactly which routes are static (cheap, CDN-served) and which are dynamic (a function invocation per request). A route you expected to be static showing up as ƒ (Dynamic) usually means you accidentally used cookies(), headers(), or a no-store fetch in it.

    The build artifacts land in .next/. By default this includes the full node_modules, which is fine for Vercel but heavy for a container.

    output: 'standalone'

    For self-hosting, set output: 'standalone' in your config. Next.js then traces exactly which files (including the minimal subset of node_modules) the server actually needs and copies them into .next/standalone/. This is the key to small, fast Docker images.

    // next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "standalone",
};

export default nextConfig;

    After next build, .next/standalone/ contains a self-contained app with its own minimal server.js. You start it with node server.js — no next start, and you don't need to ship the whole node_modules tree.

    Standalone doesn't copy public/ or static assets

    The standalone output deliberately excludes public/ and .next/static/, because on many platforms those are served by a CDN, not the Node process. When you self-host the whole thing yourself you must copy them in manually — the Dockerfile in section 4 does exactly that.

    3. Self-hosting on a Node server

    Next.js doesn't need Vercel. Any environment that can run Node can run a Next.js app with two commands:

    next build     # produce the production build
next start     # start the production server (default: port 3000)

    // package.json
{
  "scripts": {
    "build": "next build",
    "start": "next start -p 3000"
  }
}

    next start is a long-lived Node process that serves static assets, renders dynamic routes, runs Route Handlers, handles ISR regeneration, and performs image optimization — everything Vercel splits across services, in one process. Keep it alive with a process manager (pm2, systemd) and run multiple instances behind a load balancer to use all CPU cores and survive restarts.

    output: 'export' is not the same thing

    output: 'export' produces a pure static site (HTML/CSS/JS only) you can drop on any static host. But it disables everything that needs a server: SSR, ISR, Route Handlers, Server Actions, Middleware, and on-demand image optimization. Use it only for fully static sites; for everything else use next start or the standalone server.

    Reverse proxy notes

    In production you almost always put Nginx (or Caddy/a cloud load balancer) in front of the Node process to terminate TLS, serve as a single entry point, and optionally cache static assets:

    # /etc/nginx/conf.d/nextjs.conf
server {
  listen 443 ssl http2;
  server_name example.com;

  # ... ssl_certificate / ssl_certificate_key ...

  location / {
    proxy_pass         http://127.0.0.1:3000;
    proxy_http_version 1.1;
    proxy_set_header   Host              $host;
    proxy_set_header   X-Real-IP         $remote_addr;
    proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header   X-Forwarded-Proto $scheme;   # so Next.js knows it's HTTPS
    proxy_set_header   X-Forwarded-Host  $host;
  }

  # Long-cache immutable build assets
  location /_next/static/ {
    proxy_pass http://127.0.0.1:3000;
    add_header Cache-Control "public, max-age=31536000, immutable";
  }
}

    Key points: forward X-Forwarded-* headers so request-host/protocol detection works, don't buffer responses if you rely on streaming (proxy_buffering off; for streamed routes), and let /_next/static/ (content-hashed, immutable) be cached aggressively.

    ISR & caching when self-hosting

    This is the part that surprises people. ISR works out of the box with next start, but the default cache is on the local filesystem of that single instance. With more than one instance (a typical horizontally-scaled or containerized deploy) that breaks down:

    • Instance A regenerates a page; instance B still serves the stale copy.
    • On platforms with ephemeral disks (most container hosts), the cache is lost on every redeploy or restart, forcing full regeneration.

    The fix is a shared cache handler backed by something all instances share — Redis, S3, etc. Point Next.js at a custom handler in your config:

    // next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "standalone",
  cacheHandler: require.resolve("./cache-handler.js"),
  // disable the in-memory LRU when using an external store
  cacheMaxMemorySize: 0,
};

export default nextConfig;

    // cache-handler.js — a minimal Redis-backed CacheHandler
// Implements the interface Next.js expects: get / set / revalidateTag.
const { createClient } = require("redis");

const client = createClient({ url: process.env.REDIS_URL });
client.connect().catch(console.error);

module.exports = class CacheHandler {
  constructor(options) {
    this.options = options;
  }

  async get(key) {
    const value = await client.get(`nextcache:${key}`);
    return value ? JSON.parse(value) : null;
  }

  async set(key, data, ctx) {
    const entry = { value: data, lastModified: Date.now(), tags: ctx.tags };
    await client.set(`nextcache:${key}`, JSON.stringify(entry));
    // Maintain a tag → keys index so revalidateTag can find entries
    for (const tag of ctx.tags ?? []) {
      await client.sAdd(`nextcache:tag:${tag}`, key);
    }
  }

  async revalidateTag(tags) {
    const tagList = Array.isArray(tags) ? tags : [tags];
    for (const tag of tagList) {
      const keys = await client.sMembers(`nextcache:tag:${tag}`);
      if (keys.length) {
        await client.del(keys.map((k) => `nextcache:${k}`));
        await client.del(`nextcache:tag:${tag}`);
      }
    }
  }
};

    With this in place, all instances read and write the same cache, regeneration is shared, and revalidatePath/revalidateTag invalidate consistently across the fleet. (For common stores you can use the community @neshca/cache-handler package instead of hand-rolling this.)

    Image optimization when self-hosting

    next/image optimizes images on the fly using sharp — install it (npm i sharp) for the standalone server, or it falls back to a slower path. If a CDN already handles image resizing, you can set images.unoptimized: true or configure a custom loader to skip the Node-side work.

    4. Self-hosting with Docker

    A production Dockerfile uses the standalone output and a multi-stage build so the final image contains only the runtime, not the full toolchain or node_modules. This typically produces an image in the low hundreds of MB rather than over a gigabyte.

    # syntax=docker/dockerfile:1

# 1) Install dependencies only when lockfile changes
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json* ./
RUN npm ci

# 2) Build the app (requires output: 'standalone' in next.config.ts)
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
ENV NEXT_TELEMETRY_DISABLED=1
RUN npm run build

# 3) Minimal runtime image
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1

# Run as an unprivileged user
RUN addgroup --system --gid 1001 nodejs \
 && adduser  --system --uid 1001 nextjs

# Copy the standalone server, then the assets it does NOT include
COPY --from=builder /app/public ./public
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

USER nextjs
EXPOSE 3000
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"

# The standalone build emits its own server.js entrypoint
CMD ["node", "server.js"]

    # Build and run
docker build -t my-next-app .
docker run -p 3000:3000 --env-file .env.production my-next-app

    Notes that trip people up:

    • The two COPY lines for public and .next/static are mandatory — the standalone output omits them (see section 2).
    • Pass runtime secrets with --env-file / -e at docker run, not baked into the image. Only NEXT_PUBLIC_ values that exist at build time are inlined; anything read on the server at runtime should be injected when the container starts.
    • Set HOSTNAME=0.0.0.0 so the server listens on all interfaces inside the container.
    • For ISR across replicas, combine this with the shared cache handler from section 3.

    A docker-compose.yml ties the app to its cache/database for local production parity:

    services:
  web:
    build: .
    ports: ["3000:3000"]
    env_file: .env.production
    depends_on: [redis]
  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]

    5. Environment variables in depth

    Next.js loads .env files automatically and distinguishes server-only variables from browser-exposed ones.

    .env file loading order

    Next.js reads these files, with later ones overriding earlier ones (and .env.local always overriding except in test):

    File Loaded when Commit to git?
    .env always ✅ defaults / non-secrets
    .env.local always (except next test) ❌ never — real secrets
    .env.development next dev ✅
    .env.production next build / next start ✅ non-secrets only 
    # .env.local (gitignored)
DATABASE_URL=postgres://localhost:5432/dev
AUTH_SECRET=super-secret-value
NEXT_PUBLIC_SITE_URL=http://localhost:3000

    Always add .env*.local to .gitignore (create-next-app does this for you).

    Server-only vs NEXT_PUBLIC_

    • A variable without the NEXT_PUBLIC_ prefix is available only in server code — Server Components, Route Handlers, Server Actions, middleware.ts, and next.config. If you read it in a Client Component it is undefined.
    • A variable with NEXT_PUBLIC_ is inlined into the JavaScript bundle so the browser can read it.
    // Server Component / Route Handler — both work
const db = process.env.DATABASE_URL;             // server-only secret ✅
const url = process.env.NEXT_PUBLIC_SITE_URL;    // also available ✅

// Client Component ("use client")
const url = process.env.NEXT_PUBLIC_SITE_URL;    // ✅ inlined at build
const db  = process.env.DATABASE_URL;            // ❌ undefined in the browser

    NEXT_PUBLIC_ is permanent, public, and build-time

    NEXT_PUBLIC_ variables are replaced with their literal value during next build. Two consequences: (1) they are visible to every user — never use the prefix for secrets; (2) changing one requires a rebuild, not just a restart, because the old value is already baked into the bundle. Treat them as compile-time constants.

    Build-time vs runtime

    This distinction is the source of most "it works locally but not in prod" bugs:

    • Build-time — every NEXT_PUBLIC_ value and anything read while statically generating a page is fixed at next build. In a Docker image these are baked in.
    • Runtime — plain (non-public) process.env.* reads inside dynamic server code are evaluated on each request in the running process, so you can change them by restarting with new values (no rebuild).

    To force a value to be read at runtime instead of inlined at build, read it inside a dynamic code path. The App Router also exposes a connection() / dynamic API to opt a render out of static generation when it depends on runtime env. The practical rule: inject runtime secrets when the container starts; only build-time public config goes in the image.

    6. Caching & CDN considerations

    Performance and cost both come down to serving as much as possible from a CDN and invoking functions as little as possible.

    • Static & ISR pages are cacheable files — put a CDN in front of your deployment (Vercel does this automatically; self-hosters use Cloudflare, CloudFront, or the proxy cache). This is what lets a static route absorb unlimited traffic for near-zero cost.
    • Immutable assets — everything under /_next/static/ is content-hashed, so it's safe to serve with Cache-Control: public, max-age=31536000, immutable.
    • Set cache behavior intentionally with Cache-Control headers via next.config (or the proxy):
    // next.config.ts — example header rules
const nextConfig: NextConfig = {
  async headers() {
    return [
      {
        source: "/_next/static/:path*",
        headers: [
          { key: "Cache-Control", value: "public, max-age=31536000, immutable" },
        ],
      },
    ];
  },
};
    • ISR is stale-while-revalidate at the edge — the CDN serves the cached page instantly and Next.js regenerates in the background. On self-host this only works correctly with a shared cache handler (section 3).
    • Don't cache personalized responses — routes that read cookies()/headers() are dynamic; make sure your proxy/CDN doesn't cache them across users.
    • Middleware runs before the cache — it's the right place for redirects and auth gating, but heavy work there is paid on every request, even for cached pages.

    7. Monitoring & analytics

    A deployed app needs observability — to catch errors, watch real-user performance, and trace slow requests.

    Vercel Analytics & Speed Insights

    On Vercel, two first-party, privacy-friendly tools drop in with a single component each:

    • Analytics — page-view and visitor metrics (no cookies, GDPR-friendly).
    • Speed Insights — real-user Core Web Vitals (LCP, INP, CLS) measured in the field, not a lab.
    npm i @vercel/analytics @vercel/speed-insights

    // app/layout.tsx
import { Analytics } from "@vercel/analytics/next";
import { SpeedInsights } from "@vercel/speed-insights/next";

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en">
      <body>
        {children}
        <Analytics />
        <SpeedInsights />
      </body>
    </html>
  );
}

    OpenTelemetry

    For traces and metrics that work anywhere (Vercel or self-hosted), Next.js has built-in OpenTelemetry support. It auto-instruments rendering and data fetching so you can export spans to Datadog, Honeycomb, Grafana, Jaeger, etc.

    npm i @vercel/otel

    // instrumentation.ts — at the project root; runs once on server startup
import { registerOTel } from "@vercel/otel";

export function register() {
  registerOTel({ serviceName: "my-next-app" });
}

    The register() export in instrumentation.ts is Next.js's official startup hook — useful for OTel, error reporters (e.g. Sentry's init), and other one-time server setup.

    Logging

    • Server-side console.log/console.error from Server Components, Route Handlers, and Server Actions go to the server process — captured by Vercel's runtime logs, or by docker logs/your log aggregator when self-hosting. They never reach the browser.
    • Prefer structured (JSON) logging (e.g. pino) in production so logs are queryable in your aggregator.
    • Pair an error-tracking service (Sentry, etc.) with instrumentation.ts and an app/error.tsx / global-error.tsx boundary so client-side and rendering errors are reported, not just lost.
    // app/global-error.tsx — top-level error boundary ("use client")
"use client";

export default function GlobalError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  // report(error)  ← send to Sentry / your tracker here
  return (
    <html>
      <body>
        <h2>Something went wrong</h2>
        <button onClick={reset}>Try again</button>
      </body>
    </html>
  );
}
