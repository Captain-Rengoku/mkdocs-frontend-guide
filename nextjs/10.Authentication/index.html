    Authentication (Auth.js v5)

    Authentication answers who is making this request; authorization answers what they may do. In the App Router both questions are best answered on the server, before any HTML is sent — there's no client to trust, no token to leak, and no protected UI to flash.

    This chapter uses Auth.js v5 (the package formerly called NextAuth.js, now published as next-auth@5). v5 is a substantial rewrite for the App Router: a single auth.ts config exports everything (handlers, auth, signIn, signOut), the universal auth() helper reads the session anywhere on the server, and the same config powers the route handler, middleware, Server Components, Server Actions, and Route Handlers. The result is far less boilerplate than v4 and a model that fits RSC naturally.

    📚 For more: https://authjs.dev · Next.js — Authentication

    The security throughline of this guide

    The React Authentication chapter established the rules this chapter keeps: tokens/sessions live in HttpOnly, Secure, SameSite cookies (never localStorage); client-side checks are UX, not security; and every authorization decision is re-enforced on the server. Auth.js follows all three by default — its session cookie is HttpOnly and the auth() check runs server-side. The one thing it adds that you must internalize: middleware is not a security boundary (see Protecting routes).

    1. Why Auth.js v5

    Building auth by hand (the custom system in the React chapter) is instructive but error-prone — you own password hashing, OAuth flows, CSRF, cookie flags, session rotation, and breach response. Auth.js gives you:

    • Dozens of OAuth/OIDC providers (GitHub, Google, Apple, Microsoft, …) with one-line configs.
    • A Credentials provider for email/password, if you must.
    • HttpOnly session cookies with CSRF protection out of the box.
    • JWT or database sessions, switchable with one option.
    • First-class App Router integration — auth() works in Server Components, Server Actions, Route Handlers, and middleware.

    Auth.js vs NextAuth vs next-auth@5

    They're the same project. NextAuth.js was the original Next-only name. The project generalized into Auth.js (with SvelteKit, Express, etc. adapters), but the Next.js package is still installed as next-auth — version 5 is what this chapter calls "Auth.js v5." At time of writing v5 is published as next-auth@beta; the API below is the current beta API.

    2. Setup

    Install

    npm install next-auth@beta

    Generate a strong secret used to sign/encrypt the session cookie and JWTs:

    npx auth secret   # writes AUTH_SECRET to .env.local

    AUTH_SECRET is mandatory in production

    Without it, sessions can't be signed and v5 will throw at runtime. Keep it server-only (no NEXT_PUBLIC_ prefix) and rotate it as you would any signing key — rotating invalidates existing sessions.

    The central auth.ts

    The heart of v5 is a single config file at the project root that you call NextAuth() on and re-export its pieces. Everything else imports from here.

    // auth.ts
import NextAuth from "next-auth";
import GitHub from "next-auth/providers/github";

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [GitHub], // reads AUTH_GITHUB_ID / AUTH_GITHUB_SECRET from env
});

    What each export is for:

    Export What it is Where you use it
    handlers { GET, POST } route handlers app/api/auth/[...nextauth]/route.ts
    auth Universal session reader and middleware wrapper Server Components, Route Handlers, Server Actions, middleware.ts
    signIn Server-side sign-in trigger Server Actions / Route Handlers
    signOut Server-side sign-out trigger Server Actions / Route Handlers

    Env var naming convention

    v5 auto-detects provider credentials from env vars named AUTH_<PROVIDER>_ID and AUTH_<PROVIDER>_SECRET (e.g. AUTH_GITHUB_ID). You can still pass clientId/clientSecret explicitly, but the convention means many providers need zero config in auth.ts.

    The catch-all Route Handler

    Auth.js needs HTTP endpoints for the OAuth callbacks, the CSRF token, session JSON, etc. In v5 you don't write them — you re-export the generated handlers:

    // app/api/auth/[...nextauth]/route.ts
import { handlers } from "@/auth";

export const { GET, POST } = handlers;

    This one file wires up /api/auth/signin, /api/auth/callback/:provider, /api/auth/session, /api/auth/csrf, /api/auth/signout, and so on.

    middleware.ts (optional but common)

    Exporting auth as middleware keeps the session fresh and lets you gate routes at the edge. The minimal form:

    // middleware.ts
export { auth as middleware } from "@/auth";

export const config = {
  // Run on everything except static assets and Next internals
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
};

    A richer form runs your own logic with the session attached to req.auth:

    // middleware.ts
import { auth } from "@/auth";
import { NextResponse } from "next/server";

export default auth((req) => {
  const isLoggedIn = !!req.auth;            // session decoded for you
  const isOnDashboard = req.nextUrl.pathname.startsWith("/dashboard");

  if (isOnDashboard && !isLoggedIn) {
    const url = new URL("/login", req.nextUrl);
    url.searchParams.set("callbackUrl", req.nextUrl.pathname);
    return NextResponse.redirect(url);
  }
  return NextResponse.next();
});

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
};

    Keep auth.ts edge-compatible if it runs in middleware

    Middleware runs on the Edge runtime, which has no Node APIs. If your auth.ts imports a database adapter or Node-only crypto (common with the Credentials provider or database sessions), it cannot run in middleware as-is. The fix is the split config pattern below.

    Split config for the Edge (auth.config.ts)

    Put the edge-safe parts (providers that don't need Node, callbacks, pages) in auth.config.ts, and assemble the full config — with the database adapter — in auth.ts. Middleware imports only the edge-safe slice.

    // auth.config.ts — edge-safe, no DB adapter, no Node-only providers
import type { NextAuthConfig } from "next-auth";
import GitHub from "next-auth/providers/github";

export const authConfig = {
  pages: { signIn: "/login" },
  providers: [GitHub], // OAuth providers are edge-safe
  callbacks: {
    authorized({ auth, request: { nextUrl } }) {
      const isLoggedIn = !!auth?.user;
      const isOnDashboard = nextUrl.pathname.startsWith("/dashboard");
      if (isOnDashboard) return isLoggedIn; // false → redirect to signIn page
      return true;
    },
  },
} satisfies NextAuthConfig;

    // auth.ts — full config, runs on Node (Server Components / Actions / Route Handler)
import NextAuth from "next-auth";
import { PrismaAdapter } from "@auth/prisma-adapter";
import Credentials from "next-auth/providers/credentials";
import { prisma } from "@/lib/prisma";
import { authConfig } from "./auth.config";

export const { handlers, auth, signIn, signOut } = NextAuth({
  ...authConfig,
  adapter: PrismaAdapter(prisma),
  session: { strategy: "jwt" },
  providers: [
    ...authConfig.providers,
    Credentials({ /* … see below … */ }),
  ],
});

    // middleware.ts — imports ONLY the edge-safe config
import NextAuth from "next-auth";
import { authConfig } from "./auth.config";

export const { auth: middleware } = NextAuth(authConfig);

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
};

    The authorized callback is the idiomatic v5 way to gate routes from middleware: return true to allow, false to bounce to the sign-in page, or a Response/NextResponse for custom handling.

    3. Providers

    OAuth (GitHub, Google, …)

    OAuth providers are the easy, recommended path: the provider verifies the user, Auth.js completes the code exchange on the server, and sets your HttpOnly session cookie. No password ever touches your app.

    // auth.ts
import NextAuth from "next-auth";
import GitHub from "next-auth/providers/github";
import Google from "next-auth/providers/google";

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [
    GitHub,                       // AUTH_GITHUB_ID / AUTH_GITHUB_SECRET
    Google({                      // explicit form, if you prefer
      clientId: process.env.AUTH_GOOGLE_ID,
      clientSecret: process.env.AUTH_GOOGLE_SECRET,
    }),
  ],
});

    Register the callback URL with each provider. For Auth.js it is:

    https://yourapp.com/api/auth/callback/github
https://yourapp.com/api/auth/callback/google

    This mirrors the React chapter's OAuth advice

    The React chapter insisted the token exchange happens on the backend so provider tokens never reach the browser. Auth.js does exactly that — the callback handler runs server-side and only an HttpOnly session cookie comes back to the client.

    Credentials (email/password) — with caveats

    The Credentials provider lets you authenticate against your own user table. Auth.js supports it but deliberately makes it harder, because rolling your own password auth means owning hashing, rate limiting, and breach response yourself.

    // auth.ts (Credentials provider)
import Credentials from "next-auth/providers/credentials";
import { z } from "zod";
import bcrypt from "bcryptjs";
import { getUserByEmail } from "@/lib/data/user";

const credentialsSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
});

Credentials({
  async authorize(rawCredentials) {
    const parsed = credentialsSchema.safeParse(rawCredentials);
    if (!parsed.success) return null; // null → "invalid credentials", no error detail leaked

    const { email, password } = parsed.data;
    const user = await getUserByEmail(email); // your DB query
    if (!user || !user.passwordHash) return null;

    const ok = await bcrypt.compare(password, user.passwordHash);
    if (!ok) return null;

    // What you return becomes the session user — return ONLY safe fields.
    return { id: user.id, name: user.name, email: user.email, role: user.role };
  },
});

    Credentials provider constraints — read before using

    • You must hash passwords yourself (bcrypt/argon2). Auth.js stores nothing for you and does no hashing.
    • The Credentials provider forces session.strategy: "jwt" — database sessions are not supported with it, because there's no OAuth account row to link a DB session to.
    • bcrypt and most DB drivers are Node-only, so a Credentials auth.ts is not edge-compatible — use the split config so middleware never imports it.
    • Return null for any failure (bad email, wrong password, validation error). Throwing leaks which step failed and enables user enumeration.
    • You still owe rate limiting / brute-force protection on the sign-in route — Auth.js does not provide it.

    Prefer OAuth or email magic links

    For most apps, GitHub/Google OAuth or the Nodemailer/Resend email (magic link) provider is safer and less work than passwords. Reach for Credentials only when you genuinely need self-managed accounts.

    Triggering sign-in / sign-out

    From a Server Component, render a form whose action is a Server Action calling signIn/signOut:

    // app/components/auth-buttons.tsx (Server Component)
import { signIn, signOut, auth } from "@/auth";

export async function SignIn() {
  return (
    <form
      action={async () => {
        "use server";
        await signIn("github", { redirectTo: "/dashboard" });
      }}
    >
      <button type="submit">Sign in with GitHub</button>
    </form>
  );
}

export async function SignOut() {
  return (
    <form
      action={async () => {
        "use server";
        await signOut({ redirectTo: "/" });
      }}
    >
      <button type="submit">Sign out</button>
    </form>
  );
}

    For Credentials, pass the form data and handle the thrown error:

    // app/login/actions.ts
"use server";
import { signIn } from "@/auth";
import { AuthError } from "next-auth";

export async function login(_prev: unknown, formData: FormData) {
  try {
    await signIn("credentials", {
      email: formData.get("email"),
      password: formData.get("password"),
      redirectTo: "/dashboard",
    });
  } catch (error) {
    if (error instanceof AuthError) {
      // CredentialsSignin is the type for a failed authorize()
      return error.type === "CredentialsSignin"
        ? "Invalid email or password."
        : "Something went wrong.";
    }
    throw error; // re-throw the redirect Next.js uses internally
  }
}

    signIn/signOut redirect by throwing

    On success they throw a special NEXT_REDIRECT to perform the navigation. Never wrap them in a try/catch that swallows everything — re-throw anything that isn't an AuthError, or the redirect won't happen.

    4. Accessing the session

    On the server — await auth()

    The canonical, secure way to read the session is auth() in any server context. It reads and verifies the HttpOnly cookie server-side — exactly the model this guide recommends.

    // app/dashboard/page.tsx (Server Component)
import { auth } from "@/auth";
import { redirect } from "next/navigation";

export default async function Dashboard() {
  const session = await auth();
  if (!session?.user) redirect("/login");

  return <h1>Welcome, {session.user.name}</h1>;
}

    It works the same in a Route Handler and a Server Action:

    // app/api/profile/route.ts (Route Handler)
import { auth } from "@/auth";
import { NextResponse } from "next/server";

export const GET = auth((req) => {
  if (!req.auth) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  return NextResponse.json({ user: req.auth.user });
});

    // a Server Action
"use server";
import { auth } from "@/auth";

export async function updateProfile(formData: FormData) {
  const session = await auth();
  if (!session?.user) throw new Error("Not authenticated"); // re-check, always
  // …mutate using session.user.id…
}

    On the client — useSession + <SessionProvider>

    When a Client Component needs the session (e.g. to show an avatar), wrap the app in <SessionProvider> and read it with useSession. The provider fetches the session from /api/auth/session over HTTP.

    // app/providers.tsx
"use client";
import { SessionProvider } from "next-auth/react";

export function Providers({ children }: { children: React.ReactNode }) {
  return <SessionProvider>{children}</SessionProvider>;
}

    // app/layout.tsx — keep "use client" out of the layout; just render the provider
import { Providers } from "./providers";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  );
}

    // app/components/user-menu.tsx
"use client";
import { useSession, signIn, signOut } from "next-auth/react";

export function UserMenu() {
  const { data: session, status } = useSession();

  if (status === "loading") return <span>Loading…</span>;
  if (!session) return <button onClick={() => signIn("github")}>Sign in</button>;

  return (
    <div>
      <span>{session.user?.name}</span>
      <button onClick={() => signOut()}>Sign out</button>
    </div>
  );
}

    useSession is for UX, never authorization

    This is the same rule as the React chapter's "client-side authorization is UX, not security". useSession data comes from the client and can be faked by a user editing your bundle. Use it to render an avatar or hide a menu item — never to decide whether a mutation or data read is allowed. That decision belongs to auth() on the server.

    Prefer the server when you can

    In the App Router you often don't need <SessionProvider> at all: read the session with auth() in a Server Component and pass only the fields you need down as props. Reach for useSession only for genuinely client-driven UI that must react to sign-in/out without a navigation.

    Typing the session (roles, ids)

    Augment the Auth.js types so session.user.role and session.user.id are typed everywhere. Populate them in the jwt/session callbacks.

    // types/next-auth.d.ts
import { DefaultSession } from "next-auth";

declare module "next-auth" {
  interface Session {
    user: { id: string; role: "user" | "admin" } & DefaultSession["user"];
  }
  interface User {
    role: "user" | "admin";
  }
}

    // auth.ts — copy role/id through the callbacks (JWT strategy shown)
callbacks: {
  async jwt({ token, user }) {
    if (user) token.role = user.role; // runs on sign-in; user is the DB/authorize result
    return token;
  },
  async session({ session, token }) {
    session.user.id = token.sub!;          // sub is the user id
    session.user.role = token.role as "user" | "admin";
    return session;
  },
},

    5. Protecting routes

    There are two layers, and you need both.

    Layer 1 — Middleware (optimistic, fast)

    Middleware runs at the edge before the request hits a route, so it's the cheapest place to bounce obviously-unauthenticated traffic and avoid rendering a protected page at all. Either the minimal re-export or the authorized callback from §2 handles this.

    Layer 2 — Per-page server-side checks in the DAL (authoritative)

    Middleware alone is NOT sufficient — this is the critical point

    Next.js's own guidance is explicit: do not rely on middleware as your only auth check. Reasons:

    • Middleware runs an optimistic check (it typically reads/decodes the session cookie) — it is convenient, not a guarantee. A 2025 middleware-bypass CVE (CVE-2025-29927) let crafted headers skip middleware entirely. If middleware were your only gate, every protected page was exposed.
    • Middleware doesn't cover every entry point uniformly — Server Actions, Route Handlers, and data reads can be hit directly.
    • The secure pattern is defense in depth: middleware for UX/redirects, plus an authoritative check close to the data.

    Treat middleware as a redirect convenience and put the real check in your Data Access Layer.

    The recommended structure is a centralized DAL (Data Access Layer) that verifies the session and is the only path to sensitive data. Every Server Component / Action calls it; the check can't be skipped.

    // lib/dal.ts
import "server-only"; // build error if this is ever imported into client code
import { cache } from "react";
import { redirect } from "next/navigation";
import { auth } from "@/auth";

// `cache` dedupes this within a single render pass.
export const verifySession = cache(async () => {
  const session = await auth();
  if (!session?.user) redirect("/login");
  return { userId: session.user.id, role: session.user.role };
});

export async function getCurrentUserData() {
  const { userId } = await verifySession();        // authoritative gate
  return prisma.user.findUnique({ where: { id: userId } });
}

    // app/dashboard/page.tsx — the page can't render data without passing the gate
import { getCurrentUserData } from "@/lib/dal";

export default async function Dashboard() {
  const user = await getCurrentUserData(); // redirects if not authed
  return <h1>Welcome, {user?.name}</h1>;
}

    Why import \"server-only\" matters here

    Just like the React/SSR chapter's guard against leaking server modules to the browser, marking the DAL server-only means an accidental client import fails the build instead of shipping your DB query and auth logic to the browser.

    Role-based access

    Layer role checks on top of verifySession — fail closed (deny by default):

    // lib/dal.ts (continued)
export async function requireRole(role: "admin") {
  const session = await verifySession();
  if (session.role !== role) redirect("/403"); // or notFound()
  return session;
}

    // app/admin/page.tsx (Server Component)
import { requireRole } from "@/lib/dal";

export default async function AdminPage() {
  await requireRole("admin"); // server-side, authoritative
  return <AdminDashboard />;
}

    // guarding a Server Action by role
"use server";
import { requireRole } from "@/lib/dal";

export async function deleteUser(id: string) {
  await requireRole("admin"); // re-checked here too — actions are a separate entry point
  await prisma.user.delete({ where: { id } });
}

    You can also hide the admin link in the UI with useSession/session.user.role, but remember that's cosmetic — the requireRole calls above are what actually enforce it. This is the same two-place model (route-level + component-level, both backed by a server check) as the React authorization section.

    6. Session strategies: JWT vs database

    Auth.js stores the session one of two ways. Set it with session.strategy.

    // auth.ts
session: { strategy: "jwt" }       // default for OAuth-only; required for Credentials
// or
session: { strategy: "database" }  // requires an adapter (Prisma/Drizzle/…)
    JWT (default) Database
    Where session lives Encrypted JWT inside the HttpOnly cookie Row in your DB; cookie holds only an opaque session id
    Needs an adapter? No Yes (Prisma, Drizzle, etc.)
    Read cost Decrypt cookie — no DB hit DB lookup per request
    Revoke a session instantly? ❌ Hard — valid until expiry (same limitation as stateless JWTs in the React chapter) ✅ Delete the row → session dead immediately
    Edge-friendly ✅ Yes (no DB) ⚠️ Needs DB access from the edge
    Works with Credentials provider ✅ (forced) ❌ Not supported 
    // Database strategy needs an adapter:
import { PrismaAdapter } from "@auth/prisma-adapter";
import { prisma } from "@/lib/prisma";

export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: PrismaAdapter(prisma),
  session: { strategy: "database" },
  providers: [GitHub],
});

    JWT sessions can't be force-revoked — keep them short

    With the JWT strategy, "log out everywhere" and "ban this user now" don't take effect until the token expires. Mirror the React chapter's advice: keep session.maxAge modest, and if you need instant revocation (admin lockout, password reset, breach response), use the database strategy or maintain a server-side denylist checked in the jwt/session callback.

    How to choose

    • OAuth-only app, want edge speed, no instant-revoke requirement → JWT (the default).
    • Need instant revocation, an audit trail, or to query active sessions → database.
    • Using the Credentials provider → you're on JWT whether you like it or not; add a denylist if you need revocation.

    7. Alternatives

    Auth.js is the open-source default, but it isn't the only fit.

    Option Best for Trade-off
    Auth.js (NextAuth) Most Next.js apps; you want self-hosted sessions + many providers You own the user table, emails, and UI
    Clerk B2C/SaaS that wants to ship fast Hosted (vendor + pricing), but prebuilt components, orgs/multi-tenancy, MFA, and Next middleware helpers out of the box
    Lucia / your own Full control, learning, unusual requirements You implement sessions, cookies, CSRF, rotation yourself — most work, most responsibility 
    // Clerk — guarding is declarative; middleware + components are provided
// middleware.ts
import { clerkMiddleware, createRouteMatcher } from "@clerk/nextjs/server";

const isProtected = createRouteMatcher(["/dashboard(.*)", "/admin(.*)"]);

export default clerkMiddleware(async (auth, req) => {
  if (isProtected(req)) await auth.protect(); // redirects if signed out
});

    // Clerk — server-side check in a Server Component (still the authoritative layer)
import { auth } from "@clerk/nextjs/server";

export default async function Page() {
  const { userId } = await auth();
  if (!userId) redirect("/sign-in");
  // …
}

    On Lucia

    Lucia is well known as a lightweight, type-safe session library, but the maintainer has been winding it down as a library, repositioning it as a learning resource for implementing sessions yourself. If you want the "own your auth" path today, treat Lucia as a reference for the patterns (session table, cookie handling, validation) and pair it with your own code or a maintained primitive like Oslo. For most teams, Auth.js or Clerk is the lower-risk choice.

    When to build vs buy (same call as the React chapter)

    Buy when you need MFA, SSO, compliance, or social login soon — auth is not where you want hand-rolled bugs. Auth.js is the middle ground: open-source and self-hosted, but it still handles the dangerous parts (OAuth, CSRF, cookie flags) for you. Roll your own only with simple requirements and the appetite to own rotation and revocation.

    🔹 Concept 🧠 Description

    app/api/auth/[...nextauth]/route.ts One-line re-export of handlers — wires all auth endpoints
    middleware.ts export { auth as middleware } — optimistic, edge gate (UX, not the boundary)
    Split config (auth.config.ts) Edge-safe slice so middleware avoids Node-only adapters/Credentials
    OAuth providers GitHub/Google/… — server-side code exchange, HttpOnly cookie out
    Credentials provider Email/password; forces JWT, you hash + rate-limit, return null on any failure
    await auth() Authoritative session read on the server (Components, Actions, Handlers)
    useSession + <SessionProvider> Client session for UX only — never authorization
    DAL + verifySession Per-page/-action server check close to the data — the real security layer
    requireRole Fail-closed role gate, re-checked in every entry point
    JWT vs database session JWT = edge-fast, no DB, hard to revoke · DB = revocable, needs adapter
    AUTH_SECRET Mandatory signing/encryption key; server-only
    Alternatives Clerk (hosted, fast), Lucia/own (full control, most work)
