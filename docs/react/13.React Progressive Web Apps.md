# :material-progress-download: React Progressive Web Apps (PWAs)

A **Progressive Web App** is a regular web app enhanced with browser capabilities that let it behave like a native app: it can be installed to the home screen, work offline, load instantly from a cache, and receive push notifications—while still being just HTML, CSS, and JavaScript served over HTTPS.

This chapter covers what PWAs are, the three building blocks (Service Worker, Web App Manifest, and HTTPS), and how to add them to a React app. The **recommended, modern path** for a Vite React app is the [`vite-plugin-pwa`](https://vite-pwa-org.netlify.app) plugin (which wraps Google's **Workbox**), so we lead with that. We also walk through the raw Service Worker API so you understand what the tooling generates under the hood.

> 📚 For more: [web.dev/explore/progressive-web-apps](https://web.dev/explore/progressive-web-apps) · [Vite PWA Docs](https://vite-pwa-org.netlify.app) · [MDN: Progressive Web Apps](https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps)

---

## 1. What are PWAs?

### Definition and Characteristics

A PWA is a web application that uses modern web platform features to deliver an app-like experience. There is no single API called "PWA"—the term describes a set of best practices and three technical requirements.

The defining characteristics:

- **Progressive** — works for every user regardless of browser, then layers on enhancements where supported.
- **Responsive** — adapts to any screen: phone, tablet, desktop.
- **Connectivity-independent** — works offline or on flaky networks via a Service Worker cache.
- **App-like** — feels like a native app: standalone window, no browser chrome, smooth navigation.
- **Installable** — can be added to the home screen / app launcher without an app store.
- **Re-engageable** — can send push notifications even when the tab is closed.
- **Linkable** — shareable via a URL, no installation required to try it.
- **Safe** — served over HTTPS so requests can't be tampered with.

The three hard technical requirements for installability are:

1. **HTTPS** (or `localhost` during development).
2. A **registered Service Worker** with a `fetch` handler.
3. A **Web App Manifest** with the required fields (name, icons, `start_url`, `display`).

!!! note "PWAs are not a framework"
    There is nothing React-specific about a PWA. The Service Worker, manifest, and Cache API are browser platform features. React just renders the UI; the PWA layer wraps the built output. That is also why tooling like `vite-plugin-pwa` lives in the build pipeline, not in your component tree.

### Benefits of Building PWAs

- **Instant repeat loads** — pre-cached assets load from disk, not the network.
- **Offline resilience** — the app shell renders even with no connection.
- **No app store friction** — install straight from the browser; updates ship instantly on deploy.
- **One codebase** — the same app runs on web, mobile, and desktop installs.
- **Smaller footprint** — a PWA install is typically a fraction of an equivalent native binary.
- **Re-engagement** — push notifications bring users back without an app store presence.

!!! warning "Platform limitations"
    PWA support is strong on Chromium browsers and Android. On **iOS/Safari**, support has historically lagged: install ("Add to Home Screen") is manual, and **Web Push on iOS requires the app to be installed to the home screen** (Safari 16.4+). Always treat PWA features as progressive enhancements and feature-detect before using them.

## 2. Service Workers

### What is a Service Worker?

A **Service Worker** is a JavaScript file the browser runs in a **separate background thread**, independent of your page. It acts as a programmable **network proxy**: it can intercept every outgoing request from pages within its scope and decide how to respond—from the network, from a cache, or with a generated response.

Key properties that shape how you write one:

- **No DOM access** — it cannot touch `window` or `document`. It communicates with pages via `postMessage`.
- **Event-driven & terminable** — the browser can kill an idle Service Worker and restart it on the next event, so it keeps **no in-memory state** between events. Persist data in the Cache API or IndexedDB.
- **HTTPS only** — required (except on `localhost`), because a proxy that could be MITM'd would be a security hole.
- **Scope-bound** — it can only control pages under its **scope** (see below).

### Registering a Service Worker in React

Registration is done from your app code (not inside the worker). Register **after load** so it doesn't compete with critical rendering, and feature-detect first:

```tsx
// src/registerSW.ts
export function registerServiceWorker() {
  if (!('serviceWorker' in navigator)) {
    console.warn('Service workers are not supported in this browser.');
    return;
  }

  window.addEventListener('load', async () => {
    try {
      const registration = await navigator.serviceWorker.register('/sw.js', {
        scope: '/',
      });
      console.log('SW registered with scope:', registration.scope);
    } catch (err) {
      console.error('SW registration failed:', err);
    }
  });
}
```

```tsx
// src/main.tsx
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import App from './App';
import { registerServiceWorker } from './registerSW';

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>
);

registerServiceWorker();
```

!!! warning "Create React App's service worker is deprecated"
    Older tutorials reference CRA's `src/serviceWorkerRegistration.ts` and `service-worker.js`. **Create React App itself is deprecated** (the React team now recommends Vite or a framework like Next.js), and CRA's offline-first service worker was opt-out/legacy. Do not start a new PWA from CRA. Use `vite-plugin-pwa` (Vite) or Serwist/`next-pwa` (Next.js).

### Scope

**Scope** determines which URLs a Service Worker controls. By default, the scope is the directory the worker file is served from. A worker at `/sw.js` controls the **whole origin** (`/`); a worker at `/app/sw.js` would only control `/app/...`.

This is why the Service Worker file must sit at the **root** of your site (or you must send the `Service-Worker-Allowed` header to widen scope). Build tools place it correctly for you.

```bash
/sw.js          → scope "/"      controls everything
/admin/sw.js    → scope "/admin" controls only /admin/*
```

### Lifecycle: install → activate → fetch

A Service Worker moves through a well-defined lifecycle. Understanding it is the key to avoiding "stale app" bugs.

```js
// sw.js (raw API — what Workbox generates for you)
const CACHE_NAME = 'app-shell-v1';
const APP_SHELL = ['/', '/index.html', '/offline.html', '/styles.css'];

// 1) INSTALL — fired once when a new SW is downloaded.
//    Pre-cache the app shell here.
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => cache.addAll(APP_SHELL))
  );
  // Activate this SW immediately instead of waiting (use with care).
  self.skipWaiting();
});

// 2) ACTIVATE — fired when the SW takes control.
//    Clean up old caches from previous versions here.
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(
        keys.filter((k) => k !== CACHE_NAME).map((k) => caches.delete(k))
      )
    )
  );
  // Start controlling open pages without a reload.
  self.clients.claim();
});

// 3) FETCH — fired for every network request in scope.
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => cached ?? fetch(event.request))
  );
});
```

The lifecycle states in order:

1. **Installing** — `install` event runs; pre-cache assets in `event.waitUntil()`.
2. **Installed / Waiting** — a new worker waits until **all tabs using the old worker are closed**, unless you call `skipWaiting()`.
3. **Activating** — `activate` event runs; this is where you delete outdated caches.
4. **Activated** — now intercepting `fetch`. Existing tabs are controlled only after a reload, unless you call `clients.claim()`.
5. **Redundant** — superseded or failed; discarded.

!!! tip "`skipWaiting()` + `clients.claim()` can break in-flight pages"
    Forcing an immediate takeover means a page that already loaded the *old* JS chunks may suddenly request *new* hashed chunks that no longer exist, causing runtime errors. The safer pattern is a **"new version available" prompt** that asks the user to reload. `vite-plugin-pwa` gives you this with the `autoUpdate` / `prompt` strategies (shown later).

## 3. The Web App Manifest

The **Web App Manifest** is a JSON file that tells the browser how your app should appear when installed: its name, icons, colors, and window mode. Link it from your HTML `<head>`:

```html
<!-- index.html -->
<link rel="manifest" href="/manifest.webmanifest" />
<meta name="theme-color" content="#0f172a" />
```

```json
// public/manifest.webmanifest
{
  "name": "Acme Tasks",
  "short_name": "Tasks",
  "description": "Plan and track your work, online or offline.",
  "start_url": "/",
  "scope": "/",
  "display": "standalone",
  "background_color": "#0f172a",
  "theme_color": "#0f172a",
  "icons": [
    {
      "src": "/icons/icon-192.png",
      "sizes": "192x192",
      "type": "image/png"
    },
    {
      "src": "/icons/icon-512.png",
      "sizes": "512x512",
      "type": "image/png"
    },
    {
      "src": "/icons/maskable-512.png",
      "sizes": "512x512",
      "type": "image/png",
      "purpose": "maskable"
    }
  ]
}
```

### Icons

- Provide at least **192×192** and **512×512** PNGs—these are the minimum for installability.
- Add a **`"purpose": "maskable"`** icon so Android can crop it into its adaptive shapes (circle, squircle) without clipping content. Keep important art inside the central "safe zone."
- Use [Maskable.app](https://maskable.app) to preview maskable icons; use a PWA asset generator to produce all sizes from one source.

### Display modes

The `display` field controls how much browser UI is shown:

| Mode          | Appearance                                                                |
| ------------- | ------------------------------------------------------------------------- |
| `fullscreen`  | No browser UI at all; uses the whole screen (games, kiosks).              |
| `standalone`  | Looks like a native app: own window, no address bar. **Most common.**    |
| `minimal-ui`  | Standalone plus a minimal set of navigation controls (back/reload).      |
| `browser`     | A normal browser tab (effectively "not installed").                      |

### Install prompt and `beforeinstallprompt`

Chromium browsers fire a `beforeinstallprompt` event when the app meets installability criteria. You can **capture** it, show your own "Install" button, and trigger the prompt on a user gesture:

```tsx
// src/hooks/useInstallPrompt.ts
import { useEffect, useState } from 'react';

interface BeforeInstallPromptEvent extends Event {
  prompt: () => Promise<void>;
  userChoice: Promise<{ outcome: 'accepted' | 'dismissed' }>;
}

export function useInstallPrompt() {
  const [promptEvent, setPromptEvent] = useState<BeforeInstallPromptEvent | null>(null);
  const [installed, setInstalled] = useState(false);

  useEffect(() => {
    const onBeforeInstall = (e: Event) => {
      e.preventDefault(); // stop the default mini-infobar
      setPromptEvent(e as BeforeInstallPromptEvent);
    };
    const onInstalled = () => {
      setInstalled(true);
      setPromptEvent(null);
    };

    window.addEventListener('beforeinstallprompt', onBeforeInstall);
    window.addEventListener('appinstalled', onInstalled);
    return () => {
      window.removeEventListener('beforeinstallprompt', onBeforeInstall);
      window.removeEventListener('appinstalled', onInstalled);
    };
  }, []);

  const install = async () => {
    if (!promptEvent) return;
    await promptEvent.prompt();
    await promptEvent.userChoice;
    setPromptEvent(null); // a prompt event can only be used once
  };

  return { canInstall: !!promptEvent && !installed, install };
}
```

```tsx
// Usage in a component
function InstallButton() {
  const { canInstall, install } = useInstallPrompt();
  if (!canInstall) return null;
  return <button onClick={install}>Install app</button>;
}
```

!!! note "`beforeinstallprompt` is Chromium-only"
    Safari/Firefox do not fire this event. On iOS, installation is the manual **Share → Add to Home Screen** flow. Detect standalone mode with `window.matchMedia('(display-mode: standalone)').matches` (or `navigator.standalone` on iOS) and show platform-appropriate instructions.

### Handling updates with the virtual module

The plugin exposes a virtual module, `virtual:pwa-register/react`, with a hook that reports update/offline-ready state:

```tsx
// src/components/ReloadPrompt.tsx
import { useRegisterSW } from 'virtual:pwa-register/react';

export default function ReloadPrompt() {
  const {
    offlineReady: [offlineReady, setOfflineReady],
    needRefresh: [needRefresh, setNeedRefresh],
    updateServiceWorker,
  } = useRegisterSW();

  const close = () => {
    setOfflineReady(false);
    setNeedRefresh(false);
  };

  if (!offlineReady && !needRefresh) return null;

  return (
    <div role="alert" className="pwa-toast">
      <span>
        {offlineReady ? 'App ready to work offline.' : 'A new version is available.'}
      </span>
      {needRefresh && (
        <button onClick={() => updateServiceWorker(true)}>Reload</button>
      )}
      <button onClick={close}>Close</button>
    </div>
  );
}
```

!!! tip "Customizing the generated worker"
    `generateSW` (the default) builds the worker from config—great for static caching. When you need custom logic (push handlers, background sync, your own routes), switch to `strategies: 'injectManifest'` and write a `src/sw.ts`. Workbox injects the precache manifest into your file, and you keep full control. Next.js users get the same model via **[Serwist](https://serwist.pages.dev)** (the maintained successor to `next-pwa`).

## 5. Offline Support

Offline support comes from the Service Worker's `fetch` handler plus the **Cache API**, a key-value store of `Request`/`Response` pairs that persists across sessions.

### Caching assets for offline access

Pre-cache the **app shell** (the minimal HTML/CSS/JS to render the UI) at install time. With `vite-plugin-pwa` this is automatic via `globPatterns`. With the raw API, you call `cache.addAll()` in the `install` event (see §2).

```js
// Inspecting the Cache API directly (works in DevTools console too)
const cache = await caches.open('app-shell-v1');
await cache.add('/index.html');           // fetch + store
await cache.put('/data', new Response('{}')); // store a custom response
const hit = await cache.match('/index.html'); // read back
```

### Implementing an offline fallback page

When both the network and cache miss, serve a friendly fallback instead of the browser's dino error. Pre-cache an `offline.html`, then return it for failed navigations:

```js
// sw.js — offline fallback for page navigations
const OFFLINE_URL = '/offline.html';

self.addEventListener('fetch', (event) => {
  // Only handle top-level navigations (HTML documents).
  if (event.request.mode === 'navigate') {
    event.respondWith(
      fetch(event.request).catch(async () => {
        const cache = await caches.open('app-shell-v1');
        return (await cache.match(OFFLINE_URL)) ?? Response.error();
      })
    );
  }
});
```

In `vite-plugin-pwa`, `navigateFallback: '/index.html'` handles SPA routes; for a dedicated offline page, add it to `globPatterns`/`additionalManifestEntries` and reference it in your worker logic.

### Handling offline requests

For API/data requests you have two jobs: serve something useful when offline, and avoid caching things you shouldn't.

- **Detect offline state in React** with the `navigator.onLine` flag and the `online`/`offline` events, and reflect it in the UI:

```tsx
// src/hooks/useOnlineStatus.ts
import { useEffect, useState } from 'react';

export function useOnlineStatus() {
  const [online, setOnline] = useState(navigator.onLine);
  useEffect(() => {
    const up = () => setOnline(true);
    const down = () => setOnline(false);
    window.addEventListener('online', up);
    window.addEventListener('offline', down);
    return () => {
      window.removeEventListener('online', up);
      window.removeEventListener('offline', down);
    };
  }, []);
  return online;
}
```

!!! warning "Never blindly cache POST/PUT/DELETE responses"
    The Cache API is for `GET` requests. Mutating requests should not be served from cache; instead queue them for **Background Sync** (see §6). Also avoid caching authenticated/personalized responses unless you scope and invalidate them carefully.

## 6. Caching Strategies and Background Sync

A **caching strategy** is the policy your `fetch` handler uses to decide between cache and network. Choose per resource type.

### The main strategies

| Strategy                  | Behavior                                                                 | Best for                                  |
| ------------------------- | ------------------------------------------------------------------------ | ----------------------------------------- |
| **Cache First**           | Return cache if present; only hit network on a miss.                     | Hashed/immutable static assets, fonts.    |
| **Network First**         | Try network; fall back to cache on failure/timeout.                      | Frequently-changing API data.             |
| **Stale-While-Revalidate**| Return cache immediately, fetch in background to update for next time.   | Avatars, feeds—speed over freshness.      |
| **Network Only**          | Always go to network.                                                    | Non-cacheable / sensitive requests.       |
| **Cache Only**            | Always serve from cache.                                                 | Assets known to be pre-cached.            |

Raw implementations:

```js
// Cache First
async function cacheFirst(request) {
  const cached = await caches.match(request);
  if (cached) return cached;
  const response = await fetch(request);
  const cache = await caches.open('static-v1');
  cache.put(request, response.clone());
  return response;
}

// Network First (with cache fallback)
async function networkFirst(request) {
  const cache = await caches.open('api-v1');
  try {
    const response = await fetch(request);
    cache.put(request, response.clone());
    return response;
  } catch {
    return (await cache.match(request)) ?? Response.error();
  }
}

// Stale-While-Revalidate
async function staleWhileRevalidate(request) {
  const cache = await caches.open('swr-v1');
  const cached = await cache.match(request);
  const networkPromise = fetch(request).then((response) => {
    cache.put(request, response.clone());
    return response;
  });
  return cached ?? networkPromise; // serve cache now, update in background
}
```

With Workbox / `vite-plugin-pwa`, the same strategies are declarative via `runtimeCaching`:

```ts
// inside VitePWA({ workbox: { ... } })
runtimeCaching: [
  {
    urlPattern: ({ request }) => request.destination === 'image',
    handler: 'StaleWhileRevalidate',
    options: { cacheName: 'images', expiration: { maxEntries: 60 } },
  },
  {
    urlPattern: ({ url }) => url.pathname.startsWith('/api/'),
    handler: 'NetworkFirst',
    options: {
      cacheName: 'api',
      networkTimeoutSeconds: 5,
      expiration: { maxAgeSeconds: 60 * 5 },
    },
  },
],
```

!!! tip "Managing cached data"
    Version your cache names (`static-v1` → `static-v2`) and delete old ones in `activate`. Cap growth with Workbox's `expiration` plugin (`maxEntries` / `maxAgeSeconds`). Remember caches count against the origin's storage quota; request `navigator.storage.persist()` for data you must not lose to eviction.

### Background Sync for deferred requests

**Background Sync** lets you defer an action made while offline and have the browser retry it automatically once connectivity returns—even if the user has navigated away. You register a sync tag from the page; the Service Worker handles the matching `sync` event.

```tsx
// Page: queue a request when the network fails
async function submitWithSync(payload: unknown) {
  await saveToOutbox(payload); // store in IndexedDB
  const reg = await navigator.serviceWorker.ready;
  if ('sync' in reg) {
    await (reg as any).sync.register('outbox-sync');
  } else {
    await flushOutbox(); // fallback: try immediately
  }
}
```

```js
// sw.js: drain the queue when the browser fires 'sync'
self.addEventListener('sync', (event) => {
  if (event.tag === 'outbox-sync') {
    event.waitUntil(flushOutbox()); // re-attempt each queued request
  }
});
```

!!! note "Use Workbox's Background Sync queue"
    Hand-rolling an IndexedDB outbox is error-prone. Workbox's `BackgroundSyncPlugin` provides a battle-tested queue with automatic replay. The Background Sync API is Chromium-only; design a graceful fallback (retry on `online`) for other browsers.

## 7. Web Push Notifications

Push notifications re-engage users even when your app isn't open. They combine three APIs:

- **Notification API** — displays the notification (also works in the foreground).
- **Push API** — lets a server send a message to the Service Worker via a browser **push service**.
- **Web Push protocol + VAPID** — how your server authenticates pushes to the browser's push service.

### The flow at a glance

1. The app asks the user for **notification permission**.
2. The app **subscribes** to push, getting a `PushSubscription` (an endpoint URL + keys).
3. The app sends that subscription to **your server** to store.
4. Your server sends a push (signed with your **VAPID** private key) to the subscription endpoint.
5. The browser wakes the Service Worker with a `push` event; the worker calls `showNotification`.

### VAPID keys

**VAPID** (Voluntary Application Server Identification) is a public/private key pair that identifies your server to the push service. The **public** key goes in the browser subscription; the **private** key stays on your server and signs each push.

```bash
# Generate a VAPID key pair with the web-push CLI (Node)
npx web-push generate-vapid-keys
# → Public Key:  BJ...   (ship to the client)
# → Private Key: 7q...   (keep secret on the server)
```

### Requesting permission and subscribing (client)

```tsx
// src/push/subscribe.ts
const VAPID_PUBLIC_KEY = import.meta.env.VITE_VAPID_PUBLIC_KEY as string;

// Push subscriptions need the key as a Uint8Array.
function urlBase64ToUint8Array(base64: string) {
  const padding = '='.repeat((4 - (base64.length % 4)) % 4);
  const raw = atob((base64 + padding).replace(/-/g, '+').replace(/_/g, '/'));
  return Uint8Array.from([...raw].map((c) => c.charCodeAt(0)));
}

export async function subscribeToPush() {
  if (!('serviceWorker' in navigator) || !('PushManager' in window)) {
    throw new Error('Push is not supported in this browser.');
  }

  const permission = await Notification.requestPermission();
  if (permission !== 'granted') throw new Error('Notification permission denied.');

  const registration = await navigator.serviceWorker.ready;

  const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true, // required by browsers: every push must show UI
    applicationServerKey: urlBase64ToUint8Array(VAPID_PUBLIC_KEY),
  });

  // Persist the subscription on your server.
  await fetch('/api/push/subscribe', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(subscription),
  });

  return subscription;
}
```

!!! warning "Always ask permission from a user gesture"
    Browsers penalize sites that request notification permission on page load—many now block it entirely. Trigger `requestPermission()` from a click (e.g. an "Enable notifications" button) and explain the value first.

### Handling push events in the Service Worker

The `push` event delivers the server's payload; render it with `showNotification`. Handle clicks with `notificationclick` to focus or open your app:

```js
// sw.js (or src/sw.ts with injectManifest)
self.addEventListener('push', (event) => {
  const data = event.data?.json() ?? {};
  const title = data.title ?? 'New notification';
  const options = {
    body: data.body ?? '',
    icon: '/icons/icon-192.png',
    badge: '/icons/badge-72.png',
    data: { url: data.url ?? '/' },
  };
  event.waitUntil(self.registration.showNotification(title, options));
});

self.addEventListener('notificationclick', (event) => {
  event.notification.close();
  const url = event.notification.data?.url ?? '/';
  event.waitUntil(
    self.clients.matchAll({ type: 'window' }).then((clients) => {
      const existing = clients.find((c) => c.url === url);
      if (existing) return existing.focus();
      return self.clients.openWindow(url);
    })
  );
});
```

### Sending push notifications from a server

Use the `web-push` library, which implements the Web Push protocol and VAPID signing for you:

```js
// server/push.js (Node / Express)
import webpush from 'web-push';

webpush.setVapidDetails(
  'mailto:admin@example.com',
  process.env.VAPID_PUBLIC_KEY,
  process.env.VAPID_PRIVATE_KEY
);

export async function sendPush(subscription, payload) {
  try {
    await webpush.sendNotification(
      subscription,
      JSON.stringify(payload) // becomes event.data in the SW
    );
  } catch (err) {
    // 404/410 means the subscription expired — delete it from your DB.
    if (err.statusCode === 404 || err.statusCode === 410) {
      await deleteSubscription(subscription.endpoint);
    } else {
      throw err;
    }
  }
}

// Example route
app.post('/api/notify', async (req, res) => {
  const subs = await getAllSubscriptions();
  await Promise.allSettled(
    subs.map((s) => sendPush(s, { title: 'Hello', body: 'A new update!', url: '/' }))
  );
  res.json({ sent: subs.length });
});
```

!!! note "Prune dead subscriptions"
    Push services return **404/410** when a subscription is no longer valid (user uninstalled, cleared data, or it expired). Delete those from your database on send to keep your subscription list clean and your sends efficient.

## 8. Testing and Debugging PWAs

- **Lighthouse** (Chrome DevTools → Lighthouse) audits installability and best practices.
- **Application panel** (DevTools) inspects the manifest, registered Service Workers, the Cache Storage, and IndexedDB.
- Use **"Update on reload"** and **"Bypass for network"** in the Service Workers section while developing to avoid serving stale workers.
- Toggle **Offline** in the Network panel to test offline behavior and fallbacks.
- `vite-plugin-pwa`'s `devOptions.enabled` lets you exercise the worker during `vite dev`.

!!! tip "The #1 PWA gotcha: stale Service Workers"
    A cached, never-updating Service Worker is the most common PWA frustration. Always version caches, clean up old ones in `activate`, and ship an explicit update flow (the `ReloadPrompt` pattern from §4). When in doubt, "Unregister" in the Application panel and hard-reload.

### 🧭 PWA Core Concepts at a Glance

| 🔹 Concept                | 🧠 Description                                                                                       |
| ------------------------- | --------------------------------------------------------------------------------------------------- |
| Service Worker            | Background script that proxies network requests; enables offline, caching, and push.                |
| Web App Manifest          | JSON describing name, icons, colors, and display mode for installation.                              |
| Scope                     | The URL subtree a Service Worker controls; defined by where the SW file is served.                   |
| Lifecycle                 | `install` → `installed/waiting` → `activate` → `activated` (fetch) → `redundant`.                    |
| Cache API                 | Persistent store of `Request`/`Response` pairs used for offline assets.                              |
| `beforeinstallprompt`     | Chromium event you capture to show a custom install button.                                          |
| Caching strategies        | Cache-first, network-first, stale-while-revalidate, network-only, cache-only.                        |
| Background Sync           | Defers failed/offline requests and retries them when connectivity returns.                          |
| Push API + VAPID          | Server-to-SW messaging; VAPID keys authenticate your server to the push service.                     |
| Notification API          | Displays notifications from a `push` event (or in the foreground).                                   |
| `vite-plugin-pwa`         | Recommended Vite plugin wrapping Workbox to generate the SW and manifest.                            |
| Serwist / `next-pwa`      | The equivalent tooling for Next.js apps.                                                             |
