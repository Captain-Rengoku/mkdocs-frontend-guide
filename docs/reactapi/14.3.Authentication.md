## {++ Fundamentals of Authentication & Authorization ++}

This section covers the essential concepts of authentication and authorization in React and modern web applications.

### Understanding Authentication

Authentication means **verifying who the user is**.  
It ensures the user is a real, valid identity before giving access.

#### Authentication Examples

- Email + password login  
- OTP login  
- Google / GitHub login  
- Magic link login  
- Biometric login (FaceID, fingerprint)

```jsx
// Login.jsx
import { useState } from "react";
import axios from "axios";

export default function Login() {
  const [email, setEmail] = useState("");
  const [password, setPassword] = useState("");

  async function handleLogin(e) {
    e.preventDefault();
    await axios.post("/api/login", { email, password }, { withCredentials: true });
  }

  return (
    <form onSubmit={handleLogin}>
      <input 
        value={email} 
        onChange={(e) => setEmail(e.target.value)} 
        placeholder="email" 
      />
      <input 
        type="password" 
        value={password} 
        onChange={(e) => setPassword(e.target.value)} 
        placeholder="password" 
      />
      <button type="submit">Login</button>
    </form>
  );
}
```

### Understanding Authorization

Authorization means **deciding what an authenticated user is allowed to do**.

It determines:

- Which pages users can view  
- Which actions they can perform  
- What information they can access  

#### Authorization Examples

- Admin can delete users  
- Students can view only their own marks  
- Logged-out users cannot access dashboards  

### Authentication vs Authorization

| Feature  | Authentication    | Authorization          |
| -------- | ----------------- | ---------------------- |
| Meaning  | Verifies identity | Verifies permissions   |
| Question | "Who are you?"    | "What can you access?" |
| Happens  | First             | Second                 |
| Example  | Login form        | Accessing admin panel  |

#### Access Flow Diagram

```text
User → Authentication (identity check) → Authorization (permissions check) → Access Granted
```

### Types of Authentication Methods

#### Session-Based Authentication Overview

- Server stores a session ID  
- Browser stores a session cookie  

```text
Client → sessionID cookie → Server checks session → Access Granted
```

#### Token-Based Authentication Overview (JWT)

- Common in React apps  
- Server generates a JWT  
- Token stored in **HttpOnly cookies**

```text
Client → JWT token → Server verifies signature → Access Granted
```

#### OAuth Login Overview

- Used for “Login with Google”  
- Third-party authentication provider  

#### OpenID Connect Overview

- Extension on OAuth  
- Provides identity info (email, profile, etc.)

#### Passwordless Login Overview

- OTP login  
- Magic link login  
- Biometric login  

#### Multi-Factor Authentication Overview

- Requires more than one verification step (password + OTP)

### SPA Authentication Challenges

React apps run entirely in the browser, so they face unique issues.

#### Token Storage Concerns

- ❌ Avoid using localStorage due to XSS  
- ✔️ Prefer HttpOnly cookies  

#### Route Protection Logic

React must hide private content from unauthenticated users.

!!! warning "Never read the token cookie from JavaScript"
    A common mistake is to check `document.cookie.includes("token=")`. This
    **cannot work** with the secure approach this chapter recommends: the token
    lives in an **HttpOnly cookie**, which is intentionally invisible to
    `document.cookie`. Instead, derive `isAuthenticated` from **React state**
    that is populated by asking the server who you are (a `/me` request). The
    browser still sends the cookie automatically; you just never try to read it.

```jsx
// ProtectedRoute.jsx
import { Navigate } from "react-router-dom";
import { useAuth } from "./auth/AuthContext";

export default function ProtectedRoute({ children }) {
  const { isAuthenticated, isLoading } = useAuth();

  // Wait for the initial /me check before deciding — avoids redirecting
  // a logged-in user on a hard refresh while auth is still hydrating.
  if (isLoading) return <p>Checking session…</p>;

  return isAuthenticated ? children : <Navigate to="/login" replace />;
}
```

The `isAuthenticated` flag comes from an `AuthContext` that calls a `/me`
endpoint on mount. The HttpOnly cookie is sent automatically with that request
(`withCredentials: true`); if the server answers `200`, the user is logged in.
See [State Management for Auth](#state-management-for-auth) for the full context
implementation.

#### Token Expiry Handling

SPAs must manage:

- Token expiration  
- Silent re-login  
- Refresh tokens  

#### UI Flash Problem

Protected UI may show briefly before auth loads.

Solutions:

- Loading placeholders  
- Initial auth checking  

#### CORS Configuration Concerns

When frontend & backend have different origins:

- Enable `withCredentials: true`  
- Configure backend CORS properly

## {++ Authentication Methods in React ++}

This section explains the main ways authentication is implemented in React applications, from traditional cookies to modern passwordless login.

### Cookie-Based Authentication

In cookie-based auth, the **server stores the session**, and the browser stores a **session cookie**.  
React automatically sends cookies with requests (if `withCredentials` is enabled).

#### Cookie Authentication Characteristics

- Server-managed session  
- HttpOnly cookies recommended  
- Automatic cookie handling  
- Protected from XSS (when HttpOnly)

#### Cookie Authentication Example

```js
// login route (server)
res.cookie("sessionId", "abc123", {
  httpOnly: true,
  secure: true,
});
res.json({ message: "Logged in" });
```

### Token-Based Authentication (JWT)

In JWT authentication, the server returns a **signed token**.  
React sends this token back with each request (usually stored in cookies).

#### JWT Authentication Characteristics

- Stateless (no server session)  
- Token contains user data (id, role)  
- Works well with SPAs  
- Must be stored securely (HttpOnly cookie recommended)

#### JWT Authentication Example

```js
// server
const token = createJwt({ id: user.id });
res.cookie("token", token, { httpOnly: true });
```

### OAuth2 Authentication

OAuth2 is used when logging in with **third-party providers** (e.g., Google, GitHub).  
The provider verifies identity and redirects back with an authorization code.

#### OAuth2 Characteristics

- User authenticates at provider  
- Backend exchanges code for access token  
- No password needed in your app  
- Very secure and standardized  

#### OAuth2 Flow Example

```text
React → Provider Login Page → User Authorizes → Backend → Token Issued
```

### OpenID Connect Authentication

OpenID Connect (OIDC) sits on top of OAuth2 and provides **identity information** (email, name, picture).

#### OIDC Characteristics

- Uses ID Token (JWT)  
- Provides verified user identity  
- Ideal for modern “Login with …” systems  

### Social Login Authentication

React apps commonly support login using:

- Google  
- GitHub  
- Facebook  
- Twitter  
- Microsoft  

#### Social Login Characteristics

- Quick login  
- No password needed  
- Uses OAuth or OIDC under the hood  

### Passwordless Authentication

Passwordless login removes passwords entirely.  
User verifies using a one-time method.

#### Passwordless Characteristics

- No password stored  
- No password to forget  
- Uses email or phone  

#### Passwordless Login Example

```text
User enters email → App sends OTP/magic link → User clicks → Logged in
```

### Magic Link and OTP Authentication

Magic links and OTPs are modern and widely used.  
User receives a one-time login link or code.

#### Magic Link / OTP Characteristics

- Secure and user-friendly  
- No passwords required  
- Common in SaaS and mobile apps  

#### OTP Example Flow

```text
Enter phone/email → Receive OTP → Submit OTP → Verify → Logged in
```

## {++ Handling JWT in React ++}

JWT (JSON Web Token) is one of the most common authentication methods in modern React applications.  
This section explains how JWT works, how to store it safely, and how to handle expiry.

### Understanding Access Tokens

Access Tokens are short-lived tokens used to access protected APIs.  
They contain user identity and permissions in a signed format.

#### Access Token Characteristics

- Short-lived (5–15 minutes recommended)  
- Sent automatically by cookies or manually via headers  
- Validated on every request  
- Used only for authorization, not identity storage in React state  

#### Access Token Request Example

```js
// Axios request sending access token (if stored manually)
api.get("/profile", {
  headers: {
    Authorization: `Bearer ${accessToken}`,
  },
});
```

### Understanding Refresh Tokens

Refresh Tokens are long-lived tokens used to **get new access tokens** when they expire.  
React does not store refresh tokens — the **server stores it in HttpOnly cookies**.

#### Refresh Token Characteristics

- Long-lived (days or weeks)  
- Must be stored in HttpOnly cookies  
- Never exposed to JavaScript  
- Used only on a `/refresh` endpoint  

#### Refresh Token Flow Example

```text
Access Token expired → React calls /refresh → Server verifies refresh token → New access token returned
```

### Handling Token Expiry and Auto-Refresh

React must detect when the Access Token expires and automatically refresh it.

#### Auto Refresh Flow

```text
Request → 401 Unauthorized → React triggers /refresh → Retry original request
```

#### Axios Auto-Refresh Example

!!! danger "The naive version causes an infinite loop"
    Retrying on **any** `401` with no guard is dangerous: when `/refresh` itself
    returns `401` (expired/rotated refresh token), the interceptor refreshes,
    retries, gets `401` again, refreshes again… forever. The fix is three rules:
    **(1)** mark each request with a `_retry` flag so it is retried **at most
    once**, **(2)** never try to refresh when the failing request *is* the
    `/refresh` call, and **(3)** if refresh fails, give up and send the user to
    login.

```js
// api.js
import axios from "axios";

const api = axios.create({
  baseURL: "/api",
  withCredentials: true, // always send the HttpOnly cookie
});

api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const original = error.config;
    const status = error.response?.status;

    // Only act on 401s, and only once per request.
    const isUnauthorized = status === 401;
    const alreadyRetried = original?._retry;
    const isRefreshCall = original?.url?.includes("/refresh");

    if (isUnauthorized && !alreadyRetried && !isRefreshCall) {
      original._retry = true; // guard: this request gets exactly one retry
      try {
        await api.post("/refresh"); // server reads refresh cookie, sets new access cookie
        return api(original); // replay the original request
      } catch (refreshError) {
        // Refresh failed (refresh token expired/revoked) → hard logout.
        window.location.assign("/login");
        return Promise.reject(refreshError);
      }
    }

    return Promise.reject(error);
  }
);

export default api;
```

!!! tip "Avoid a refresh stampede"
    If several requests fail with `401` at the same time, the snippet above fires
    one `/refresh` per request. In production, queue concurrent failures behind a
    single in-flight refresh promise — see
    [Refresh-Token Rotation](#refresh-token-rotation-and-the-stampede-problem).

### Secure Token Storage in React

Choosing where to store tokens is critical for security.

#### Token Storage Comparison

**localStorage:**  

- ❌ Vulnerable to XSS  
- ❌ Token can be stolen  
- ❌ Not recommended for JWT auth  

**HttpOnly Cookies:**  

- ✔️ Cannot be read by JavaScript  
- ✔️ Protection from XSS  
- ✔️ Automatically sent with requests  
- ⭐ Recommended for React apps  

### HttpOnly Cookies in JWT Auth

An HttpOnly cookie prevents JavaScript from accessing the token.

#### HttpOnly Cookie Characteristics

- Invisible to JS (`document.cookie` cannot read it)  
- Safe from XSS attacks  
- Sent automatically with `withCredentials: true`  
- Should always be paired with `SameSite` and `Secure` flags  

#### HttpOnly Cookie Example

```js
// server sets HttpOnly cookie
res.cookie("token", jwt, {
  httpOnly: true,
  secure: true,
  sameSite: "strict",
});
res.json({ message: "Logged in" });
```

## {++ Authorization Concepts ++}

Authentication tells you **who** the user is. Authorization decides **what they
may do**. In a React SPA, authorization happens in two places at once:

- **Route level** — can this user even open `/admin`?
- **Component/UI level** — should this user see the "Delete" button?

!!! danger "Client-side authorization is UX, not security"
    Hiding a button or redirecting a route only improves the *experience*. A
    determined user can edit your bundle and call any endpoint. **Every
    authorization rule must be re-enforced on the server.** The frontend rules
    in this section assume the backend independently rejects forbidden requests.

### Role-Based Access Control (RBAC)

Each user has one or more **roles** (`admin`, `editor`, `viewer`). Permissions
are attached to roles, not to individual users. Simple and the most common model.

```ts
// auth/roles.ts
export type Role = "admin" | "editor" | "viewer";

export function hasRole(userRoles: Role[], allowed: Role[]) {
  return userRoles.some((r) => allowed.includes(r));
}
```

```jsx
// Show only to admins/editors
{hasRole(user.roles, ["admin", "editor"]) && (
  <button onClick={publish}>Publish</button>
)}
```

### Permission-Based Access Control (PBAC)

Roles are coarse; permissions are fine-grained (`post:create`, `post:delete`,
`user:ban`). Roles become *bundles* of permissions, and the UI checks the
permission rather than the role — so re-mapping which role can do what never
touches your components.

```ts
// auth/can.ts
export function can(user, permission: string) {
  return user.permissions?.includes(permission) ?? false;
}

// usage
{can(user, "post:delete") && <DeleteButton />}
```

### Attribute-Based Access Control (ABAC)

The most flexible model: decisions depend on **attributes** of the user, the
resource, and the environment ("an editor may edit a post **only if they own
it** and it is **not published**"). Great for ownership rules RBAC can't express.

```ts
// auth/policies.ts
export const canEditPost = (user, post) =>
  user.id === post.authorId && post.status === "draft";

export const canViewSalary = (user, employee) =>
  user.role === "hr" || user.id === employee.managerId;
```

```jsx
{canEditPost(user, post) && <EditPostButton postId={post.id} />}
```

| Model | Granularity | Best for |
| ----- | ----------- | -------- |
| RBAC  | Role        | Small apps, clear job titles |
| PBAC  | Permission  | Apps where roles change often |
| ABAC  | Attribute   | Ownership / contextual rules |

### Route-Level vs Component-Level Authorization

- **Route-level** — wrap a route so the whole page is gated (see
  [Protecting Routes in React](#protecting-routes-in-react)).
- **Component-level** — render-or-hide individual controls with `hasRole` /
  `can`. A small declarative helper keeps JSX clean:

```jsx
// auth/Can.jsx
import { useAuth } from "./AuthContext";

export function Can({ permission, role, children, fallback = null }) {
  const { user } = useAuth();
  if (!user) return fallback;

  const ok =
    (role ? user.roles?.includes(role) : true) &&
    (permission ? user.permissions?.includes(permission) : true);

  return ok ? children : fallback;
}

// usage
<Can permission="user:ban" fallback={<p>Not allowed</p>}>
  <BanUserButton />
</Can>
```

## {++ State Management for Auth ++}

The browser holds the **token** (in an HttpOnly cookie it can't read). React
holds the **derived auth state** — `user`, `isAuthenticated`, `isLoading` —
which it learns by asking the server. This section builds that state.

### Auth Context API

A single `AuthProvider` fetches `/me` on mount and exposes login/logout helpers.

```jsx
// auth/AuthContext.jsx
import { createContext, useContext, useEffect, useState, useCallback } from "react";
import api from "../api"; // axios instance with withCredentials: true

const AuthContext = createContext(null);

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(true); // true during initial /me

  // Hydrate auth state on first load. The HttpOnly cookie rides along
  // automatically; we never read it ourselves.
  useEffect(() => {
    let active = true;
    api
      .get("/me")
      .then((res) => active && setUser(res.data.user))
      .catch(() => active && setUser(null)) // 401 → not logged in
      .finally(() => active && setIsLoading(false));
    return () => {
      active = false;
    };
  }, []);

  const login = useCallback(async (email, password) => {
    await api.post("/login", { email, password }); // server sets the cookie
    const { data } = await api.get("/me"); // pull fresh profile
    setUser(data.user);
  }, []);

  const logout = useCallback(async () => {
    await api.post("/logout"); // server clears the cookie
    setUser(null);
  }, []);

  const value = {
    user,
    isAuthenticated: !!user,
    isLoading,
    login,
    logout,
  };

  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}

export function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error("useAuth must be used within <AuthProvider>");
  return ctx;
}
```

Wrap your app once:

```jsx
// main.jsx
<AuthProvider>
  <BrowserRouter>
    <App />
  </BrowserRouter>
</AuthProvider>
```

### Zustand Auth Store

For larger apps, a Zustand store avoids prop/context re-render concerns and is
easy to read outside React (e.g. inside an axios interceptor).

```ts
// auth/useAuthStore.ts
import { create } from "zustand";
import api from "../api";

interface AuthState {
  user: User | null;
  isLoading: boolean;
  fetchMe: () => Promise<void>;
  login: (email: string, password: string) => Promise<void>;
  logout: () => Promise<void>;
}

export const useAuthStore = create<AuthState>((set) => ({
  user: null,
  isLoading: true,
  fetchMe: async () => {
    try {
      const { data } = await api.get("/me");
      set({ user: data.user });
    } catch {
      set({ user: null });
    } finally {
      set({ isLoading: false });
    }
  },
  login: async (email, password) => {
    await api.post("/login", { email, password });
    const { data } = await api.get("/me");
    set({ user: data.user });
  },
  logout: async () => {
    await api.post("/logout");
    set({ user: null });
  },
}));
```

### TanStack Query for Auth

`/me` is just server state, so TanStack Query is a natural fit — it gives you
caching, `isLoading`, refetch-on-focus, and a cache to invalidate on logout for
free.

```tsx
// auth/useUser.ts
import { useQuery, useQueryClient, useMutation } from "@tanstack/react-query";
import api from "../api";

export function useUser() {
  return useQuery({
    queryKey: ["me"],
    queryFn: async () => (await api.get("/me")).data.user,
    retry: false,        // a 401 is an answer ("not logged in"), not a failure
    staleTime: 5 * 60_000,
  });
}

export function useLogout() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: () => api.post("/logout"),
    onSuccess: () => qc.clear(), // drop every cached query, including ["me"]
  });
}
```

```tsx
function App() {
  const { data: user, isLoading } = useUser();
  if (isLoading) return <Splash />;
  return user ? <Dashboard /> : <Login />;
}
```

### Persisting Auth and Hydration

!!! note "Don't persist the user object to localStorage"
    With HttpOnly cookies you do **not** need to persist auth yourself — the
    cookie *is* the persistence. On every page load, call `/me`; the cookie
    proves the session and the server returns the fresh profile. Mirroring the
    user into localStorage only risks showing stale data after a server-side
    revocation. The single source of truth is the cookie + the `/me` response.

The brief window between "page loaded" and "`/me` resolved" is **hydration**.
Always render a loading state during it so you never flash the wrong UI:

```jsx
function App() {
  const { isLoading } = useAuth();
  if (isLoading) return <FullPageSpinner />;
  return <Routes>{/* ... */}</Routes>;
}
```

## {++ Protecting Routes in React ++}

With auth state in place, route guards become small wrapper components. These
examples use **React Router v6/v7**.

### Private (Protected) Routes

Already shown in [Route Protection Logic](#route-protection-logic): wait for
hydration, then render the page or redirect. Use the `<Outlet />` form to guard
a whole branch of nested routes at once:

```jsx
// routes/RequireAuth.jsx
import { Navigate, Outlet, useLocation } from "react-router-dom";
import { useAuth } from "../auth/AuthContext";

export default function RequireAuth() {
  const { isAuthenticated, isLoading } = useAuth();
  const location = useLocation();

  if (isLoading) return <FullPageSpinner />;

  // Remember where the user was headed so we can return them after login.
  return isAuthenticated ? (
    <Outlet />
  ) : (
    <Navigate to="/login" replace state={{ from: location }} />
  );
}
```

### Public-Only Routes

Stop already-logged-in users from seeing `/login` or `/register`:

```jsx
// routes/PublicOnly.jsx
import { Navigate, Outlet } from "react-router-dom";
import { useAuth } from "../auth/AuthContext";

export default function PublicOnly() {
  const { isAuthenticated, isLoading } = useAuth();
  if (isLoading) return null;
  return isAuthenticated ? <Navigate to="/dashboard" replace /> : <Outlet />;
}
```

### Role-Based Routes

Compose on top of `RequireAuth` to gate by role:

```jsx
// routes/RequireRole.jsx
import { Navigate, Outlet } from "react-router-dom";
import { useAuth } from "../auth/AuthContext";

export default function RequireRole({ allow }) {
  const { user } = useAuth();
  const ok = user?.roles?.some((r) => allow.includes(r));
  return ok ? <Outlet /> : <Navigate to="/403" replace />;
}
```

### Wiring the Router

```jsx
// App.jsx
<Routes>
  {/* public */}
  <Route element={<PublicOnly />}>
    <Route path="/login" element={<Login />} />
    <Route path="/register" element={<Register />} />
  </Route>

  {/* authenticated */}
  <Route element={<RequireAuth />}>
    <Route path="/dashboard" element={<Dashboard />} />
    <Route path="/profile" element={<Profile />} />

    {/* admin-only branch */}
    <Route element={<RequireRole allow={["admin"]} />}>
      <Route path="/admin" element={<AdminPanel />} />
    </Route>
  </Route>

  <Route path="/403" element={<Forbidden />} />
</Routes>
```

### Redirects with Return-To

After a successful login, send the user back to the page that bounced them:

```jsx
// Login.jsx (relevant part)
import { useNavigate, useLocation } from "react-router-dom";
import { useAuth } from "../auth/AuthContext";

function Login() {
  const navigate = useNavigate();
  const location = useLocation();
  const { login } = useAuth();
  const from = location.state?.from?.pathname || "/dashboard";

  async function handleLogin(email, password) {
    await login(email, password);
    navigate(from, { replace: true }); // back where they wanted to go
  }
  // ...
}
```

## {++ Secure API Communication ++}

### Attaching Credentials

With HttpOnly cookies you attach **nothing manually** — the browser sends the
cookie as long as the axios instance opts in:

```js
const api = axios.create({ baseURL: "/api", withCredentials: true });
```

If you are forced into the header approach (token in memory, *not* localStorage),
attach it in a request interceptor — but cookies remain the safer default:

```js
api.interceptors.request.use((config) => {
  const token = getAccessTokenFromMemory(); // module variable, never localStorage
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});
```

### CSRF Protection with Cookies

Cookie auth is the right call for XSS safety, but cookies are sent automatically
on *cross-site* requests too — which is exactly what **CSRF** exploits. Two
defenses, used together:

1. **`SameSite=Strict` (or `Lax`)** on the auth cookie blocks most cross-site
   sends. This alone covers a lot of cases.
2. **Double-submit CSRF token** for state-changing requests: the server sets a
   readable (non-HttpOnly) `XSRF-TOKEN` cookie; the client echoes it back in a
   header. axios does this automatically via `xsrfCookieName` / `xsrfHeaderName`.

```js
const api = axios.create({
  baseURL: "/api",
  withCredentials: true,
  xsrfCookieName: "XSRF-TOKEN",   // axios reads this cookie...
  xsrfHeaderName: "X-XSRF-TOKEN", // ...and sends it as this header
});
```

!!! note "Why double-submit works"
    An attacker's site can *trigger* a request that carries your cookies, but it
    **cannot read** the `XSRF-TOKEN` cookie value (same-origin policy) to place
    it in the header. The server requires header == cookie, so forged requests
    fail.

### Refresh-Token Rotation and the Stampede Problem

A secure setup uses a **short-lived access token** and a **long-lived refresh
token**, and **rotates** the refresh token on every use (each `/refresh` issues
a new refresh token and invalidates the old one — a reused old token signals
theft and revokes the whole family).

The interceptor from [Axios Auto-Refresh Example](#axios-auto-refresh-example)
already retries safely once. The remaining issue: if ten requests fail with
`401` at the same instant, you fire ten `/refresh` calls. Queue them behind one
shared promise instead:

```js
// api.js (refresh queue)
let refreshPromise = null;

api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const original = error.config;
    const is401 = error.response?.status === 401;
    const isRefresh = original?.url?.includes("/refresh");

    if (is401 && !original._retry && !isRefresh) {
      original._retry = true;

      // Reuse one in-flight refresh for all concurrent failures.
      refreshPromise = refreshPromise ?? api.post("/refresh");
      try {
        await refreshPromise;
        return api(original);
      } catch (e) {
        window.location.assign("/login");
        return Promise.reject(e);
      } finally {
        refreshPromise = null; // reset for the next expiry cycle
      }
    }
    return Promise.reject(error);
  }
);
```

## {++ Security Best Practices ++}

### Token Storage Tradeoffs

| Storage | XSS-readable? | CSRF-exposed? | Verdict |
| ------- | ------------- | ------------- | ------- |
| `localStorage` / `sessionStorage` | ✔️ Yes (any injected script reads it) | ❌ No | **Avoid** |
| In-memory (JS variable) | Only while running; lost on refresh | ❌ No | OK for access token |
| **HttpOnly cookie** | ❌ No | ✔️ Yes → mitigate with SameSite + CSRF token | **Recommended** |

The practical winner: **access + refresh tokens in HttpOnly, `Secure`,
`SameSite` cookies**, with CSRF defenses for mutations.

### XSS — the Real Threat to Tokens

If an attacker can run JavaScript in your origin, HttpOnly cookies save your
*token* but the session is still abusable (the attacker rides the cookie via
your own `api` instance). So XSS prevention is non-negotiable:

- React escapes by default — **never** use `dangerouslySetInnerHTML` with
  untrusted input; sanitize with DOMPurify if you must.
- Apply a strict **Content-Security-Policy** header.
- Audit third-party scripts; they run with full access to your origin.

### Short-Lived Access + Refresh Tokens

- **Access token: 5–15 minutes.** Limits the damage window if it leaks.
- **Refresh token: days/weeks, rotated on every use, revocable server-side.**
- Refresh transparently via the interceptor; the user never notices.

### Logout and Revocation

Because the token is HttpOnly, the client **cannot** delete it — logout must hit
the server, which clears the cookie *and* invalidates the refresh token.

```js
// server logout
app.post("/logout", (req, res) => {
  revokeRefreshToken(req.cookies.refreshToken); // kill it server-side
  res.clearCookie("token");
  res.clearCookie("refreshToken");
  res.json({ message: "Logged out" });
});
```

!!! warning "Stateless JWTs can't be force-revoked"
    A signed access token is valid until it expires, even after "logout." Keep
    access tokens short, and maintain server-side state for **refresh** tokens
    (a denylist or rotation family) so a logout/breach response is immediate.

## {++ Building a Custom Auth System ++}

A complete email/password flow tying the chapter together: context for state,
cookies for tokens, an interceptor for refresh, and guarded routes.

### 1. The axios instance

```js
// api.js — single shared instance (interceptor from §Secure API Communication)
import axios from "axios";
export default axios.create({ baseURL: "/api", withCredentials: true });
```

### 2. Signup with validation

```jsx
// Register.jsx
import { useState } from "react";
import api from "./api";
import { useNavigate } from "react-router-dom";

export default function Register() {
  const [form, setForm] = useState({ email: "", password: "" });
  const [error, setError] = useState(null);
  const navigate = useNavigate();

  function validate({ email, password }) {
    if (!/^[^@\s]+@[^@\s]+\.[^@\s]+$/.test(email)) return "Invalid email";
    if (password.length < 8) return "Password must be 8+ characters";
    return null;
  }

  async function handleSubmit(e) {
    e.preventDefault();
    const msg = validate(form);
    if (msg) return setError(msg);
    try {
      await api.post("/register", form); // server creates user + sets cookies
      navigate("/dashboard");
    } catch (err) {
      setError(err.response?.data?.message ?? "Registration failed");
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        value={form.email}
        onChange={(e) => setForm({ ...form, email: e.target.value })}
        placeholder="email"
      />
      <input
        type="password"
        value={form.password}
        onChange={(e) => setForm({ ...form, password: e.target.value })}
        placeholder="password"
      />
      {error && <p role="alert">{error}</p>}
      <button type="submit">Create account</button>
    </form>
  );
}
```

### 3. Global state + auto-login on refresh

Use the `AuthProvider` from
[State Management for Auth](#state-management-for-auth). Its `useEffect` call to
`/me` *is* the auto-login: on every page load the HttpOnly cookie is validated
and the user is restored — no extra code, nothing in localStorage.

### 4. Login

```jsx
// Login.jsx — uses login() + return-to from earlier sections
const { login } = useAuth();
await login(email, password);
navigate(location.state?.from?.pathname || "/dashboard", { replace: true });
```

### 5. Logout

```jsx
function LogoutButton() {
  const { logout } = useAuth();
  const navigate = useNavigate();
  return (
    <button
      onClick={async () => {
        await logout();          // server clears + revokes
        navigate("/login", { replace: true });
      }}
    >
      Log out
    </button>
  );
}
```

That's an end-to-end system: register → cookie set → state hydrated via `/me` →
routes guarded → access token auto-refreshed → logout revokes server-side.

## {++ Handling Edge Cases ++}

### Token Expiry Mid-Session

The user is filling a form when the access token expires. They should *not* be
kicked out — the refresh interceptor transparently mints a new token and replays
the failed request. Only if **refresh** also fails do you redirect to login.
This is exactly the behaviour built in
[Secure API Communication](#secure-api-communication).

### Multi-Tab Synchronization

Log out in one tab and the others should follow. Broadcast the change and have
every tab listen — `BroadcastChannel` (or a `localStorage` "signal" key for the
classic `storage`-event approach):

```jsx
// auth/useAuthSync.js
import { useEffect } from "react";
import { useAuth } from "./AuthContext";

const channel = new BroadcastChannel("auth");

export function broadcastLogout() {
  channel.postMessage("logout");
}

export function useAuthSync() {
  const { logout } = useAuth();
  useEffect(() => {
    const onMessage = (e) => {
      if (e.data === "logout") logout(); // clears state in THIS tab too
    };
    channel.addEventListener("message", onMessage);
    return () => channel.removeEventListener("message", onMessage);
  }, [logout]);
}
```

!!! note "storage-event fallback"
    Older browsers without `BroadcastChannel` can flip a flag —
    `localStorage.setItem("logout", Date.now())` — and listen via
    `window.addEventListener("storage", ...)`. The `storage` event fires in
    *other* tabs only, which is precisely what cross-tab sync needs. Store a
    timestamp signal, never the token itself.

### Silent Refresh

Rather than wait for a `401`, refresh **proactively** just before expiry so the
user never sees a failed request. Schedule it from the access token's lifetime
(returned by `/login`/`/me`, e.g. `expiresIn`):

```js
// auth/scheduleSilentRefresh.js
import api from "../api";

export function scheduleSilentRefresh(expiresInMs, onFail) {
  // refresh 60s before the access token actually expires
  const delay = Math.max(expiresInMs - 60_000, 0);
  return setTimeout(async () => {
    try {
      await api.post("/refresh");
    } catch {
      onFail?.();
    }
  }, delay);
}
```

### Preventing the Protected-UI Flash

Never render protected content before hydration finishes. Gate on `isLoading`
(see [Persisting Auth and Hydration](#persisting-auth-and-hydration)) so the
dashboard never flickers for a split second before a redirect.

## {++ Advanced Topics ++}

### OAuth 2.0 / OIDC Flow Overview

For third-party login, the modern, SPA-safe flow is **Authorization Code with
PKCE**. The token exchange should happen on **your backend** so client secrets
and tokens never touch the browser:

```text
1. User clicks "Login with Google"
2. Browser → redirected to Google's consent screen (with PKCE code_challenge)
3. User approves → Google redirects back to /callback?code=...
4. Your BACKEND exchanges code (+ code_verifier) for tokens
5. Backend sets your own HttpOnly session cookie
6. Browser is now logged into YOUR app (no provider token in JS)
```

!!! warning "Don't do the token exchange in the browser"
    The SPA's job is to start the redirect and hand the `code` to your backend.
    Letting the frontend hold the provider's access token reintroduces the very
    XSS exposure HttpOnly cookies were chosen to avoid.

### Social Login (button → backend)

```jsx
// the frontend just kicks off the server-driven flow
function GoogleLoginButton() {
  return (
    <a href="/api/auth/google">
      <button>Continue with Google</button>
    </a>
  );
}
// /api/auth/google (backend) redirects to Google;
// /api/auth/google/callback (backend) finishes the exchange and sets the cookie.
```

### Single Sign-On (SSO)

SSO lets one identity provider authenticate the user across many apps (typically
**SAML** or **OIDC** in enterprises). To the React app it looks like any OAuth
login — you redirect to the IdP and receive a session cookie back; the IdP keeps
the user signed in across the suite.

### Authorization Guards as HOCs

A higher-order component is an alternative to wrapper routes when you want to
guard a *component* anywhere it's used:

```jsx
// auth/withRole.jsx
import { useAuth } from "./AuthContext";
import { Navigate } from "react-router-dom";

export function withRole(Component, allowed) {
  return function Guarded(props) {
    const { user, isLoading } = useAuth();
    if (isLoading) return <FullPageSpinner />;
    const ok = user?.roles?.some((r) => allowed.includes(r));
    return ok ? <Component {...props} /> : <Navigate to="/403" replace />;
  };
}

const AdminPanel = withRole(AdminPanelBase, ["admin"]);
```

### Server-Side Authorization (Next.js)

In Next.js App Router, check auth in a Server Component / middleware **before**
the page renders — no flash, no client-side bypass:

```tsx
// app/admin/page.tsx (Server Component)
import { cookies } from "next/headers";
import { redirect } from "next/navigation";
import { verifySession } from "@/lib/auth";

export default async function AdminPage() {
  const session = await verifySession(cookies().get("token")?.value);
  if (!session || !session.roles.includes("admin")) redirect("/login");
  return <AdminDashboard />;
}
```

## {++ Testing Auth ++}

Use **Vitest/Jest + React Testing Library** for components and **MSW** to mock
the auth API. The goal: assert behaviour (redirects, what renders) without a
real backend.

### Mocking Auth State

Render components inside a fake provider so you control `isAuthenticated`/`user`:

```jsx
// test-utils.jsx
import { AuthContext } from "../auth/AuthContext";

export function renderWithAuth(ui, { auth } = {}) {
  const value = {
    user: null,
    isAuthenticated: false,
    isLoading: false,
    login: vi.fn(),
    logout: vi.fn(),
    ...auth,
  };
  return render(
    <AuthContext.Provider value={value}>{ui}</AuthContext.Provider>
  );
}
```

!!! note
    Export the `AuthContext` object (not just the provider) so tests can supply
    their own value directly.

### Protected-Route Tests

```jsx
// RequireAuth.test.jsx
import { screen } from "@testing-library/react";
import { MemoryRouter, Routes, Route } from "react-router-dom";
import RequireAuth from "../routes/RequireAuth";
import { renderWithAuth } from "../test-utils";

function setup(auth) {
  return renderWithAuth(
    <MemoryRouter initialEntries={["/dashboard"]}>
      <Routes>
        <Route element={<RequireAuth />}>
          <Route path="/dashboard" element={<p>Dashboard</p>} />
        </Route>
        <Route path="/login" element={<p>Login Page</p>} />
      </Routes>
    </MemoryRouter>,
    { auth }
  );
}

test("redirects anonymous users to login", () => {
  setup({ isAuthenticated: false });
  expect(screen.getByText("Login Page")).toBeInTheDocument();
});

test("renders the page for authenticated users", () => {
  setup({ isAuthenticated: true, user: { id: 1 } });
  expect(screen.getByText("Dashboard")).toBeInTheDocument();
});

test("shows a spinner while auth is hydrating", () => {
  setup({ isLoading: true });
  expect(screen.queryByText("Dashboard")).not.toBeInTheDocument();
  expect(screen.queryByText("Login Page")).not.toBeInTheDocument();
});
```

### Mocking the Auth API with MSW

For integration tests of the real `AuthProvider`, intercept HTTP instead of
faking context:

```js
// mocks/handlers.js
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("/api/me", () =>
    HttpResponse.json({ user: { id: 1, roles: ["admin"] } })
  ),
  http.post("/api/login", () => HttpResponse.json({ ok: true })),
  http.post("/api/logout", () => HttpResponse.json({ ok: true })),
];

// return a 401 in a specific test to assert the logged-out path:
// server.use(http.get("/api/me", () => new HttpResponse(null, { status: 401 })));
```

!!! tip "Test the loop, not the token"
    You can't read HttpOnly cookies in tests either — and that's fine. Assert
    the *observable behaviour*: a `401` from `/me` shows the login screen, a
    `200` shows the dashboard, and a mid-session `401` triggers exactly one
    `/refresh` then a retry.

### What to Cover

- `useAuth` throws outside its provider.
- Login success navigates to the return-to path.
- Logout clears state and the TanStack Query cache.
- The refresh interceptor retries **once** and stops on a `/refresh` 401.
- Role guards redirect to `/403` for the wrong role.
