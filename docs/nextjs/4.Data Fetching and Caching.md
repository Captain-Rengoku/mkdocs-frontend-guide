# :simple-nextdotjs: Data Fetching & Caching

Data fetching in the App Router is deceptively simple — you `await` data right inside a Server Component — but the *caching* behind it is where most of the surprises (and the performance) live. Next.js layers **four distinct caches** between your `fetch` call and the user's screen, and the defaults for those caches **changed significantly in Next.js 15**.

This chapter covers fetching in Server Components, the full four-cache model, every flavour of revalidation, static generation with `generateStaticParams` and ISR, the newer `unstable_cache` / `'use cache'` directions, when to drop down to client-side fetching with SWR or TanStack Query, and how to handle errors and missing data gracefully.

> 📚 Builds on [React SSR & SSG](../react/12.React%20SSR%20and%20SSG.md) and the *Rendering* chapter. For the canonical reference: [nextjs.org/docs — Caching](https://nextjs.org/docs/app/building-your-application/caching).

---

## 1. Fetching in Server Components

In the App Router, the default data-fetching mechanism is **`async`/`await` directly inside a Server Component**. There is no `getServerSideProps`, no `getStaticProps`, no hook — the component itself is `async` and you `await` the data it needs.

```tsx
// app/posts/page.tsx — a Server Component (no "use client")
type Post = { id: string; title: string; excerpt: string };

export default async function PostsPage() {
  const res = await fetch("https://api.example.com/posts");
  if (!res.ok) throw new Error("Failed to load posts");
  const posts: Post[] = await res.json();

  return (
    <main>
      <h1>Posts</h1>
      <ul>
        {posts.map((p) => (
          <li key={p.id}>
            <h2>{p.title}</h2>
            <p>{p.excerpt}</p>
          </li>
        ))}
      </ul>
    </main>
  );
}
```

Because this runs **only on the server**, you can also skip HTTP entirely and call your database, ORM, or internal services directly — credentials and query code never reach the browser.

```tsx
// app/dashboard/page.tsx
import { db } from "@/lib/db";

export default async function Dashboard() {
  const stats = await db.metric.findMany({ take: 20 }); // direct DB access
  return <StatsTable rows={stats} />;
}
```

### Parallel vs. sequential fetching

Each `await` is a potential waterfall. If two requests don't depend on each other, fire them **in parallel** with `Promise.all` instead of awaiting one after the other.

```tsx
// ❌ Sequential — total time = artist time + albums time (a waterfall)
const artist = await getArtist(id);
const albums = await getAlbums(id);

// ✅ Parallel — total time = max(artist, albums)
const [artist, albums] = await Promise.all([getArtist(id), getAlbums(id)]);
```

For independent slow regions, prefer **streaming**: render each in its own `<Suspense>` boundary so the page shell flushes immediately and each region streams in when ready (covered in the *Rendering* chapter, and again under error handling below).

!!! tip "Fetch where the data is used"
    You don't need to lift fetching to the top of the tree and thread props down. Fetch in the component that actually needs the data — even if two sibling components request the *same* URL, **Request Memoization** (next section) collapses them into one call. Co-locating fetches keeps components self-contained.

## 2. The Next.js caching model

Next.js sits four caches between your code and the response. Understanding which one is doing what is the key to predictable behaviour:

| Cache | What it stores | Where | Lifetime | Cleared by |
| ----- | -------------- | ----- | -------- | ---------- |
| **Request Memoization** | Return value of `fetch` (and `React.cache`) | Server | A single render pass | Automatically, when the render finishes |
| **Data Cache** | Result of data fetches | Server (persistent) | Across requests & deploys | `revalidate`, `revalidateTag`, `revalidatePath` |
| **Full Route Cache** | Rendered HTML + RSC payload of a route | Server (persistent) | Across requests, until revalidated | Revalidating its data, redeploy |
| **Router Cache** | RSC payload of visited routes | Client (in-memory) | Session / a short time | `router.refresh()`, navigation, server action revalidation |

They run in roughly that order: a `fetch` first checks **memoization** (same render), then the **Data Cache** (persistent), and the rendered output lands in the **Full Route Cache** (server) and **Router Cache** (client).

### 2.1 Request Memoization

During a **single render pass**, if you call `fetch` with the same URL and options more than once, Next.js runs it **only once** and replays the result to every caller. This is React's deduplication, not a persistent cache — it's discarded as soon as the render completes.

```tsx
// Both of these run during the same request → fetch executes ONCE
async function getUser(id: string) {
  const res = await fetch(`https://api.example.com/users/${id}`);
  return res.json();
}

// Called in the layout...
const user = await getUser("42");
// ...and again in a deeply nested component — no second network call.
const sameUser = await getUser("42");
```

Memoization only applies to `GET` `fetch` calls within the React render. For non-`fetch` work (a DB query you want deduped within a render), wrap it in React's `cache`:

```tsx
import { cache } from "react";
import { db } from "@/lib/db";

// Deduped per-request, just like fetch memoization
export const getProject = cache(async (id: string) => {
  return db.project.findUnique({ where: { id } });
});
```

### 2.2 The Data Cache

The **Data Cache** is a *persistent* server-side store that survives across requests and even deployments. This is the cache whose **default flipped in Next.js 15**.

!!! danger "Next.js 15: `fetch` is NOT cached by default"
    This is the single most important change to internalize.

    - **Next.js 14 and earlier:** `fetch()` defaulted to `cache: 'force-cache'` — results were cached in the Data Cache *indefinitely* unless you opted out. People were frequently surprised by stale data.
    - **Next.js 15:** `fetch()` defaults to **`cache: 'no-store'`** — it is **not** cached, and the request runs every time. You must **opt in** to caching.

    The same uncached-by-default shift applies to **GET Route Handlers** and to **client-side navigations** (the Router Cache — see §2.4). Always set caching **explicitly** rather than relying on whatever the installed version's default happens to be.

```tsx
// Next.js 15 defaults & how to opt back into caching:

// (default) Not cached — runs on every request → dynamic / SSR-like
await fetch("https://api.example.com/feed");
await fetch("https://api.example.com/feed", { cache: "no-store" }); // explicit, same thing

// Cached indefinitely in the Data Cache → static / SSG-like
await fetch("https://api.example.com/page", { cache: "force-cache" });

// Cached, but revalidated in the background at most every 60s → ISR
await fetch("https://api.example.com/products", { next: { revalidate: 60 } });

// Cached AND tagged for targeted on-demand invalidation
await fetch("https://api.example.com/products", {
  next: { tags: ["products"], revalidate: 3600 },
});
```

!!! note "Reading `cookies()`/`headers()` forces dynamic"
    Even with `force-cache` fetches, calling a dynamic API like `await cookies()`, `await headers()`, `await draftMode()`, or reading `searchParams` opts the **route** into dynamic rendering. The Data Cache entry can still be reused, but the route's HTML won't be statically frozen. (In Next.js 15 these are all async — `await` them.)

### 2.3 The Full Route Cache

When a route renders **without any dynamic input**, Next.js caches its rendered output (the HTML and the RSC payload) at **build time** into the **Full Route Cache**. Subsequent requests are served from this cache with no re-render — this is what makes a route "static."

A route is statically rendered (and thus Full-Route-Cached) **unless** it:

- uses a dynamic function (`cookies()`, `headers()`, `connection()`, uncached `searchParams`), or
- contains an **uncached** `fetch` (the Next.js 15 default), or
- sets `export const dynamic = "force-dynamic"` (or `revalidate = 0`).

```tsx
// app/about/page.tsx — fully static, lands in the Full Route Cache at build
export default function About() {
  return <article>Static marketing copy, served from cache.</article>;
}
```

```tsx
// Force a route to render dynamically on every request (opt out of the Full Route Cache)
export const dynamic = "force-dynamic";
// Or force it static and error if anything dynamic is used:
export const dynamic = "force-static";
```

The Full Route Cache is invalidated when the underlying Data Cache entries are revalidated, or on redeploy.

### 2.4 The Router Cache (client-side)

The **Router Cache** lives **in the browser**. As the user navigates with `<Link>` or `router.push`, Next.js stores the RSC payload of visited segments so that going *back* (or revisiting) is instant — no server round-trip.

!!! warning "Next.js 15: client navigations are not cached by default"
    In Next.js 14, the Router Cache held **page** segments for a default `staleTime` of 30 seconds even for dynamic pages, which led to users seeing stale data after navigating. In **Next.js 15**, the default `staleTime` for dynamically-rendered pages is **`0`** — pages are refetched on navigation by default. (Statically-rendered pages still cache for ~5 minutes.) You can tune this explicitly:

```ts
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30, // cache dynamic pages in the Router Cache for 30s
      static: 180, // and static pages for 180s
    },
  },
};

export default nextConfig;
```

The Router Cache is cleared by `router.refresh()`, by a Server Action that calls `revalidatePath`/`revalidateTag`, or on a full page reload.

### Putting the four together

```text
fetch(url)
  │
  ├─▶ Request Memoization   (same render only — dedupes duplicate calls)
  │
  ├─▶ Data Cache            (persistent — survives requests/deploys; opt-in in v15)
  │
  ▼
render route
  │
  ├─▶ Full Route Cache      (server — cached HTML/RSC for static routes)
  │
  ▼
client navigation
  │
  └─▶ Router Cache          (browser — instant back/forward; v15 default 0 for dynamic)
```

## 3. Revalidation

Revalidation is how cached data becomes fresh again without a full rebuild. There are two models: **time-based** and **on-demand**.

### 3.1 Time-based revalidation

Set a `revalidate` window and Next.js serves the cached value, then regenerates it in the background after the window elapses — the *stale-while-revalidate* pattern. The first visitor after expiry gets the stale page instantly; the *next* visitor gets fresh content.

```tsx
// Per-fetch — only this request revalidates on its own clock
await fetch("https://api.example.com/products", { next: { revalidate: 60 } });
```

```tsx
// Per-segment — applies to the whole route (and is the floor for fetches in it)
// app/products/page.tsx
export const revalidate = 3600; // seconds

export default async function Products() {
  const products = await fetch("https://api.example.com/products", {
    cache: "force-cache",
  }).then((r) => r.json());
  return <ProductList products={products} />;
}
```

!!! note "How segment vs. fetch `revalidate` interact"
    The **lowest** `revalidate` wins for the route's freshness. A segment `revalidate = 3600` with a fetch at `revalidate: 60` means that fetch refreshes every 60s, but the *route's* Full Route Cache follows the segment value. Setting `export const revalidate = 0` makes the route fully dynamic.

### 3.2 On-demand revalidation

When data changes on an *event* (a CMS publish, a successful mutation) rather than on a schedule, invalidate exactly what changed with `revalidatePath` or `revalidateTag`. Both come from `next/cache` and run on the server only — in a Route Handler or a Server Action.

```ts
// app/api/revalidate/route.ts — e.g. called by a CMS webhook
import { revalidatePath, revalidateTag } from "next/cache";
import { NextResponse } from "next/server";

export async function POST(request: Request) {
  const secret = new URL(request.url).searchParams.get("secret");
  if (secret !== process.env.REVALIDATE_SECRET) {
    return NextResponse.json({ message: "Invalid token" }, { status: 401 });
  }

  const { path, tag } = await request.json();
  if (path) revalidatePath(path); // refresh a specific route, e.g. "/products"
  if (tag) revalidateTag(tag);     // refresh every fetch tagged with this tag

  return NextResponse.json({ revalidated: true, now: Date.now() });
}
```

- **`revalidateTag(tag)`** — purges every Data Cache entry tagged with `tag`. Tag your fetches (`next: { tags: ["products"] }`) and you can invalidate a whole logical group from anywhere. This is the most precise tool.
- **`revalidatePath(path)`** — purges the cached data and route for a given path. Pass a route pattern with a type for dynamic segments: `revalidatePath("/blog/[slug]", "page")`.

```ts
// Inside a Server Action: revalidate right after a mutation
"use server";
import { revalidateTag } from "next/cache";

export async function createProduct(data: FormData) {
  await db.product.create({ data: parse(data) });
  revalidateTag("products"); // any page listing products is now stale → refreshes
}
```

!!! tip "Tag, then invalidate by tag"
    Reaching for `revalidatePath` everywhere couples your cache logic to your URL structure. Prefer tagging fetches by the *data* they return (`tags: ["products"]`, `tags: [`product:${id}`]`) and invalidating by tag — it's decoupled from routing and far easier to reason about as the app grows.

## 4. Static generation: `generateStaticParams` & ISR

### 4.1 `generateStaticParams` (SSG for dynamic routes)

To pre-render a *set* of dynamic routes at build time (one page per blog post, per product, etc.), export `generateStaticParams`. It's the App Router replacement for the Pages Router's `getStaticPaths`.

```tsx
// app/blog/[slug]/page.tsx
type Post = { slug: string; title: string; body: string };

// 1. Which slugs to pre-render at build time
export async function generateStaticParams() {
  const posts: Post[] = await fetch("https://api.example.com/posts").then((r) =>
    r.json()
  );
  return posts.map((post) => ({ slug: post.slug }));
}

// 2. Render each slug (runs per-slug at build, async params in v15)
export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params; // Next.js 15: params is a Promise
  const post: Post = await fetch(`https://api.example.com/posts/${slug}`, {
    cache: "force-cache",
  }).then((r) => r.json());

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  );
}
```

!!! note "`params` & `searchParams` are async in Next.js 15"
    In Next.js 15, `params`, `searchParams`, `cookies()`, and `headers()` are **Promises** — `await` them (or read with `use()` in a Client Component). In Next.js 14 they were synchronous. All examples here use the v15 async form.

### 4.2 `dynamicParams` — handling the long tail

By default `dynamicParams = true`: a slug **not** returned by `generateStaticParams` is rendered **on first request** and then cached (on-demand ISR). Set it to `false` to return a 404 for any unlisted slug.

```tsx
export const dynamicParams = false; // unlisted slugs → 404 instead of on-demand render
```

This lets you pre-build only your most popular pages and generate the rest lazily — keep the `generateStaticParams` list bounded and let `dynamicParams` cover the tail.

### 4.3 Incremental Static Regeneration (ISR)

ISR = static pages that **regenerate in the background**. Add `revalidate` to a statically-generated route and you have it:

```tsx
// app/products/[id]/page.tsx
export const revalidate = 600; // regenerate at most every 10 minutes

export async function generateStaticParams() {
  const ids: string[] = await getTopProductIds();
  return ids.map((id) => ({ id }));
}

export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const product = await fetch(`https://api.example.com/products/${id}`, {
    next: { revalidate: 600, tags: [`product:${id}`] },
  }).then((r) => r.json());
  return <ProductDetail product={product} />;
}
```

You get CDN speed plus data that stays reasonably fresh. Combine the time-based `revalidate` with tag-based on-demand invalidation (`revalidateTag(`product:${id}`)`) so a product edit refreshes instantly *and* a daily window catches anything that slipped through.

### 4.4 `unstable_cache` for non-`fetch` data

`fetch` caching only covers HTTP. To put a **direct DB query** (or any async function) into the **Data Cache** with the same `revalidate`/`tags` semantics, wrap it in `unstable_cache`.

```ts
// lib/products.ts
import { unstable_cache } from "next/cache";
import { db } from "@/lib/db";

export const getCachedProducts = unstable_cache(
  async (category: string) => db.product.findMany({ where: { category } }),
  ["products-by-category"],          // key prefix parts
  { revalidate: 3600, tags: ["products"] }
);

// Usage in a Server Component:
const electronics = await getCachedProducts("electronics");
```

The function arguments are automatically folded into the cache key, so `getCachedProducts("books")` and `getCachedProducts("toys")` are cached separately. `revalidateTag("products")` invalidates them all.

!!! warning "`unstable_cache` arguments & closures"
    `unstable_cache` keys on the **arguments** you pass and the key array — **not** on variables captured from an outer scope. Always pass anything that affects the result as an argument, or you'll get cache collisions. (It also can't read dynamic data like `cookies()` inside.)

### 4.5 The `'use cache'` direction

Next.js is moving toward an explicit, composable caching directive: **`'use cache'`**. Placed at the top of a file, a function, or a component, it marks that scope as cacheable, replacing the implicit-defaults model and eventually `unstable_cache`. It pairs with `cacheLife` (named revalidation profiles) and `cacheTag`.

```tsx
// A cached async function (Next 15 canary / 16 — behind a flag today)
import { unstable_cacheLife as cacheLife, unstable_cacheTag as cacheTag } from "next/cache";

async function getProducts(category: string) {
  "use cache";
  cacheLife("hours");      // named profile: revalidate ~hourly
  cacheTag("products");    // invalidate via revalidateTag("products")
  return db.product.findMany({ where: { category } });
}
```

Enable it in config while it stabilizes:

```ts
// next.config.ts
const nextConfig = {
  experimental: { useCache: true }, // or dynamicIO, depending on release
};
export default nextConfig;
```

!!! note "Still stabilizing — pin your version"
    As of mid-2026, `'use cache'`, `cacheLife`, and `cacheTag` are **experimental** and gated behind a flag (`useCache`/`dynamicIO`); the API surface is still shifting between releases. For production today, prefer explicit `fetch` options and `unstable_cache`. Treat `'use cache'` as the clearly-signposted future direction and adopt it once it's stable in your pinned Next.js version.

## 5. Client-side fetching: when to reach for SWR / TanStack Query

Server Components handle the *initial* render. Reach for **client-side fetching** when the data is genuinely client-driven and doesn't belong on the server:

- Data that changes rapidly and needs **polling** or **focus refetching** (live dashboards, presence).
- **User-specific, highly interactive** data behind authentication where SEO is irrelevant.
- **Infinite scroll / pagination** driven by user interaction.
- Anything needing optimistic updates with automatic rollback on the client.

For one-off client fetches, plain `fetch` in an event handler is fine. For caching, deduping, revalidation, and retries on the client, use a library.

### 5.1 SWR

Lightweight, hook-first, from Vercel. The name is the pattern: *stale-while-revalidate*.

```tsx
"use client";
import useSWR from "swr";

const fetcher = (url: string) => fetch(url).then((r) => r.json());

export default function Profile() {
  const { data, error, isLoading } = useSWR("/api/me", fetcher, {
    refreshInterval: 30_000, // poll every 30s
  });

  if (isLoading) return <p>Loading…</p>;
  if (error) return <p>Failed to load.</p>;
  return <p>Welcome, {data.name}</p>;
}
```

### 5.2 TanStack Query (v5)

More powerful (mutations, infinite queries, fine-grained cache control). **TanStack Query v5** changed several APIs — note the single-object signatures and the renamed flags below.

First, a provider (a Client Component) mounted in your root layout:

```tsx
// app/providers.tsx
"use client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { useState } from "react";

export default function Providers({ children }: { children: React.ReactNode }) {
  // One client per browser session; useState avoids re-creating it on render
  const [queryClient] = useState(() => new QueryClient());
  return (
    <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
  );
}
```

```tsx
// app/layout.tsx
import Providers from "./providers";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  );
}
```

Querying — note v5's **object form** and `isPending` (v4's `isLoading` was renamed; `isLoading` now means "pending *and* fetching"):

```tsx
"use client";
import { useQuery } from "@tanstack/react-query";

type Todo = { id: string; title: string };

async function fetchTodos(): Promise<Todo[]> {
  const res = await fetch("/api/todos");
  if (!res.ok) throw new Error("Network error");
  return res.json();
}

export default function Todos() {
  const { data, error, isPending } = useQuery({
    queryKey: ["todos"],
    queryFn: fetchTodos,
    staleTime: 60_000,
  });

  if (isPending) return <p>Loading…</p>;
  if (error) return <p>Error: {error.message}</p>;
  return (
    <ul>
      {data.map((t) => (
        <li key={t.id}>{t.title}</li>
      ))}
    </ul>
  );
}
```

Mutations — v5 also takes a single object, and the cache-callback signatures are object-based:

```tsx
"use client";
import { useMutation, useQueryClient } from "@tanstack/react-query";

export default function AddTodo() {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: (title: string) =>
      fetch("/api/todos", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ title }),
      }).then((r) => r.json()),
    onSuccess: () => {
      // v5: invalidateQueries takes a filters object
      queryClient.invalidateQueries({ queryKey: ["todos"] });
    },
  });

  return (
    <button onClick={() => mutation.mutate("New todo")} disabled={mutation.isPending}>
      {mutation.isPending ? "Adding…" : "Add"}
    </button>
  );
}
```

!!! note "Server-render, then hydrate (best of both)"
    You can prefetch on the server and hand the cache to TanStack Query so the client mounts already-populated — no loading flash, no double fetch. Prefetch into a `QueryClient` on the server, then wrap the tree in `<HydrationBoundary state={dehydrate(queryClient)}>`. This pattern (v5's `HydrationBoundary`, replacing v4's `Hydrate`) is the recommended way to combine RSC data loading with client-side caching.

!!! tip "Don't reach for a client library by default"
    In the App Router, **most** data should be fetched in Server Components — it ships less JS and is simpler. Add SWR/TanStack Query only for the genuinely client-interactive cases above. Fetching everything on the client re-introduces the waterfalls and SEO gaps the App Router was designed to remove. (See the *State Management* chapter for combining server data with client state.)

## 6. Error & not-found handling

### 6.1 `error.tsx` — segment error boundaries

An `error.tsx` file wraps its route segment in a React Error Boundary. When a Server (or Client) Component in that segment throws, the boundary renders instead of the page. It **must be a Client Component** and receives the error plus a `reset` function to retry the render.

```tsx
// app/dashboard/error.tsx
"use client";
import { useEffect } from "react";

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    // Log to your monitoring service
    console.error(error);
  }, [error]);

  return (
    <div role="alert">
      <h2>Something went wrong.</h2>
      <button onClick={() => reset()}>Try again</button>
    </div>
  );
}
```

- `error.tsx` catches errors in its segment and **below**, but **not** in the segment's own `layout.tsx` (the layout sits above the boundary). To catch layout errors, put an `error.tsx` in the *parent* segment.
- For the root layout itself, use `app/global-error.tsx` (it must render its own `<html>` and `<body>`).
- In production the actual error message is hidden from the client for security; you get a `digest` you can correlate with server logs.

### 6.2 `not-found.tsx` and `notFound()`

Call `notFound()` from a Server Component to render the nearest `not-found.tsx` and send a **404** status. This is the idiomatic way to handle "this record doesn't exist."

```tsx
// app/blog/[slug]/page.tsx
import { notFound } from "next/navigation";

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const res = await fetch(`https://api.example.com/posts/${slug}`);

  if (res.status === 404) notFound(); // throws → renders not-found.tsx, 404 status
  if (!res.ok) throw new Error("Failed to load post"); // → error.tsx

  const post = await res.json();
  return <article>{post.title}</article>;
}
```

```tsx
// app/blog/[slug]/not-found.tsx
import Link from "next/link";

export default function NotFound() {
  return (
    <div>
      <h2>Post not found</h2>
      <Link href="/blog">Back to all posts</Link>
    </div>
  );
}
```

!!! tip "Pair `loading.tsx` + `error.tsx` + Suspense for streaming"
    Wrap slow fetches in `<Suspense>` and give the segment a `loading.tsx`; pair it with `error.tsx` and `notFound()`. The result: an instant skeleton, content streamed in as it resolves, a clean 404 for missing data, and a friendly fallback for unexpected failures — all without blocking the rest of the page.

```tsx
// app/products/page.tsx — streaming a slow region with graceful states
import { Suspense } from "react";

export default function ProductsPage() {
  return (
    <section>
      <h1>Products</h1>
      <Suspense fallback={<ProductsSkeleton />}>
        <ProductList /> {/* async; may notFound() or throw → caught by not-found/error.tsx */}
      </Suspense>
    </section>
  );
}
```

## 📚 Summary

| 🔹 Concept | 🧠 Description |
| ---------- | -------------- |
| **Server Component fetch** | `async`/`await` (or direct DB) inside the component; no `getServerSideProps` |
| **Request Memoization** | Dedupes identical `fetch`/`cache()` calls **within one render** |
| **Data Cache** | Persistent server cache for fetch results; **opt-in** in Next 15 |
| **Full Route Cache** | Server cache of rendered static route HTML/RSC |
| **Router Cache** | Client cache of visited routes; dynamic default `staleTime` is **0** in Next 15 |
| **`cache: 'force-cache'`** | Opt a fetch back into the Data Cache (was the v14 default) |
| **`next: { revalidate }`** | Time-based ISR — stale-while-revalidate |
| **`revalidateTag` / `revalidatePath`** | On-demand invalidation; prefer tags |
| **`generateStaticParams`** | Pre-render dynamic routes at build (replaces `getStaticPaths`) |
| **`dynamicParams`** | `true` = render unlisted slugs on demand; `false` = 404 |
| **`unstable_cache`** | Put non-`fetch` (DB) results in the Data Cache |
| **`'use cache'`** | Experimental future direction; `cacheLife` + `cacheTag` |
| **SWR / TanStack Query v5** | Client-side caching for interactive/polling data |
| **`error.tsx` / `notFound()`** | Segment error boundary; idiomatic 404 handling |

!!! abstract "The one rule to remember"
    In **Next.js 15**, **nothing is cached unless you ask for it** — `fetch`, GET Route Handlers, and client navigations are all uncached by default. Set `cache`/`revalidate`/`tags` explicitly, tag your data, and invalidate by tag. Default to the server; reach for client libraries only for genuinely interactive data.
