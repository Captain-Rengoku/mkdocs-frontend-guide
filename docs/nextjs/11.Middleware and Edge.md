# :simple-nextdotjs: Middleware & Edge

**Middleware** is code that runs *before* a request finishes routing—after the request hits your deployment but before it reaches a page, layout, or Route Handler. It's your one chance to inspect, redirect, rewrite, or decorate every matching request in a single place: auth gates, locale redirects, A/B routing, geo-personalization, and security headers all live here.

In Next.js, middleware runs on the **Edge runtime** by default—a lightweight, Web-standard environment distributed close to users. That makes it fast, but it also constrains what you can do (no Node.js APIs, a limited package surface). This chapter covers the `middleware.ts` API end to end, the common use cases, the *correct* way to use middleware for auth (a lightweight optimistic check only—never your real authorization), the Edge runtime and its trade-offs, and the static `next.config` networking primitives (`redirects`, `rewrites`, `headers`) that handle the cases where you don't need code at all.

> 📚 For more: [https://nextjs.org/docs/app/building-your-application/routing/middleware](https://nextjs.org/docs/app/building-your-application/routing/middleware) · [https://nextjs.org/docs/app/api-reference/file-conventions/middleware](https://nextjs.org/docs/app/api-reference/file-conventions/middleware)

## 1. `middleware.ts` basics

Create a single `middleware.ts` (or `.js`) file at the **root of your project**—next to `app/`, or inside `src/` if you use that layout. Next.js automatically detects it; there is no import or registration step. You get **one** middleware file per project.

```ts
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

// Runs before every matching request (see `config.matcher` below)
export function middleware(request: NextRequest) {
  console.log("Incoming:", request.method, request.nextUrl.pathname);
  return NextResponse.next(); // continue to the intended route
}

export const config = {
  matcher: ["/dashboard/:path*"], // only run for these paths
};
```

The file exports two things:

- **`middleware`** — the function that runs per request. It can be sync or `async` and receives a `NextRequest`.
- **`config`** (optional but recommended) — configuration, most importantly the `matcher` that scopes *which* requests trigger it.

### `NextRequest`

`NextRequest` extends the Web `Request` with Next.js conveniences:

```ts
export function middleware(request: NextRequest) {
  // Parsed, mutable URL — prefer this over building URLs by hand
  const { pathname, searchParams } = request.nextUrl;

  // Typed cookie helpers (read here; write on the response — see §4)
  const session = request.cookies.get("session")?.value;

  // Standard headers
  const ua = request.headers.get("user-agent");

  // Geo & IP are provided by the host (e.g. Vercel) at the edge
  const country = request.headers.get("x-vercel-ip-country"); // "US", "DE", …

  return NextResponse.next();
}
```

!!! note "`request.geo` / `request.ip` moved to headers"
    In older Next.js, `NextRequest` exposed `request.geo` and `request.ip`. In Next.js 15 these were **removed**; on Vercel, read the equivalent values from request headers (`x-vercel-ip-country`, `x-vercel-ip-city`, `x-forwarded-for`, …) or use `@vercel/functions`' `geolocation(request)` / `ipAddress(request)` helpers. Other hosts inject their own headers.

## 2. The response: `next`, `redirect`, `rewrite`

A middleware function returns a `NextResponse` (or a plain `Response`, or nothing—which is treated as "continue"). The three workhorse factories on `NextResponse` decide what happens to the request.

### `NextResponse.next()` — continue

Let the request proceed to its intended destination. Return this when you only want to *observe* the request or *attach* something (a header, a cookie) on the way through.

```ts
return NextResponse.next();
```

You can also pass request-header overrides so the downstream route sees a modified request (covered in §4).

### `NextResponse.redirect()` — change the URL the user sees

Sends a real HTTP redirect (307 by default). The browser's address bar changes to the new URL.

```ts
// Bounce unauthenticated users to /login
return NextResponse.redirect(new URL("/login", request.url));
```

!!! tip "Always build URLs with `new URL(path, request.url)`"
    Passing a bare string like `"/login"` is not a valid absolute URL for a redirect. `new URL("/login", request.url)` resolves it against the current request's origin, so it works in dev, preview, and production without hardcoding a domain.

### `NextResponse.rewrite()` — change what's served, not the URL

Serves the content of a *different* path while the browser keeps showing the **original** URL. The user never sees the rewrite; it's invisible proxying.

```ts
// User sees /blog, but we serve the content at /cms/blog internally
return NextResponse.rewrite(new URL("/cms/blog", request.url));
```

| Method | Browser URL changes? | HTTP status | Typical use |
| ------ | -------------------- | ----------- | ----------- |
| `next()` | No | (continues) | Observe / attach headers & cookies |
| `redirect()` | **Yes** (new URL shown) | 307 / 308 | Auth gate, moved pages, locale switch |
| `rewrite()` | No (original URL kept) | 200 (of target) | Proxying, A/B, masking internal paths |

## 3. The `config.matcher`

Without a matcher, middleware runs on **every** request—including static assets, images, and `_next` internals—which is wasteful and slow. The `matcher` scopes it to exactly the paths you care about. It's statically analyzed at build time, so it must be a literal value (not computed at runtime).

```ts
export const config = {
  // A single path, or an array of paths
  matcher: ["/dashboard/:path*", "/account/:path*"],
};
```

Matcher syntax (powered by `path-to-regexp`):

- `/about` — exact path.
- `/dashboard/:path*` — `/dashboard` and everything beneath it.
- `/users/:id` — a single dynamic segment.
- Full regex via a named group: `/((?!api|_next/static|_next/image|favicon.ico).*)`.

A very common pattern is "run on everything **except** static assets and image optimization":

```ts
export const config = {
  matcher: [
    /*
     * Match all request paths except:
     *  - /api            (API routes handle their own logic)
     *  - /_next/static   (static files)
     *  - /_next/image    (image optimization)
     *  - /favicon.ico, /robots.txt, /sitemap.xml, common image extensions
     */
    "/((?!api|_next/static|_next/image|favicon.ico|robots.txt|sitemap.xml|.*\\.(?:png|jpg|jpeg|gif|svg|webp|ico)$).*)",
  ],
};
```

For finer control, a matcher entry can be an **object** that also conditions on method, headers, or query params:

```ts
export const config = {
  matcher: [
    {
      source: "/dashboard/:path*",
      // Skip prefetch requests (Next.js sends these for <Link> hover prefetch)
      missing: [{ type: "header", key: "next-router-prefetch" }],
    },
  ],
};
```

!!! note "Matcher first, guard clauses second"
    Even with a matcher, start your function with early `return NextResponse.next()` guards for paths you don't want to touch. The matcher narrows *which requests run the function*; in-function checks keep the logic readable and let you short-circuit cheaply. Middleware runs on the hot path of every matching request, so keep it lean.

## 4. Reading & setting cookies and headers

Cookies and headers are read from the **request** and written on the **response**. Mixing these up is the most common middleware mistake.

### Cookies

```ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  // READ from the incoming request
  const theme = request.cookies.get("theme")?.value ?? "light";
  const all = request.cookies.getAll();
  const hasSession = request.cookies.has("session");

  const response = NextResponse.next();

  // WRITE on the outgoing response (Set-Cookie)
  response.cookies.set("theme", theme, {
    httpOnly: true,
    secure: process.env.NODE_ENV === "production",
    sameSite: "lax",
    path: "/",
    maxAge: 60 * 60 * 24 * 365, // 1 year
  });

  // Delete a cookie
  // response.cookies.delete("stale-flag");

  return response;
}
```

### Response headers

Attach headers (e.g. for security or tracing) to what the client receives:

```ts
export function middleware(request: NextRequest) {
  const response = NextResponse.next();
  response.headers.set("x-request-id", crypto.randomUUID());
  response.headers.set("X-Frame-Options", "DENY");
  return response;
}
```

### Request headers (passed *into* the route)

To make the downstream page/Route Handler see a modified request—say, to forward a derived value—clone the request headers and pass them through `NextResponse.next({ request })`:

```ts
export function middleware(request: NextRequest) {
  const requestHeaders = new Headers(request.headers);
  requestHeaders.set("x-user-locale", "en-US"); // visible to the route via headers()
  requestHeaders.set("x-pathname", request.nextUrl.pathname);

  return NextResponse.next({
    request: { headers: requestHeaders },
  });
}
```

The route can then read it server-side:

```ts
// app/some-page/page.tsx
import { headers } from "next/headers";

export default async function Page() {
  const locale = (await headers()).get("x-user-locale"); // "en-US"
  return <p>Locale: {locale}</p>;
}
```

!!! note "Request headers vs response headers"
    `requestHeaders` (set via `next({ request })`) are sent **forward** to your route—the browser never sees them. `response.headers.set(...)` are sent **back** to the browser. Use the former to pass derived data inward, the latter for security/caching headers outward.

## 5. Common use cases

### Auth gate (lightweight check)

The canonical example: redirect unauthenticated users away from protected sections. Note this is an *optimistic* check only—§6 explains why.

```ts
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

const PROTECTED = ["/dashboard", "/account", "/settings"];

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;
  const isProtected = PROTECTED.some((p) => pathname.startsWith(p));
  const hasSession = request.cookies.has("session");

  if (isProtected && !hasSession) {
    const url = new URL("/login", request.url);
    url.searchParams.set("from", pathname); // remember where they wanted to go
    return NextResponse.redirect(url);
  }

  // Already logged in? Keep them off the auth pages.
  if (pathname.startsWith("/login") && hasSession) {
    return NextResponse.redirect(new URL("/dashboard", request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/dashboard/:path*", "/account/:path*", "/settings/:path*", "/login"],
};
```

### i18n locale redirect

Detect the user's preferred locale and redirect to the localized path when none is present in the URL.

```ts
// middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

const LOCALES = ["en", "de", "fr"] as const;
const DEFAULT_LOCALE = "en";

function getLocale(request: NextRequest): string {
  // 1. Prefer a previously chosen locale cookie
  const cookieLocale = request.cookies.get("NEXT_LOCALE")?.value;
  if (cookieLocale && LOCALES.includes(cookieLocale as (typeof LOCALES)[number])) {
    return cookieLocale;
  }
  // 2. Fall back to Accept-Language (first matching locale)
  const accept = request.headers.get("accept-language") ?? "";
  const preferred = accept.split(",")[0]?.split("-")[0];
  return LOCALES.includes(preferred as (typeof LOCALES)[number])
    ? (preferred as string)
    : DEFAULT_LOCALE;
}

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // Skip if the path already has a supported locale prefix
  const hasLocale = LOCALES.some(
    (l) => pathname === `/${l}` || pathname.startsWith(`/${l}/`)
  );
  if (hasLocale) return NextResponse.next();

  const locale = getLocale(request);
  const url = new URL(`/${locale}${pathname}`, request.url);
  return NextResponse.redirect(url);
}

export const config = {
  // Skip internals & files (anything with a dot)
  matcher: ["/((?!api|_next|.*\\..*).*)"],
};
```

### A/B testing (rewrite to a variant)

Assign a sticky bucket via cookie, then **rewrite** so each user consistently sees one variant—at the same URL.

```ts
export function middleware(request: NextRequest) {
  if (request.nextUrl.pathname !== "/") return NextResponse.next();

  let bucket = request.cookies.get("ab-bucket")?.value;
  if (!bucket) {
    bucket = Math.random() < 0.5 ? "a" : "b"; // assign once
  }

  // Serve /home-a or /home-b, but the URL stays "/"
  const response = NextResponse.rewrite(
    new URL(`/home-${bucket}`, request.url)
  );
  response.cookies.set("ab-bucket", bucket, { maxAge: 60 * 60 * 24 * 30 });
  return response;
}

export const config = { matcher: ["/"] };
```

### Geo-personalization

Route or redirect based on the visitor's country (provided by the host at the edge).

```ts
export function middleware(request: NextRequest) {
  const country = request.headers.get("x-vercel-ip-country") ?? "US";

  // Send EU visitors to the EU storefront
  if (["DE", "FR", "ES", "IT"].includes(country)) {
    if (!request.nextUrl.pathname.startsWith("/eu")) {
      return NextResponse.rewrite(
        new URL(`/eu${request.nextUrl.pathname}`, request.url)
      );
    }
  }
  return NextResponse.next();
}
```

## 6. Using middleware for auth correctly

Middleware is a tempting place to put authorization because it runs before everything. It is the right place for a **fast, optimistic gate**—and the wrong place to be your *only* line of defense.

!!! warning "Don't rely on middleware alone for auth"
    Treat middleware as an **optimistic check only**: read the session cookie's *presence* and redirect obvious cases (e.g. no cookie → `/login`) for a fast UX. **Do not** make it your sole authorization layer. Per Next.js guidance, perform the **real** authentication and authorization close to the data—in your **Data Access Layer (DAL)**, Server Components, Server Actions, and Route Handlers—where you can verify the session and check permissions before reading or mutating data. A middleware-only check can be bypassed and can read stale or unverified state, and middleware on the Edge often can't safely do the heavy work (DB lookups, full token verification) anyway. Defense in depth: optimistic gate at the edge, authoritative check at the data layer.

Why this split:

- **Middleware runs on the Edge** with tight CPU/time budgets and a limited package surface—doing a database round-trip or a heavy crypto verification on every request is slow and often unsupported.
- **A cookie's presence is not proof of a valid session.** Cookies can be stale, forged, or revoked. The middleware check is a UX optimization (skip rendering a protected page just to bounce the user), not a security boundary.
- **The data layer is the real boundary.** Whatever actually reads or writes user data must verify identity and permissions itself, so a request that slips past middleware (or comes from a non-browser client) still can't access protected data.

```ts
// middleware.ts — OPTIMISTIC gate only: just check the cookie exists
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export function middleware(request: NextRequest) {
  const hasSession = request.cookies.has("session");
  if (!hasSession && request.nextUrl.pathname.startsWith("/dashboard")) {
    return NextResponse.redirect(new URL("/login", request.url));
  }
  return NextResponse.next();
}

export const config = { matcher: ["/dashboard/:path*"] };
```

```ts
// lib/dal.ts — AUTHORITATIVE check, run wherever data is accessed
import "server-only";
import { cookies } from "next/headers";
import { cache } from "react";
import { redirect } from "next/navigation";
import { decrypt } from "@/lib/session"; // verify & decrypt the session token

// `cache` dedupes this within a single render pass
export const verifySession = cache(async () => {
  const token = (await cookies()).get("session")?.value;
  const session = await decrypt(token); // throws/returns null if invalid or expired

  if (!session?.userId) {
    redirect("/login"); // not authenticated
  }
  return { userId: session.userId, role: session.role };
});

// Authorize a specific action close to the data
export async function getUserOrders() {
  const { userId } = await verifySession(); // real check, every time
  return db.order.findMany({ where: { userId } });
}
```

The page just calls the DAL; the verification happens there, not in middleware:

```ts
// app/dashboard/page.tsx
import { getUserOrders } from "@/lib/dal";

export default async function Dashboard() {
  const orders = await getUserOrders(); // verifies session + scopes to the user
  return <OrdersList orders={orders} />;
}
```

!!! tip "Where each layer belongs"
    - **Middleware** → optimistic redirect on missing/obvious cases. Fast, runs everywhere.
    - **DAL / Server Components / Server Actions / Route Handlers** → verify the session and check authorization *every time data is read or mutated*. This is the real gate.

## 7. The Edge runtime

### What it is

The **Edge runtime** is a lightweight JavaScript environment based on **Web standard APIs** (the same APIs as Service Workers / Cloudflare Workers), *not* Node.js. Middleware runs here by default, and you can opt individual Route Handlers / pages into it. Code is deployed to many locations and runs **geographically close to the user**, with near-instant cold starts.

- ✅ Very fast cold starts, globally distributed, ideal for latency-sensitive cross-cutting logic (redirects, header rewriting, optimistic auth, geo).
- ❌ A constrained environment: limited APIs, smaller deployment size limits, short execution budgets.

### Constraints — no Node.js APIs

The Edge runtime exposes Web APIs (`fetch`, `Request`/`Response`, `Headers`, `URL`, `crypto.subtle`, `TextEncoder`, `structuredClone`, …) but **not** the Node.js standard library:

```ts
// ❌ These do NOT work in the Edge runtime
import fs from "node:fs";          // no filesystem
import path from "node:path";      // no node:path
import crypto from "node:crypto";  // no node:crypto (use Web Crypto: globalThis.crypto)
import net from "node:net";        // no raw TCP sockets

// ✅ Web-standard equivalents that DO work
const id = crypto.randomUUID();             // Web Crypto
const hash = await crypto.subtle.digest(/* … */);
const res = await fetch("https://api.example.com");
```

Practical consequences:

- **No native database drivers** that rely on TCP sockets (e.g. the classic `pg` driver). Use HTTP-based / serverless-driver clients (Neon, PlanetScale, Upstash, Turso) if you must hit a DB at the edge—though for middleware you usually shouldn't.
- **Many npm packages won't run** if they import Node built-ins. Crypto/JWT libraries often ship Edge-compatible builds (e.g. `jose` works on the Edge; some others don't).
- **`process.env`** is available for build-time-inlined env vars, but dynamic Node-only globals are not.

### When to opt into the Node.js runtime

If middleware genuinely needs Node APIs or an incompatible package, Next.js 15 lets you run **middleware on the Node.js runtime** (stabilized for middleware in 15.x). Opt in via the `runtime` flag in config:

```ts
// middleware.ts
export const config = {
  matcher: ["/dashboard/:path*"],
  runtime: "nodejs", // run middleware on Node.js instead of the Edge
};
```

For **Route Handlers and pages**, the runtime is chosen per file via the segment export (Node.js is the default there):

```ts
// app/api/report/route.ts
export const runtime = "nodejs"; // default — full Node APIs, DB drivers, big libs
// export const runtime = "edge"; // opt this handler into the Edge runtime instead
```

!!! note "Default runtimes differ by context"
    **Middleware** defaults to the **Edge** runtime; **Route Handlers and pages** default to the **Node.js** runtime. Reach for the Edge when you want global, low-latency execution and only need Web APIs; stay on / switch to Node.js when you need the full Node standard library, native drivers, or heavier packages. Don't opt into the Edge "for speed" if your code needs Node—measure first.

## 8. `next.config` networking: `redirects`, `rewrites`, `headers`

Not everything needs middleware. For **static, declarative** URL handling, `next.config.ts` offers three async functions. They run as part of routing without your own code on the hot path, so prefer them when the rule is fixed (doesn't depend on cookies, geo, or runtime logic).

```ts
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  async redirects() {
    return [
      // Permanent move (308): /old-blog/:slug → /blog/:slug
      { source: "/old-blog/:slug", destination: "/blog/:slug", permanent: true },
      // Temporary (307)
      { source: "/promo", destination: "/sales/summer", permanent: false },
      // Conditional on a header / cookie / query
      {
        source: "/legacy",
        has: [{ type: "cookie", key: "beta", value: "true" }],
        destination: "/new",
        permanent: false,
      },
    ];
  },

  async rewrites() {
    return [
      // Proxy /api/v1/* to an external API (URL stays /api/v1/* for the client)
      { source: "/api/v1/:path*", destination: "https://api.example.com/:path*" },
      // Mask an internal docs path
      { source: "/help/:slug*", destination: "/cms/help/:slug*" },
    ];
  },

  async headers() {
    return [
      {
        // Apply security headers to every route
        source: "/:path*",
        headers: [
          { key: "X-Frame-Options", value: "DENY" },
          { key: "X-Content-Type-Options", value: "nosniff" },
          { key: "Referrer-Policy", value: "strict-origin-when-cross-origin" },
        ],
      },
      {
        // Long-cache static assets
        source: "/assets/:path*",
        headers: [
          { key: "Cache-Control", value: "public, max-age=31536000, immutable" },
        ],
      },
    ];
  },
};

export default nextConfig;
```

- **`redirects`** — declarative HTTP redirects. `permanent: true` → 308, `false` → 307. Supports path params, wildcards (`:path*`), and `has`/`missing` conditions on headers, cookies, host, or query.
- **`rewrites`** — declarative proxying: serve a different (possibly external) destination while keeping the user's URL. Same matching power as redirects.
- **`headers`** — attach response headers to matching routes (security headers, caching, CORS).

### `next.config` networking vs middleware

| | `next.config` (`redirects`/`rewrites`/`headers`) | Middleware |
| --- | --- | --- |
| Logic | **Static / declarative** (path patterns + `has`/`missing`) | **Dynamic** — arbitrary code per request |
| Inputs | URL, host, headers, cookies, query (pattern match) | Anything: parse cookies, call APIs, geo, randomize |
| Performance | No user code on the path; very cheap | Runs your function on every matching request |
| Use when | The rule is fixed and known at build time | The decision depends on runtime logic |

!!! tip "Reach for config first"
    If a rule can be expressed as a static pattern—"these old URLs 308 to new ones", "add these security headers everywhere", "proxy `/api/v1/*` to an external host"—put it in `next.config`. Use middleware only when the decision needs real per-request logic (auth state, locale detection, A/B bucketing, geo). It's cheaper and harder to get wrong.

## 📚 Summary

| 🔹 Concept | 🧠 Description |
| ---------- | -------------- |
| `middleware.ts` | One root-level file; runs before matching requests reach a route |
| `NextRequest` | Web `Request` + `nextUrl`, typed `cookies`, headers |
| `NextResponse.next()` | Continue to the route (optionally attach headers/cookies) |
| `NextResponse.redirect()` | Change the URL the browser shows (307/308) |
| `NextResponse.rewrite()` | Serve a different path; browser URL stays the same |
| `config.matcher` | Statically-analyzed scope: which paths trigger middleware |
| Read cookies/headers | From the **request** (`request.cookies` / `request.headers`) |
| Write cookies/headers | On the **response** (`response.cookies.set` / `response.headers.set`) |
| `next({ request: { headers } })` | Forward modified headers *into* the route (read via `headers()`) |
| Use cases | Auth gate, i18n locale redirect, A/B (rewrite), geo-personalization |
| Auth in middleware | **Optimistic check only** — real authz lives in the DAL/server |
| Edge runtime | Web-standard APIs, global, fast cold start; **no Node.js APIs** |
| Node runtime for middleware | `config.runtime = "nodejs"` (Next 15) when you need Node/native pkgs |
| `redirects` / `rewrites` / `headers` | Static, declarative networking in `next.config` — prefer over middleware when the rule is fixed |
