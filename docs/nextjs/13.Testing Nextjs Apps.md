# Testing Next.js Apps

Testing a Next.js app is *mostly* testing React — but the App Router adds wrinkles that
plain-React tooling doesn't cover out of the box: components that run on the **server**,
the `next/navigation` and `next/headers` modules that only work inside a request, Route
Handlers that are just exported functions, and Server Actions that look like ordinary
async functions but really cross the network. This chapter covers the Next.js-specific
angle on each.

!!! tip "Read the React testing chapter first"
    This chapter assumes you know the **general** testing toolkit — the testing pyramid,
    Vitest setup, React Testing Library queries (`getByRole`/`findBy*`), `user-event`,
    MSW handler basics, and Playwright fundamentals. All of that lives in
    **[React Testing and Debugging](../react/16.React%20Testing%20and%20Debugging.md)**
    and is **not** repeated here. We build on it and focus only on what's different in
    Next.js.

---

## 1. The Lay of the Land

Next.js code falls into a few buckets, and each is tested a little differently:

| What you're testing | Best tool | Why |
|---|---|---|
| Pure logic / utilities | Vitest | No DOM, no framework — same as anywhere |
| **Client** Components (`"use client"`) | Vitest + RTL | Render in jsdom like ordinary React |
| **Async Server** Components | **E2E (Playwright)** | RTL can't fully render them yet (see below) |
| Route Handlers (`route.ts`) | Vitest | Call the exported `GET`/`POST` with a `Request` |
| Server Actions (`"use server"`) | Vitest | Call the function directly with mocked deps |
| Whole user flows | Playwright / Cypress | Real browser, real server, real RSC |

The single biggest Next.js testing decision is the line between **unit/component tests**
(fast, in jsdom, with mocks) and **E2E tests** (slow, in a real browser, against a real
build). Server-rendered behavior — Server Components, streaming, the actual data
fetching — is most reliably verified by E2E. Logic and Client Components are unit-tested.

!!! warning "Async Server Components aren't fully supported by RTL yet"
    React Testing Library renders into jsdom, which has **no React Server Components
    runtime**. An `async` Server Component returns a promise that RTL can't render the way
    React does on the server, and Next-specific server APIs (`fetch` caching,
    `cookies()`, `headers()`) don't run there. The official guidance is:

    - **For async Server Components, prefer end-to-end tests** (Playwright) that exercise
      the real rendered output.
    - Or **extract and unit-test the logic separately** — pull data fetching, mapping,
      and formatting into plain async functions you can call directly, and keep the
      component a thin shell. You then test the functions with Vitest and the rendered
      result with E2E.

    **Synchronous** Server Components (no `await`, no server-only APIs) can often be
    rendered by RTL because they're just functions returning JSX — but treat that as a
    convenience, not a guarantee.

## 2. Setting Up Vitest for Next.js

The Vitest + RTL + jsdom setup is the same as in the React chapter, with two Next.js
additions: the React plugin must understand JSX/TSX in your `app/` files, and you'll want
path aliases (`@/`) to resolve. Next.js's docs use Vitest with `@vitejs/plugin-react`.

```bash
npm install -D vitest @vitejs/plugin-react jsdom \
  @testing-library/react @testing-library/user-event @testing-library/jest-dom \
  vite-tsconfig-paths
```

```ts
// vitest.config.ts
import { defineConfig } from "vitest/config";
import react from "@vitejs/plugin-react";
import tsconfigPaths from "vite-tsconfig-paths";

export default defineConfig({
  plugins: [tsconfigPaths(), react()], // tsconfigPaths resolves your "@/*" aliases
  test: {
    globals: true,
    environment: "jsdom",
    setupFiles: ["./vitest.setup.ts"],
  },
});
```

```ts
// vitest.setup.ts
import "@testing-library/jest-dom/vitest";
```

```json
// package.json
{
  "scripts": {
    "test": "vitest",
    "test:run": "vitest run"
  }
}
```

!!! note "Why not Jest?"
    Next.js ships first-class config for **both** Jest (`next/jest`, which wires up SWC,
    CSS modules, and `next/font` mocks automatically) and Vitest. This chapter uses
    **Vitest** to stay consistent with the rest of the guide; if your team prefers Jest,
    run `npm install -D jest jest-environment-jsdom @testing-library/jest-dom` and use the
    `next/jest` transformer — the *tests themselves* are nearly identical.

!!! warning "Don't put server-API tests in the jsdom environment"
    Route Handler and Server Action tests don't need a DOM and may import server-only
    modules. Either keep them in the default config (they ignore jsdom) or give them the
    `node` environment with a per-file directive:

    ```ts
    // @vitest-environment node
    ```

## 3. Testing Client Components

A `"use client"` component is ordinary React — render it with RTL exactly as the React
chapter describes. The only Next.js-specific issue is that Client Components often import
from `next/navigation` and `next/link`, which throw outside a Next runtime. You mock them.

```tsx
// components/SearchBox.tsx
"use client";

import { useRouter, useSearchParams } from "next/navigation";
import { useState } from "react";

export default function SearchBox() {
  const router = useRouter();
  const params = useSearchParams();
  const [q, setQ] = useState(params.get("q") ?? "");

  return (
    <form
      onSubmit={(e) => {
        e.preventDefault();
        router.push(`/search?q=${encodeURIComponent(q)}`);
      }}
    >
      <label htmlFor="q">Search</label>
      <input id="q" value={q} onChange={(e) => setQ(e.target.value)} />
      <button type="submit">Go</button>
    </form>
  );
}
```

### Mocking `next/navigation`

`useRouter`, `usePathname`, and `useSearchParams` all come from `next/navigation`. Replace
the module with `vi.mock` and hand back controllable stand-ins. Note that
`useSearchParams` must return a real `URLSearchParams` (or `ReadonlyURLSearchParams`) so
`.get()` works.

```tsx
// components/SearchBox.test.tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { describe, it, expect, vi, beforeEach } from "vitest";
import SearchBox from "./SearchBox";

const push = vi.fn();

vi.mock("next/navigation", () => ({
  useRouter: () => ({ push, replace: vi.fn(), refresh: vi.fn(), prefetch: vi.fn() }),
  usePathname: () => "/search",
  useSearchParams: () => new URLSearchParams("q=react"),
}));

beforeEach(() => vi.clearAllMocks());

describe("SearchBox", () => {
  it("prefills from the URL and pushes a new query on submit", async () => {
    const user = userEvent.setup();
    render(<SearchBox />);

    const input = screen.getByLabelText(/search/i);
    expect(input).toHaveValue("react"); // came from useSearchParams

    await user.clear(input);
    await user.type(input, "next.js");
    await user.click(screen.getByRole("button", { name: /go/i }));

    expect(push).toHaveBeenCalledWith("/search?q=next.js");
  });
});
```

!!! tip "Mock once, override per test"
    If you need different return values across tests, expose the mocked functions and
    reassign them with `vi.mocked(...)`:

    ```tsx
    import { usePathname } from "next/navigation";

    vi.mock("next/navigation", () => ({
      usePathname: vi.fn(),
      useRouter: () => ({ push: vi.fn() }),
      useSearchParams: () => new URLSearchParams(),
    }));

    it("highlights the active link", () => {
      vi.mocked(usePathname).mockReturnValue("/dashboard");
      // ...render and assert
    });
    ```

!!! note "`<Link>` usually just works"
    `next/link` renders a plain `<a>` in tests, so you rarely need to mock it. If a
    component calls `router.push` *via* a `<Link>`, query the anchor by role
    (`getByRole("link", { name: ... })`) and assert its `href` instead of spying on the
    router.

### Mocking `next/headers`

`cookies()` and `headers()` from `next/headers` only run on the server. If a Client
Component's *child* path or a shared helper imports them in a unit test, mock the module:

```ts
vi.mock("next/headers", () => ({
  cookies: () => ({
    get: (name: string) => (name === "theme" ? { value: "dark" } : undefined),
    getAll: () => [{ name: "theme", value: "dark" }],
    has: (name: string) => name === "theme",
  }),
  headers: () => new Headers({ "x-forwarded-for": "127.0.0.1" }),
}));
```

!!! warning "`cookies()` and `headers()` are async in Next.js 15"
    In Next.js 15 these APIs became **asynchronous** — you `await cookies()` /
    `await headers()` in app code. When you mock them, make the mock match: either return
    the object directly and `await` a non-promise (which is fine), or return
    `Promise.resolve(...)` to mirror real usage precisely. Prefer mirroring real usage so
    your test catches a missing `await`.

## 4. Testing Route Handlers

A Route Handler (`app/api/.../route.ts`) exports functions named after HTTP verbs. The
beauty of this is that a handler is **just a function** — you import the exported `GET`
or `POST` and call it with a standard `Request`. No server needs to be running.

```ts
// app/api/users/route.ts
import { NextResponse } from "next/server";

const db = { users: [{ id: 1, name: "Ada Lovelace" }] };

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const q = searchParams.get("q")?.toLowerCase();
  const users = q
    ? db.users.filter((u) => u.name.toLowerCase().includes(q))
    : db.users;
  return NextResponse.json({ users });
}

export async function POST(request: Request) {
  const body = (await request.json()) as { name?: string };
  if (!body.name) {
    return NextResponse.json({ error: "name is required" }, { status: 400 });
  }
  const user = { id: db.users.length + 1, name: body.name };
  db.users.push(user);
  return NextResponse.json(user, { status: 201 });
}
```

```ts
// app/api/users/route.test.ts
// @vitest-environment node
import { describe, it, expect } from "vitest";
import { GET, POST } from "./route";

describe("GET /api/users", () => {
  it("returns all users", async () => {
    const res = await GET(new Request("http://localhost/api/users"));
    expect(res.status).toBe(200);
    expect(await res.json()).toEqual({ users: [{ id: 1, name: "Ada Lovelace" }] });
  });

  it("filters by the q query param", async () => {
    const res = await GET(new Request("http://localhost/api/users?q=ada"));
    const { users } = await res.json();
    expect(users).toHaveLength(1);
  });
});

describe("POST /api/users", () => {
  it("creates a user", async () => {
    const req = new Request("http://localhost/api/users", {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify({ name: "Grace Hopper" }),
    });
    const res = await POST(req);
    expect(res.status).toBe(201);
    expect(await res.json()).toMatchObject({ name: "Grace Hopper" });
  });

  it("400s when name is missing", async () => {
    const req = new Request("http://localhost/api/users", {
      method: "POST",
      body: JSON.stringify({}),
    });
    const res = await POST(req);
    expect(res.status).toBe(400);
  });
});
```

!!! note "`NextResponse` is a real `Response`"
    `NextResponse.json(...)` extends the web `Response`, so `res.status`,
    `await res.json()`, and `res.headers.get(...)` all work in the test with no special
    handling.

### Dynamic segments and the context argument

A dynamic route like `app/api/users/[id]/route.ts` receives a second `context` argument
whose `params` you must supply. In **Next.js 15, `params` is a `Promise`**, so wrap it.

```ts
// app/api/users/[id]/route.ts
import { NextResponse } from "next/server";

export async function GET(
  _request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;
  return NextResponse.json({ id: Number(id) });
}
```

```ts
// app/api/users/[id]/route.test.ts
// @vitest-environment node
import { describe, it, expect } from "vitest";
import { GET } from "./route";

it("reads the dynamic id param", async () => {
  const res = await GET(new Request("http://localhost/api/users/42"), {
    params: Promise.resolve({ id: "42" }), // params is a Promise in Next 15
  });
  expect(await res.json()).toEqual({ id: 42 });
});
```

!!! tip "Need `NextRequest`, cookies, or headers?"
    If your handler uses `request.cookies` or `request.nextUrl`, construct a
    `NextRequest` (from `next/server`) instead of a plain `Request`, or set headers on the
    `Request` you pass in:

    ```ts
    import { NextRequest } from "next/server";
    const req = new NextRequest("http://localhost/api/me", {
      headers: { cookie: "session=abc123" },
    });
    ```

## 5. Testing Server Actions

A Server Action (`"use server"`) is, from a test's point of view, **an exported async
function**. You import it and call it directly, mocking the things it touches —
the database, `revalidatePath`, `redirect`, and auth helpers. You are testing the action's
*logic*, not the React form integration (that's an E2E concern).

```ts
// app/actions.ts
"use server";

import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
import { z } from "zod";
import { db } from "@/lib/db";
import { getCurrentUser } from "@/lib/auth";

const schema = z.object({
  title: z.string().min(1, "Title is required"),
});

export async function createPost(_prev: unknown, formData: FormData) {
  const user = await getCurrentUser();
  if (!user) return { error: "Unauthorized" };

  const parsed = schema.safeParse({ title: formData.get("title") });
  if (!parsed.success) {
    return { error: parsed.error.issues[0].message };
  }

  const post = await db.post.create({
    data: { title: parsed.data.title, authorId: user.id },
  });

  revalidatePath("/posts");
  redirect(`/posts/${post.id}`);
}
```

The action mixes pure validation with three side-effecting dependencies. Mock each so the
test stays isolated and you can assert how the action *used* them.

```ts
// app/actions.test.ts
// @vitest-environment node
import { describe, it, expect, vi, beforeEach } from "vitest";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
import { db } from "@/lib/db";
import { getCurrentUser } from "@/lib/auth";
import { createPost } from "./actions";

vi.mock("next/cache", () => ({ revalidatePath: vi.fn() }));
vi.mock("next/navigation", () => ({
  // redirect throws internally in Next; a no-op mock is enough for unit tests
  redirect: vi.fn(),
}));
vi.mock("@/lib/auth", () => ({ getCurrentUser: vi.fn() }));
vi.mock("@/lib/db", () => ({
  db: { post: { create: vi.fn() } },
}));

function form(fields: Record<string, string>) {
  const fd = new FormData();
  for (const [k, v] of Object.entries(fields)) fd.set(k, v);
  return fd;
}

beforeEach(() => vi.clearAllMocks());

describe("createPost", () => {
  it("rejects unauthenticated users", async () => {
    vi.mocked(getCurrentUser).mockResolvedValue(null);

    const result = await createPost(null, form({ title: "Hello" }));

    expect(result).toEqual({ error: "Unauthorized" });
    expect(db.post.create).not.toHaveBeenCalled();
  });

  it("returns a validation error for an empty title", async () => {
    vi.mocked(getCurrentUser).mockResolvedValue({ id: "u1" });

    const result = await createPost(null, form({ title: "" }));

    expect(result).toEqual({ error: "Title is required" });
    expect(db.post.create).not.toHaveBeenCalled();
  });

  it("creates the post, revalidates, and redirects", async () => {
    vi.mocked(getCurrentUser).mockResolvedValue({ id: "u1" });
    vi.mocked(db.post.create).mockResolvedValue({ id: 7, title: "Hello" });

    await createPost(null, form({ title: "Hello" }));

    expect(db.post.create).toHaveBeenCalledWith({
      data: { title: "Hello", authorId: "u1" },
    });
    expect(revalidatePath).toHaveBeenCalledWith("/posts");
    expect(redirect).toHaveBeenCalledWith("/posts/7");
  });
});
```

!!! warning "`redirect()` throws by design"
    In real Next.js, `redirect()` works by **throwing** a special `NEXT_REDIRECT` error
    that the framework catches. In a unit test that's noise, so mock it to a no-op (as
    above) and assert it was *called*. If you test the **un-mocked** action, wrap the call
    in `expect(...).rejects` or a `try/catch` and inspect the thrown digest — but mocking
    is simpler and tests your intent more directly.

!!! tip "Keep actions thin and testable"
    The easiest Server Actions to test are the ones that delegate: validate input, call a
    well-defined data-access function, revalidate, redirect. Push real database logic into
    a `lib/` data layer so you can mock it cleanly here and test *it* separately. This is
    the same data-access-layer pattern from the Database Integration chapter.

## 6. API Mocking with MSW

When a **Client** Component (or a SWR/TanStack Query hook) fetches from your own API or a
third party, mock the network with **MSW v2** instead of stubbing `fetch`. The handler
setup is identical to the React chapter — same `http`/`HttpResponse` API, same
`setupServer` for Node — so this section only highlights the Next.js specifics.

```ts
// test/handlers.ts
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("https://api.example.com/posts", () =>
    HttpResponse.json([{ id: 1, title: "Hello from MSW" }])
  ),
];
```

```ts
// test/server.ts
import { setupServer } from "msw/node";
import { handlers } from "./handlers";

export const server = setupServer(...handlers);
```

```ts
// vitest.setup.ts
import "@testing-library/jest-dom/vitest";
import { beforeAll, afterEach, afterAll } from "vitest";
import { server } from "./test/server";

beforeAll(() => server.listen({ onUnhandledRequest: "error" }));
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

!!! warning "Use absolute URLs in handlers for server-side fetches"
    In jsdom/Node there is **no origin**, so a relative `fetch("/api/posts")` has nothing
    to resolve against. For Client-Component tests, configure a base URL or use absolute
    handler patterns. For data fetched **inside Server Components**, MSW-in-Node can
    intercept those server `fetch` calls too — but because async Server Components are
    awkward to render in jsdom (Section 1), it's usually cleaner to assert that behavior
    with **Playwright** against the real server, mocking upstream APIs with `page.route`.

!!! note "MSW shines for third-party APIs"
    The highest-value MSW use in Next.js is mocking **external** services (Stripe, a CMS,
    an auth provider) in component and integration tests, so you don't hit real endpoints.
    For your *own* Route Handlers, prefer calling the exported function directly
    (Section 4) — it's faster and doesn't need interception.

## 7. End-to-End Testing with Playwright

E2E is where Next.js testing gets its real confidence: a Playwright run drives a **real
browser** against a **real production build**, so Server Components, streaming, caching,
Server Actions, and middleware all execute for real. This is the recommended way to test
async Server Components and full user flows.

```bash
npm init playwright@latest
```

The key Next.js detail is the `webServer` block: build and start the app the way
production does (`next build && next start`), not `next dev`. The dev server is slower,
behaves differently around caching, and isn't what you ship.

```ts
// playwright.config.ts
import { defineConfig, devices } from "@playwright/test";

export default defineConfig({
  testDir: "./e2e",
  use: { baseURL: "http://localhost:3000", trace: "on-first-retry" },
  webServer: {
    command: "npm run build && npm run start", // next build && next start
    url: "http://localhost:3000",
    reuseExistingServer: !process.env.CI,
    timeout: 120_000, // a production build can take a while on first run
  },
  projects: [{ name: "chromium", use: { ...devices["Desktop Chrome"] } }],
});
```

!!! tip "Speed up local iteration"
    A full `next build` before every run is slow while developing tests. Locally you can
    point `webServer.command` at `npm run dev` and switch to the build command in CI via
    an env check. Just remember dev-mode caching differs from production, so the
    build-based config is the source of truth.

A test exercising a Server-Component page plus a Server Action — the exact things unit
tests can't cover:

```ts
// e2e/posts.spec.ts
import { test, expect } from "@playwright/test";

test("renders server-fetched posts", async ({ page }) => {
  await page.goto("/posts");
  // This HTML was rendered on the server from a real data fetch.
  await expect(page.getByRole("heading", { name: /posts/i })).toBeVisible();
  await expect(page.getByRole("listitem")).not.toHaveCount(0);
});

test("creates a post via a Server Action", async ({ page }) => {
  await page.goto("/posts/new");
  await page.getByLabel(/title/i).fill("My first post");
  await page.getByRole("button", { name: /create/i }).click();

  // The action redirected to the new post's page.
  await expect(page).toHaveURL(/\/posts\/\d+/);
  await expect(page.getByRole("heading", { name: "My first post" })).toBeVisible();
});
```

### Mocking upstream APIs and faking auth

Use `page.route` to stub the **external** services your server calls, and seed an
authenticated session by setting a cookie before navigating (the exact mechanism depends
on your auth setup — see the Authentication chapter).

```ts
test("shows weather from a mocked upstream API", async ({ page }) => {
  await page.route("https://api.weather.com/**", (route) =>
    route.fulfill({ json: { tempC: 21 } })
  );
  await page.goto("/weather");
  await expect(page.getByText("21°C")).toBeVisible();
});

test("dashboard requires a session", async ({ page, context }) => {
  await context.addCookies([
    { name: "session", value: "test-token", url: "http://localhost:3000" },
  ]);
  await page.goto("/dashboard");
  await expect(page.getByRole("heading", { name: /dashboard/i })).toBeVisible();
});
```

!!! tip "Reuse a logged-in state with storage state"
    Logging in through the UI in every test is slow. Do it **once** in a Playwright
    *setup project* (or global setup), save `context.storageState()` to a file, and load
    it via `use: { storageState: "e2e/.auth/user.json" }`. The Playwright auth docs cover
    this pattern in full — it's the standard way to test authenticated Next.js apps.

## 8. Cypress (Alternative)

**Cypress** is a solid alternative to Playwright for Next.js E2E, with its signature
time-travel runner. The setup mirrors any app: point it at your running server and write
specs. As with Playwright, run against a **production build** for trustworthy results.

```bash
npm install -D cypress
npx cypress open
```

```ts
// cypress/e2e/posts.cy.ts
describe("posts", () => {
  it("creates a post", () => {
    cy.visit("/posts/new");
    cy.findByLabelText(/title/i).type("My first post");
    cy.findByRole("button", { name: /create/i }).click();

    cy.url().should("match", /\/posts\/\d+/);
    cy.contains("My first post").should("be.visible");
  });
});
```

You start the server yourself before running Cypress (Cypress has no built-in `webServer`
like Playwright); the `start-server-and-test` package is the common helper:

```json
// package.json
{
  "scripts": {
    "e2e": "start-server-and-test 'npm run build && npm run start' http://localhost:3000 'cypress run'"
  }
}
```

!!! note "Component testing in Cypress"
    Cypress also has a **Component Testing** mode that can mount Next.js Client Components
    in a real browser. It's a middle ground between RTL and full E2E, but for Next.js
    most teams stick with Vitest + RTL for components and reserve Cypress/Playwright for
    flows. See the comparison table in the
    **[React Testing chapter](../react/16.React%20Testing%20and%20Debugging.md#5-end-to-end-e2e-testing)**.

## Summary

- **Pick the tool by what runs where.** Logic and Client Components → Vitest + RTL; Route
  Handlers and Server Actions → call the exported function directly; Server Components and
  full flows → Playwright.
- **Async Server Components aren't fully supported by RTL** — prefer E2E for them, or
  extract their logic into plain functions you can unit-test and keep the component thin.
- **Mock `next/navigation`** (`useRouter`/`usePathname`/`useSearchParams`) and
  **`next/headers`** (`cookies`/`headers`) with `vi.mock` for Client Component tests;
  remember `cookies()`/`headers()` are **async in Next 15**.
- **Route Handlers are just functions** — import `GET`/`POST`, pass a `Request`, assert on
  the returned `Response`. Dynamic `params` is a **Promise** in Next 15.
- **Server Actions are just functions** too — call them with a `FormData` and mock `db`,
  `revalidatePath`, `redirect`, and auth; remember `redirect()` throws by design.
- **MSW v2** mocks the network for Client Components and third-party APIs; for your own
  API, calling the handler directly is faster.
- **Playwright is the recommended E2E tool** — configure `webServer` to run
  `next build && next start` so tests hit a real production build; **Cypress** is the
  established alternative.
- For all the **general** testing fundamentals (queries, `user-event`, the testing
  pyramid, debugging, error boundaries), see
  **[React Testing and Debugging](../react/16.React%20Testing%20and%20Debugging.md)**.
