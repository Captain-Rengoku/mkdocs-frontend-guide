# :simple-nextdotjs: Routing in Next.js

The App Router is **file-system based**: the folders you create inside `app/` *are* your URL structure, and a handful of reserved file names (`page`, `layout`, `loading`, `error`, …) decide what each segment renders. There is no central route config to keep in sync—the directory tree is the routing table.

This chapter covers how URLs map to files, how layouts nest, the full range of dynamic and advanced route conventions (route groups, parallel and intercepting routes), the navigation APIs, and the special files that give each segment loading, error, and not-found UI. Throughout, the stack is **Next.js 15 + TypeScript + React 18/19**, App Router only.

> 📚 For more: [https://nextjs.org/docs/app/building-your-application/routing](https://nextjs.org/docs/app/building-your-application/routing)

---

## 1. File-based routing in `app/`

A **route** is created when a folder contains a `page.tsx`. The folder path (relative to `app/`) becomes the URL path; the `page.tsx` inside it makes that path publicly reachable.

```text
app/
├── layout.tsx          → required root layout (wraps every page)
├── page.tsx            → /
├── about/
│   └── page.tsx        → /about
└── blog/
    ├── page.tsx        → /blog
    └── [slug]/
        └── page.tsx    → /blog/:slug
```

A folder **without** a `page.tsx` is just a path segment used for organisation—it doesn't become a navigable URL on its own. Files you co-locate next to a `page.tsx` (components, styles, tests, utilities) are *not* routable; only the reserved file names are special.

!!! note "Reserved file names at a glance"
    | File | Role |
    | ---- | ---- |
    | `page.tsx` | The unique UI of a route (makes the segment public) |
    | `layout.tsx` | Shared shell that wraps a segment **and its children** |
    | `template.tsx` | Like a layout, but re-mounts on every navigation |
    | `loading.tsx` | Suspense fallback for the segment |
    | `error.tsx` | Error boundary for the segment (Client Component) |
    | `not-found.tsx` | UI for `notFound()` / unmatched routes |
    | `route.ts` | An API endpoint (Route Handler) instead of UI |

### `page.tsx` — the leaf UI

A `page.tsx` exports a default React component. It's a **Server Component by default** and receives `params` and `searchParams` as props.

```tsx
// app/page.tsx — the home page, rendered at "/"
export default function HomePage() {
  return (
    <main>
      <h1>Welcome</h1>
      <p>This is the root route.</p>
    </main>
  );
}
```

```tsx
// app/dashboard/page.tsx — a Server Component reading the URL's query string
export default async function DashboardPage({
  searchParams,
}: {
  searchParams: Promise<{ tab?: string }>;
}) {
  const { tab = "overview" } = await searchParams; // Next.js 15: searchParams is async
  return <h1>Dashboard — {tab}</h1>;
}
```

!!! note "`params` and `searchParams` are Promises in Next.js 15"
    In Next.js 15, `params` and `searchParams` (and `cookies()`/`headers()`) are **async** and must be `await`ed in Server Components, or unwrapped with `use()` in Client Components. In Next.js 14 they were plain synchronous objects. Every example here uses the v15 async form.

### `layout.tsx` — the shared shell

A `layout.tsx` wraps the `page.tsx` in its segment **and everything nested below it**. It receives a `children` prop—the active page or child layout—and must render it. The **root layout** (`app/layout.tsx`) is required, must render `<html>` and `<body>`, and is the only place those tags live.

```tsx
// app/layout.tsx — the required root layout
import type { Metadata } from "next";
import type { ReactNode } from "react";

export const metadata: Metadata = {
  title: "My App",
  description: "Built with the Next.js App Router",
};

export default function RootLayout({ children }: { children: ReactNode }) {
  return (
    <html lang="en">
      <body>
        <header>Global header</header>
        {children}
        <footer>Global footer</footer>
      </body>
    </html>
  );
}
```

!!! tip "Layouts preserve state across navigation"
    When you navigate between routes that share a layout, the layout **does not re-render or unmount**—its DOM and React state are preserved, and only the changing `children` swap out. That's why scroll position in a persistent sidebar or the contents of an `<input>` in a layout survive page changes. (Use `template.tsx` when you need the opposite.)

## 2. Nested routes & nested layouts

Nesting folders nests both the **URL** and the **layouts**. Each segment can have its own `layout.tsx`, and Next.js composes them from the root inward—the root layout wraps the section layout, which wraps the page.

```text
app/
├── layout.tsx              → RootLayout
└── dashboard/
    ├── layout.tsx          → DashboardLayout (sidebar, tabs…)
    ├── page.tsx            → /dashboard
    ├── settings/
    │   └── page.tsx        → /dashboard/settings
    └── analytics/
        └── page.tsx        → /dashboard/analytics
```

For `/dashboard/settings`, the rendered tree is:

```text
RootLayout
  └── DashboardLayout
        └── SettingsPage
```

```tsx
// app/dashboard/layout.tsx — wraps every route under /dashboard
import Link from "next/link";
import type { ReactNode } from "react";

export default function DashboardLayout({ children }: { children: ReactNode }) {
  return (
    <div className="dashboard">
      <nav>
        <Link href="/dashboard">Home</Link>
        <Link href="/dashboard/settings">Settings</Link>
        <Link href="/dashboard/analytics">Analytics</Link>
      </nav>
      <section>{children}</section>
    </div>
  );
}
```

!!! note "Layouts nest; pages don't"
    A layout always wraps its descendants, so deeper layouts stack. Pages are leaves—only one `page.tsx` renders for a given URL, at the deepest matched segment.

## 3. Dynamic routes

When part of a URL is data (a post slug, a user id), wrap the folder name in **square brackets**. The bracketed name becomes a key on `params`.

### `[id]` — a single dynamic segment

```text
app/users/[id]/page.tsx   → /users/1, /users/2, /users/anything
```

```tsx
// app/users/[id]/page.tsx
export default async function UserPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params; // e.g. "42" for /users/42
  const user = await fetch(`https://api.example.com/users/${id}`).then((r) =>
    r.json()
  );
  return <h1>{user.name}</h1>;
}
```

Multiple dynamic segments in one path each get their own key:

```text
app/[category]/[product]/page.tsx   → /shoes/runner  →  { category, product }
```

```tsx
// app/[category]/[product]/page.tsx
export default async function ProductPage({
  params,
}: {
  params: Promise<{ category: string; product: string }>;
}) {
  const { category, product } = await params;
  return <h1>{category} / {product}</h1>;
}
```

### `[...slug]` — catch-all segments

A catch-all matches **one or more** path segments and gives you an **array**. It does *not* match the parent route with no segments.

```text
app/docs/[...slug]/page.tsx
  /docs/a          → { slug: ["a"] }
  /docs/a/b/c      → { slug: ["a", "b", "c"] }
  /docs            → ❌ not matched (needs at least one segment)
```

```tsx
// app/docs/[...slug]/page.tsx
export default async function DocsPage({
  params,
}: {
  params: Promise<{ slug: string[] }>;
}) {
  const { slug } = await params; // always an array, length >= 1
  return <h1>Docs path: {slug.join(" / ")}</h1>;
}
```

### `[[...slug]]` — optional catch-all

Double brackets make the catch-all **optional**: it *also* matches the parent route, where `slug` is then `undefined`. Use this when one component should handle both `/shop` and `/shop/any/depth`.

```text
app/shop/[[...slug]]/page.tsx
  /shop            → { slug: undefined }
  /shop/sale       → { slug: ["sale"] }
  /shop/men/shoes  → { slug: ["men", "shoes"] }
```

```tsx
// app/shop/[[...slug]]/page.tsx
export default async function ShopPage({
  params,
}: {
  params: Promise<{ slug?: string[] }>;
}) {
  const { slug } = await params; // undefined at /shop
  if (!slug) return <h1>Shop home</h1>;
  return <h1>Filtering by: {slug.join(", ")}</h1>;
}
```

!!! note "Catch-all vs optional catch-all"
    | Convention | Matches parent (no segment)? | `params` value |
    | ---------- | --------------------------- | -------------- |
    | `[...slug]` | ❌ No | `string[]` (≥ 1) |
    | `[[...slug]]` | ✅ Yes | `string[]` or `undefined` |

### Pre-rendering dynamic routes with `generateStaticParams`

To statically generate dynamic routes at build time (SSG), export `generateStaticParams`—it's the App Router replacement for the legacy `getStaticPaths`.

```tsx
// app/blog/[slug]/page.tsx
export async function generateStaticParams() {
  const posts: { slug: string }[] = await fetch(
    "https://api.example.com/posts"
  ).then((r) => r.json());
  return posts.map((post) => ({ slug: post.slug })); // one entry per pre-rendered page
}

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = await fetch(`https://api.example.com/posts/${slug}`).then((r) =>
    r.json()
  );
  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  );
}
```

## 4. Route groups `(group)`

Wrapping a folder name in **parentheses** creates a *route group*. The folder name is used purely for organisation and is **stripped from the URL**. Route groups let you:

1. Group routes by feature/team without affecting the path.
2. Apply a **different layout to a subset of routes** at the same URL level.
3. Split one app into **multiple root layouts**.

```text
app/
├── (marketing)/
│   ├── layout.tsx       → marketing layout
│   ├── page.tsx         → /          (NOT /marketing)
│   └── about/page.tsx   → /about
└── (shop)/
    ├── layout.tsx       → shop layout
    ├── cart/page.tsx    → /cart
    └── checkout/page.tsx→ /checkout
```

Both groups sit at the URL root, but each gets its own `layout.tsx`. A visitor on `/about` gets the marketing layout; on `/cart` they get the shop layout—even though neither "marketing" nor "shop" appears in the URL.

!!! warning "Two root layouts → full page reload between them"
    You can give each group its own root layout (each rendering `<html>`/`<body>`) by **omitting** a shared `app/layout.tsx`. But navigating *between* two different root layouts triggers a **full page reload**, not a client-side transition. Also avoid creating two routes that resolve to the **same URL** across groups (e.g. `(a)/page.tsx` and `(b)/page.tsx` both at `/`)—that's a build error.

## 5. Parallel routes `@slot`

Parallel routes let a single layout render **multiple pages in the same view simultaneously**, each in its own *slot*. A slot is a folder prefixed with `@`. Slots are passed to the layout as **named props** (the `@` folder name), alongside the usual `children`.

```text
app/dashboard/
├── layout.tsx
├── page.tsx          → the implicit "children" slot
├── @team/
│   └── page.tsx      → the "team" slot
└── @analytics/
    └── page.tsx      → the "analytics" slot
```

```tsx
// app/dashboard/layout.tsx — slots arrive as named props
import type { ReactNode } from "react";

export default function DashboardLayout({
  children,
  team,
  analytics,
}: {
  children: ReactNode;
  team: ReactNode;
  analytics: ReactNode;
}) {
  return (
    <div className="grid">
      <section>{children}</section>
      <aside>{team}</aside>
      <aside>{analytics}</aside>
    </div>
  );
}
```

Like `(group)` folders, `@slot` folders **do not** add anything to the URL—`@team/page.tsx` is *not* reachable at `/dashboard/team`. Each slot can have its own `loading.tsx` and `error.tsx`, so the sections stream and fail independently.

### `default.tsx` for unmatched slots

When you navigate to a URL that matches one slot but not another, the unmatched slot falls back to its `default.tsx` (and to the currently-rendered content on soft navigation). Providing `default.tsx` avoids a 404 on hard reload.

```tsx
// app/dashboard/@team/default.tsx — fallback when this slot has no match
export default function Default() {
  return null; // render nothing for the team slot
}
```

!!! tip "What parallel routes are for"
    Dashboards with independently-loading panels, **modals** (a slot that overlays content), and conditional UI (e.g. render `@login` vs `@dashboard` based on auth) all map naturally onto parallel routes.

## 6. Intercepting routes `(.)`, `(..)`

Intercepting routes let you **load a route's content within the current layout** instead of navigating away to it—most famously to show a route as a **modal** when reached via in-app navigation, while a hard refresh or direct visit shows the full standalone page.

The convention mirrors relative imports, using parentheses:

| Convention | Intercepts |
| ---------- | ---------- |
| `(.)` | the **same** level |
| `(..)` | **one level up** |
| `(..)(..)` | **two levels up** |
| `(...)` | from the **root** `app` |

A common pattern is a photo gallery where clicking a thumbnail opens the photo in a modal, but `/photos/3` loaded directly renders the full page. Combine an intercepting route with a parallel `@modal` slot:

```text
app/
├── layout.tsx
├── @modal/
│   ├── default.tsx
│   └── (.)photos/
│       └── [id]/page.tsx   → intercepts /photos/[id] → shows a modal
├── photos/
│   └── [id]/page.tsx       → the full standalone page
└── page.tsx                → the gallery with <Link>s to /photos/[id]
```

```tsx
// app/@modal/(.)photos/[id]/page.tsx — shown when /photos/[id] is reached via <Link>
import { Modal } from "@/components/modal";

export default async function PhotoModal({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  return (
    <Modal>
      <img src={`/photos/${id}.jpg`} alt={`Photo ${id}`} />
    </Modal>
  );
}
```

```tsx
// app/layout.tsx — the @modal slot renders alongside children
import type { ReactNode } from "react";

export default function RootLayout({
  children,
  modal,
}: {
  children: ReactNode;
  modal: ReactNode;
}) {
  return (
    <html lang="en">
      <body>
        {children}
        {modal}
      </body>
    </html>
  );
}
```

!!! note "Soft vs hard navigation"
    The interception only happens on **soft (client-side) navigation**—a `<Link>` click from inside the app. A **hard navigation** (typing the URL, refreshing, or arriving from outside) bypasses the interceptor and renders the real `photos/[id]/page.tsx`. That's exactly the modal-vs-full-page behaviour you want.

## 7. Navigation

Navigation between routes is **client-side by default**: Next.js prefetches and swaps only the changed segments, preserving shared layouts. There are declarative (`<Link>`) and imperative (`useRouter`) APIs, plus server-side `redirect`, and hooks to read the current URL.

### `<Link>` — the primary navigation primitive

`<Link>` renders an `<a>` but performs a client-side transition and **prefetches** the target in the background when it enters the viewport. It works in both Server and Client Components.

```tsx
// Works in a Server Component — no "use client" needed
import Link from "next/link";

export default function Nav() {
  return (
    <nav>
      <Link href="/">Home</Link>
      <Link href="/blog">Blog</Link>
      <Link href="/users/42">User 42</Link>
      {/* Object form with query string */}
      <Link href={{ pathname: "/search", query: { q: "next" } }}>Search</Link>
      <Link href="/dashboard" prefetch={false}>
        Dashboard (no prefetch)
      </Link>
    </nav>
  );
}
```

### `useRouter` — imperative navigation (Client Component)

For programmatic navigation (after a form submit, a timeout, etc.), use `useRouter` from **`next/navigation`** (not the legacy `next/router`). This is a hook, so its component **must be a Client Component**.

```tsx
"use client"; // required: useRouter is a hook
import { useRouter } from "next/navigation";

export default function LoginForm() {
  const router = useRouter();

  async function onSubmit(formData: FormData) {
    await fetch("/api/login", { method: "POST", body: formData });
    router.push("/dashboard");   // navigate, adds a history entry
    // router.replace("/dashboard"); // navigate without adding history
    // router.back();                // history back
    // router.refresh();             // re-fetch server data for the current route
  }

  return <form action={onSubmit}>{/* fields */}</form>;
}
```

!!! note "`next/navigation`, not `next/router`"
    In the App Router, all navigation hooks (`useRouter`, `usePathname`, `useSearchParams`, `useParams`) come from **`next/navigation`**. The Pages Router's `next/router` does **not** work in `app/`.

### `redirect` — server-side redirect

`redirect()` (from `next/navigation`) is for **Server Components, Server Actions, and Route Handlers**. It throws to interrupt rendering, so call it *outside* try/catch blocks and after any work that must complete first.

```tsx
// app/dashboard/page.tsx — redirect on the server before rendering
import { redirect } from "next/navigation";
import { getSession } from "@/lib/auth";

export default async function DashboardPage() {
  const session = await getSession();
  if (!session) {
    redirect("/login"); // 307 redirect; nothing below runs
  }
  return <h1>Hello, {session.user.name}</h1>;
}
```

Use `permanentRedirect()` for a 308 (permanent) instead of a 307.

### `usePathname` & `useSearchParams` — reading the URL (Client Component)

Both are hooks from `next/navigation`, so they require `"use client"`. `usePathname` returns the current path string; `useSearchParams` returns a read-only `URLSearchParams`.

```tsx
"use client"; // required: both are hooks
import { usePathname, useSearchParams } from "next/navigation";
import Link from "next/link";

export default function ActiveLink({ href, label }: { href: string; label: string }) {
  const pathname = useSearchParamsAware(href);
  return (
    <Link href={href} aria-current={pathname ? "page" : undefined}>
      {label}
    </Link>
  );
}

function useSearchParamsAware(href: string) {
  const pathname = usePathname();             // e.g. "/blog"
  const searchParams = useSearchParams();     // e.g. ?page=2
  const page = searchParams.get("page");      // "2" | null
  return pathname === href && !page;
}
```

!!! warning "`useSearchParams` needs a Suspense boundary"
    A Client Component calling `useSearchParams` should be wrapped in `<Suspense>`. Without it, the route opts out of static rendering and is forced fully dynamic during the build. To **read** query params on the server instead, use the `searchParams` prop on `page.tsx`—no hook required.

!!! tip "Updating the URL without a navigation"
    To change query params in place (filters, tabs) you typically build a new query string and call `router.replace(\`?\${params.toString()}\`)`. Libraries like `nuqs` wrap this pattern with typed, `useState`-like ergonomics for "URL as state."

## 8. Special files

Each route segment can declare reserved files that Next.js wires into the React tree automatically—loading fallbacks, error boundaries, and not-found UI—without you writing the Suspense/error-boundary plumbing.

### `loading.tsx` — instant loading UI

Adding `loading.tsx` wraps the segment's `page.tsx` in a `<Suspense>` boundary for you. While the page's async work resolves, the fallback shows immediately and the rest of the already-rendered shell (the layout) stays put.

```tsx
// app/dashboard/loading.tsx — shown while DashboardPage's data loads
export default function Loading() {
  return <p>Loading dashboard…</p>; // a skeleton goes here in real apps
}
```

### `error.tsx` — segment error boundary (Client Component)

`error.tsx` is a React **error boundary** for its segment. It **must be a Client Component** (`"use client"`) and receives the `error` plus a `reset()` function to retry rendering the segment.

```tsx
// app/dashboard/error.tsx
"use client"; // required: error boundaries are Client Components
import { useEffect } from "react";

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    console.error(error); // report to your monitoring service
  }, [error]);

  return (
    <div role="alert">
      <h2>Something went wrong.</h2>
      <button onClick={() => reset()}>Try again</button>
    </div>
  );
}
```

!!! note "`error.tsx` does not catch the layout in its own segment"
    An `error.tsx` catches errors thrown by its segment's `page` and **nested** children—but **not** the `layout.tsx` *of the same segment* (the error boundary lives inside that layout). To catch a segment's own layout errors, add an `error.tsx` one level up. For the root layout, use the special `app/global-error.tsx`, which replaces the entire page (and must render its own `<html>`/`<body>`).

### `not-found.tsx` — 404 UI

`not-found.tsx` renders when you call `notFound()` from a segment, and the root `app/not-found.tsx` also handles any unmatched URL. `notFound()`, like `redirect()`, throws to halt rendering.

```tsx
// app/users/[id]/page.tsx — trigger the nearest not-found boundary
import { notFound } from "next/navigation";

export default async function UserPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const user = await fetch(`https://api.example.com/users/${id}`).then((r) =>
    r.ok ? r.json() : null
  );
  if (!user) notFound(); // renders the nearest not-found.tsx, sends a 404
  return <h1>{user.name}</h1>;
}
```

```tsx
// app/not-found.tsx — root 404 page
import Link from "next/link";

export default function NotFound() {
  return (
    <div>
      <h2>Page not found</h2>
      <Link href="/">Go home</Link>
    </div>
  );
}
```

### `template.tsx` — a layout that re-mounts

`template.tsx` is like `layout.tsx`—it wraps `children`—but it creates a **new instance on every navigation**: DOM is recreated, state resets, and effects re-run. Reach for it when you specifically *want* a fresh start per route, e.g. enter animations or per-page analytics that should fire on each visit.

```tsx
// app/template.tsx — re-runs effects and resets state on each navigation
"use client"; // common, since templates usually animate/track on mount
import { useEffect } from "react";
import type { ReactNode } from "react";

export default function Template({ children }: { children: ReactNode }) {
  useEffect(() => {
    // fires on EVERY navigation into a route using this template
  }, []);
  return <div className="page-transition">{children}</div>;
}
```

!!! note "`layout.tsx` vs `template.tsx`"
    | | `layout.tsx` | `template.tsx` |
    | --- | --- | --- |
    | State across navigation | **Preserved** | **Reset** |
    | DOM across navigation | Reused | Re-created |
    | Effects on navigation | Run once | Run every time |
    | Use when | Persistent nav/sidebars (the default) | Per-visit animations/logging |

### Route segment config

Each `page.tsx`, `layout.tsx`, or `route.ts` can export config constants that control how the segment is rendered, cached, and run. They are **static exports** (not function calls) read at build time.

```tsx
// app/products/page.tsx — segment-level configuration
export const dynamic = "force-dynamic"; // "auto" | "force-dynamic" | "error" | "force-static"
export const revalidate = 3600;          // ISR window in seconds (or false / 0)
export const dynamicParams = true;       // allow params not in generateStaticParams
export const fetchCache = "auto";        // override default fetch caching
export const runtime = "nodejs";         // "nodejs" (default) | "edge"
export const preferredRegion = "auto";   // deployment region hint

export default async function Products() {
  const products = await fetch("https://api.example.com/products").then((r) =>
    r.json()
  );
  return <ul>{products.map((p: { id: string; name: string }) => <li key={p.id}>{p.name}</li>)}</ul>;
}
```

!!! tip "The two you'll reach for most"
    `dynamic` and `revalidate` are the everyday levers. `export const dynamic = "force-dynamic"` opts a route fully into SSR (re-rendered per request); `export const revalidate = N` turns a static route into ISR (regenerated at most every `N` seconds). Leave the rest at their defaults until you have a specific reason. See the *Data Fetching & Caching* chapter for how these interact with the cache.

## 📚 Summary

| 🔹 Convention | 🧠 What it does |
| ------------- | --------------- |
| `page.tsx` | Makes a segment a public route (Server Component by default) |
| `layout.tsx` | Shared shell wrapping a segment + children; **state preserved** |
| Nested folders | Nest both URL and layouts (root → section → page) |
| `[id]` | Single dynamic segment → `params.id` (string) |
| `[...slug]` | Catch-all → `params.slug` (string[], ≥ 1) |
| `[[...slug]]` | Optional catch-all → `string[]` or `undefined` |
| `(group)` | Route group: organise/share layouts, **no URL segment** |
| `@slot` | Parallel route rendered as a named layout prop, **no URL segment** |
| `(.)` `(..)` `(...)` | Intercepting routes (e.g. modals on soft navigation) |
| `<Link>` | Declarative client-side navigation + prefetch (any component) |
| `useRouter` | Imperative navigation — **Client Component** (`next/navigation`) |
| `redirect()` / `notFound()` | Server-side control flow (throw to interrupt render) |
| `usePathname` / `useSearchParams` | Read the URL — **Client Component** hooks |
| `loading.tsx` | Auto Suspense fallback for the segment |
| `error.tsx` | Segment error boundary — **Client Component** |
| `not-found.tsx` | 404 UI (root file also catches unmatched URLs) |
| `template.tsx` | Layout that **re-mounts** per navigation |
| Segment config | `dynamic`, `revalidate`, `runtime`, … per segment |
