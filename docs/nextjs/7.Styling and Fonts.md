# :art: Styling & Fonts

The styling **options** in Next.js are the same ones you already know from React — CSS Modules, global CSS, Sass, Tailwind, and CSS-in-JS — but the App Router changes the *rules*. Because components render on the server by default (React Server Components), anything that depends on a browser runtime has to declare itself, and a runtime CSS-in-JS library can't just be dropped in. On the upside, Next.js builds in things you'd otherwise wire up by hand: first-class CSS Modules and Sass support, a Tailwind-ready PostCSS pipeline, and `next/font`, which optimizes web fonts and eliminates font-driven layout shift automatically.

This chapter assumes you've read the React **Styling Techniques** chapter — the fundamentals of CSS Modules, Tailwind, and CSS-in-JS carry over unchanged. Here we focus on the **Next.js 15 App Router** angle: where each approach plugs into the framework, what the server/client boundary means for styling, and how to load fonts the Next.js way.

> 📚 Stack: **Next.js 15, App Router, TypeScript**. Scaffold with `npx create-next-app@latest my-app --typescript --app`.

## 1. CSS in Next.js

Next.js has a built-in CSS pipeline (powered by PostCSS and Lightning CSS / its bundler), so `.css` and `.module.css` files just work — no loader configuration. There are three flavors: **global CSS**, **CSS Modules**, and **Sass**.

### Global CSS

Global stylesheets apply everywhere and aren't scoped. Use them for resets, CSS custom properties (design tokens), base typography, and `body`-level rules. In the App Router, import your global CSS **once** into the root layout (`app/layout.tsx`):

```tsx
// app/layout.tsx
import "./globals.css";

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

```css
/* app/globals.css */
:root {
  --color-primary: #2563eb;
  --color-text: #111827;
  --radius: 6px;
}

* {
  box-sizing: border-box;
  margin: 0;
}

body {
  font-family: system-ui, sans-serif;
  color: var(--color-text);
}
```

!!! note "Where global CSS can be imported"
    In the App Router you can import a global (non-module) `.css` file from **any layout, page, or component** — not just the root layout, unlike the old Pages Router which restricted global imports to `pages/_app.tsx`. That said, importing it in the **root layout** is the convention: it guarantees the styles load on every route and keeps the import in one obvious place.

### CSS Modules

A CSS Module is a regular CSS file named `*.module.css` whose class names are **locally scoped** — at build time each class is hashed to a unique name so it can never collide. This is the React behavior you already know, and it's enabled out of the box in Next.js. Crucially, **CSS Modules work in Server Components**: they're resolved entirely at build time and ship zero JavaScript, so there's no `"use client"` requirement.

```css
/* app/components/Button.module.css */
.btn {
  padding: 8px 16px;
  border: none;
  border-radius: var(--radius);
  cursor: pointer;
}
.primary {
  background: var(--color-primary);
  color: #fff;
}
```

```tsx
// app/components/Button.tsx — a Server Component, no "use client" needed
import styles from "./Button.module.css";
import clsx from "clsx";

export default function Button({
  variant = "primary",
  children,
}: {
  variant?: "primary" | "ghost";
  children: React.ReactNode;
}) {
  return <button className={clsx(styles.btn, styles[variant])}>{children}</button>;
}
```

!!! tip "CSS Modules are the safe default in the App Router"
    Because they're zero-runtime and server-compatible, CSS Modules are the lowest-friction "real CSS" option in Next.js. You get full CSS power (pseudo-classes, media queries, keyframes), guaranteed scope isolation, and no impact on the client bundle. The tiny [`clsx`](https://github.com/lukeed/clsx) helper handles conditional classes, exactly as in plain React.

### Sass

Next.js supports `.scss`/`.sass` natively — install `sass` and the framework wires up the loader. Both global Sass and **Sass Modules** (`*.module.scss`) are supported.

```bash
npm install --save-dev sass
```

```scss
// app/components/Card.module.scss
$radius: 8px;

.card {
  border: 1px solid #e5e7eb;
  border-radius: $radius;
  padding: 16px;

  &__title {
    font-weight: 700;
  }

  &--featured {
    border-color: var(--color-primary);
  }
}
```

```tsx
// app/components/Card.tsx
import styles from "./Card.module.scss";

export default function Card({ title }: { title: string }) {
  return (
    <div className={styles.card}>
      <h3 className={styles.__title}>{title}</h3>
    </div>
  );
}
```

You can configure Sass globally via `next.config.ts` — for example to register load paths so `@use` can resolve shared partials without long relative paths:

```ts
// next.config.ts
import type { NextConfig } from "next";
import path from "node:path";

const nextConfig: NextConfig = {
  sassOptions: {
    includePaths: [path.join(process.cwd(), "styles")],
  },
};

export default nextConfig;
```

!!! note "Modern Sass syntax"
    Dart Sass deprecated `@import` in favor of the module system (`@use` / `@forward`). New projects should reach for `@use "variables" as vars;` rather than `@import`. The deprecation warnings you may see come from Sass itself, not Next.js.

## 2. Tailwind CSS

[Tailwind CSS](https://tailwindcss.com) is the most common styling choice for new Next.js apps — `create-next-app` even offers to set it up for you. It's **utility-first**: you compose small classes (`flex`, `pt-4`, `bg-blue-600`) in your markup, and the build emits only the CSS you actually use. Tailwind is **zero-runtime** and works in Server Components without `"use client"`, which makes it a natural fit for the App Router.

The setup differs by major version. Tailwind **v4** (the current default) is CSS-first and configured through a PostCSS plugin; **v3.4** uses the older `tailwind.config.js` + directives model.

=== "Tailwind v4 (recommended)"

    Install Tailwind and its PostCSS plugin:

    ```bash
    npm install tailwindcss @tailwindcss/postcss
    ```

    Register the plugin in PostCSS. Next.js auto-detects `postcss.config.mjs`:

    ```js
    // postcss.config.mjs
    const config = {
      plugins: {
        "@tailwindcss/postcss": {},
      },
    };

    export default config;
    ```

    A **single import** replaces the old `@tailwind base/components/utilities` directives:

    ```css
    /* app/globals.css */
    @import "tailwindcss";
    ```

    That's it — there's no JS config file and no `content` array by default; v4 auto-detects your source files. Import `globals.css` in the root layout (as in §1) and you're done.

=== "Tailwind v3.4"

    The v3 flow uses three dev dependencies and an init step:

    ```bash
    npm install -D tailwindcss postcss autoprefixer
    npx tailwindcss init -p
    ```

    Point the `content` array at your App Router source so the JIT engine knows where to scan for classes:

    ```js
    // tailwind.config.js
    /** @type {import('tailwindcss').Config} */
    export default {
      content: [
        "./app/**/*.{js,ts,jsx,tsx,mdx}",
        "./components/**/*.{js,ts,jsx,tsx,mdx}",
      ],
      theme: { extend: {} },
      plugins: [],
    };
    ```

    ```css
    /* app/globals.css */
    @tailwind base;
    @tailwind components;
    @tailwind utilities;
    ```

!!! warning "v4 changed the PostCSS package"
    In Tailwind **v3** the PostCSS plugin was the `tailwindcss` package itself. In **v4** the PostCSS integration moved to a separate **`@tailwindcss/postcss`** package, and the v3 `tailwind.config.js` is replaced by CSS-based configuration. Mixing the two (e.g. v4's import with a v3 PostCSS setup) is a common source of "Tailwind classes aren't applying" bugs. (Tailwind v4 also ships a first-party Vite plugin, but in a Next.js project you use the **PostCSS** plugin, since Next.js owns the bundler.)

### Using utilities and customizing the theme

Utilities, responsive prefixes (`md:`), and state prefixes (`hover:`, `focus:`, `dark:`) behave exactly as in any Tailwind project:

```tsx
// app/components/Hero.tsx — a Server Component
export default function Hero() {
  return (
    <section className="mx-auto max-w-2xl p-8 text-center">
      <h1 className="mb-2 text-3xl font-bold text-gray-900 dark:text-white">
        Build fast
      </h1>
      <button className="mt-4 rounded-md bg-blue-600 px-4 py-2 text-white hover:bg-blue-700">
        Get started
      </button>
    </section>
  );
}
```

In **v4**, you customize design tokens in CSS with `@theme`; in **v3** you extend `tailwind.config.js`:

=== "v4 — CSS `@theme`"

    ```css
    @import "tailwindcss";

    @theme {
      --color-brand: #2563eb;
      --font-display: "Inter", sans-serif;
    }
    ```

    These tokens generate utilities automatically (`bg-brand`, `font-display`).

=== "v3 — `tailwind.config.js`"

    ```js
    export default {
      content: ["./app/**/*.{js,ts,jsx,tsx,mdx}"],
      theme: {
        extend: {
          colors: { brand: "#2563eb" },
          fontFamily: { display: ["Inter", "sans-serif"] },
        },
      },
      plugins: [],
    };
    ```

For conditional and merge-aware class composition, the `clsx` + [`tailwind-merge`](https://github.com/dcastil/tailwind-merge) pairing is standard (see the React chapter). And as you'll see in §5, Tailwind utilities combine perfectly with `next/font` CSS variables.

## 3. CSS-in-JS with the App Router

This is where Next.js diverges most sharply from a plain React app. Runtime CSS-in-JS libraries like **styled-components** and **Emotion** rely on React Context and generate styles in the browser — both of which assume a *client* runtime. In the App Router, components are **Server Components by default**, and Server Components have no client runtime, no Context, and no `useState`. So a runtime CSS-in-JS library cannot be used directly in a Server Component.

Two things are needed to make it work:

1. The components that use CSS-in-JS must be **Client Components** (`"use client"`).
2. A **style registry** must collect the styles rendered on the server and inject them into the initial HTML, so the first paint isn't unstyled. Next.js exposes the `useServerInsertedHTML` hook for exactly this.

### The style registry pattern (styled-components)

```bash
npm install styled-components
```

Create a client-side registry component. It buffers the styles styled-components generates during SSR and flushes them into `<head>` via `useServerInsertedHTML`:

```tsx
// app/lib/styled-components-registry.tsx
"use client";

import { useState } from "react";
import { useServerInsertedHTML } from "next/navigation";
import { ServerStyleSheet, StyleSheetManager } from "styled-components";

export default function StyledComponentsRegistry({
  children,
}: {
  children: React.ReactNode;
}) {
  // Create the stylesheet only once, lazily, per render.
  const [styledComponentsStyleSheet] = useState(() => new ServerStyleSheet());

  useServerInsertedHTML(() => {
    const styles = styledComponentsStyleSheet.getStyleElement();
    styledComponentsStyleSheet.instance.clearTag();
    return <>{styles}</>;
  });

  // On the client, render children as-is (the browser takes over).
  if (typeof window !== "undefined") return <>{children}</>;

  return (
    <StyleSheetManager sheet={styledComponentsStyleSheet.instance}>
      {children}
    </StyleSheetManager>
  );
}
```

Wrap the app with the registry in the root layout. The registry is a Client Component, but it can wrap Server Component `children` — the boundary stays at the registry:

```tsx
// app/layout.tsx
import StyledComponentsRegistry from "./lib/styled-components-registry";

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en">
      <body>
        <StyledComponentsRegistry>{children}</StyledComponentsRegistry>
      </body>
    </html>
  );
}
```

You'll also want SWC's styled-components transform on for stable class names and SSR correctness:

```ts
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  compiler: {
    styledComponents: true,
  },
};

export default nextConfig;
```

Now any **Client Component** can use styled-components normally:

```tsx
// app/components/FancyButton.tsx
"use client";

import styled from "styled-components";

const Button = styled.button<{ $primary?: boolean }>`
  padding: 8px 16px;
  border-radius: 6px;
  border: none;
  cursor: pointer;
  background: ${(p) => (p.$primary ? "#2563eb" : "#e5e7eb")};
  color: ${(p) => (p.$primary ? "white" : "#111827")};
`;

export default function FancyButton() {
  return <Button $primary>Click me</Button>;
}
```

!!! note "Emotion follows the same shape"
    Emotion needs an equivalent registry built on its `@emotion/cache` + `CacheProvider`, again flushed through `useServerInsertedHTML`. The mechanics differ but the pattern is identical: a `"use client"` registry that injects server-collected styles. As with styled-components, every component that calls Emotion's `css` prop or styled API must itself be a Client Component.

!!! warning "RSC limitations — CSS-in-JS opts you out of the server"
    The catch is fundamental, not cosmetic: **runtime CSS-in-JS does not work in Server Components**, so adopting it pushes a `"use client"` boundary across your styled UI and ships those components (plus the library runtime) to the browser. That undercuts a core App Router benefit — shipping less JavaScript. There's also a per-render serialization cost in the browser, the same one called out in the React chapter.

!!! tip "Prefer zero-runtime styling in the App Router"
    For new App Router projects, the Next.js docs (and the broader ecosystem) recommend **zero-runtime** styling: **CSS Modules**, **Tailwind**, or a **compile-time CSS-in-JS** library that emits static CSS — [Pigment CSS](https://github.com/mui/pigment-css), [vanilla-extract](https://vanilla-extract.style/), or [Linaria](https://linaria.dev/). These work in Server Components, ship no styling runtime, and keep the client bundle small. Reach for styled-components / Emotion mainly when you're **migrating** an existing app or already standardized on them (e.g. a heavy MUI codebase).

## 4. Fonts with `next/font`

Custom web fonts are a classic source of two problems: a **network request to Google's servers** (a privacy and latency cost) and **layout shift** when the web font swaps in for the fallback (hurting CLS). The built-in `next/font` module solves both.

!!! note "What `next/font` does for you"
    - **Self-hosts the font** — at build time Google Fonts files are downloaded and served from your own deployment. No request ever goes to `fonts.googleapis.com` at runtime (better privacy, no extra DNS/connection round-trips).
    - **Zero layout shift** — it computes the font's metrics and applies a matched `size-adjust` on the fallback, so the swap doesn't reflow text (improving CLS).
    - **Self-optimizes** — fonts are preloaded and scoped automatically; no manual `<link rel="preload">`.

### Google fonts

Import the font from `next/font/google`, call it as a function, and apply the generated `className`. Do this in a layout (commonly the root) so it's loaded once:

```tsx
// app/layout.tsx
import { Inter } from "next/font/google";
import "./globals.css";

const inter = Inter({
  subsets: ["latin"],
  display: "swap",
});

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en" className={inter.className}>
      <body>{children}</body>
    </html>
  );
}
```

!!! tip "Variable fonts need no weight array"
    `Inter` is a **variable font**, so you don't list weights — the whole axis ships in one file. For a **non-variable** font you must specify the weights (and styles) you use, which keeps the download small:

    ```tsx
    import { Roboto } from "next/font/google";

    const roboto = Roboto({
      weight: ["400", "700"],
      style: ["normal", "italic"],
      subsets: ["latin"],
      display: "swap",
    });
    ```

### CSS variables (and pairing fonts with Tailwind)

Instead of (or alongside) `className`, expose the font as a **CSS custom property** with the `variable` option. This lets you reference it from CSS or wire it into a Tailwind theme — and use multiple fonts cleanly (e.g. a body font and a display font):

```tsx
// app/layout.tsx
import { Inter, Roboto_Mono } from "next/font/google";

const inter = Inter({
  subsets: ["latin"],
  variable: "--font-sans",
  display: "swap",
});

const robotoMono = Roboto_Mono({
  subsets: ["latin"],
  variable: "--font-mono",
  display: "swap",
});

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en" className={`${inter.variable} ${robotoMono.variable}`}>
      <body>{children}</body>
    </html>
  );
}
```

Now `--font-sans` and `--font-mono` are available everywhere. Reference them in CSS:

```css
/* app/globals.css */
body {
  font-family: var(--font-sans), system-ui, sans-serif;
}
code,
pre {
  font-family: var(--font-mono), monospace;
}
```

Or register them in Tailwind so they drive `font-sans` / `font-mono` utilities:

=== "Tailwind v4 (`@theme`)"

    ```css
    @import "tailwindcss";

    @theme {
      --font-sans: var(--font-sans);
      --font-mono: var(--font-mono);
    }
    ```

=== "Tailwind v3 (config)"

    ```js
    export default {
      content: ["./app/**/*.{js,ts,jsx,tsx,mdx}"],
      theme: {
        extend: {
          fontFamily: {
            sans: ["var(--font-sans)"],
            mono: ["var(--font-mono)"],
          },
        },
      },
    };
    ```

### Local fonts

For fonts you ship yourself (brand fonts, licensed files), use `next/font/local` and point at the files in your project. The same optimization and zero-shift benefits apply:

```tsx
// app/layout.tsx
import localFont from "next/font/local";

const brand = localFont({
  src: [
    { path: "./fonts/Brand-Regular.woff2", weight: "400", style: "normal" },
    { path: "./fonts/Brand-Bold.woff2", weight: "700", style: "normal" },
  ],
  variable: "--font-brand",
  display: "swap",
});

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en" className={brand.variable}>
      <body>{children}</body>
    </html>
  );
}
```

!!! warning "Call the font loader at module scope"
    A `next/font` loader **must** be called in the top-level module scope with literal arguments — not inside a component, a loop, or with values computed at runtime. Next.js evaluates these calls at build time to download and inline the font; a dynamic call can't be analyzed and will error. Define each font once per module (typically in a layout) and reuse its `className`/`variable`.

## :material-table: Choosing an approach

| Approach | Works in Server Components? | Runtime cost | Notes for Next.js |
| --- | --- | --- | --- |
| **Global CSS** | ✅ | None | Resets, tokens; import once in the root layout |
| **CSS Modules** | ✅ | None | Scoped, zero-runtime — the safe default |
| **Sass / Sass Modules** | ✅ | None | Install `sass`; global or `*.module.scss` |
| **Tailwind CSS** | ✅ | None (build-time JIT) | v4 uses `@tailwindcss/postcss` + `@import "tailwindcss"` |
| **styled-components / Emotion** | ❌ (client-only) | Medium (runtime) | Needs a `"use client"` registry; opts components out of RSC |
| **Pigment / vanilla-extract / Linaria** | ✅ | None (compiled) | Zero-runtime CSS-in-JS; recommended if you want the `css` ergonomics |

!!! note "TL;DR for the App Router"
    - **Default to zero-runtime:** CSS Modules, Sass Modules, or Tailwind — all work in Server Components and ship no styling JS.
    - **Want CSS-in-JS ergonomics?** Prefer a **compile-time** library (Pigment CSS, vanilla-extract) over runtime styled-components/Emotion.
    - **Migrating an existing styled-components/Emotion app?** Use the **style registry** pattern and accept the `"use client"` boundary.
    - **For fonts, always use `next/font`** — self-hosting, preloading, and zero layout shift come for free; expose a `variable` to combine fonts and feed Tailwind.
