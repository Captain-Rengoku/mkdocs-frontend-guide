# :simple-react: Rendering: Server & Client Components

The App Router is built on **React Server Components (RSC)**. The single biggest mental shift coming from plain React or the Pages Router is this: **components run on the server by default, and you opt *into* the client**, not the other way around. Getting the server/client split right is what determines how much JavaScript ships, how fast the page paints, and whether your data access stays safely on the server.

This chapter covers the RSC model, when each kind of component is appropriate, the `"use client"` boundary and the composition patterns that keep it small, how Next.js decides between static and dynamic rendering, streaming with `<Suspense>`, and the Node.js vs Edge runtime choice.

> 📚 For more: [https://react.dev/reference/rsc/server-components](https://react.dev/reference/rsc/server-components) · [https://nextjs.org/docs/app/getting-started/server-and-client-components](https://nextjs.org/docs/app/getting-started/server-and-client-components)

---

## 1. The React Server Components model

React Server Components are components that **render only on the server** and never ship their code to the browser. They were designed by the React team and are surfaced to most developers through a framework — in our case, the Next.js App Router.

A useful way to picture it: there are now **two kinds of React components** living in one tree.

- **Server Components** run during the request (or at build time). They can be `async`, talk to a database, read files, and use secrets. Their *rendered output* — not their source — is sent to the browser as a special serialized description of the UI (the "RSC payload").
- **Client Components** are the React you already know. They are server-rendered to HTML for the first paint, then their **JavaScript is sent to the browser and hydrated** so they can hold state and respond to events.

```text
Request ──▶ Server renders the whole tree
            ├─ Server Components → RSC payload (description of UI, no JS)
            └─ Client Components → initial HTML + their JS bundle
                                   ↓
Browser ──▶ paints HTML, then hydrates only the Client Components
```

The RSC payload is React's serialized output, not HTML. The browser uses it to reconcile the tree and to slot Client Components into the right places. On navigation, Next.js fetches a fresh RSC payload for the new route instead of a full HTML document, which is why client-side navigation stays fast while still re-running server code.

!!! note "RSC ≠ SSR"
    They are related but distinct. **SSR** turns a React tree into an HTML string on each request (it has existed for years). **RSC** is a component *type* that runs only on the server and ships zero JS for that component. In the App Router you get **both**: Server Components produce no client JS, and the whole tree is server-rendered to HTML for the first paint. See the [React SSR & SSG](../react/12.React%20SSR%20and%20SSG.md) chapter for the SSR/SSG/ISR fundamentals this builds on.

## 2. Server vs Client Components

### Server Components are the default

In the App Router, **every component is a Server Component unless the file (or one it's imported from) has `"use client"`**. There is no directive for "this is a server component" — server is simply the default.

```tsx
// app/products/page.tsx — a Server Component (no directive needed)
// This code, the DB call, and any secrets never reach the browser.
import { db } from "@/lib/db";

export default async function ProductsPage() {
  const products = await db.product.findMany({ take: 20 }); // direct data access

  return (
    <main>
      <h1>Products</h1>
      <ul>
        {products.map((p) => (
          <li key={p.id}>{p.name}</li>
        ))}
      </ul>
    </main>
  );
}
```

What Server Components can do that Client Components can't:

- Be `async` and `await` data **directly** — no `useEffect`, no `getServerSideProps`.
- Access server-only resources: the database, the filesystem (`fs`), environment secrets, internal services.
- Keep large dependencies (markdown parsers, date libraries, SDKs) on the server so they **never inflate the client bundle**.

What they **cannot** do:

- Use state or lifecycle: `useState`, `useReducer`, `useEffect`, `useRef` for DOM, etc.
- Attach event handlers (`onClick`, `onChange`, `onSubmit`).
- Use browser-only APIs (`window`, `document`, `localStorage`, `navigator`).
- Use most React Context (`useContext`) or hooks that depend on the above.

### Client Components opt in with `"use client"`

When a component needs interactivity, mark its file with `"use client"` as the very first line.

```tsx
// app/components/Counter.tsx
"use client";
import { useState } from "react";

export default function Counter({ start }: { start: number }) {
  const [count, setCount] = useState(start);
  return (
    <button onClick={() => setCount((n) => n + 1)}>
      Clicked {count} times
    </button>
  );
}
```

Reach for a Client Component when you need any of: `useState`/`useReducer`, `useEffect`/lifecycle, event handlers, browser APIs, React Context consumers/providers, or class components, and most third-party libraries that use those internally (animation, charts, form libraries, anything touching `window`).

### Choosing — a quick reference

| You need to… | Component type |
| ------------ | -------------- |
| Fetch data / query the DB | **Server** |
| Use secrets / server-only env vars | **Server** |
| Keep a heavy dependency off the client | **Server** |
| Read `cookies()` / `headers()` | **Server** |
| Use `useState`, `useEffect`, `useRef` | **Client** |
| Handle events (`onClick`, `onChange`) | **Client** |
| Use `window` / `localStorage` / `navigator` | **Client** |
| Consume/provide React Context | **Client** |

!!! tip "Default to Server, opt into Client at the leaves"
    Start everything as a Server Component. Only convert the **smallest interactive piece** to a Client Component. A "page" is almost always a Server Component that renders a few small client islands (a like button, a search box, a menu toggle).

## 3. The `"use client"` boundary

`"use client"` doesn't just mark one component — it marks the **boundary** between the server-only world and the client bundle. Everything **imported** from a `"use client"` file (and everything *it* imports) becomes part of the client bundle.

```text
app/page.tsx                (Server)
└─ "use client" Header.tsx  ← boundary starts here
   ├─ Nav.tsx               (Client — imported by Header, no directive needed)
   └─ SearchBox.tsx         (Client — imported by Header)
```

You only write `"use client"` **once**, at the top of the boundary. Files imported *into* the client subtree are automatically Client Components — adding the directive to every file is redundant but harmless.

!!! warning "Push the boundary down the tree"
    Marking a `layout.tsx` or a big page `"use client"` drags its entire subtree into the client bundle and forfeits the benefits of Server Components. Keep `"use client"` on the smallest leaf that actually needs it.

### Common mistake: importing a Server Component into a Client Component

This is the single most common RSC mistake. Once a file is a Client Component, anything it **imports** is bundled and run on the client — so a Server Component imported there **loses its server-ness** (and will error the moment it does something server-only like `await db…`).

```tsx
// ❌ Wrong: importing a Server Component into a Client Component
"use client";
import { useState } from "react";
import ServerData from "./ServerData"; // an async Server Component

export default function Panel() {
  const [open, setOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setOpen((o) => !o)}>Toggle</button>
      {open && <ServerData />} {/* ServerData is now treated as client code */}
    </div>
  );
}
```

The fix is **composition**: pass the Server Component *in* from a server parent, as `children` or a prop, instead of importing it.

### The fix: pass Server Components as `children` / props

A Client Component can **render** Server Components it receives as props — it just can't **import** them. The Server Component is rendered on the server; the Client Component only ever receives its already-rendered output and drops it into a "hole" in its own tree.

```tsx
// app/components/Panel.tsx
"use client";
import { useState, type ReactNode } from "react";

export default function Panel({ children }: { children: ReactNode }) {
  const [open, setOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setOpen((o) => !o)}>Toggle</button>
      {open && children} {/* rendered on the server, just placed here */}
    </div>
  );
}
```

```tsx
// app/page.tsx — a Server Component composes the two
import Panel from "./components/Panel";
import ServerData from "./components/ServerData"; // async Server Component

export default function Page() {
  return (
    <Panel>
      <ServerData /> {/* stays a Server Component: rendered on the server */}
    </Panel>
  );
}
```

This is the **interleaving** pattern: server → client → server again, alternating down the tree. The rule is simply *who imports whom*. A Server Component may import and render Client Components; a Client Component may only **receive** Server Components as props/children, never import them.

### Only serializable props cross the boundary

Props passed from a Server Component to a Client Component are serialized into the RSC payload, so they must be **serializable**.

```tsx
// app/page.tsx (Server)
import Counter from "./components/Counter";

export default async function Page() {
  const start = await getInitialCount();

  return (
    <>
      {/* ✅ OK: numbers, strings, booleans, arrays, plain objects, */}
      {/*        null, Dates, Maps/Sets, Promises, and JSX (as children) */}
      <Counter start={start} />

      {/* ❌ Not OK: functions, class instances, and other non-serializable values */}
      {/* <Counter onChange={() => save()} /> */}
    </>
  );
}
```

- ✅ Allowed: primitives, plain objects/arrays, `Date`, `Map`, `Set`, `Promise` (React can pass a promise and you `use()` it on the client), and JSX/Server Components passed as `children`.
- ❌ Not allowed: functions/event handlers, class instances, symbols, anything with methods or non-plain prototypes.

If a Client Component needs to *trigger server work*, don't pass a function down — call a **Server Action** (`"use server"`), covered in the Server Actions chapter, or hit a Route Handler.

!!! note "The `server-only` and `client-only` guards"
    To make accidental cross-boundary imports fail at **build time** instead of leaking, mark modules explicitly:

    ```ts
    // lib/db.ts — must never reach the browser
    import "server-only"; // build error if imported into a Client Component
    ```

    `npm i server-only client-only`. Use `server-only` for anything with secrets or a DB/`fs` dependency; use `client-only` for code that touches `window`.

### `"use client"` doesn't mean "no SSR"

A common misread: Client Components still render to HTML on the server for the first paint — `"use client"` only means *this component's JS is also sent to the browser and hydrated*. It does **not** turn off server-side rendering, and it does **not** make the component a client-only/CSR component. (For genuinely client-only rendering, see `next/dynamic` with `ssr: false` in the Optimizations chapter.)

## 4. Static vs Dynamic rendering

Independently of the server/client split, each **route** is rendered in one of two modes:

- **Static rendering (SSG)** — the route is rendered **at build time** (or in the background during revalidation) and served as cached HTML. This is the default.
- **Dynamic rendering (SSR)** — the route is rendered **on each request**, because it depends on request-time information.

Next.js chooses automatically: a route stays static until it uses something that **can't be known at build time**, at which point it becomes dynamic.

### What triggers dynamic rendering

A route opts into dynamic rendering when it uses any **Dynamic API** or an uncached request:

- **`cookies()`**, **`headers()`**, **`draftMode()`** from `next/headers`.
- The **`searchParams`** prop of a page (it depends on the actual URL).
- An **uncached `fetch`** — `fetch(url, { cache: "no-store" })`, or in Next.js 15 simply a `fetch` you didn't opt into caching.
- An explicit **route-segment config** forcing it (below).

```tsx
// app/dashboard/page.tsx — DYNAMIC: reads a request-time cookie
import { cookies } from "next/headers";

export default async function Dashboard() {
  const cookieStore = await cookies(); // Next.js 15: cookies() is async
  const theme = cookieStore.get("theme")?.value ?? "light";
  return <main data-theme={theme}>Welcome back</main>;
}
```

```tsx
// app/blog/[slug]/page.tsx — STATIC: no dynamic APIs, cached data
export default async function Post({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params; // Next.js 15: params is async
  const post = await fetch(`https://api.example.com/posts/${slug}`, {
    cache: "force-cache",
  }).then((r) => r.json());
  return <article>{post.title}</article>;
}
```

### Forcing the mode with route-segment config

You can override the heuristic per route segment:

```ts
// at the top of a page.tsx / layout.tsx / route.ts
export const dynamic = "force-dynamic";  // always render per request (SSR)
// export const dynamic = "force-static"; // always static; Dynamic APIs return defaults
// export const revalidate = 60;          // ISR: static, refreshed at most every 60s
// export const dynamic = "error";        // build error if anything makes it dynamic
```

!!! note "Caching defaults changed in Next.js 15"
    In Next.js **14**, `fetch` was cached by default (routes leaned static). In Next.js **15**, `fetch` is **uncached by default**, so a route with a plain `fetch` renders dynamically unless you opt in with `cache: "force-cache"` or `next: { revalidate }`. Be explicit about caching rather than relying on the version default — see the **Data Fetching & Caching** chapter for the full model.

!!! tip "You don't choose one mode for the whole app"
    Static vs dynamic is **per route**, and with streaming you can keep a static shell while rendering a dynamic region inside `<Suspense>` (the Partial Prerendering direction). Default to static; go dynamic only where the data is genuinely per-request.

## 5. Streaming & `<Suspense>`

The App Router can **stream** a response: send the static shell of the page immediately, then flush slower parts as their data resolves — instead of blocking the entire response on the slowest query. This improves TTFB and lets users see and interact with the fast parts first.

Wrap a slow `async` Server Component in **`<Suspense>`** with a `fallback`. React sends the fallback in the initial HTML, then streams the real content in when it's ready and swaps it in.

```tsx
// app/dashboard/page.tsx
import { Suspense } from "react";
import RevenueChart from "./RevenueChart"; // slow async Server Component
import Greeting from "./Greeting";         // fast

export default function Dashboard() {
  return (
    <main>
      <Greeting /> {/* renders immediately */}
      <Suspense fallback={<p>Loading revenue…</p>}>
        <RevenueChart /> {/* streamed in when its data resolves */}
      </Suspense>
    </main>
  );
}
```

```tsx
// app/dashboard/RevenueChart.tsx — its slow await doesn't block the rest of the page
export default async function RevenueChart() {
  const data = await getRevenue(); // e.g. a 2s query
  return <Chart data={data} />;
}
```

Each `<Suspense>` boundary streams independently, so you can have several slow regions resolve in parallel without any one blocking the others.

### `loading.tsx` — automatic Suspense for a whole segment

Dropping a `loading.tsx` file into a route segment wraps that segment's `page` in a Suspense boundary for you, giving the **whole route** an instant loading UI on navigation.

```tsx
// app/dashboard/loading.tsx
export default function Loading() {
  return <p>Loading dashboard…</p>; // shown instantly while the segment renders
}
```

!!! note "Streaming and dynamic rendering work together"
    A route can serve a **static shell** instantly and stream a **dynamic** region (one that reads `cookies()` or does a `no-store` fetch) inside a `<Suspense>`. That's the essence of Partial Prerendering: cached shell + streamed dynamic islands. See the SSR & SSG chapter for how this maps onto the broader rendering spectrum.

## 6. Runtimes: Node.js vs Edge

Your server code (Server Components, Route Handlers, Server Actions, Middleware) runs in one of two **runtimes**, and you can choose per route segment.

```ts
// at the top of a page.tsx / layout.tsx / route.ts
export const runtime = "nodejs"; // default
// export const runtime = "edge";
```

| | **Node.js runtime** (default) | **Edge runtime** |
| --- | --- | --- |
| APIs | Full Node.js APIs | Web-standard APIs only (`fetch`, `Request`, `Response`, Web Crypto) |
| Native modules / DB drivers | ✅ Supported | ❌ Many TCP-based DB drivers won't run |
| Cold start | Larger | Near-zero |
| Distribution | Regional | Deployed close to the user (edge network) |
| Best for | Data fetching, ORMs, heavy libraries, most app code | Auth checks, redirects, lightweight personalization, geolocation |

- **Node.js** is the default and the right choice for the vast majority of work: anything using a database driver, an ORM (Prisma/Drizzle), the filesystem, or a substantial dependency.
- **Edge** runs in a constrained, Web-standard environment with very fast cold starts and runs geographically close to the user. It's ideal for **Middleware** (which runs on Edge), auth gating, redirects, and lightweight per-user logic — but it can't use Node-only APIs or most traditional database drivers.

!!! warning "Edge is constrained — verify your dependencies"
    The Edge runtime is **not** Node.js. Code that imports `fs`, `crypto` (the Node module, not Web Crypto), native addons, or a TCP database client will fail there. Use Edge deliberately for small, latency-sensitive routes; keep your data-heavy routes on Node.js. The **Middleware & Edge** chapter covers the constraints in depth.

## 📚 Summary

| 🔹 Concept | 🧠 Description |
| ---------- | -------------- |
| **Server Component** | Default in the App Router; runs server-only, can be `async`, ships **zero JS** |
| **Client Component** | Opt in with `"use client"`; SSR'd for first paint, then hydrated for interactivity |
| `"use client"` | Marks the **boundary**; everything imported below it joins the client bundle |
| **Composition** | Pass Server Components into Client Components as `children`/props, never **import** them |
| **Serializable props** | Only primitives, plain objects, Dates, Promises, JSX cross the boundary — **no functions** |
| `server-only` / `client-only` | Build-time guards against leaking server code to the client (and vice-versa) |
| **Static rendering** | Default; rendered at build time and cached (SSG) |
| **Dynamic rendering** | Per-request (SSR); triggered by `cookies()`/`headers()`, `searchParams`, uncached `fetch` |
| `export const dynamic` / `revalidate` | Route-segment config to force static/dynamic/ISR |
| **Streaming + `<Suspense>`** | Send the shell first, stream slow regions in as data resolves |
| `loading.tsx` | Automatic Suspense fallback for a whole route segment |
| **Node.js runtime** | Default; full Node APIs, DB drivers, heavy libraries |
| **Edge runtime** | Web-standard only; near-zero cold start, close to the user, constrained |

!!! warning "The mistakes to avoid"
    - **Importing a Server Component into a Client Component** → pass it as `children`/props instead.
    - **Using `useState`/`useEffect`/`onClick`/`window` in a Server Component** → it will error; add `"use client"` to the smallest leaf.
    - **Passing a function as a prop** across the boundary → use a Server Action or Route Handler.
    - **Marking a layout/page `"use client"`** → bloats the bundle; push the directive down the tree.
    - **Forgetting `cache`/`revalidate`** → surprising stale-or-always-dynamic behavior after the 14→15 default flip.
