# :simple-nextdotjs: Introduction to Next.js

Plain React gives you a component model and nothing else. The moment you need routing, server rendering, data fetching, bundling, and a deployment story, you assemble those yourself from a dozen libraries. **Next.js** is the framework that packages all of it—and, crucially, lets you render on the server so your HTML ships fast and is fully crawlable.

This chapter introduces Next.js: what it is, what it adds over plain React, when to reach for it, a quick recap of the rendering strategies (CSR / SSR / SSG / ISR), the two routers (App vs Pages), and how to scaffold and run a project. We target **Next.js 15** with the **App Router** and **TypeScript** throughout.

> 📚 For more: [https://nextjs.org/docs](https://nextjs.org/docs) · [https://react.dev/learn/start-a-new-react-project](https://react.dev/learn/start-a-new-react-project)

!!! note "Builds on the React section"
    This assumes you're comfortable with React components, hooks, and the rendering-strategy concepts. If SSR/SSG/ISR or React Server Components are new, read **React → SSR & SSG** first; this chapter recaps them but doesn't re-teach them from scratch.

## 1. What is Next.js?

**Next.js** is a full-stack React framework, maintained by **Vercel**, built on top of React. Where a tool like Vite gives you a fast dev server and a bundler and leaves the rest to you, Next.js is *opinionated*: it provides a complete, integrated set of answers to the questions every real React app eventually asks.

You still write React components. Next.js adds the framework *around* them.

### What it adds over plain React

A "plain React" app (Vite, or the old Create React App) is a **client-side single-page app**: the browser downloads a JS bundle and React builds the page on the user's device. That's it—no server, no routing, no rendering choices. Next.js layers the following on top:

| Capability | Plain React (Vite/CRA) | Next.js |
| ---------- | ---------------------- | ------- |
| **Routing** | Add React Router, configure routes manually | File-system routing—folders *are* routes |
| **Rendering** | Client-side only (CSR) | CSR, SSR, SSG, ISR—**chosen per route** |
| **Server code** | Separate backend project | Route Handlers & Server Actions in the same app |
| **Data fetching** | `useEffect` + a client library | `await` directly inside Server Components |
| **Server Components** | Not available | Default—components that ship **zero JS** |
| **Image optimization** | DIY / external service | `next/image` (resize, lazy-load, modern formats) |
| **Font optimization** | Manual `<link>` tags | `next/font` (self-hosted, no layout shift) |
| **Code splitting** | Manual `React.lazy` | Automatic per-route + `next/dynamic` |
| **SEO / metadata** | `react-helmet` or manual | Built-in Metadata API |
| **Deployment** | Build static files, host an API yourself | Zero-config on Vercel; adapters elsewhere |

The headline additions are **server rendering** and **React Server Components**. Plain React can only run in the browser; Next.js can run your components on the server—at build time, at request time, or a blend—so users (and crawlers) receive real HTML instead of an empty `<div id="root">`.

### Why use it

- **Routing** — Create a folder and a `page.tsx` file and you have a route. Nested layouts, dynamic segments, and loading/error states are all file conventions, not boilerplate.
- **Rendering** — Each route picks how its HTML is produced (static, dynamic, or streamed), so you get the right trade-off of speed, freshness, and cost per page.
- **Data** — Server Components fetch data with plain `async`/`await`, with a built-in caching layer. Mutations use Server Actions—no hand-written API for forms.
- **Optimizations** — Images, fonts, scripts, and code splitting are optimized by default, which directly improves Core Web Vitals.
- **Deployment** — A Next.js app deploys to Vercel with zero config (SSR, ISR, and API routes become serverless/edge functions automatically), and to other hosts via adapters or a Node server.

!!! tip "Next.js is React, not an alternative to it"
    You don't choose *between* React and Next.js. Next.js **is** a way to build React apps—it adds the framework scaffolding that React deliberately leaves out. If you know React, you already know most of Next.js.

## 2. Why and when to use it

Next.js is the default choice for most production React apps today, but it isn't mandatory. Match the tool to the job.

### Reach for Next.js when…

- **SEO matters** — marketing sites, blogs, docs, e-commerce, landing pages. Pre-rendered HTML gives crawlers and social scrapers complete content and Open Graph tags immediately.
- **You want fast first paint** — server-rendered or static HTML shows real content before any app JS runs, which helps LCP on slow networks and low-end devices.
- **You need a backend too** — Route Handlers and Server Actions let you build the API in the same project, no separate server.
- **You're deploying full-stack** — auth, databases, server-side data fetching, and a frontend in one deployable unit.
- **You want batteries included** — image/font optimization, routing, and caching without wiring up libraries.

### A lighter tool may be enough when…

- **It's a purely client-side, auth-gated app** with no SEO needs—an internal admin dashboard behind a login. A Vite SPA is simpler and cheaper to host.
- **You're building a component library or embeddable widget**, not an application.
- **You need an existing non-React backend** and just want a frontend bundler—Vite is leaner.

!!! note "SSR is not automatically faster end-to-end"
    Server rendering adds server work on every request plus a hydration step on the client. A poorly-cached SSR page with a heavy data fetch can have a *worse* time-to-first-byte than a static SPA. Next.js wins on **first contentful paint and SEO**, not unconditionally on total cost—prefer static/ISR whenever the data doesn't truly need to be per-request.

## 3. Rendering strategies recap

The **rendering strategy** answers: *who turns my components into HTML, and at what moment?* Next.js supports all four of the common strategies, and in the App Router you choose **per route**—not once for the whole app.

### CSR — Client-Side Rendering

The browser downloads a near-empty HTML shell plus a JS bundle, then React builds the DOM on the user's device. The classic SPA model.

- ✅ Cheap to host, great for highly interactive, auth-gated apps where SEO is irrelevant.
- ❌ Blank screen until JS loads and runs; weaker SEO; performance depends on the user's device.

### SSR — Server-Side Rendering

The server runs your components **on every request** and returns fully-formed HTML, which the browser then **hydrates** into an interactive page.

- ✅ Always-fresh, personalized content; good SEO.
- ❌ Server compute on every request → higher cost and latency than serving a static file.

### SSG — Static Site Generation

HTML is generated **once at build time** and served from a CDN. Every visitor gets the same pre-built file.

- ✅ Fastest delivery, cheapest hosting, excellent SEO.
- ❌ Data is frozen at build time; rebuilding a huge site is slow.

### ISR — Incremental Static Regeneration

SSG's pragmatic upgrade: pages are static but **regenerate in the background** on a schedule or on demand, without a full rebuild. CDN speed with data that stays reasonably fresh.

- ✅ Static performance with content that updates—ideal for blogs, docs, catalogs.
- ❌ Visitors may briefly see stale content until the next regeneration completes.

### Where Next.js fits

| Strategy | HTML generated | Data freshness | Best for |
| -------- | -------------- | -------------- | -------- |
| **CSR**  | In the browser | Live (client fetch) | Auth-only apps, dashboards |
| **SSR**  | Per request on the server | Always fresh | Personalized, request-time data |
| **SSG**  | Once at build | Frozen at build | Marketing, docs, rarely-changing content |
| **ISR**  | At build, refreshed in background | Stale-then-fresh | Blogs, catalogs, content that changes sometimes |

In the App Router you don't pick one of these for the whole site. A route is **static by default** and becomes dynamic when it reads request-time data (cookies, headers, an uncached `fetch`); ISR is a per-route or per-`fetch` `revalidate` setting; and you can even stream a static shell with dynamic regions inside it. Think of CSR/SSR/SSG/ISR as a **spectrum**, not four separate buckets.

!!! tip "The throughline: ship less JavaScript"
    Across SEO and Core Web Vitals, the recurring win is shipping less JS to the client. React Server Components (the App Router default) render data-only components on the server and send zero JS for them—making "less JS" the default rather than an optimization you bolt on later.

> The full mechanics of these strategies—hydration, caching, `generateStaticParams`, `revalidate`—are covered in **Rendering** and **Data Fetching & Caching** later in this section.

## 4. App Router vs Pages Router

Next.js has **two routers**, and which one you use determines the entire programming model. You'll meet both, but they are not equal in 2025.

### Pages Router (legacy)

The original model, living in a `pages/` directory. Routing is file-based, but rendering is configured with special exported functions—`getServerSideProps` (SSR), `getStaticProps` + `getStaticPaths` (SSG/ISR)—and API endpoints live in `pages/api/`. It predates React Server Components. It's still fully supported and extremely common in existing codebases.

### App Router (default & recommended)

Introduced in Next.js 13, stable since 13.4, and the **default in 14 and 15**. It lives in an `app/` directory and is built on **React Server Components**.

- **Components are Server Components by default**—they run only on the server, can be `async`, can `await` data directly, and ship **zero JavaScript** to the browser. Opt into the client with `"use client"`.
- **Data fetching is just `await`** inside a component—no `getServerSideProps`/`getStaticProps`.
- **Nested layouts** let route segments share UI that persists across navigation without re-rendering.
- **Streaming and Suspense** are first-class via `loading.tsx` and `<Suspense>`.
- **Mutations** use Server Actions (`"use server"`), often without any hand-written API endpoint.

### Why the App Router is recommended for new code

| Concern | Pages Router | App Router |
| ------- | ------------ | ---------- |
| Data fetching | `getServerSideProps` / `getStaticProps` | `async` Server Component + `await` |
| Server Components | Not available | Default |
| Layouts | One `_app.tsx`; manual per-page wrappers | Nested `layout.tsx`, persist across navigation |
| Loading / error UI | Manual | `loading.tsx` / `error.tsx` conventions |
| Streaming | Limited | Built in via Suspense |
| Mutations | API route + client `fetch` | Server Actions |
| Status | Maintained, legacy | **Default & actively developed** |

!!! warning "Don't mix them carelessly—and prefer one model"
    A single project *can* contain both `app/` and `pages/` (Next.js routes `app/` first), which is useful for incrementally migrating. But for **new** projects, use the App Router exclusively. The Pages Router data functions (`getServerSideProps`, etc.) **do not exist** in the App Router—they're replaced by `async` components, `fetch` caching, `generateStaticParams`, and `revalidate`. This chapter and the rest of the section are App Router-first; Pages Router is noted only where it still matters for existing code.

## 5. Creating a project

The official scaffolding tool is **`create-next-app`**. It sets up the project structure, TypeScript, linting, and dependencies for you.

### `create-next-app` with TypeScript

Run the interactive scaffolder with `npx` (no global install needed):

```bash
npx create-next-app@latest my-app
```

It will ask a series of questions. For a modern App Router + TypeScript project, answer roughly like this:

```text
✔ Would you like to use TypeScript? … Yes
✔ Would you like to use ESLint? … Yes
✔ Would you like to use Tailwind CSS? … (your choice)
✔ Would you like your code inside a `src/` directory? … (your choice)
✔ Would you like to use App Router? (recommended) … Yes
✔ Would you like to use Turbopack? (recommended) … Yes
✔ Would you like to customize the import alias (`@/*`)? … No
```

To skip the prompts and accept those defaults non-interactively, pass flags:

```bash
npx create-next-app@latest my-app \
  --typescript \
  --eslint \
  --app \
  --src-dir \
  --import-alias "@/*" \
  --use-npm
```

Then start the dev server:

```bash
cd my-app
npm run dev
```

Open **http://localhost:3000** and you'll see the starter page.

!!! note "The `@/*` import alias"
    `create-next-app` configures a path alias in `tsconfig.json` so you can write `import { db } from "@/lib/db"` instead of brittle `../../../lib/db` relative paths. With `--src-dir`, `@/*` points at `src/*`; without it, at the project root. Examples in this guide use the `@/` alias.

### Project structure & file conventions

A fresh App Router project (with `src/`) looks roughly like this:

```text
my-app/
├── src/
│   └── app/                 # App Router — folders are routes
│       ├── layout.tsx       # Root layout (required) — wraps <html>/<body>
│       ├── page.tsx         # UI for "/"
│       ├── globals.css      # Global styles
│       └── favicon.ico
├── public/                  # Static assets served from "/"
├── next.config.ts           # Next.js configuration
├── tsconfig.json            # TypeScript config (incl. the @/* alias)
├── eslint.config.mjs        # ESLint flat config
├── package.json
└── next-env.d.ts            # Next.js TS types (auto-generated, don't edit)
```

**`app/`** is where your routing and UI live. Folders define URL segments; specially-named files give those segments behavior. **`public/`** holds static files (images, fonts, `robots.txt`) served from the root—`public/logo.png` is reachable at `/logo.png`.

The power of the App Router is in its **special files**. Each is a convention: drop a file with the right name into a route folder and Next.js wires it up automatically.

| File | Purpose |
| ---- | ------- |
| `layout.tsx` | Shared UI wrapping a segment and its children; persists across navigation. The **root layout is required**. |
| `page.tsx` | The unique UI for a route (makes the segment publicly routable). |
| `loading.tsx` | Instant loading UI for a segment, shown via an automatic Suspense boundary while it loads. |
| `error.tsx` | Error boundary for a segment (must be a Client Component). |
| `not-found.tsx` | UI rendered for `notFound()` calls or unmatched routes. |
| `template.tsx` | Like a layout, but re-mounts on every navigation (fresh state each time). |
| `route.ts` | A **Route Handler**—an API endpoint, using Web `Request`/`Response`. |
| `[folder]` | A **dynamic** route segment, e.g. `[slug]` → `/blog/:slug`. |
| `(folder)` | A **route group**—organizes files without affecting the URL. |
| `_folder` | A **private folder**—opted out of routing (for colocated helpers/components). |

Here's the minimal pair that every App Router app starts with—a root layout and a home page:

```tsx
// src/app/layout.tsx — the required root layout
import type { Metadata } from "next";
import "./globals.css";

export const metadata: Metadata = {
  title: "My App",
  description: "Built with Next.js 15 and the App Router",
};

export default function RootLayout({
  children,
}: Readonly<{ children: React.ReactNode }>) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  );
}
```

```tsx
// src/app/page.tsx — a Server Component (no "use client"), the UI for "/"
export default function HomePage() {
  return (
    <main>
      <h1>Hello, Next.js 15 👋</h1>
      <p>Edit this file and the page updates instantly.</p>
    </main>
  );
}
```

!!! warning "The root layout is mandatory and owns `<html>`/`<body>`"
    Every App Router app **must** have `app/layout.tsx`, and it must render the `<html>` and `<body>` tags—Next.js does not add them for you. Don't add a `<head>` tag manually; use the **Metadata API** (the `metadata` export above) so Next.js manages it. Routing, layouts, and the full set of special files get their own deep dive in the **Routing** chapter.

### Dev / build / start scripts and Turbopack

`create-next-app` adds these scripts to `package.json`:

```json
{
  "scripts": {
    "dev": "next dev --turbopack",
    "build": "next build",
    "start": "next start",
    "lint": "next lint"
  }
}
```

| Command | What it does |
| ------- | ------------ |
| `npm run dev` | Starts the **development server** (default port 3000) with Hot Module Replacement and Fast Refresh. Use this while coding. |
| `npm run build` | Creates an **optimized production build** in `.next/`—bundles, minifies, and pre-renders static routes. |
| `npm run start` | Serves the **production build** (`next start`). Run `build` first; this is what runs in production on a Node server. |
| `npm run lint` | Runs ESLint over the project. |

The dev and production servers are distinct: `next dev` prioritizes fast feedback (no minification, instant updates), while `next build` + `next start` produces the optimized output you actually ship.

**Turbopack** is Next.js's Rust-based bundler, the successor to Webpack. It's the default for `next dev` in Next.js 15 (hence the `--turbopack` flag in the `dev` script) and delivers dramatically faster cold starts and HMR on large apps. Production builds (`next build --turbopack`) are also available and increasingly stable in Next.js 15.

```bash
# Dev with Turbopack (the create-next-app default)
next dev --turbopack

# Production build with Turbopack
next build --turbopack
```

!!! tip "If a tool chokes on Turbopack, drop the flag"
    Turbopack is the recommended default, but a few legacy Webpack plugins and `next.config` customizations aren't fully supported yet. If something breaks, remove `--turbopack` to fall back to the stable Webpack pipeline while you investigate—your app code doesn't change either way.

!!! note "Node version"
    Next.js 15 requires **Node.js 18.18 or later**. Check with `node -v`; if you're behind, use a version manager like `nvm` or `fnm` to upgrade before scaffolding.

## 📚 Summary

| 🔹 Concept | 🧠 Description |
| ---------- | -------------- |
| **Next.js** | A full-stack React framework (by Vercel) adding routing, rendering, data, and optimization to React |
| **vs plain React** | Plain React is client-only; Next.js adds server rendering, Server Components, file routing, and a backend |
| **When to use** | SEO-sensitive, fast-first-paint, or full-stack apps; a Vite SPA may suffice for auth-only dashboards |
| **CSR / SSR / SSG / ISR** | Per-route rendering strategies—where and when HTML is generated |
| **App Router** | Default `app/` model built on Server Components; recommended for all new projects |
| **Pages Router** | Legacy `pages/` model with `getServerSideProps`/`getStaticProps`; known for existing code |
| **`create-next-app`** | Official scaffolder; use `--typescript --app` for a modern setup |
| **`app/`** | Routing + UI; folders are routes, special files add behavior |
| **`public/`** | Static assets served from `/` |
| **Special files** | `layout`, `page`, `loading`, `error`, `not-found`, `template`, `route` |
| **Root layout** | Required `app/layout.tsx`; renders `<html>`/`<body>` |
| **dev / build / start** | Dev server, production build, production server |
| **Turbopack** | Rust bundler, default for `next dev` in Next.js 15; faster than Webpack |
